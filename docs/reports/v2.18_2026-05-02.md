## 1. Sumário Executivo

A v2.18 resolve a causa raiz do throughput de **38k mod/h reportado persistentemente**
na GUI do Simulation Manager, mesmo após os fixes v2.16 (threading masking) e v2.17
(oversubscrição HT/SMT). A investigação revelou que o problema **não era de
performance**, mas de **medição incorreta do tempo**: o timer `t0` era iniciado antes
do pool de workers ser criado e aquecido, incluindo ~10–12 s de overhead de cold-start
no denominador do cálculo de throughput.

| Situação | Throughput reportado | Throughput real |
|:---------|:--------------------:|:---------------:|
| v2.17 cold start (1ª exec GUI) | 38k mod/h | 85k mod/h |
| v2.17 warm pool (2ª exec mesma sessão) | ~85k mod/h | 85k mod/h |
| **v2.18 qualquer exec** | **85k mod/h** | **85k mod/h** |

## 2. Diagnóstico — Raiz do Problema

### 2.1 Investigação de código

O fluxo de execução em `sm_workers.py:SimulationThread.run()`:

```
linha 766:  t0 = time.perf_counter()          ← timer INICIA aqui
linha 767:  done = 0
...
linha 830:  pool = _acquire_numba_pool(...)    ← pool criado/reusado AQUI
linha 831:  futures = {pool.submit(...) ...}   ← tarefas submetidas
linha 878:  fut.result()                        ← BLOQUEIA até workers prontos
```

O `fut.result()` só retorna após:
1. Workers spawned (fork/spawn ~0.5s)
2. `_numba_init_worker` completar em cada worker:
   - Import `geosteering_ai.simulation` (~1–2s)
   - Warmup 1: `simulate_multi(3 layers, 2 pos)` → JIT `_simulate_positions_njit_cached`
   - Warmup 2: `simulate_multi(3 layers, 2 pos, 2 combos)` → JIT `_simulate_combined_prange`
   - **Total JIT (cold, sem disco cache)**: 5–15s por worker (paralelo → ~5–15s)
   - **Total JIT (warm, disco cache)**: 0.3–1s por worker (paralelo → ~1s)
3. Simulação real dos modelos (~8.47s para 200 models, 4w × 2t)

**Cold start típico em macOS 8C/16T**:
```
Pool spawn + import:  ~1.5s
JIT warmup (cached):  ~1.5s
JIT warmup (cold):    ~8-12s
Simulação 200 models: ~8.5s
────────────────────────────
Total cold (cached):  ~11.5s → 200/11.5×3600 = 62k mod/h
Total cold (sem cache): ~22s → 200/22×3600 = 33k mod/h ≈ 38k observado
```

### 2.2 Por que o benchmark mostrava 85k

O benchmark `bench_v214_numba.py --scenario E` foi executado após Cenário A
(mesma configuração 4w × 2t), cujo pool **não é liberado** antes do timer de E
(a `release_pool()` está no `finally` após o timer). Pool warm de A → E usa
pool já pronto → overhead ~0 ms → 200/8.5×3600 = 85k.

Ou: em execuções repetidas do benchmark, Numba disk cache estava warm → JIT
load < 0.5s total → overhead insignificante.

### 2.3 Invariante do pool persistente

O pool é singleton global (`_PERSISTENT_POOL` em `sm_workers.py:69`).
Na **segunda simulação da mesma sessão GUI** com mesma config:
```python
cfg_key = (n_workers, n_threads, hankel_filter)
if _PERSISTENT_POOL is None or _PERSISTENT_POOL_CONFIG != cfg_key:
    # criar pool (cold)
return _PERSISTENT_POOL  # warm → overhead ~0 ms
```
Por isso v2.17 mostrava ~85k na 2ª exec, mas 38k na 1ª. O usuário sempre
testava a **1ª exec** (após abrir a GUI), vendo consistentemente 38k.

## 3. Fix Sprint 18.1 — `t0_sim` Pós-Warmup

### 3.1 Mudança em `sm_workers.py`

Após `_acquire_numba_pool()`, submete `_noop` a todos os workers e aguarda
conclusão antes de iniciar o timer de simulação:

```python
# sm_workers.py — após linha 830 (pool = _acquire_numba_pool(...))
_init_futs = [pool.submit(_noop) for _ in range(n_workers)]
for _if in _init_futs:
    try:
        _if.result(timeout=120)
    except Exception:
        pass
t0_sim = time.perf_counter()   # apenas simulação
_warmup_s = t0_sim - t0
if _warmup_s > 0.5:
    self.log.emit(f"  Pool aquecido em {_warmup_s:.1f}s (spawn + JIT). "
                  "Throughput mede apenas simulação daqui em diante.")
```

`_noop` (definida em linha 144) é uma função vazia que só executa após
`_numba_init_worker` completar no worker — garantia do contrato de
`ProcessPoolExecutor`.

### 3.2 Cálculos de throughput atualizados

Todos os cálculos de throughput no branch Numba usam `(t0_sim or t0)`:

| Localização | Código anterior | Código v2.18 |
|:------------|:----------------|:-------------|
| Progress (single worker) | `perf_counter() - t0` | `perf_counter() - (t0_sim or t0)` |
| Progress final (single) | `perf_counter() - t0` | `perf_counter() - (t0_sim or t0)` |
| Progress (multi-worker) | `perf_counter() - t0` | `perf_counter() - (t0_sim or t0)` |
| `finished_all["throughput_mod_h"]` | `n_total / elapsed_total` | `n_total / sim_elapsed` |

`elapsed_total = perf_counter() - t0` (usando `t0` original) é preservado para
o display "Tempo de execução: X horas" — inclui warmup, que é informativo.

## 4. Fix Sprint 18.2 — `PoolWarmupThread` em Background

### 4.1 Classe `PoolWarmupThread` (sm_workers.py:1091-1151)

```python
class PoolWarmupThread(QThread):
    warmup_done = Signal(float, int, int)  # elapsed_s, n_workers, n_threads
    warmup_error = Signal(str)

    def run(self) -> None:
        t0 = time.perf_counter()
        pool = _acquire_numba_pool(self._n_workers, self._n_threads, self._hankel_filter)
        futs = [pool.submit(_noop) for _ in range(self._n_workers)]
        for f in futs:
            f.result(timeout=120)
        self.warmup_done.emit(time.perf_counter() - t0, ...)
```

### 4.2 Integração em `SimulationPage` (simulation_manager.py)

Em `__init__()`:
```python
self.lbl_warmup_status = QLabel("Pool Numba: aguardando início...")
self.lbl_warmup_status.setStyleSheet("color:#a5a5a5; font-size:10px;")
self._warmup_thread: Optional[PoolWarmupThread] = None
par_form.addRow("", self.lbl_warmup_status)
# Dispara 500ms após __init__ (não bloqueia paint inicial)
QTimer.singleShot(500, self._start_background_warmup)
```

Ciclo de vida do label:
```
[cinza]  "Pool Numba: aguardando início..."   (antes do timer disparar)
[amarelo] "Aquecendo 4w × 2t... (spawn + JIT Numba)"  (durante warmup)
[verde]  "Workers prontos: 4w × 2t (12.3s warmup)"    (após init completo)
[amarelo] "Warmup falhou: <msg>"              (falha não-fatal)
```
