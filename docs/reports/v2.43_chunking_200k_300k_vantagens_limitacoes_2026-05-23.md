## Sumário Executivo

Trabalhar com 200k–300k modelos via chunking é **totalmente viável** para os
cenários relevantes ao treinamento de ML (A, B, C, D, E, F). O cenário G (full
multi-curriculum 4×4×4) é viável mas exige cuidado com dois recursos do host:
RAM e I/O. A tabela abaixo resume os tempos e viabilidades:

| Cen | Caso de uso ML | Chunk T4 | Chunks (200k) | Tempo 200k | Tempo 300k | RAM acumul. 200k |
|:---:|:--------------|:--------:|:-------------:|:----------:|:----------:|:----------------:|
| A | Pré-treino single-pos | 5.000 | 40 | **0,01h** | **0,01h** | 0,03 GB |
| B | Baseline multi-pos | 500 | 400 | **0,76h** | **1,14h** | 2,7 GB |
| C | Multi-freq curriculum | 150 | 1.334 | **1,31h** | **1,96h** | 10,7 GB |
| D | Multi-TR curriculum | 2.000 | 100 | **0,02h** | **0,03h** | 0,1 GB |
| E | Perfil completo 600 pts | 100 | 2.000 | **4,54h** | **6,81h** | 16,1 GB |
| F | LWD multi-config (real) | 35 | 5.715 | **5,46h** | **8,19h** | 42,9 GB ⚠️ |
| G | Full curriculum SurrogateNet | 8 | 25.000 | **21,67h** | **32,50h** | 171,7 GB ❌ |
| H | Stress test (não prático) | 1 | 200.000 | 1,83h* | 2,74h* | 2,7 GB |

> *H com chunk=1: muito lento por overhead/call. Melhor com lax.map sequencial.

**Veredicto prático**: para o dataset de treinamento do SurrogateNet (cenário F ou G),
o chunking é a única arquitetura possível. F é direto (RAM 42 GB, requer HDF5 streaming).
G com 200k modelos está **no limite superior viável** — requer streaming obrigatório
e ~22 horas de GPU T4 (ou ~22 horas no A100, pois o throughput mal muda com GPU
mais rápida quando o chunk size é pequeno).

## 1. Como o Chunking Funciona (Revisão Técnica)

O chunking divide `n_models_total` em fatias de `CHUNK_SIZE` e executa
`simulate_multi_jax_batched` sequencialmente para cada fatia:

```python
# Pseudocódigo do loop de chunking (Sprint A3 vai encapsular isso)
results = []
for start in range(0, n_models_total, CHUNK_SIZE):
    end = min(start + CHUNK_SIZE, n_models_total)
    batch = simulate_multi_jax_batched(
        rho_h_batch[start:end],      # (chunk, n_cam) float64
        rho_v_batch[start:end],      # (chunk, n_cam) float64
        esp_batch[start:end],        # (chunk, n_cam-2) float64
        positions_z,                  # (n_pos,) float64 — CONSTANTE
        frequencies_hz=freqs,         # constante
        tr_spacings_m=trs,            # constante
        dip_degs=dips,                # constante
    )
    results.append(batch.H_tensor)   # acumula em RAM ou streaming para disco

H_full = np.concatenate(results, axis=0)  # (n_total, nTR, nAng, n_pos, nf, 9)
```

**Por que funciona nativamente sem nenhuma mudança no código de produção**:
cada chunk é uma chamada independente e completa à API atual. As posições e
configurações físicas (freqs, TRs, dips) são constantes compartilhadas.
O XLA cache key é `(n_cam, npt)` — não depende de `CHUNK_SIZE` — então o
mesmo XLA program é reusado para todos os chunks (exceto o último, ver §3.3).

## 2. Vantagens

### 2.1 VRAM Constante e Controlada

A vantagem mais fundamental: o consumo de VRAM é determinístico e limitado ao
tamanho de um único chunk, independentemente de `n_models_total`.

```
VRAM(chunk) ≈ CHUNK_SIZE × unidades_vmap × 218 KB + overhead_fixo (~1 GB)

Para G com CHUNK_SIZE=8:
  VRAM = 8 × (4 × 4 × 100 × 4) × 218 KB + 1 GB
       = 8 × 6.400 × 218 KB + 1 GB
       ≈ 11,2 GB + 1 GB = 12,2 GB → cabe na T4 (15 GB)

O mesmo hardware que rodou G com n=5 agora processa 200k modelos.
```

**Não há limite superior em `n_models_total` do ponto de vista de VRAM.**
Se CHUNK_SIZE for calibrado para o hardware, qualquer volume de dados é processável.

### 2.2 XLA Cache Reutilizado em Todos os Chunks

O `_UNIFIED_JIT_CACHE` é keyed por `(n_cam, npt)` — invariante ao número de
modelos e ao chunk size. O XLA program compilado no warmup é reutilizado em
todos os chunks do loop:

```
Chunk 1: compila XLA (se não foi feito no warmup)  → 20–30 s
Chunk 2: cache hit → executa em < 0,1 ms de overhead de JIT
Chunk 3..N: cache hit → cada um executa sem overhead de compilação
```

Para 25.000 chunks (G, 200k), há exatamente **1 compilação** e 24.999 cache
hits. O custo de warmup é pago uma única vez, não por chunk.

**Exceção**: o último chunk pode ter `n_models < CHUNK_SIZE`. Se isso acontecer,
o XLA verá um batch de tamanho diferente e **recompilará**. Mitigação: usar
padding (completar com modelos fictícios no último chunk) ou dimensionar
`n_models_total` para ser múltiplo de `CHUNK_SIZE`.

### 2.3 Paridade Numérica Preservada Perfeitamente

Cada chunk é matematicamente idêntico a um subconjunto de uma chamada única:

```
simulate_multi_jax_batched(batch_full[0:N])  ==
concatenate([
    simulate_multi_jax_batched(batch_full[0:K]).H_tensor,
    simulate_multi_jax_batched(batch_full[K:N]).H_tensor,
])
```

Isso segue diretamente do fato de que `jax.vmap` não cria interação entre
os elementos do batch — cada modelo é computado de forma completamente
independente. A paridade Fortran `<1e-12` (164/164 PASS) é preservada
integralmente, por modelo, para qualquer divisão de chunks.

### 2.4 Throughput Próximo ao Ótimo para a Maioria dos Cenários

O overhead de chunking (19,3 ms por chamada) é baixo em relação ao compute
para a maioria dos cenários práticos:

| Cenário | n_chunks (200k) | Overhead total | % do tempo total |
|:-------:|:---------------:|:--------------:|:----------------:|
| A | 40 | 0,8 s | 2,8% |
| B | 400 | 7,7 s | 0,3% |
| C | 1.334 | 25,7 s | 0,5% |
| E | 2.000 | 38,6 s | 0,2% |
| F | 5.715 | 110 s | 0,6% |
| G | 25.000 | 482 s (8 min) | **0,6%** |

Em todos os casos, o overhead acumulado é **< 3% do tempo total** — a
eficiência do chunking é > 97% do throughput máximo teórico.

### 2.5 Paralelismo com Geração de Modelos Geológicos

O loop de chunking permite sobreposição entre geração de parâmetros e
execução GPU:

```python
# Sprint A3 — versão com prefetch assíncrono
import concurrent.futures

def generate_models(seed_start, n):
    """Gera n modelos geológicos aleatórios (CPU, rápido)."""
    rng = np.random.default_rng(seed_start)
    return {
        'rho_h': rng.uniform(1., 100., (n, N_CAM)),
        'rho_v': ...,
        'esp':   ...
    }

executor = concurrent.futures.ThreadPoolExecutor(max_workers=1)
results = []
prefetch = executor.submit(generate_models, seed_start=i+1, n=CHUNK_SIZE)

for i in range(n_chunks):
    current_params = prefetch.result()  # parâmetros prontos
    if i < n_chunks - 1:               # enquanto GPU processa, CPU prepara próximo
        prefetch = executor.submit(generate_models, seed_start=i+2, n=CHUNK_SIZE)

    chunk_result = simulate_multi_jax_batched(**current_params, positions_z, ...)
    results.append(chunk_result.H_tensor)
```

Com `generate_models` levando < 1 ms e o chunk de GPU levando > 100 ms,
a sobreposição é quase total — custo de geração de parâmetros vira zero.

### 2.6 Checkpointing Natural

O loop de chunks permite salvar checkpoints intermediários sem risco de perder
todo o progresso em caso de falha de runtime (Colab desconecta, quota de GPU):

```python
# Sprint A3 — com checkpointing e retomada
import h5py, os

CHECKPOINT_FILE = "/content/drive/MyDrive/dataset_partial.h5"

with h5py.File(CHECKPOINT_FILE, "a") as f:
    if "H" not in f:
        f.create_dataset("H", shape=(N_TOTAL, nTR, nAng, n_pos, nf, 9),
                         dtype="complex128", chunks=(CHUNK_SIZE, nTR, nAng, n_pos, nf, 9))
        f.attrs["chunks_done"] = 0

    start_chunk = int(f.attrs["chunks_done"])

    for i in range(start_chunk, n_chunks):
        start, end = i * CHUNK_SIZE, min((i+1) * CHUNK_SIZE, N_TOTAL)
        result = simulate_multi_jax_batched(rho_h[start:end], ...)
        f["H"][start:end] = result.H_tensor
        f.attrs["chunks_done"] = i + 1  # checkpoint atômico
        f.flush()
```

Se o Colab desconectar no chunk 15.000 de 25.000, a próxima sessão retoma do
chunk 15.001. Nenhum dado é perdido.

### 2.7 Flexibilidade de Configuração por Chunk

Cada chunk pode ter configurações físicas diferentes — permitindo datasets
com curriculum de dificuldade crescente:

```python
# Exemplo: curriculum dip 0° → 30° → 90° por terço do dataset
for i, (start, end, dip) in enumerate([
    (0,        66_666, [0.]),    # fase 1: vertical
    (66_666,  133_333, [30.]),   # fase 2: oblíquo
    (133_333, 200_000, [87.5]), # fase 3: horizontal (JAX cap)
]):
    result = simulate_multi_jax_batched(
        rho_h[start:end], rho_v[start:end], esp[start:end],
        positions_z, dip_degs=dip, ...  # ← dip muda por fase
    )
```

**Atenção**: mudar `dip_degs` entre chunks **não** causa recompilação XLA
(dips são arrays de valores, não shapes). Mudar o número de dips (e.g., 1→4)
**causaria** recompilação (muda `nAngles` que afeta o shape do vmap).

## 3. Limitações

### 3.1 RAM do Host para Acúmulo do H_tensor (Limitação Estrutural)

O H_tensor completo de 200k modelos precisa ser mantido em RAM do host
(CPU) para ser salvo ou passado ao pipeline de treinamento:

| Cen | H_tensor (200k, RAM) | H_tensor (300k, RAM) | Viável em Colab (25 GB RAM)? |
|:---:|:--------------------:|:--------------------:|:----------------------------:|
| A | 0,03 GB | 0,05 GB | ✅ |
| B | 2,7 GB | 4,0 GB | ✅ |
| C | 10,7 GB | 16,1 GB | ✅ (mas apertado) |
| D | 0,1 GB | 0,2 GB | ✅ |
| E | 16,1 GB | 24,1 GB | ⚠️ marginal |
| **F** | **42,9 GB** | **64,4 GB** | **❌ OOM RAM** |
| **G** | **171,7 GB** | **257,6 GB** | **❌ OOM RAM** |

**Para F e G, a única estratégia viável é streaming para disco (HDF5/Zarr)**
durante a geração. Tentar acumular em RAM causará `MemoryError` no host
antes mesmo de a GPU terminar.

A solução (detalhada em §2.6) é escrever cada chunk diretamente no arquivo
HDF5 via `h5py`, mantendo na RAM apenas o buffer de um único chunk:

```
Chunk G (n=8):   H_tensor = 0.11 MB em RAM (por chunk) → totalmente seguro
Dataset G final: 171,7 GB em disco → requer storage no Google Drive
```

### 3.2 Loop Python de rho_h_at_obs (Limitação de Performance)

A função `simulate_multi_jax_batched` (multi_forward.py:1121-1135) contém
um loop Python pós-GPU que não é vetorizado:

```python
for i_model in range(n_models):     # O(n_models) = O(CHUNK_SIZE) por chunk
    for i_ang in range(nAngles):    # O(nAngles)
        for i_pos in range(n_pos):  # O(n_pos)
            lay = layer_at_depth(...)
            rho_h_at_obs[i_model, i_ang, i_pos] = rho_h_m[lay]
```

Para o dataset completo de 200k modelos, o tempo acumulado deste loop é:

| Cen | Iterações por chunk | n_chunks | Iterações totais | Tempo Python (~200 ns/iter) |
|:---:|:-------------------:|:--------:|:----------------:|:---------------------------:|
| A | 8 × 1 = 8 | 40 | 320 | < 1 ms — negligível |
| B | 8 × 100 = 800 | 400 | 320.000 | ~64 ms — negligível |
| E | 8 × 600 = 4.800 | 2.000 | 9.600.000 | **~1,9 s** |
| G | 8 × 4 × 100 = 3.200 | 25.000 | 80.000.000 | **~16 s** |

> Nota: o `rho_h_at_obs` só é relevante para a inversão (conhecer qual camada
> cada posição observa). Para geração de dataset puro do SurrogateNet, esse
> campo pode ser **ignorado** — apenas `H_tensor` é necessário. A solução de
> curto prazo é passar `return_rho_at_obs=False` (quando essa flag existir,
> Sprint A3), que eliminaria o loop completamente.

### 3.3 Último Chunk com Tamanho Diferente → Recompilação XLA

O XLA usa `CHUNK_SIZE` como dimensão estática do vmap. Se `n_models_total`
não for múltiplo de `CHUNK_SIZE`, o último chunk terá tamanho diferente:

```
n_models_total = 200.000, CHUNK_SIZE = 8 (cenário G):
200.000 / 8 = 25.000 chunks exatos → sem problema

n_models_total = 200.001, CHUNK_SIZE = 8:
Chunk 25.000: n_models = 1 → recompilação XLA (~30 s de custo)
```

**Mitigações** (em ordem de preferência):
1. Garantir que `n_models_total % CHUNK_SIZE == 0` (recomendado)
2. Padding: completar o último chunk com modelos duplicados e descartar após
3. Aceitar 1 recompilação extra no último chunk (custo único de ~30 s)

### 3.4 Overhead de Tempo Real por Número de Chamadas

Cada chamada Python à API custa ~19,3 ms fixos (kernel launch GPU, sync).
Para volumes grandes com chunk size pequeno:

| Cen | n_chunks (200k) | Overhead total | Impacto |
|:---:|:---------------:|:--------------:|:-------:|
| A | 40 | 0,8 s | Negligível |
| G | 25.000 | 482 s (8 min) | Notável mas <3% do total |
| H (chunk=1) | 200.000 | 64 min | **Proibitivo** — usar `lax.map` |

O cenário H com chunk=1 acumula 200.000 × 19,3 ms = 64 minutos só de
overhead. Para H, `jax.lax.map` é obrigatório (zero overhead entre steps,
pois a execução é intra-XLA).

### 3.5 Ausência de Progresso Observável (UX)

25.000 chamadas sequenciais sem feedback visual são problemáticas em sessões
Colab longas. Sem `tqdm` ou logging por chunk, o usuário não sabe se o processo
está travado ou progredindo.

**Solução mínima** (Sprint A3):
```python
from tqdm.auto import tqdm

for i, chunk_slice in enumerate(tqdm(chunk_slices, desc=f"Gerando {N_TOTAL:,} modelos")):
    result = simulate_multi_jax_batched(...)
    ...
```

### 3.6 Reprodutibilidade com Chunks (Semente por Chunk)

A semente aleatória para `_build_models` precisa ser derivada de forma
determinística por chunk para garantir reprodutibilidade:

```python
# Errado: mesma semente em todos os chunks → modelos duplicados
for i in range(n_chunks):
    models = _build_models(CHUNK_SIZE, seed=42)  # ← mesmo seed!

# Correto: semente derivada do índice do chunk
BASE_SEED = 42
for i in range(n_chunks):
    models = _build_models(CHUNK_SIZE, seed=BASE_SEED + i)  # ← único por chunk
```

Sem essa separação, todos os chunks gerariam os mesmos modelos geológicos,
produzindo um dataset com `n_models_unique = CHUNK_SIZE`, não `n_models_total`.

### 3.7 I/O e Storage para Cenários G/F

Para cenários G e F, o dataset de 200k modelos ocupa:
- F (200k): **42,9 GB em disco**
- G (200k): **171,7 GB em disco**
- G (300k): **257,6 GB em disco**

Implicações práticas:
- Colab: `/content` tem ~70-100 GB de armazenamento temporário → F cabe, G não
- Google Drive: `My Drive` tem limite de cota (tipicamente 15-100 GB dependendo
  do plano) → G requer Drive com expansão ou armazenamento alternativo (GCS, S3)
- Transferência para treinamento: streaming do HDF5 no Drive via `h5py` tem
  latência de ~100-300 MB/s → para G, carregar o dataset completo leva ~10 min.

**Alternativa prática**: gerar e treinar em streaming (`tf.data` lendo do HDF5
durante o treinamento, sem carregar tudo na RAM).

## 4. Comparação de Estratégias para 200k–300k Modelos

### 4.1 Estratégia 1 — Python Loop + np.concatenate (Atual, Sprint A3 Fase 1)

```python
results = []
for i, start in enumerate(range(0, N_TOTAL, CHUNK)):
    batch = simulate_multi_jax_batched(params[start:start+CHUNK], ...)
    results.append(batch.H_tensor)
H_full = np.concatenate(results, axis=0)
```

**Pros**: implementação trivial, zero mudança na API core, funciona agora
**Contras**: RAM cresce linearmente com n_total; OOM para F/G; sem checkpointing

### 4.2 Estratégia 2 — Python Loop + HDF5 Streaming (Recomendada Sprint A3)

```python
with h5py.File("dataset.h5", "w") as f:
    H_ds = f.create_dataset("H", shape=(N_TOTAL, ...), dtype="complex128")
    for i, start in enumerate(range(0, N_TOTAL, CHUNK)):
        batch = simulate_multi_jax_batched(params[start:start+CHUNK], ...)
        H_ds[start:start+CHUNK] = batch.H_tensor  # ← escreve direto no disco
```

**Pros**: RAM = O(1 chunk); checkpointing automático; suporta qualquer N_TOTAL
**Contras**: requer h5py; I/O adicional (~0,1-1 ms/chunk, negligível)

### 4.3 Estratégia 3 — jax.lax.map Intra-XLA (Futura Sprint A3+)

```python
# Um único kernel XLA, execução sequencial interna, 0 overhead entre steps
H_all = jax.lax.map(
    lambda params: simulate_single_model_jax(params),
    stacked_params  # (N_TOTAL, ...)
)
```

**Pros**: overhead zero entre modelos (1 kernel launch total); ideal para H
**Contras**: requer refatoração de `simulate_multi_jax_batched`; mais complexo;
throughput por step menor que vmap (sem paralelismo entre modelos)

### 4.4 Estratégia 4 — Híbrida: vmap por chunk + lax.map entre chunks

```python
# Melhor dos dois mundos: vmap para paralelismo, lax.map para eliminar overhead
chunked_params = stacked_params.reshape(n_chunks, CHUNK, ...)
H_all = jax.lax.map(
    lambda chunk: simulate_multi_jax_batched_jax_internal(chunk),
    chunked_params
)
```

**Pros**: paralelismo dentro do chunk + zero overhead entre chunks
**Contras**: mais complexo de implementar; requer que `simulate_multi_jax_batched`
seja um callable JAX puro (sem Python runtime no loop interno)

```

  ├── ThreadPoolExecutor para gerar parâmetros em background
  ├── Sobreposição GPU compute + CPU geração de modelos

Fase 3 (Sprint A4+, ~6h): lax.map para cenário H
  ├── Refatorar _one_model como callable JAX puro
  ├── jax.lax.map sobre eixo de modelos
  └── Eliminação dos 64 min de overhead acumulado no cenário H
```
