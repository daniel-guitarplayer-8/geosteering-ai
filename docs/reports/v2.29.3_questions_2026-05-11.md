## Pergunta 1 — Como utilizar o CLI do Geosteering AI?

### 1.1 Entry Point e Instalação

A CLI é declarada em [`pyproject.toml:62-64`](../../pyproject.toml#L62-L64):

```toml
[project.scripts]
geosteering-cli = "geosteering_ai.cli:main"
```

**Instalação (uma vez por venv)**:

```bash
cd ~/Geosteering_AI
source ~/Geosteering_AI_venv/bin/activate
pip install -e .   # modo editable — auto-detecta mudanças no código
```

Após instalação, o binário fica em `~/Geosteering_AI_venv/bin/geosteering-cli`
(ativo enquanto o venv estiver ativado).

**Três formas equivalentes de invocar**:

```bash
# Forma 1 — binário (preferida)
geosteering-cli version

# Forma 2 — módulo Python (não exige pip install)
python -m geosteering_ai.cli version

# Forma 3 — import Python (programático)
python -c "from geosteering_ai.cli import main; main(['version'])"
```

### 1.2 Subcomandos Disponíveis

A CLI tem 3 subcomandos. Listagem via `geosteering-cli --help`:

| Subcomando | Handler | Descrição |
|:-----------|:--------|:----------|
| `version` | inline em [`main.py`](../../geosteering_ai/cli/main.py) | Exibe versão atual (v2.24) |
| `simulate` | [`simulate.py`](../../geosteering_ai/cli/simulate.py) | Gera N modelos 1D 5-camada randomizados + executa simulação |
| `benchmark` | [`benchmark.py`](../../geosteering_ai/cli/benchmark.py) | Executa cenário canônico (A/B/C/D/E/F) com diferentes combinações `npos/freq/TR` |

### 1.3 `geosteering-cli simulate` — Argumentos

```bash
geosteering-cli simulate --help
```

| Flag | Tipo | Default | Descrição |
|:-----|:----:|:-------:|:----------|
| `--models` | int | 10 | Número de modelos sintéticos (1D 5-camada) |
| `--n-pos` | int | 600 | Posições de profundidade por modelo |
| `--workers` | int | None | Workers paralelos (None = auto-detect) |
| `--threads` | int | None | Threads Numba/worker (None = auto-detect) |
| `--seed` | int | 42 | Semente PRNG (reprodutibilidade) |
| `--out` | str | None | Diretório de saída `.npz` (None = só memória) |
| `--quiet` | flag | False | Suprime logs |

**Exemplo prático**:

```bash
# Simula 100 modelos com 2w × 4t e salva resultado em /tmp/sim
geosteering-cli simulate \
    --models 100 \
    --n-pos 600 \
    --workers 2 \
    --threads 4 \
    --seed 42 \
    --out /tmp/sim
```

**Saída esperada**:

```
[2026-05-11 17:23:00] Gerando 100 modelos sintéticos (seed=42, n_pos=600)...
[2026-05-11 17:23:00] Paralelismo: 2 workers × 4 threads/worker
[2026-05-11 17:23:15] OK: 100 modelos simulados em 15.27s (23,529 mod/h)
[2026-05-11 17:23:15] Resultados gravados em: /tmp/sim/simulate_results.npz
```

### 1.5 Testando o Simulador Numba JIT

**Teste rápido (warmup, ~5–10s)**:

```bash
geosteering-cli benchmark --scenario A --n 100
```

**Teste de produção — configuração padrão canônica do usuário**
(1 freq 20 kHz, dip 0°, 1 TR, n_pos=600, tj=120m, n=2000):

```bash
# Cold cache (primeira execução, ~80s)
find geosteering_ai -name "__pycache__" -type d -exec rm -rf {} +
geosteering-cli benchmark --scenario E --n 2000

# Warm cache (subsequente, ~75s)
geosteering-cli benchmark --scenario E --n 2000

# Steady-state (3ª+ execução, mediana estável)
geosteering-cli benchmark --scenario E --n 2000
```

**Variáveis de ambiente Numba úteis** (override do auto-detect):

```bash
# Forçar threads Numba (override auto-detect):
NUMBA_NUM_THREADS=2 geosteering-cli benchmark --scenario E --n 100

# Desabilitar JIT (modo debug, MUITO LENTO):
NUMBA_DISABLE_JIT=1 geosteering-cli benchmark --scenario A --n 1
```

### 1.6 Pitfalls Comuns

1. **Cache JIT não persiste entre processos**:
   Cada `python -m geosteering_ai.cli` é um processo novo. Para benchmarks
   precisos, rode 2 vezes consecutivas e descarte a 1ª medição (cold cache).

2. **Auto-detect pode falhar em containers Docker**:

   ```bash
   # Se falhar, especifique manualmente:
   geosteering-cli simulate --models 10 --workers 1 --threads 1
   ```

3. **Throughput varia por `n_models`** (warmup amortizado):

   | n_models | Throughput Cenário E |
   |:--------:|:--------------------:|
   | 200 | ~65k mod/h |
   | 1000 | ~92k mod/h |
   | 2000 | ~95k mod/h (steady) |

4. **Fortran ausente causa skip silencioso** em testes de paridade. Compile
   `Fortran_Gerador/tatu.x` antes de rodar paridade.

### 1.7 Solução: Workflow Recomendado para o Usuário

**Diário (validação rápida ~30s)**:

```bash
# Activate venv
source ~/Geosteering_AI_venv/bin/activate

# Smoke teste CLI
geosteering-cli version  # 1s

# Validação rápida do simulador
geosteering-cli benchmark --scenario A --n 100  # 5-10s
```

**Sprint validation (medição completa ~5min)**:

```bash

find geosteering_ai -name "__pycache__" -type d -exec rm -rf {} +

# 3 cenários representativos
geosteering-cli benchmark --scenario A --n 1000  # single-pos
geosteering-cli benchmark --scenario E --n 2000  # config padrão
geosteering-cli benchmark --scenario F --n 500   # multi-freq+TR
```

## Pergunta 3 — Por que só CodeRabbit é acionado?

### 3.1 Inventário Completo de Revisores

| `geosteering-code-reviewer` | Sonnet 4.6 | high | PEP 8, type hints, D1-D14, errata física |
| `geosteering-physics-reviewer` | Opus 4.7 | extra-high | Paridade Fortran <1e-12, simetria Maxwell |
| `geosteering-perf-reviewer` | Haiku 4.5 | high | Benchmarks Cenários A-K, regressão |
| `geosteering-security-auditor` | Sonnet 4.6 | high | Secrets, CVEs, path traversal |

#### Camada 2 — Agentes Task (Claude Code built-in)

### 3.2 Por que apenas CodeRabbit Parece Automático?

Investigando os fluxos de revisão:

1. **CodeRabbit CLI está instalado** (`~/.local/bin/coderabbit`) e tem
   slash command implícito via skill `coderabbit:code-review`. Quando você
   digita "revisar código" ou após implementação grande, o Claude tende a
   acionar essa skill por padrão.

2. **Skills locais NÃO têm trigger automático**: nenhum hook ou prompt
   pré-configurado invoca `geosteering-code-reviewer` automaticamente. O
   usuário ou Claude precisa solicitar explicitamente via:

   ```text
   /geosteering-code-reviewer
   ```

3. **Hooks PreToolUse/PostToolUse rodam silenciosamente**: hooks como
   `check-anti-patterns.sh` validam mas não geram review estruturado. Eles
   apenas bloqueiam ou avisam.

4. **Agentes Task (feature-dev:code-reviewer) precisam ser invocados via
   `Task tool`**: o Claude pode chamar `feature-dev:code-reviewer` se
   identificar contexto apropriado, mas não há automação determinística.

5. **Não há `/review` ou `/code-review` slash command global no projeto**:
   apenas externos (plugins). O `geosteering-orchestrator.md` menciona
   `coderabbit review --agent` no workflow de sprint.

### 3.3 Por que Apenas CodeRabbit é Acionado pelo Claude

**Hipótese**: o Claude tende a preferir CodeRabbit porque:

- É **integrado via plugin** (`coderabbit:code-review` skill)
- Tem **trigger semântico claro** ("revisar código", "review", "qualidade")
- Tem **CLI binário pronto** (executa imediatamente)
- **Retorna feedback estruturado** em JSON (parseable)

Skills locais como `geosteering-code-reviewer` exigem:

- Carregar skill no contexto (latência)
- Modelo Sonnet 4.6 dedicado (custo de tokens)
- Análise profunda contextual (5-10 min)
- Output formatado em markdown (revisão narrativa)

**O Claude opta pelo caminho de menor resistência (CodeRabbit) a menos que
o usuário solicite explicitamente skill especializada.**

### 3.4 Solução: Workflow Multi-Reviewer

**Para garantir cobertura COMPLETA, use TODOS os reviewers em sequência:**

```bash
# 1. Validação automática (sempre roda — hooks)
# Ao editar arquivo: PreToolUse + PostToolUse hooks executam

# 2. Revisão genérica rápida (CodeRabbit, ~60s)
coderabbit review --agent -t uncommitted

# 3. Revisão especializada Geosteering (skills locais)
# No Claude Code, digite:
/geosteering-code-reviewer        # PEP 8 + D1-D14 (~3 min)
/geosteering-physics-reviewer      # Paridade Fortran (~5 min)
/geosteering-perf-reviewer         # Throughput regression (~3 min)
/geosteering-security-auditor      # Secrets + CVEs (~1 min)

# 4. Para mudanças GRANDES (>5 arquivos): orquestrador
/geosteering-orchestrator
# (coordena os 4 reviewers acima em paralelo)

# 5. Validação final automática (Stop hook)
# Ao terminar a sessão: run-pytest.sh executa toda a suite
```

**Recomendação para o usuário**:

- **Para correção rápida**: deixe CodeRabbit (padrão atual) — bom para
  bugs sintáticos, type hints, dead code
- **Para revisão de domínio físico**: invoque `/geosteering-physics-reviewer`
  EXPLICITAMENTE — CodeRabbit não conhece Geosteering AI
- **Para mudanças no simulador hot-path**: invoque
  `/geosteering-perf-baseline` para garantir não-regressão de throughput
- **Antes de PR major**: invoque `/geosteering-orchestrator` para fan-out
  paralelo

## Pergunta 4 — Workers nativos no Simulador Python Numba JIT

### 4.1 Resposta Direta: JÁ ESTÁ IMPLEMENTADO

A feature solicitada **JÁ EXISTE** desde **Sprint v2.12 (2026-04-30)**.
Implementação em [`geosteering_ai/simulation/_workers.py`](../../geosteering_ai/simulation/_workers.py)
(1.010 LOC) e exposta via [`simulate_multi()`](../../geosteering_ai/simulation/multi_forward.py)
em `multi_forward.py`.

### 4.2 API Pública

```python
from geosteering_ai.simulation import simulate_multi
from geosteering_ai.simulation._workers import (
    recommend_default_parallelism,
    detect_cpu_topology,
)

# Detectar topologia do hardware
logical, physical, has_ht = detect_cpu_topology()
print(f"CPU: {physical} cores físicos, {logical} lógicos, HT={has_ht}")

# Recomendar paralelismo ótimo
n_workers, threads_per_worker = recommend_default_parallelism(n_models_hint=30000)
print(f"Recomendado: {n_workers} workers × {threads_per_worker} threads")

# Executar simulação batch
result = simulate_multi(
    models=[{"rho_h": ..., "rho_v": ..., "thicknesses": ...}, ...],
    positions_z=positions_z,
    frequencies_hz=[20000.0],
    tr_spacings_m=[1.0],
    dip_degs=[0.0],
    n_workers=n_workers,            # ← AQUI: workers explícitos
    threads_per_worker=threads_per_worker,  # ← AQUI: threads explícitas
)

print(f"Throughput: {result.throughput_mod_per_h:.0f} mod/h")
print(f"Modo: {result.mode}  # A=single, B=multi-thread, C=multi-worker, D=hybrid")
```

### 4.3 Modos de Execução

A função `_detect_mode()` retorna A/B/C/D baseado em `n_workers` e
`threads_per_worker`:

| Modo | n_workers | threads | Uso | Quando escolhe |
|:----:|:---------:|:-------:|:----|:---------------|
| **A** | 1 | 1 | Debug / 1 modelo | `recommend_default_parallelism(n_models_hint=1)` |
| **B** | 1 | N | 1 modelo grande | n_pos ≥ 100, threads do host |
| **C** | M | 1 | Batch paralelo | n_pos baixo |
| **D** | M | K | **DEFAULT PRODUÇÃO** | Batch >= 100, CPU 8C → 4w×2t |

**Anti-oversubscription**: `n_workers × threads_per_worker ≤ physical_cores`
(documentado em [`_workers.py:283-307`](../../geosteering_ai/simulation/_workers.py#L283-L307)).

### 4.4 CLI Já Expõe Workers

```bash
# Especificar workers e threads explicitamente
geosteering-cli benchmark --scenario E --n 1000 --workers 4 --threads 2
```

### 4.7 Solução: Como o Usuário Pode Usar

**Programaticamente** (Python script ou notebook):

```python
import numpy as np
from geosteering_ai.simulation import simulate_multi

# Gerar modelos
rng = np.random.default_rng(42)
models = []
for _ in range(2000):
    rho_h = rng.uniform(1.0, 100.0, size=5).astype(np.float64)
    rho_v = rho_h * rng.uniform(1.0, 3.0, size=5).astype(np.float64)
    esp = rng.uniform(2.0, 10.0, size=3).astype(np.float64)
    models.append({"rho_h": rho_h, "rho_v": rho_v, "thicknesses": esp})

# Executar com 4 workers × 2 threads
result = simulate_multi(
    models=models,
    positions_z=np.linspace(-5.0, 5.0, 600).astype(np.float64),
    frequencies_hz=[20000.0],
    tr_spacings_m=[1.0],
    dip_degs=[0.0],
    n_workers=4,
    threads_per_worker=2,
)

print(f"Throughput: {result.throughput_mod_per_h:.0f} mod/h")
```

**Via CLI** (mais simples):

```bash
geosteering-cli benchmark --scenario E --n 2000 --workers 4 --threads 2
```

## 7. Conclusão

Os 5 itens investigados revelam um projeto em **estado maduro e bem
arquitetado** para o simulador Numba JIT (com workers nativos já existindo
desde v2.12), mas com **dívida técnica acumulada no Simulation Manager
GUI** (monolítico 10.759 LOC). Os mecanismos de revisão são **abundantes
mas pouco automatizados** — Claude prefere CodeRabbit por menor latência,
mas skills locais (`geosteering-code-reviewer`, `geosteering-physics-reviewer`)
são superiores em domínio específico e precisam ser invocadas explicitamente.

**Próximos passos críticos**:

1. **Sprint v2.27**: pytest-qt (desbloquear MVC)
2. **Sprint v2.30**: split `simulation_manager.py` (ganho de tokens + preparação MVC)
3. **Sprint v3.0+**: MVC completo + Studio

**Sobre 150k mod/h**: continua não-atingido em medições atuais (~95k em
n=2000). Investigação em Sprint v2.30+ com hardware monitor + isolation
de processos é necessária.

**Eficiência tokens**: `simulation_manager.py` consome 16% de uma janela
500k. Split em sub-módulos reduz para ~3% por consulta. Recomendação:
priorizar split na Sprint v2.27 (paralelo ao pytest-qt).
