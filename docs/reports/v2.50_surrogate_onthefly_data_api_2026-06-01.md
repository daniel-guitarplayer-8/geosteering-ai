## 1. O que era a dívida

O Sprint C validou o treino fim-a-fim, mas a geração de pares ficou **hard-rolled
num benchmark** + round-trip `.npz` offline. `data/surrogate_data.py` só fazia
**extração offline** (`extract_surrogate_pairs(data_22col)`); a geração via
`generate_batch` tinha um **laço Python O(N×L)** montando `dat_22col` — "inviável
a 30k×600". Esta sprint produtiza o caminho on-the-fly como **API reusável e testada**.

---

## 2. Arquitetura da API (3 níveis)

```
┌──────────────────────────────────────────────────────────────────────┐
│  surrogate_pairs_from_batch(batch, cfg)   ← ponte (GeneratedBatch→DS)  │
│        ▲ vetorizado: H_tensor → (N,L,22) → extract_surrogate_pairs     │
│  generate_surrogate_dataset(cfg, n_models, ...)  ← one-shot on-the-fly │
│        ▲ amostra + simula (dispatcher) + ponte; build_dat_22col=False  │
│  iter_surrogate_batches(cfg, batch_size, ...)    ← stream fresco/época │
│        ▲ seed+i por batch → modelos novos; refresh entre épocas        │
└──────────────────────────────────────────────────────────────────────┘
```

| Função | Assinatura (resumo) | Reuso |
|:-------|:--------------------|:------|
| `surrogate_pairs_from_batch` | `(batch, config, *, apply_decoup=True) → SurrogateDataset` | `_layer_at_batch`, `extract_surrogate_pairs` |
| `generate_surrogate_dataset` | `(config, *, n_models, n_positions=600, n_layers=5, …, seed, generator=None, **gen_kwargs)` | `SyntheticDataGenerator.generate_batch` |
| `iter_surrogate_batches` | `(config, *, batch_size, n_batches=None, seed=0, …) → Iterator[SurrogateDataset]` | `generate_surrogate_dataset` |
| `_resolve_generator` | reroteia `fortran_f2py → auto` (default do config) | `dataclasses.replace` (não muta caller) |

## 5. Throughput — SEM regressão (verificado)

A produtização da Fase A roteia pelo caminho **grouped** (dispatcher Sprint B) em
vez dos blocos-por-geometria hand-tuned. Investigação rigorosa:

| Gate / medição | Resultado | Veredito |
|:---------------|:----------|:---------|
| **DG perf gate** (`test_simulation_jax_perf_baseline`) | **6/6 PASS** (A/B/E/G/**DG** + estrutura) | **autoritativo: 0 regressão de pacote** |
| **A/B grouped vs blocks** @128/grupo (regime 30k) | NEW_sim **155.6s** vs OLD blocks **186.8s** → **ratio 0.83×** | grouped é **17% mais rápido** que blocos diretos |
| `surrogate_pairs_from_batch` (extração) | **0.41s** @ 2048×600 | desprezível (não é gargalo) |

**Conclusão**: NÃO há regressão estrutural — o caminho da API (grouped) é
**equivalente-ou-mais-rápido** que os blocos diretos no regime de ocupação de
produção (≈128 modelos/grupo). O gate DG (que mede throughput do pacote) está verde.

**Nota honesta sobre o 30k absoluto**: um run isolado de 30k×600 em processo FRIO
mediu 1962s (vs baseline de sessão anterior 491s). Isso é **ambiental** (cache JIT
frio / estado de máquina) e afeta **ambos** os caminhos IGUALMENTE — o A/B controlado
(mesmo processo, com warmup) prova que grouped ≤ blocks (0.83×). O simulador/kernel
**não foi tocado**, então não há como introduzir regressão de throughput de pacote
(confirmado pelo gate DG). Não foi necessário manter um fallback `--blocked`.
