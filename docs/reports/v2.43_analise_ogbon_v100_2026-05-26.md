## Sumário Executivo

O OGBON (Petrobras/SENAI CIMATEC) possui **312 NVIDIA V100 SXM2 (16 GB)** em 78 nós com 4 GPUs cada, conectados por InfiniBand EDR. A descoberta crítica desta análise é que o **V100 tem FP64 ratio 1:2** — o mesmo tier arquitetural que o A100, e **25× mais rápido que a RTX A6000 em FP64**. Para o simulador `complex128` do Geosteering AI (que exige `jax_enable_x64=True`), o OGBON representa um salto qualitativo sem precedentes.

Porém, há um obstáculo arquitetural fundamental: **Claude Code não pode acessar diretamente os nós de computação do OGBON** porque esses nós tipicamente não têm acesso à internet (necessário para a API Anthropic). O fluxo recomendado é **desenvolvimento híbrido**: Claude Code na workstation local (A6000) + execução de testes/benchmarks no OGBON via SLURM.

| Cenário | Viabilidade | Impacto |
|:--------|:-----------:|:-------:|
| Claude Code acessando GPU V100 diretamente | ❌ Internet bloqueada em compute nodes | — |
| Claude Code no nó de login, submetendo jobs | ✅ Parcial (sem GPU no login node) | MÉDIO |
| Desenvolvimento local (A6000) + execução OGBON | ✅ Recomendado | ALTO |
| PINN training multi-GPU (4× V100, pmap) | ✅ Viável | ESTRATÉGICO |
| Benchmark JAX V100 (FP64 puro, 4-GPU) | ✅ Viável | ALTO |

## Pergunta 1: Claude Code no OGBON — Arquitetura e Diferenças vs Colab

### 1.1 Arquitetura do OGBON

O OGBON é um supercomputador da plataforma **Bull Sequana X1000** (Atos/Eviden), operado pelo SENAI CIMATEC em Salvador, BA, para a Petrobras. Especificações confirmadas:

| Componente | OGBON |
|:-----------|:------|
| **Total de GPUs** | **312 NVIDIA V100 SXM2** |
| **VRAM por GPU** | **16 GB HBM2** |
| **Nós GPU** | **78 nós × 4 GPUs/nó** |
| **FP64 por GPU** | **7.8 TFLOPS (ratio 1:2)** |
| **Bandwidth HBM2** | **900 GB/s por GPU** |
| **NVLink intra-nó** | V100 SXM2 suporta NVLink 2.0 (confirmação pendente por nó) |
| **Inter-nó** | **Mellanox InfiniBand EDR** (~25 Gb/s por link) |
| **Job scheduler** | Provavelmente SLURM (padrão Bull/Atos) |
| **Filesystem** | Lustre paralelo (padrão Bull Sequana) |

**Referência TOP500**: O sistema está registrado no TOP500 como entrada #179703.

### 1.2 Por que Claude Code NÃO pode acessar GPU V100 diretamente

Claude Code requer três condições simultâneas:
1. **Conexão estável à internet** → `api.anthropic.com` (HTTPS/443)
2. **Sessão interativa persistente** → bash em tempo real
3. **Acesso de leitura/escrita ao codebase** → filesystem local

Em supercomputadores HPC, nós de computação (onde as GPUs ficam) **tipicamente não têm acesso à internet outbound**. Esta é uma política de segurança universal em HPC de produção — isola o ambiente de computação de ataques externos e evita consumo de banda da rede de interconexão para tráfego não-HPC.

```
┌─────────────────────────────────────────────────────────────────┐
│  Arquitetura típica OGBON (Bull Sequana X1000)                  │
│                                                                 │
│  Internet ←→ Nó de Login (SSH, internet, sem GPU)              │
│                    ↕ InfiniBand EDR / Ethernet                  │
│              SLURM scheduler / Lustre filesystem                │
│                    ↕                                            │
│         ┌──────────────────────────────────┐                   │
│         │  Nós de computação GPU           │  ← SEM INTERNET   │
│         │  78 nós × 4 × V100 SXM2 16 GB   │                   │
│         │  NVLink intra-nó                 │                   │
│         └──────────────────────────────────┘                   │
└─────────────────────────────────────────────────────────────────┘
```

### 1.3 Quatro cenários de uso do Claude Code com OGBON

#### Cenário A: Desenvolvimento Híbrido Local + OGBON ✅ (RECOMENDADO)

```
Workstation Local (A6000, Claude Code ativo)
    │
    ├── Edita código Python/Fortran (Claude Code lê/escreve)

```

**Vantagem**: Claude Code opera inteiramente na workstation (internet garantida). Os V100s executam os testes em paralelo. O ciclo de iteração é: editar → push → sbatch → wait → pull → analisar.

#### Cenário B: Claude Code no Nó de Login ⚠️ (Viável com restrições)

```bash

ssh daniel@ogbon.senaicimatec.edu.br

```

**Funciona para**: edição de código, submissão de jobs, leitura de resultados.
**NÃO funciona para**: execução direta de GPU (login node não tem GPU).

**Riscos**: nós de login têm política de uso justo (fair use). Processos pesados (compilação Python, pip install) podem ser interrompidos. Claude Code em si (processo leve de CLI) deve ser tolerado.

#### Cenário C: Nó de Computação Interativo ❌ (Improvável na maioria dos HPC)

```bash

     --time=4:00:00 --partition=gpu --pty bash
# Dentro do nó: sem internet → Claude Code não pode contatar API
```

**Pode funcionar SE**: OGBON tem proxy HTTPS para internet nos compute nodes (incomum) OU se usar `anthropic_api_url` apontando para proxy interno.

#### Cenário D: VSCode Remote + Claude Code Extension ✅ (Alternativa ergonômica)

VSCode suporta "Remote SSH" para conectar ao nó de login. A extensão Claude Code do VSCode funciona no servidor remoto. Combina a ergonomia IDE com acesso SSH à infraestrutura.

## Pergunta 2: Configuração do Geosteering AI no OGBON

### 2.1 Viabilidade de ambiente Python customizável

O OGBON usa sistema de módulos (Lmod ou Environment Modules — padrão Bull/Atos). O usuário **não precisa de sudo** para instalar pacotes Python: conda/mamba em espaço do usuário é o caminho padrão.

**Pré-requisitos verificar com o admin OGBON**:
- `module avail python` → versão Python 3.13 disponível?
- `module avail cuda` → CUDA 12.x ou apenas CUDA 11.x?
- Política de uso de conda/mamba em user-space
- Cota de espaço em `/home` e `/scratch`
- Acesso a PyPI durante jobs (nós de computação têm internet para pip install?)

### 2.2 Sequência de setup no OGBON


**Scripts helper para automatizar o workflow:**

```python
# scripts/submit_and_wait.py
# Claude Code pode usar Bash tool para rodar este script:
import subprocess, time, pathlib, sys

def submit_and_wait(script: str, poll_interval: int = 30) -> dict:
    """Submete job SLURM e aguarda conclusão, retornando resultados."""
    result = subprocess.run(
        ["ssh", "ogbon", f"cd ~/geosteering-ai && sbatch {script}"],
        capture_output=True, text=True
    )
    job_id = result.stdout.strip().split()[-1]
    print(f"Job submetido: {job_id}")

    while True:
        status = subprocess.run(
            ["ssh", "ogbon", f"squeue -j {job_id} -h -o '%T'"],
            capture_output=True, text=True
        ).stdout.strip()
        if not status:  # job completou, saiu da fila
            break
        print(f"  Status {job_id}: {status}")
        time.sleep(poll_interval)

    # Copiar resultados
    subprocess.run(["rsync", "-avz",
                    f"ogbon:~/geosteering-ai/results/",
                    "./results/"])
    return {"job_id": job_id, "status": "COMPLETED"}
```

## Pergunta 3: Considerações Importantes que Você Pode Estar Esquecendo

### 3.1 REVELAÇÃO CRÍTICA: V100 FP64 — O Game Changer

A descoberta mais importante desta análise: o **V100 SXM2 é um GPU datacenter com FP64 ratio 1:2**, igual ao A100. Isso coloca o OGBON em uma categoria completamente diferente da workstation A6000 para o simulador `complex128`.

| GPU | FP64 (TFLOPS) | Ratio FP64:FP32 | VRAM | Bandwidth | Tier para simulador |
|:----|:-------------:|:---------------:|:----:|:---------:|:-------------------:|
| T4 (Colab) | ~0.25 | **1:128** | 15 GB | 300 GB/s | Inferência (limitado) |
| RTX A6000 (local) | **~0.31** | **1:128** | 48 GB | 768 GB/s | Inferência (limitado) |
| **V100 SXM2 (OGBON)** | **7.8** | **1:2** | **16 GB** | **900 GB/s** | **Datacenter FP64** |
| A100 SXM4 (Colab) | 9.7 | 1:2 | 40 GB | 2 TB/s | Datacenter FP64 |

**Implicação direta**: Para o simulador JAX com `complex128`, OGBON V100 é **25× mais poderoso que a workstation A6000** em computação FP64 pura. A comparação correta é V100 com A100, não com T4 ou A6000.

### 3.2 ALERTA: V100 tem apenas 16 GB VRAM (mesmo que T4)

O V100 SXM2 do OGBON tem **16 GB HBM2** — apenas 1 GB a mais que o T4 (15 GB). Cenários que causavam OOM no T4 podem causar OOM no V100:

| Cenário | VRAM estimada | T4 (15 GB) | V100 (16 GB) | A100 (40 GB) |
|:--------|:-------------:|:----------:|:------------:|:------------:|
| E (n=50, pos=600) | ~2 GB | ✅ | ✅ | ✅ |
| F (n=20, 4×4×1) | ~5 GB | ✅ com n=20 | ✅ com n=20 | ✅ com n=50 |
| G (n=5, 4×4×4) | ~8 GB | ✅ com n=5 | ✅ com n=5 | ✅ com n=50 |
| H (n=5, 8×8×8) | **>15 GB** | ❌ OOM | ⚠️ Risco OOM | ✅ com n=5 |

**Solução**: Cenário H no OGBON requer multi-GPU (`jax.pmap` com 2-4 V100s), pois 2 GPUs = 32 GB total.

### 3.3 Projeções de Throughput JAX V100 (simulador `complex128`)

Com base nos baselines T4 e A100 e nos specs do V100:

**Situação atual (pré-Sprint O2 — kernel overhead-bound a ~7% FP64 utilization):**

```
V100 vs T4 (cenário E, FP64):
  - Overhead launch latency domina → V100 ≈ A100 ≈ T4 (1.0-1.2× T4)
  - Estimativa: ~43,000–55,000 mod/h (similar ao T4 43,021 mod/h)
  - O gargalo NÃO é FP64 compute: é kernel launch overhead do JAX

V100 vs T4 (cenário A, B — vmap-heavy):
  - Mais paralelismo, melhor ocupação → V100 > T4
  - Estimativa A: ~8-10M mod/h (+15-45% vs T4 6.9M)
  - Estimativa B: ~280-350k mod/h (+9-36% vs T4 257k)
```

**Após Sprint O2 (fori_loop + FilterLoader module-scope — recupera 20-25%):**

```
V100 cenário E (pós-O2): ~55,000–80,000 mod/h
V100 cenário A (pós-O2): ~10-14M mod/h
V100 cenário B (pós-O2): ~350-480k mod/h
```

**Com 4× V100 (pmap no nó OGBON — Sprint A2):**

```
jax.pmap over n_models (split batch por GPU):
  E (4× V100, pós-O2): ~220k–320k mod/h
  B (4× V100, pós-O2): ~1.4M–1.9M mod/h
  A (4× V100, pós-O2): ~40M–56M mod/h

→ 4× V100 em E se aproxima/supera Numba 7970X (~500k mod/h)!
```

**Com complex64 opt-in (Sprint O4) — Tensor Cores V100 (125 TFLOPS FP16):**

```
V100 Tensor Cores FP16: 125 TFLOPS
V100 FP64:              7.8 TFLOPS (16× diferença)

Aceleração complex64 vs complex128: estimada 8-16×

E (1× V100, complex64, pós-O4): ~440k–640k mod/h
→ 1 V100 sozinho supera Numba 7970X em E com complex64!
4× V100: ~1.7M–2.5M mod/h em E (3-4× Numba 7970X)
```

### 3.4 Multi-GPU pmap: Sprint A2 Imediatamente Viável no OGBON

O OGBON tem 4 V100 SXM2 por nó com NVLink 2.0. Isso torna a Sprint A2 (`jax.pmap`) imediatamente testável, algo que o Colab Pro+ não oferece (1 GPU por sessão).

```python
# Sprint A2 — jax.pmap sobre n_models (batch split por GPU)
# 4 V100 × 16 GB = 64 GB VRAM disponível via pmap

from jax import pmap
import jax.numpy as jnp

# n_models deve ser múltiplo de n_gpus
n_gpus = len(jax.devices("gpu"))  # 4 no OGBON
n_models_per_gpu = n_models // n_gpus  # e.g., 200 // 4 = 50

# Reshape batch para (n_gpus, n_models_per_gpu, ...)
rho_h_pmap = rho_h_batch.reshape(n_gpus, n_models_per_gpu, -1)

# pmap distribui automaticamente entre 4 V100s
simulate_pmap = pmap(simulate_single_gpu_batch, axis_name="gpus")
result = simulate_pmap(rho_h_pmap, rho_v_pmap, esp_pmap, ...)
```

### 3.5 CUDA Version — Risco de Incompatibilidade

O V100 foi lançado em 2017 e suporta CUDA 10.1–12.x. Mas o driver CUDA instalado no OGBON determina o que é possível:

```bash
# No nó de login, verificar:
nvidia-smi | grep "CUDA Version"
module avail cuda  # versões disponíveis

# Se CUDA Driver < 12.0:
# jax[cuda12] NÃO funciona
# Usar: pip install "jax[cuda11_pip]" -f <url>

# Se CUDA Driver >= 12.0:
# jax[cuda12] funciona normalmente
```

**Risco médio**: Clusters HPC corporativos às vezes têm drivers CUDA desatualizados (CUDA 11.7-11.8 ainda comum em 2026 para sistemas V100 legacy). Verificar **antes** de qualquer setup.

### 3.6 XLA Compilation Cache no Lustre

O Lustre é otimizado para **arquivos grandes** (I/O sequencial de GB), não para **muitos arquivos pequenos** (o padrão do cache XLA). O cache XLA cria centenas de arquivos `.pb` de ~1-10 MB cada.

```bash
# PROBLEMA: JAX_COMPILATION_CACHE_DIR no Lustre → lento, metadata overhead
export JAX_COMPILATION_CACHE_DIR=$LUSTRE_HOME/jax_cache  # ❌ lento

# SOLUÇÃO 1: Cache em /tmp local do nó (apagado quando job termina)
export JAX_COMPILATION_CACHE_DIR=/tmp/jax_cache_$SLURM_JOB_ID  # ✅ rápido, volátil

# SOLUÇÃO 2: Cache em NVMe local do nó (se disponível)
export JAX_COMPILATION_CACHE_DIR=/local_nvme/jax_cache  # ✅ rápido, volátil

# SOLUÇÃO 3: Lustre com striping para small files
lfs setstripe -c 1 $LUSTRE_HOME/jax_cache  # ✅ melhor metadata em Lustre
export JAX_COMPILATION_CACHE_DIR=$LUSTRE_HOME/jax_cache
```

**Recomendação**: Use `/tmp` para o cache durante jobs. O custo do cold-start XLA (~30-180s) é pago uma vez por job; o benefício de persistência inter-job é menor do que a latência Lustre em small files.

### 3.7 Impacto no Treinamento TF/Keras (SurrogateNet + INN)

Além do simulador, o OGBON tem impacto direto no pipeline de treinamento de modelos de inversão:

| Modelo | VRAM (FP32 train) | 1× V100 (16 GB) | 4× V100 (64 GB) | A6000 (48 GB) |
|:-------|:-----------------:|:---------------:|:---------------:|:-------------:|
| ResNet-18 (~12M params) | ~2-4 GB | ✅ batch 2048 | ✅ batch 8192 | ✅ batch 4096 |
| TCN Surrogate (~127M params) | ~8-16 GB | ⚠️ batch pequeno | ✅ | ✅ |
| SurrogateNet ModernTCN (~204M params) | ~16-32 GB | ❌ (limit 16GB) | ✅ (64 GB pmap) | ✅ (48 GB) |
| INN (~200M params + jacobiano) | ~30-60 GB | ❌ | ✅ (64 GB pmap) | ⚠️ apertado |

### 3.12 Latência de Iteração — O Trade-off Principal

A maior diferença qualitativa entre OGBON e Colab é a velocidade de iteração:

| Fluxo | Colab Pro+ | OGBON |
|:------|:----------:|:-----:|
| Submeter teste GPU | 0 min (interativo) | 5-60 min (fila SLURM) |
| Ver resultado | Imediato | 5-60 min + 2 min job |
| Ciclo completo edit→test | ~5-15 min | **30-120 min** |
| Iterações/dia | ~20-50 | **5-15** |

**Para o desenvolvimento do simulador JAX GPU com Claude Code**, a velocidade de iteração é crítica. A solução é:
1. **Testes unitários rápidos** em CPU local (sem SLURM) para 90% das iterações
2. **Jobs SLURM** apenas para benchmarks definitivos e validação GPU
3. **Suite de GPU tests** bem estruturada: testes rápidos (<5 min) separados de benchmarks longos (>1h)
