## 1. Por Que Existe Tanto Espaço para Ganho?

A A100 opera a ~7% do pico FP64 teórico — não por limitação de hardware, mas por
três causas estruturais independentes identificadas na investigação multi-agente:

### 1.1 Gargalo n.º 1 — Loop Python fora da GPU (causa dominante em E/G)

```
multi_forward.py:1123-1135
for iAng in range(n_ang):
    for iPos in range(n_pos):
        rho_h_at_obs[...] = np.searchsorted(...)   ← CPU Numba
```

Para cenário E (n_models=50, n_pos=600, n_ang=2): **60.000 chamadas Numba individuais**
executam na CPU *depois* que a GPU termina. Estimativa: 15–60 ms acumulados por call,
comparável ao tempo de compute JAX. A GPU fica ociosa esperando a CPU iterar o loop.

**Este é provavelmente o maior ganho isolado de toda a iniciativa.**

### 1.2 Gargalo n.º 2 — Kernel launches excessivos via `fori_loop`

O `jax.lax.fori_loop` sobre `n_cam` camadas emite **1 kernel CUDA por iteração**
por modelo × posição × frequência. Com overhead de ~5–10 μs por launch:

```
n_cam=5 → 4 launches = 20–40 μs de overhead puro por unidade vmap
Cenário E: 50 modelos × 600 posições × 1 freq = 30.000 unidades → 600 ms–1,2 s desperdiçados
```

`lax.scan(unroll=n_cam)` funde todas as iterações em 1 kernel, eliminando esse overhead.

### 1.3 Gargalo n.º 3 — Overcompute de 5× nos dipoles

`_hmd_tiv_descent_body` e `_vmd_descent_body` em `dipoles_unified.py:200-285` calculam
**todos os 5 candidatos geométricos** a cada iteração do `fori_loop` via `jnp.where`,
descartando 4/5 do resultado. Isso representa ~40% do compute real dos kernels dipole
desperdiçado em aritmética inútil — eliminável com `lax.cond` binarizado.

### 1.4 Causa estrutural — complex128 é bandwidth-bound, não compute-bound

```
FP64 bandwidth A100: 1.555 GB/s HBM
FP64 compute A100 : 9,7  TFLOPS
FP32 compute A100 : 19,5 TFLOPS (mas sem Tensor Cores para complex)
complex128 tamanho : 16 bytes/elemento
complex64  tamanho :  8 bytes/elemento (50% menor → 2× mais modelos/bandwidth)
```

O simulador EM TIV é **bandwidth-bound** (muitas leituras/escritas de arrays de camadas
pequenos), não compute-bound. Migrar para complex64 (Sprint O4) reduz pressão no HBM em
2× *e* libera 2× mais VRAM para batches maiores — dois multiplicadores simultâneos.

### 2.4 Sprint O4 — complex64 Opt-In (estimativa: 3–4 semanas, R&D estratégica)

**Ganho adicional: 3–8× no hot path · REQUER ADR formal · NÃO muda o modo default**

Esta é a otimização de maior impacto isolado, mas com uma restrição absoluta:
**a paridade Fortran `<1e-12` proíbe complex64 no modo default para sempre**.
O Sprint O4 implementa complex64 como **opt-in explícito** (`cfg.dtype = "complex64"`),
destinado à geração de dados de treinamento onde 6-7 dígitos de precisão são suficientes.

**Mecanismo dos 3-8× em workloads bandwidth-bound:**

```
complex128 (16 bytes/elem) → complex64 (8 bytes/elem):
  ├── 2× menos pressão HBM bandwidth → 2× mais throughput em workloads BW-bound
  ├── 2× mais modelos cabem em VRAM → batch efetivo dobra → 2× mais amortização
  ├── Multiplicador combinado: ~2× × ~2× = ~4× (com saturação por outros overheads)
  └── Range real em benchmarks EM: 3–8× (Literatura: CSEM/TEM JAX complex64 vs 128)
```

**Por que o range é amplo (3–8×):**

- Se O1-O3 já eliminaram todos os overheads de CPU/launch, o hot path restante é puro
  compute/bandwidth → complex64 dá o máximo (~8×).
- Se ainda houver overheads residuais de CPU (~20% do tempo), complex64 age só nos ~80%
  JAX, resultando em ~4-5× efetivo.

## 6. A Relação com o i9 Histórico — Contexto Correto

A comparação com o i9 histórico requer normalização:

```
i9 histórico medido: 4 workers × 2 threads = 8 núcleos físicos em paralelo
A100 medido:         1 device (single GPU)

Para comparação justa (1 GPU vs 1 CPU):
  A100 Cenário E atual: 47.812 mod/h (1 GPU)
  i9 Cenário E histórico: 122.000 mod/h (8 núcleos paralelos)
  Normalizado per-core: i9 ≈ 15.250 mod/h/núcleo → A100 já é 3,1× mais eficiente por unidade

Para igualar a produtividade TOTAL do i9 com GPU:
  Pós-O2: 115–160k mod/h em 1 GPU → equivalente a 1-1,3 "i9" sem paralelismo multi-GPU
  Com pmap em 4 GPUs: 460–640k mod/h → 3,8-5,2× a capacidade produtiva do i9
```

A meta real não é "superar o i9 por dispositivo" — é extrair o máximo de cada GPU
para que, quando disponíveis múltiplas GPUs (e.g., pmap Sprint O4+), a capacidade
produtiva total escale linearmente.
