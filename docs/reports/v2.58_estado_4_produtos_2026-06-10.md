## Sumário Executivo

O Geosteering AI é um **ecossistema de 4 produtos sobre um núcleo de física compartilhado**.
Os quatro "conversam entre si" porque **reaproveitam o mesmo núcleo** — não porque um chama o
outro. O núcleo é o pacote importável `geosteering_ai/simulation/` (rota física única
`dispatch.simulate_batch`, paridade Fortran/JAX/Numba `<1e-12`) e a fundação MVVM
`geosteering_ai/gui/`.

| # | Produto | Maturidade (est.) | Estado | Entry point principal |
|:-:|:--------|:-----------------:|:-------|:----------------------|
| ① | **Geosteering CLI** | ~75 % | MVP maduro em produção | `geosteering-cli {simulate,benchmark,version}` |
| ② | **Simulation Manager** | monólito 100 % · MVVM ~35 % | Monólito funcional; MVVM em Strangler Fig | `python -m geosteering_ai.simulation.tests.simulation_manager` / `python -m apps.sim_manager.app` |
| ③ | **API / Bibliotecas Python** | ~72 % | Núcleo maduro; API REST MVP | `import geosteering_ai` / `geosteering-api` |
| ④ | **Geosteering AI Studio** | 0 % código | Só requisitos; bloqueado pela Fase 0 | (não existe `apps/studio/`) |

**Conclusão arquitetural (resposta à pergunta-chave):** os simuladores **Numba JIT e JAX GPU
são bibliotecas/APIs Python**, não a CLI. A CLI, o SM e a API REST são *adapters finos*
(arquitetura hexagonal) sobre o mesmo núcleo. **O princípio "tudo é biblioteca acessível por
CLI/SM/import/Studio" já é a arquitetura atual** — o trabalho restante é completar os adapters
(SM MVVM, `/simulate` REST) e preservar os dois invariantes que sustentam a propriedade.

## 1. Pergunta Arquitetural — Os simuladores são bibliotecas ou a CLI?

### 1.1 Resposta

Os simuladores vivem em **`geosteering_ai/simulation/`** — um pacote Python puro, importável
(`pip install -e .`). Os dois backends são submódulos intercambiáveis selecionados por
**parâmetro**, nunca por duplicação:

```
geosteering_ai/simulation/
├── dispatch.py          ← simulate_batch(...) : ROTA FÍSICA ÚNICA (árvore de decisão JAX⇄Numba)
├── _numba/              ← backend CPU (njit + prange + ProcessPool)
├── _jax/                ← backend GPU/CPU (jit + vmap + scatter)
├── multi_forward.py     ← simulate_multi(...) : caminho pool Numba
├── filters/             ← FilterLoader (Hankel Werthmüller 201pt default)
└── __init__.py          ← superfície pública (__all__, 25 símbolos)
```

### 1.2 Prova de uso direto (executado, sem CLI)

```python
from geosteering_ai.simulation import simulate_batch, SimulationConfig
import numpy as np

rho_h = np.array([[10.0, 50.0, 10.0]]); esp = np.array([[5.0]])
pz = np.linspace(-3, 3, 5)
H, info = simulate_batch(rho_h, rho_h, esp, pz, backend="numba",
                         frequencies_hz=[20000.0], tr_spacings_m=[1.0], dip_degs=[0.0])
# → H.shape = (1, 1, 1, 5, 1, 9) ; info["backend"] == "numba"   ✓
```

A seleção de backend é `SimulationConfig(backend="numba"|"jax"|"auto")` ou
`simulate_batch(..., backend=)`. O modo `"auto"` decide via árvore medida
(`dispatch._resolve_backend`: GPU + n≥32 + geometria agrupável → JAX bucketed; senão Numba).

### 1.3 Superfície pública (`geosteering_ai.simulation.__all__`)

```
simulate · simulate_batch · simulate_multi · simulate_multi_jax ·
simulate_multi_jax_batched · simulate_multi_jax_batched_grouped ·
SimulationConfig · compute_jacobian · compute_jacobian_jax · compute_jacobian_fd_numba ·
FilterLoader · HankelFilter · group_by_geometry · detect_cpu_topology ·
recommend_default_parallelism · release_numba_cache · get_jit_cache_info ·
SimulationResult · MultiSimulationResult · MultiSimulationResultJAX ·
MultiSimulationResultBatch · MultiSimulationResultBatchedJAX · JacobianResult ·
get_numba_cache_size · release_pool
```

### 1.4 Quem consome (todos são adapters — nenhum reimplementa física)

| Consumidor | Como acessa o simulador | Física própria? |
|:-----------|:------------------------|:---------------:|
| **CLI** (`geosteering_ai/cli/`) | `from geosteering_ai.simulation.dispatch import simulate_batch` + `simulate_multi` (`_exec.py:612-699`) | **Não** |
| **SM** (monólito + MVVM) | `from geosteering_ai.simulation import simulate_batch` (`SimulationThread.run`) | **Não** |
| **API REST** (`geosteering_ai/api/`) | wrapper HTTP de `InferencePipeline` / `simulate_batch` | **Não** |
| **Studio** (futuro) | reaproveitará `gui/services/` + `simulation/` (Strangler Fig) | **Não** (planejado) |

### 1.5 Diagrama hexagonal (núcleo + adapters)

```
                 ┌──────────── ADAPTERS (sem física) ────────────┐
                 │                                                │
   $ geosteering-cli         python -m apps.sim_manager      uvicorn .api.app
        │                          │                               │
        ▼                          ▼                               ▼
   ┌─────────┐              ┌──────────────┐                ┌─────────────┐
   │ ① CLI   │              │ ② Sim Manager│                │ ③ API REST  │
   └────┬────┘              └──────┬───────┘                └──────┬──────┘
        │                          │                               │
        └──────────────┬───────────┴───────────────┬───────────────┘
                       ▼                            ▼
            ┌───────────────────────── NÚCLEO ─────────────────────────┐
            │  geosteering_ai/simulation/dispatch.simulate_batch(...)   │
            │     ├── _numba/ (CPU)        └── _jax/ (GPU/CPU)           │
            │  PARIDADE Fortran/JAX/Numba <1e-12  [INVIOLÁVEL]          │
            └──────────────────────────────────────────────────────────┘
                       ▲
        from geosteering_ai.simulation import simulate_batch   (④ Studio futuro + import direto)
```

### 1.6 Invariantes a preservar (o que mantém a propriedade verdadeira)

1. **Rota física única** — toda simulação passa por `dispatch.simulate_batch`
   (`dispatch.py:233`); nenhum adapter recalcula campos EM.
2. **Fronteira de adapter** — CLI/SM/API contêm zero física; só orquestram a lib.
   Enforçado de facto (verificado) e pelo hook `validate-no-pytorch.sh`.

## 2. Produto ① — Geosteering CLI (~75 %)

### 2.1 Estado

Adapter de linha de comando production-ready (`geosteering_ai/cli/`, ~3.6k LOC). Arquitetura
hexagonal com imports lazy nos handlers (`--help` < 5 s). Zero física própria.

### 2.2 Entry points

| Entry point | Origem | Uso |
|:------------|:-------|:----|
| `geosteering-cli` | `geosteering_ai.cli:main` (`pyproject.toml:118`) | `simulate` / `benchmark` / `version` |
| `geosteering-warmup` | `geosteering_ai.cli.warmup:main` (`pyproject.toml:120`) | pré-aquece Numba/LLVM (CI/notebooks) |
| `python -m geosteering_ai.cli ...` | módulo | idem |

### 2.3 O que funciona

- `simulate`: gera N modelos sintéticos + executa batch (ProcessPool Numba ou dispatcher JAX).
- `benchmark`: 8 cenários canônicos A–H (E = Inv0Dip 600 pos; H = estresse 8×8×8 = 512 combos).
- Multi-dimensionalidade: `--frequencies` / `--dips` / `--tr-spacings` (produto cartesiano CSV).
- Backend: `--backend {numba,jax,auto}` (`auto` = árvore do dispatcher, jax-free no pré-voo).
- Geometria: `--geometry {per-model,templates,quantized}` (batchabilidade JAX).
- `--compare-backends`: numba × jax lado-a-lado (throughput + speedup + paridade max|Δ|).
- `--repeat`, `--dtype`, `--jax-strategy`, `--jax-chunk-size`, `--list-scenarios` (v2.58).
- Saída: `.npz` / `.dat` 22-col Fortran-compat / `--json`; tabela ASCII + hardware info.
- **Guard TLS-safe** (v2.55): `_count_geometry_groups` NumPy-puro evita init CUDA no caminho Numba.

### 2.4 Gaps / dívidas

| Gap | Prioridade sugerida |
|:----|:-------------------:|
| `docs/USER_GUIDE_CLI.md` ausente (docs só em docstrings) | média |
| CI não testa `--backend auto + --geometry templates` (teste de drift vs dispatcher) | média |
| Flags `geosteering-warmup --jax/--gpu/--auto/--numba` (warmup hoje é dual-path) | baixa (backlog) |
| ADR formal da spec-0003 (`--backend` default numba→auto) — hoje referência interna em `_exec.py:70` | baixa |
| Logging stdlib `basicConfig` (sem JSON/observabilidade estruturada) | baixa |

### 2.5 Cobertura de testes (≈ 110 testes CLI)

`test_cli_mvp.py` (28) · `test_cli_backend_auto.py` (22) · `test_cli_geometry.py` (13) ·
`test_cli_jax_crash_guard.py` (9) · `test_cli_backend_table.py` (15) · `test_cli_dat_save.py` (11) ·
`test_cli_hwinfo.py` (6) · `test_cli_benchmark_flags.py` (4, v2.58) · `test_cli_warmup_*` (6).

## 3. Produto ② — Simulation Manager (monólito 100 % · MVVM ~35 %)

### 3.1 Estado — duas arquiteturas paralelas (Strangler Fig)

`dataclasses/typing/numpy/scipy/logging` — **zero Qt**.

### 3.2 Entry points

- Monólito (produção): `python -m geosteering_ai.simulation.tests.simulation_manager`
- MVVM (dev): `python -m apps.sim_manager.app`
- Fundação importável: `from geosteering_ai.gui import MainWindowBase, Perspective, VMSignal`

### 3.3 O que funciona (MVVM)

- Fatia **6a** — execução: progresso por-grupo, cancel/pause cooperativo (Numba), log→sidebar, status bar.
- Fatia **6b** — geologia: filtro Hankel selecionável, 7 perfis canônicos, editor manual de camadas.
- Fatia **6c** — experimentos: `.exp.json`, histórico de snapshots (cache LRU/reload/busca).
- Fatia **6d** — galeria: 9 componentes + 5 geosinais byte-fiéis, perfis ρ/λ, heatmap, animação.
- Shell Antigravity (activity rail + perspectivas + secondary sidebar), JAX-GPU em subprocesso TLS-safe.
