## Sumário Executivo

A resposta para "o que acontece com n_models=2000?" é **radicalmente diferente por cenário**
e contradiz a intuição de que "mais modelos = mais throughput" para toda a suíte.

A análise quantitativa — calibrada com o OOM empírico real do cenário F (17.87 GB para
n=50) — revela dois comportamentos opostos:

| Grupo | Cenários | n_models=2000 | Throughput | Por quê |
|:------|:---------|:-------------:|:----------:|:--------|
| **Limitados por overhead** | A, D | ✅ Seguro (1.4–2.7 GB) | **+2–4×** | GPU subutilizada em n=50; lote maior amortiza overhead |
| **Limitados por VRAM** | B, C, E, F, G, H | ❌ OOM (43–21.303 GB) | N/A | GPU já saturada em n=50; scaling de n_models não ajuda |

**O achado mais importante**: para os cenários gate (A, B, E), o benchmark atual **subestima

sem nenhuma mudança de código** e ainda cabe na T4. Em B e E, ao contrário, n=50 já expõe o
teto real da GPU; aumentar n_models não ajudaria mesmo que a VRAM permitisse.

## 1. Anatomia da Memória XLA: Por que n_models=2000 Causa OOM

### 1.1 A Fonte de Verdade: OOM Empírico do Cenário F

A Sprint A1.6 nos deu um dado experimental preciso: o cenário F com n_models=50 tentou
alocar **17,869,320,632 bytes = 16.64 GB** e falhou na T4 (15 GB).

O cenário F tem: nTR=4, nAng=1, n_pos=100, nf=4 → **1.600 unidades de vmap por modelo**.

Isso nos permite calibrar o custo real de memória XLA por "unidade de vmap":

```
Memória XLA total = 17.869 GB (para n_models=50, F)
Memória por modelo = 17.869 / 50 = 357 MB/modelo
Memória por unidade vmap = 357 MB / 1.600 unidades = 218 KB/unidade

Output real por unidade = 9 componentes × 16 bytes (complex128) = 144 bytes
Ratio intermediários/output = 218.000 / 144 = 1.514×
```

**O XLA aloca 1.514× mais memória em buffers intermediários do que no tensor de saída.**

### 1.2 Por que esse ratio é tão alto?

O XLA executa o `jax.vmap` compilando o mapa inteiro como um único kernel. Para cada
unidade do vmap, ele materializa simultaneamente todos os buffers intermediários do
cálculo do simulador:

```
Por unidade (pos, freq, TR, modelo):
  ┌─ Input kernel Hankel: krJ0J1 (npt=201) × wJ0/wJ1 (201) = 6.432 bytes
  ├─ common_arrays: 9 arrays × (n_cam=5) × (npt=201) × 16 bytes = 144.720 bytes
  ├─ common_factors: 6 fatores × (n_cam=5) × 16 bytes = 480 bytes
  ├─ Propagação TE/TM: 4 arrays × (n_cam=5) × 16 bytes = 320 bytes
  ├─ Kernel Hzz/Hxx/Hxz/etc.: 9 componentes × 16 bytes = 144 bytes (output)
  └─ Buffers de compilação XLA (copy-on-write, gradient checkpoints): ~65.000 bytes
  TOTAL: ~218.000 bytes ≈ 218 KB
```

O dominante é `common_arrays`: 9 arrays de shape `(n_cam, npt)` por frequência —
exatamente o que a Sprint 2.10 otimizou via cache no Numba, mas que o XLA não
pode pré-computar (porque é interno ao trace JIT).

### 1.3 Fórmula Geral de Memória XLA

```
VRAM(n_models, nTR, nAng, n_pos, nf) ≈
    n_models × nTR × nAng × n_pos × nf × 218 KB
    + overhead_fixo (~1 GB: compilação XLA, filtro Hankel, vetores globais)
```

Aplicada a cada cenário da suíte, com n_models=2000:

| Cen | nT×nA×nP×nF | Mem/modelo | Max T4 | Max A100 | VRAM (n=2000) | n=2000 viável? |
|:---:|:-----------:|:----------:|:------:|:--------:|:-------------:|:--------------:|
| **A** | 1×1×1×1 | **0.2 MB** | **67.299** | **187.476** | **1,4 GB** | **✅ T4** |
| B | 1×1×100×1 | 21 MB | 672 | 1.874 | 43 GB | ❌ OOM T4+A100 |
| C | 1×1×100×4 | 85 MB | 168 | 468 | 168 GB | ❌ OOM |
| **D** | 4×1×1×1 | **0,9 MB** | **16.824** | **46.869** | **2,7 GB** | **✅ T4** |
| E | 1×1×600×1 | 127 MB | 112 | 312 | 251 GB | ❌ OOM |
| F | 4×1×100×4 | 340 MB | 42 | 117 | 667 GB | ❌ OOM |
| G | 4×4×100×4 | 1.360 MB | 10 | 29 | 2.664 GB | ❌ OOM |
| H | 8×8×100×8 | 10.880 MB | 1 | 3 | 21.303 GB | ❌ OOM |

> **Nota sobre H**: o cenário H com n_models=2000 exigiria 21 petabytes de VRAM —
> mais do que toda a memória GPU existente no mundo. Não é uma questão de hardware
> futuro; é uma questão de arquitetura do XLA que exige reprojeto (chunking).

## 2. Escalabilidade de Throughput: Dois Regimes Completamente Diferentes

### 2.1 Modelo Analítico

O tempo de execução segue o modelo:

```
t_total = t_overhead_fixo + n_models × t_compute_por_modelo

Onde:
  t_overhead_fixo ≈ 19,3 ms (launch kernel XLA, block_until_ready, sync GPU→CPU)
  t_compute_por_modelo ≈ 135,9 μs × n_pos × f(combos)  [calibrado de A_50 e B_50]

  f(combos): fator de paralelismo vmap (não linear):
    1 combo (A, B, E)  →  f = 1,0
    4 TRs   (D)        →  f ≈ 2,5   (ganho de 1,6× vs 4×, por vmap paralelo)
    4 freqs (C)        →  f ≈ 1,7   (ganho de 2,4× vs 4×)
    4×4 (F-style)      →  f ≈ 7,2   (ganho de 2,2×)
    4×4×4 (G)          →  f ≈ 28,5  (ganho de 2,2×)
```

### 2.2 Regime 1 — GPU Subutilizada (Cenários A e D): n_models=2000 AJUDA

Quando n_pos=1 (cenário A) ou n_pos=1 com poucos combos (D), o tempo de compute
por chamada é pequeno. O overhead fixo de 19,3 ms **domina**. Isso significa que
a GPU está ociosa ~74% do tempo para A_50 (26 ms total, 19 ms overhead).

Aumentar n_models distribui o overhead fixo por mais modelos:

### 2.3 Regime 2 — GPU Saturada (Cenários B, C, E, F, G): n_models=2000 NÃO AJUDA

Para cenários com n_pos ≥ 100, o compute por chamada já é muito maior que o overhead.
A GPU está utilizando seus cores FP64 em plena capacidade.

**Cenário B — Throughput por n_models (até o limite da T4):**

**O throughput do cenário B só melhora 2,6% mesmo usando 13× mais modelos e ocupando

**Resumo do teto de throughput por cenário (T4):**

> **Insight fundamental**: o throughput por modelo depende de **qual é o gargalo**.
> Quando o gargalo é o overhead fixo (A, D), mais modelos = mais eficiência.
> Quando o gargalo é o compute (B, C, E, F, G), mais modelos = mais VRAM, sem ganho.

## 3. O Custo Oculto: Loop Python de rho_h_at_obs

Além do cálculo GPU, a função `simulate_multi_jax_batched` tem um loop Python
pós-GPU para preencher o tensor de observabilidade:

```python
# multi_forward.py:1121-1135
for i_model in range(n_models):         # O(n_models)
    for i_ang in range(nAngles):         # O(nAngles)
        for i_pos in range(n_pos):       # O(n_pos)
            lay = layer_at_depth(...)
            rho_h_at_obs[i_model, i_ang, i_pos] = ...
```

A ~200 ns por iteração Python:

| Configuração | Iterações | Tempo Python | % do tempo GPU (A_2000) | % do tempo GPU (E_50) |
|:-------------|:---------:|:------------:|:-----------------------:|:---------------------:|
| A, n=2000 | 2.000 × 1 × 1 = 2.000 | ~0,4 ms | **0,1%** | — |
| B, n=672 | 672 × 1 × 100 = 67.200 | ~13 ms | — | 1,9% |
| E, n=112 | 112 × 1 × 600 = 67.200 | ~13 ms | — | **0,3%** |
| A, n=65.000 | 65.000 × 1 × 1 = 65.000 | ~13 ms | 1,5% | — |

**Conclusão**: para a faixa de n_models e n_pos atual, o loop Python é negligenciável
(< 2% do tempo total). Só se tornaria relevante com n_models ≥ 50.000 + n_pos ≥ 600
simultaneamente — fora do escopo imediato.

**Atenção futura**: quando Sprint A3 implementar geração on-the-fly de datasets com
n_models=10.000 + n_pos=600, esse loop deverá ser substituído por NumPy vetorizado.
A versão numpy seria equivalente e ~1000× mais rápida:

```python
# Versão vetorizada futura (Sprint A3):
layer_indices = np.searchsorted(prof_mids, positions_z[None, :], side="right") - 1
rho_h_at_obs = rho_h_batch[np.arange(n_models)[:, None, None],
                             layer_indices[np.newaxis, :, :]]
```

## 4. O Que Muda e O Que Quebra com n_models=2000

### 4.1 Cenários A e D — Funcionam, Melhoram

| Aspecto | Impacto com n_models=2000 |
|:--------|:--------------------------|
| VRAM | 1,4 GB (A) / 2,7 GB (D) — seguro para T4 e A100 |
| Throughput | +3,58× em A, +2,07× em D |
| Paridade Fortran | Inalterada — vmap distribui computações independentes |
| XLA cache key | Inalterado — key é `(n_cam, npt)`, não `n_models` |
| Tempo de warmup | Aumenta ~proporcionalmente com n_models no primeiro run (JIT trace maior) |
| Tempo de run 1 (cold) | Aumenta, mas C/H ainda deve ficar próximo de 1,0 |
| Loop Python rho_h_at_obs | 2.000 iterações → < 1 ms, negligenciável |

### 4.2 Cenários B, C, E, F, G, H — OOM ou Sem Benefício

| Cenário | Com n_models=2000 |
|:--------|:-----------------|
| B | OOM: precisaria 43 GB (T4 tem 15, A100 tem 40) |
| C | OOM: precisaria 168 GB |
| E | OOM: precisaria 251 GB — mais de 16 T4s |
| F | OOM: precisaria 667 GB |
| G | OOM: precisaria 2,7 TB |
| H | OOM: precisaria 21 TB |

### 4.3 O Paradoxo de F_50 vs B_2000

É contra-intuitivo, mas F com n_models=20 usa ~7 GB de T4, enquanto B com n_models=672
usaria ~14,6 GB — mais que F. Por quê?

```
F: 4×1×100×4 = 1.600 unidades/modelo × 20 modelos × 218 KB = 6,9 GB
B: 1×1×100×1 = 100 unidades/modelo × 672 modelos × 218 KB = 14,2 GB
```

F por modelo é 16× mais caro que B, mas F usa 33× menos modelos. O produto é comparable.
**O gargalo de memória não é n_models isolado, mas o produto n_models × n_pos × combos.**

O notebook atual usa n_models=50 para todos os cenários gate (A, B, E). Para A, isso

## 6. Estratégia de Geração de Datasets com n_models Grande

### 6.1 O Problema Real da Sprint A3

Para gerar um dataset de 100.000 modelos geológicos com configuração G (4 freqs, 4 TRs,
4 dips), não é possível fazer uma única chamada `simulate_multi_jax_batched(n_models=100000)`.
A VRAM necessária seria ~136.000 GB.

### 6.2 Solução: Chunking por Lotes

```python
# Sprint A3 — geração por chunks (pseudocódigo)
CHUNK_SIZE = {
    'A': 5000,   # seguro T4, throughput ótimo
    'B': 500,    # throughput marginal mas seguro
    'C': 150,    # seguro T4
    'D': 5000,   # seguro T4, throughput ótimo
    'E': 100,    # seguro T4
    'F': 35,     # seguro T4
    'G': 8,      # seguro T4
}

results = []
for chunk_start in range(0, n_models_total, CHUNK_SIZE[scen]):
    chunk_end = min(chunk_start + CHUNK_SIZE[scen], n_models_total)
    batch = simulate_multi_jax_batched(
        rho_h[chunk_start:chunk_end], ...,
        rho_v[chunk_start:chunk_end], ...,
        esp[chunk_start:chunk_end], ...
    )
    results.append(batch.H_tensor)

H_full = np.concatenate(results, axis=0)
```

**Throughput com chunking** (igual a 1 chunk, pois é sequencial):

| Cenário | Chunk size | Throughput efetivo | Tempo para 100k modelos |
|:-------:|:----------:|:------------------:|:-----------------------:|
| A | 5.000 | ~25,8M mod/h | **14 segundos** |
| B | 500 | ~264k mod/h | **22,7 minutos** |
| C | 150 | ~153k mod/h | **39 minutos** |
| D | 5.000 | ~10,4M mod/h | **35 segundos** |
| E | 100 | ~44k mod/h | **136 minutos** |
| G | 8 | ~9,4k mod/h | **637 minutos = 10,6 horas** |

### 6.3 jax.lax.map como Alternativa ao Python for-loop

Para evitar 40 chamadas Python sequenciais, `jax.lax.map(f, batch)` executa
o mapa sequencialmente **dentro do XLA** — sem retornar ao Python por iteração.
O overhead por chunk cai de ~19,3 ms (kernel launch) para ~0,1 ms:

```python
# Alternativa com lax.map (Sprint A3+):
# Usa O(1 chunk) de memória mas mantém XLA sequencial
H_all = jax.lax.map(
    lambda model_params: simulate_single_model(model_params),
    stacked_params   # shape (n_models_total, ...)
)
```

**Trade-off**: `lax.map` tem menor throughput que `vmap` (sequencial), mas consome
VRAM constante independente de n_models_total. Adequado para cenários G/H onde
mesmo n=5 já usa 7–14 GB.

## 9. Conclusão

| Pergunta | Resposta |
|:---------|:---------|
| "n_models=2000 cabe na T4?" | **Apenas para A e D.** Todos os outros cenários estão acima de 40 GB |
| "n_models=2000 melhora o throughput?" | **Apenas para A (+3,6×) e D (+2,1×).** Para B, C, E, F, G: ganho < 3% |
| "Por que B não melhora com mais modelos?" | GPU já está saturada pela combinação n_pos × combos. Bottleneck não é n_models |
| "Como processar 2000 modelos em B/E/G?" | Chunking sequencial (40 chamadas de 50). Throughput total = mesmo |
| "A baseline atual de 6,9M mod/h para A é válida?" | **Tecnicamente, sim; praticamente, subestima o hardware em 3,6×** |
| "Qual o verdadeiro teto do T4 para A?" | **~26,4M mod/h** (n_models ≥ 5.000) |

O resultado mais contraintuitivo desta análise: **para a geração de datasets ML
(configurações multi-freq×multi-TR×multi-dip), aumentar n_models não acelera o JAX
GPU**. O gargalo real é o XLA materializar todos os buffers intermediários para
todas as combinações de (pos × freq × TR × dip) simultaneamente — o n_models só
multiplica um overhead de VRAM já dominado por esses fatores.

A solução arquitetural definitiva é `jax.lax.map` com chunking inteligente (Sprint A3),
que processa qualquer n_models com VRAM constante, ao custo de execução sequencial
intra-chunk.
