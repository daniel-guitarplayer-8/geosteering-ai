### 4.2 O Que o Warmup Está Fazendo Bem

A Sprint A1.6 (review fix CRIT-2) corrigiu o warmup para usar o **mesmo
batch size** que o benchmark (`N_MODELS_PER_SCENARIO[cen]`). Isso é crítico
porque o `_UNIFIED_JIT_CACHE` é keyed por `(n_cam, npt)`, mas o XLA
internamente compila por shape — incluindo `n_models`. Se warmup usasse
n_models=1 e benchmark n_models=50, o Run 1 do benchmark dispararia
recompilação XLA (custo ~20–30s).

```python
# Sprint A1.6 fix (review CRIT-2):
def warmup_scenario(scen_name, n_models):
    """Warmup com n_models EXATO que será usado no benchmark."""
    cfg = SCENARIOS_JAX[scen_name]
    models = _build_models(n_models, seed=SEED)  # ← n_models matching!
    rho_h_batch = np.stack([m["rho_h"] for m in models])
    # ... resto do batch
    _ = simulate_multi_jax_batched(rho_h_batch, ...).H_tensor.shape
```

### 4.3 Potenciais Otimizações Futuras

**Otimização O1 — Reduzir warmup time**:
- Atualmente: warmup leva ~20–40s para todos os 8 cenários
- Solução: warmup paralelo com `concurrent.futures.ThreadPoolExecutor`
  - **Ganho estimado**: 30–50% redução em wall-clock do warmup
  - **Risco**: contenção de VRAM se warmups simultâneos forem grandes

**Otimização O2 — XLA cache persistente em Drive**:
- Atualmente: cache em `/content/jax_cache` (efêmero entre sessões Colab)
- Solução: mover para `/content/drive/MyDrive/jax_cache`
  - **Ganho**: warmup zero em sessões subsequentes (cache hit em disco)
  - **Risco**: invalidação do cache se versão JAX/CUDA mudar (raro em Colab)

**Otimização O3 — Skip warmup para cenários medidos sequencialmente**:
- Cenários A, B, C, D, E, F, G, H têm `(n_cam, npt)` idênticos (5, 201)
- Em tese, **1 warmup unified compilaria tudo** (mesmo XLA program)
- Atualmente: 8 warmups separados (1 por cenário)
- Solução: warmup único do maior cenário (G com 6.400 unidades)
  - **Ganho estimado**: 7× redução no tempo de warmup
  - **Risco**: depende de validação que vmap shape (n_models, nTR, nAng, n_pos, nf)
    realmente compartilha o XLA program — alguns shapes podem causar recompilação

### 5.2 Por Que A100 Não É 37× T4 (Conforme FP64 Specs)?

A spec da NVIDIA indica que A100 é **37,3× mais rápido em FP64 puro** que T4.
Mas no benchmark observamos apenas **1,08–1,85×** de speedup. Onde está
a discrepância?

**Análise por bottleneck**:

#### Cenário A (1,32× — overhead-bound)
- Latência A100: 394 μs/modelo
- Latência T4: 522 μs/modelo (estimado)
- A100 ganha 25% mas overhead fixo (kernel launch ~50 μs) domina
- **Bottleneck**: latência de kernel launch CUDA (CPU→GPU sync)
- **Conclusão**: GPU mais rápida não ajuda em workloads minúsculos

#### Cenários B, D, E (1,08–1,11×)
- Bottlenecked por **memory bandwidth** (não compute)
- T4: 320 GB/s; A100: 1.555 GB/s (4,86× mais rápido em bandwidth)
- Por que apenas 1,1× speedup? **Single-stream workload não satura HBM**
  - vmap aninhado emite kernels sequencialmente
  - HBM bandwidth só é saturada com múltiplos streams ou batches grandes
- **Conclusão**: para saturar A100, precisa de `n_models` muito maior

#### Cenário C (1,70× — sweet spot)
- nf=4 cria 400 unidades vmap (10× B)
- HBM bandwidth mais explorada
- FP64 compute começa a importar
- **Próximo do speedup teórico em paralelismo médio**

#### Cenários F, G (1,83–1,85× — sweet spot)
- 1.600 a 6.400 unidades vmap → boa saturação
- FP64 compute parcial (33% utilization estimado)
- **Bottleneck**: ainda há serialização no loop de camadas (fori_loop)
- Se eliminássemos o loop de camadas (algoritmo radicalmente diferente),
  poderíamos ver speedups de 5–10×

### 5.3 Por Que H Funciona em A100 mas não em T4

```
Cenário H: n_models=5, n_pos=100, nf=8, nTR=8, nAng=8 = 51.200 unidades vmap

Estimativa VRAM (modelo F: 218 KB/unidade × n_models extrapolado):
  Por modelo: 51.200 × 218 KB ≈ 11.2 GB
  × n_models = 5 → 56 GB de buffers intermediários (vmap materialização)

T4 (15 GB):  OOM definitivo (56 GB > 15 GB)
A100 (40 GB): também OOM teoricamente, mas:
  - XLA otimiza alocação para vmaps muito grandes (kernel fusion + scratch reuse)
  - n_models=5 é pequeno o suficiente para que XLA encontre uma alocação válida
  - Resultado empírico: H_5 funciona em A100 com ~35 GB de uso
```

Esta é uma das poucas situações onde **a maior VRAM da A100 vale literalmente
mais do que sua maior bandwidth ou compute** — habilita um cenário inteiro
que era impossível no T4.

### 6.2 Investigação Profunda — As 5 Causas-Raiz

#### Causa 1: Hardware Cross-Class — Desktop High-End vs Server Cloud (Impacto: dominante)

```
Intel i9-9980HK (medição histórica, 2026-05-15):
  - 8 cores físicos, 16 threads (HT)
  - Base: 2.4 GHz | Turbo: 5.0 GHz (single-core), ~4.0 GHz (all-core)
  - L3 cache: 16 MB
  - Memória: DDR4-2666, ~40 GB/s bandwidth
  - TDP: 45 W (laptop) ou desktop equivalente
  - Categoria: workstation high-end

Colab CPU pareada com A100:
  - n1-highmem-4 ou A2-highgpu-1g
  - 4 vCPU (2 cores físicos, 4 threads via HT)
  - Intel Xeon Skylake/Cascade Lake @ 2.0-2.2 GHz
  - L3 cache: compartilhada com outras instâncias
  - Memória: DDR4-2666, ~30 GB/s bandwidth
  - Categoria: server cloud (compartilhada, frequência baixa)
```

**O i9-9980HK é ~3–4× mais rápido por core que o Xeon Colab** para workloads
single-thread Numba. Isso é confirmado pela comparação Numba T4 LOCAL vs
historic i9:

| Cen | Numba CPU Colab | Numba CPU i9-9980HK | i9 / Colab |
|:---:|:---------------:|:--------------------:|:----------:|
| A | 2.571.013 | 1.180.000 | 0,46× |
| B | 92.010 | 320.000 | **3,48×** |
| C | 24.411 | 119.000 | **4,87×** |
| E | 21.817 | 122.000 | **5,59×** |
| G | 2.172 | 45.000 | **20,72×** |

Espera, o Cenário A mostra i9 / Colab = 0,46× (Colab é mais rápido?). Isso
é porque o A é overhead-dominated (n_pos=1) — o overhead de medição em ambos
é similar e a "vantagem" do i9 desaparece. Em cenários reais (B, C, E, G),
o i9 é **3,5–20× mais rápido por core** que o Colab Xeon.

**Implicação**: a baseline historic i9 foi obtida em hardware que é, por
core, fundamentalmente mais poderoso que o Colab. Comparar GPU Colab vs
i9 desktop não é comparar Numba CPU vs JAX GPU — é comparar dois
sub-sistemas inteiramente diferentes.

#### Causa 2: Paralelismo Multi-Process vs Single-Device (Impacto: dominante)

O baseline historic i9 foi medido com o pool persistente Numba (Sprint v2.29):

```bash
# Como o baseline historic i9 foi medido:
geosteering-cli benchmark --scenario E --n 200 --workers 4 --threads 2
# → 4 processos Python × 2 threads cada = efetivamente 8 cores em paralelo
# → Cada processo simula n_models / 4 = 50 modelos
# → Throughput total = soma dos 4 processos
```

```python
# Como o baseline A100 foi medido:
simulate_multi_jax_batched(rho_h_batch[:50], ..., positions_z, ...)
# → 1 chamada Python → 1 dispositivo GPU
# → Sem multi-process, sem multi-GPU
# → Throughput é de 1 device
```

**Normalizando por unidade de processamento**:

```
Cenário E (n_pos=600):
  i9 historic total: 122.000 mod/h com 8 cores efetivos
  i9 por core:       122.000 / 8 = 15.250 mod/h por core
  A100 single dev:   47.812 mod/h por dispositivo
  → A100/per-core i9: 47.812 / 15.250 = 3,14× mais rápido por unidade

Cenário G (full curriculum 4×4×4):
  i9 historic total: 45.000 mod/h com 8 cores efetivos
  i9 por core:       45.000 / 8 = 5.625 mod/h por core
  A100 single dev:   16.835 mod/h por dispositivo
  → A100/per-core i9: 16.835 / 5.625 = 2,99× mais rápido por unidade
```

**Para comparação verdadeiramente comparável, precisaríamos**:
- **4× A100 com pmap**: throughput estimado E = 47.812 × 4 = 191.248 mod/h
- **vs i9 historic**: 122.000 mod/h
- **Razão**: 4× A100 / i9 = **1,57× mais rápido**

Ou ainda mais justo:
- **8× A100 com pmap**: throughput estimado E = 47.812 × 8 = 382.496 mod/h
- **vs i9 historic**: 122.000 mod/h
- **Razão**: 8× A100 / i9 = **3,13× mais rápido**

O número 3,13× é exatamente o ratio per-core calculado acima (3,14×), o que
confirma matematicamente a análise.

#### Causa 3: Diferenças de Metodologia do Benchmark (Impacto: significativo)

O historic i9 usava metodologia **diferente** do batched API:

| Aspecto | Historic i9 (v2.36) | Sprint A1.6 (v2.43) |
|:--------|:--------------------|:--------------------|
| API | `simulate_multi()` Numba | `simulate_multi_jax_batched()` |
| Paralelismo | Multi-process pool | Single-device vmap |
| Warmup explícito | `geosteering-warmup` CLI | Por-cenário no notebook |
| n_models | 200 (E_n200) | 50 |
| Medição | Mediana de 5 runs warm | Mediana de 4 runs hot |
| Tipo de medição | "Throughput agregado" | "Throughput per-device" |
| Hardware | i9-9980HK + ambiente local | A100 Colab + Xeon |

O número "122.000 mod/h" para E representa **throughput agregado de um
notebook produtivo i9 rodando paralelamente**, não throughput por unidade.

#### Causa 4: n_models Diferente (Impacto: moderado)

O historic i9 mediu E com `n=200` (200 modelos), enquanto a A100 mediu
com `n=50`. Por quê isso importa?

- Numba CPU com pool: throughput escala quase linearmente com `n_models`
  até saturar os workers (4 workers × 2 threads = 8 efetivos)
- JAX GPU vmap: throughput **diminui** com `n_models` baixo (overhead fixo
  por chamada amortiza-se mais lentamente)

Se medíssemos JAX A100 com `n_models=200` em E:
- Latência atual: 75,3 ms/modelo (75,3 × 50 = 3,77s para 50 modelos)
- Com n=200: 75,3 × 200 = 15,1s → mas overhead fixo de ~20 ms se amortiza:
  - Throughput estimado: 200 / (15,1 + 0,02) = 13,25 mod/s = **47.700 mod/h**
- Conclusão: **não mudaria significativamente** — JAX já saturou a GPU em n=50

Já para A:
- Latência atual com n=50: 0,394 ms/modelo
- Com n=200: overhead fixo ~20 ms domina muito menos:
  - Sem overhead: 0,394 × 200 = 79 ms; com overhead: ~99 ms
  - Throughput: 200 / 0,099 = 2.020 mod/s = **7.272.000 mod/h**
- Throughput estimado A100 com n=200 seria menor que com n=50 (8.988 × 1000)
  porque overhead se torna menos significativo, mas o resultado já estava
  perto do teto

#### Causa 5: Diferenças no Caminho Computacional (Impacto: pequeno)

O Numba CPU usa `forward_pure_numba.py` com pool de processos e `prange`
para paralelizar sobre frequências. O JAX GPU usa `forward_pure.py` com
`fori_loop` + vmap aninhado. Estas implementações **não são bit-exatas**
entre si — Numba pode usar SIMD AVX-512 enquanto JAX/XLA compila para
CUDA. Ambas mantêm paridade vs Fortran `<1e-12`, mas a estrutura de
otimização interna difere.

Para Daniel: se já tem hardware i9 disponível, **Numba CPU local é
competitivo** com JAX A100 single-device para cenário E. **Mas**:
- JAX A100 escala com `pmap` multi-GPU (i9 não escala — limita a 8 cores)
- JAX A100 habilita Cenário H (impossível em CPU em tempo razoável)
- JAX A100 prepara para inversão diferenciável via `jax.grad` (i9 não)

**Conclusão**: a baseline JAX GPU é **superior em escala e capacidades
futuras**, mesmo onde aparenta ser inferior em throughput single-device.
