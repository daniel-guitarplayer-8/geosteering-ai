**Conclusão central**: a gargalo de performance migra conforme o cenário —
para `30k+ modelos × 600 posições × 1 freq × 1 TR × 1 ângulo` (caso de produção),
o **Modo D (4 workers × 2 threads/worker)** já entrega throughput perto do
Fortran nativo. Para `multi-freq × multi-TR × multi-ângulo` (PINN, treino
massivo), **JAX GPU com vmap consolidado (Sprint 10 Phase 2)** é o caminho
correto, não Numba CUDA.

## 1. Performance — Otimizações para Multi-Ângulo / Multi-Frequência / Multi-TR

### 1.1 Estado atual da computação paralela

O caminho hot do simulador para um cenário multi-dimensional segue esta hierarquia
em [multi_forward.py:629-714](geosteering_ai/simulation/multi_forward.py#L629-L714):

```
for i_tr in range(nTR):                   # ← serial Python
    for i_ang in range(nAngles):          # ← serial Python
        # cache lookup por hordist = L·|sin(dip)|
        cache_tuple = caches[hordist_key]
        _simulate_positions_njit_cached(  # ← @njit
            positions_z, ..., H_tensor[i_tr, i_ang]
        )
        # internamente: for j in prange(n_pos): ... freq_loop interno
```

A paralelização ocorre **somente sobre posições** via `prange` em
[forward.py:133-178](geosteering_ai/simulation/forward.py#L133-L178). Loops sobre
TR, ângulo e frequência são **serializados em Python**, com a única otimização sendo
a deduplicação de cache por `hordist` (poço vertical reutiliza um cache para todos
os TR).

### 1.2 Gargalos identificados

#### 1.2.1 Loop serial sobre TR e ângulos

**Sintoma**: para `nTR=3, nAngles=10, nf=2, n_pos=600, n_models=30k`, há
3 × 10 × 30000 = 900.000 chamadas Python a `_simulate_positions_njit_cached`.
Cada chamada paga overhead de:

- transição Python → Numba (~10–50 µs);
- alocação de `H_tensor` slice (memória contígua já existente, mas Numba revalida);
- lookup no dicionário `caches[key]`.

Em produção, **isso adiciona ~30–60 segundos de overhead Python puro** para
batches grandes — pequeno em valor absoluto, mas perceptível. O custo do
prange interno escala com `n_pos`, então cenários com `n_pos < 100` veem o
overhead Python dominar.

#### 1.2.2 Loop sobre frequências interno é escalar (não vetorizado)

**Localização**: dentro de `_fields_in_freqs_kernel_cached` (em
[kernel.py](geosteering_ai/simulation/_numba/kernel.py)), o loop sobre `nf` frequências
é um for sequencial. Cada freq recomputa `zeta = jωμ₀`, depois chama
`common_factors → hmd_tiv → vmd → rotate_tensor`.

**Problema**: as recursões TE/TM (em
[propagation.py:370-466](geosteering_ai/simulation/_numba/propagation.py#L370-L466))
têm dependência sequencial entre camadas — não há paralelismo natural em camadas.
Mas as **frequências são independentes** e podem ser:

- vetorizadas com NumPy broadcasting (matrizes `(nf, n_layers)` complexas em vez
  de loops `for f in range(nf): array_1d`);
- paralelizadas com `prange` adicional (loop externo em vez de interno).

#### 1.2.3 Cache não deduplica multi-frequência

**Atual**: `caches[hordist]` armazena 9 arrays computados para um conjunto fixo
de `nf` frequências. Se o usuário fizer duas chamadas `simulate_multi` com
`frequencies_hz=[20kHz]` e depois `[20kHz, 40kHz]`, o cache é recomputado
inteiro porque a chave é `hordist`, não `(hordist, freqs_tuple)`.

**Impacto**: PINN/Picasso muitas vezes barre frequências diferentes para
mesmo perfil — sem cache cross-call, cada chamada paga setup completo.

### 1.3 Recomendações priorizadas

| Pri | Otimização                                                  | Ganho esperado          | Complexidade | Onde                                        |
|:----|:------------------------------------------------------------|:------------------------|:-------------|:--------------------------------------------|
| P0  | **Vetorizar frequências em `common_arrays`/`hmd_tiv`**      | 1.5–3× em multi-freq    | Média        | `_numba/propagation.py`, `_numba/dipoles.py` |
| P0  | **prange sobre eixo combinado** `(iTR × iAng)`              | 1.2–1.8× em multi-TR    | Baixa        | `multi_forward.py:629-714`                  |
| P1  | **Cache de `common_arrays` por `(hordist, freqs_signature)`** | 5× em chamadas repetidas | Baixa       | `multi_forward.py:272-332`                  |
| P1  | **Eliminar overhead Python por chunk de modelos**           | ~20% em N pequeno       | Média        | `multi_forward.py` (laço de modelos)        |
| P2  | **Aplicar `nogil=True` em `@njit`**                         | Permite chamadas concorrentes thread-pool externo | Baixa | Todos os `@njit` do _numba/   |
| P2  | **`fastmath=True` em decoupling/Hankel**                    | 5–10% via FMA, sem impacto numérico | Baixa | Funções que não dependem de paridade Fortran |

#### Detalhamento P0 — Vetorização de frequências

**Estado atual** ([_numba/propagation.py:243](geosteering_ai/simulation/_numba/propagation.py#L243)):

```python
@njit(cache=True)
def common_arrays(zeta, kr_freq, sigma_h, sigma_v, ...):
    # zeta: scalar complex (1 frequência)
    # kr_freq: (npt,) reais
    n_layers = sigma_h.shape[0]
    u = np.empty((npt, n_layers), dtype=np.complex128)
    for i in range(n_layers):
        kh2 = -zeta * sigma_h[i]
        u[:, i] = np.sqrt(kr_freq**2 - kh2)  # vetorizado em kr
    # ... idem para s, uh, sh
```

**Versão vetorizada multi-freq**:

```python
@njit(cache=True, parallel=True)
def common_arrays_multi(zetas, kr_freqs, sigma_h, sigma_v, ...):
    # zetas: (nf,) complex — UMA matriz para todas as frequências
    # kr_freqs: (nf, npt) — kr varia com freq via prepare_kr
    n_layers = sigma_h.shape[0]
    nf = zetas.shape[0]
    u = np.empty((nf, npt, n_layers), dtype=np.complex128)
    for f in prange(nf):                # ← novo prange
        for i in range(n_layers):
            kh2 = -zetas[f] * sigma_h[i]
            u[f, :, i] = np.sqrt(kr_freqs[f]**2 - kh2)
    return u, ...  # 9 arrays todos com axis 0 = freq
```

**Trade-off**: aumenta uso de memória 9× (uma matriz por freq) — para
`npt=201, nf=10, n_layers=22` isso é 9 × 10 × 201 × 22 × 16 B ≈ 6 MB,
totalmente tolerável.

#### Detalhamento P0 — prange sobre eixo combinado

**Estado atual**: `for i_tr × for i_ang` é serial Python.

**Refator**: combinar em índice flat e usar `prange`:

```python
@njit(parallel=True, cache=True)
def _simulate_multi_njit(
    positions_z, dz_halfs, r_halfs,    # arrays (nTR*nAngles,) já calculados
    cache_indices,                       # lookup table comprimida
    ...,
    H_tensor_flat,                       # (nTR*nAngles, n_pos, nf, 9)
):
    n_combos = dz_halfs.shape[0]
    for k in prange(n_combos):           # ← prange aqui
        # acessa caches via cache_indices[k]
        ...
```

**Cuidado**: requer materializar caches em arrays Numba (não dict Python).
Padrão: usar `(unique_caches, lookup_table)` onde `lookup_table[k]` aponta para
índice no `unique_caches`.

#### Detalhamento P1 — Cache cross-call

**Padrão proposto**:

```python
# Module-level dict (não thread-safe, mas chamadas multi-call são seriais)
_GLOBAL_HORDIST_CACHE: dict[tuple, ...] = {}

def _build_unique_hordist_caches(tr_spacings, dip_rads, freqs, ...):
    new_caches = {}
    for L, dip in itertools.product(tr_spacings, dip_rads):
        hordist = L * abs(math.sin(dip))
        freqs_sig = tuple(freqs)
        key = (round(hordist, 12), freqs_sig)
        if key in _GLOBAL_HORDIST_CACHE:
            new_caches[key] = _GLOBAL_HORDIST_CACHE[key]   # reuso
        else:
            new_caches[key] = precompute_common_arrays_cache(...)
            _GLOBAL_HORDIST_CACHE[key] = new_caches[key]   # persiste
    return new_caches
```

**Impacto**: PINN típico calcula gradiente em ~50 chamadas → primeira chamada
paga setup, próximas 49 reutilizam cache (5–10× speedup em iterações de treino
com mesma geometria).

## 2. Multithreading no Numba JIT — Análogo ao OpenMP?

### 2.1 Resposta direta

**Sim — funcionalmente equivalente, mas o backend é LLVM TBB ou OpenMP, escolhido
em tempo de instalação do Numba**. O usuário escreve `@njit(parallel=True)` e usa
`prange`, e o compilador Numba gera código nativo paralelizado análogo ao gerado
pelo `gfortran -fopenmp` em loops `!$OMP PARALLEL DO`.

### 2.2 Mecânica detalhada

#### 2.2.1 O que `@njit(parallel=True)` faz

Quando o decorador `@njit(parallel=True)` está presente:

1. **Análise estática**: o compilador Numba detecta loops `prange` e identifica
   *parallel reduction operations* (`+=`, `*=`) seguras.
2. **IR de paralelismo**: gera *Numba Parallel IR* — uma representação intermediária
   que descreve dependências entre iterações.
3. **Lowering para LLVM**: converte para LLVM IR com chamadas a `tbb_*` ou
   `omp_*` (dependendo do backend).
4. **Compilação**: LLVM gera código de máquina com chamadas a uma das três
   bibliotecas de runtime:
   - **Intel TBB** (`tbb` package): default em macOS e Windows, scheduler
     dinâmico work-stealing.
   - **OpenMP** (`-fopenmp`): default em Linux x86_64 com GCC/Clang.
   - **`workqueue`** (Numba próprio): fallback simples, fila FIFO.

[Documentação Numba sobre parallelização](https://numba.readthedocs.io/en/stable/user/parallel.html)
descreve estas três opções.

#### 2.2.2 Diferenças vs OpenMP "puro" (Fortran/C)

| Aspecto                       | OpenMP Fortran (`tatu.x`)             | Numba `@njit(parallel=True) + prange`        |
|:------------------------------|:--------------------------------------|:---------------------------------------------|
| Pragma/decorator              | `!$OMP PARALLEL DO`                   | `@njit(parallel=True)` + `prange`            |
| Diretivas explícitas          | `private`, `firstprivate`, `reduction` | Inferidas automaticamente pelo compilador   |
| Sincronização                 | `!$OMP CRITICAL`, `!$OMP BARRIER`     | Não suportado (loops devem ser independentes ou usar reduções) |
| Schedule (estático/dinâmico)  | `schedule(static, chunk)`             | Dinâmico (TBB) ou estático (OpenMP)          |
| Controle de threads           | `OMP_NUM_THREADS`                     | `NUMBA_NUM_THREADS` ou `numba.set_num_threads(n)` |
| GIL                           | N/A (Fortran nativo)                  | Liberado (Numba executa em código nativo)    |
| Chamada nested                | Permitido (via `OMP_MAX_ACTIVE_LEVELS`) | Apenas 1 nível de prange                   |
| Variável reduction            | `reduction(+:sum)`                    | Detectada automaticamente em `sum += x`      |

#### 2.2.3 Como controlar o número de threads no projeto

[multi_forward.py:623-627](geosteering_ai/simulation/multi_forward.py#L623-L627):

```python
if cfg.num_threads > 0 and HAS_NUMBA:
    import numba as _numba
    _numba.set_num_threads(cfg.num_threads)
```

E em [sm_workers.py:83-86](geosteering_ai/simulation/tests/sm_workers.py#L83-L86):

```python
def _numba_init_worker(n_threads: int, hankel_filter: str) -> None:
    os.environ["NUMBA_NUM_THREADS"] = str(n_threads)
    os.environ["OMP_NUM_THREADS"] = str(n_threads)
    os.environ.setdefault("OMP_MAX_ACTIVE_LEVELS", "2")
```

**Hierarquia de prioridade**:

1. `numba.set_num_threads(N)` em runtime (mais alta prioridade).
2. `NUMBA_NUM_THREADS=N` no env (lido na primeira chamada `@njit`).
3. `OMP_NUM_THREADS=N` no env (fallback se Numba usar OpenMP backend).
4. `os.cpu_count()` (default).

#### 2.2.4 Por que prange libera o GIL

O código gerado por `@njit` é **C-level nativo**, sem chamadas ao interpretador
Python. Antes de entrar na função compilada, Numba executa
`PyEval_SaveThread()` (libera GIL) e `PyEval_RestoreThread()` ao retornar.

Isso significa que múltiplas threads Python podem chamar `@njit` em paralelo
**sem contention** — útil quando o usuário tem `ThreadPoolExecutor` orchestrando
chamadas independentes do simulador.

### 2.3 Boas práticas para o projeto

1. **Sempre adicionar `nogil=True`** em `@njit` que será chamada de threads
   externas (`ThreadPoolExecutor`). Hoje, [propagation.py:159](geosteering_ai/simulation/_numba/propagation.py#L159)
   e similares **não têm** `nogil=True` — o GIL é liberado automaticamente
   apenas dentro do código nativo, mas `nogil=True` documenta a intenção e
   habilita otimizações adicionais.

2. **Documentar o backend ativo**: em
   [\_numba/\_\_init\_\_.py](geosteering_ai/simulation/_numba/__init__.py)
   adicionar log da biblioteca em uso:

   ```python
   import numba
   from numba.np.ufunc.parallel import threading_layer
   logger.info(f"Numba threading layer: {threading_layer()}")
   ```

3. **Evitar nested prange**: se `simulate_multi_njit` paraleliza eixo (TR×ang)
   e internamente chama outra `@njit(parallel=True)` que também usa prange,
   o Numba serializa o nível interno (não há nested parallelism). Para evitar
   surpresas, usar `parallel=False` em funções chamadas dentro de prange.

## 3. Proposta — 4 Modos de Execução com Workers Nativos

### 3.1 Definição dos modos

### 3.3 Implementação proposta

```python
def simulate_multi(..., n_workers: int = 1, models: Optional[List[dict]] = None):
    # Caso 1: 1 modelo (API legada — preserva compat)
    if models is None:
        return _simulate_multi_single_model(...)
    
    # Caso 2: lote de modelos
    n_models = len(models)
    
    if n_workers == 1:
        # Modo A ou B — loop serial sobre modelos, prange interno
        H_list = []
        for model in models:
            H_list.append(_simulate_multi_single_model(
                rho_h=model["rho_h"], rho_v=model["rho_v"],
                esp=np.array(model["thicknesses"]),
                positions_z=positions_z, ...
            ).H_tensor)
        return MultiSimulationResultBatch(H_stack=np.stack(H_list, axis=0), ...)
    
    # Modo C ou D — ProcessPoolExecutor
    chunks = _chunk_models(models, n_workers)
    eff_threads = _resolve_effective_threads(n_workers, cfg.num_threads)
    
    with ProcessPoolExecutor(
        max_workers=n_workers,
        mp_context=mp.get_context("spawn"),
        initializer=_simulate_worker_init,
        initargs=(eff_threads, cfg.hankel_filter),
    ) as pool:
        futures = [
            pool.submit(_run_simulate_chunk, chunk, positions_z,
                        frequencies_hz, tr_spacings_m, dip_degs, cfg, eff_threads)
            for chunk in chunks
        ]
        H_parts = [f.result() for f in as_completed(futures)]
    
    H_stack = np.concatenate(H_parts, axis=0)
    return MultiSimulationResultBatch(H_stack=H_stack, ...)
```

### 3.4 Detalhes críticos

#### 3.4.1 Resolução automática de threads (anti-oversubscription)

```python
def _resolve_effective_threads(n_workers: int, requested_threads: int) -> int:
    cpu = os.cpu_count() or 8
    if requested_threads <= 0:  # auto
        # Distribuir cores igualmente, mínimo 1 por worker
        return max(1, cpu // n_workers)
    # Validar
    total = n_workers * requested_threads
    if total > cpu:
        warnings.warn(
            f"Oversubscription: {n_workers}w × {requested_threads}t = {total} "
            f"> {cpu} CPUs. Considere reduzir threads ou workers."
        )
    return requested_threads
```

**Exemplo (8-core)**:

| `n_workers` | `cfg.num_threads` (input) | `eff_threads` (output) | Total CPU |
|:-----------:|:-------------------------:|:----------------------:|:---------:|
| 1           | -1 (auto)                 | 8                      | 8         |
| 4           | -1 (auto)                 | 2                      | 8         |
| 4           | 4                         | 4 + warning            | 16 ⚠️     |
| 1           | 4                         | 4                      | 4         |

### 3.7 Viabilidade — Análise de risco

| Risco                                            | Probabilidade | Mitigação                                         |
|:-------------------------------------------------|:--------------|:--------------------------------------------------|
| Oversubscription degrada perf vs Modo B          | Média         | `_resolve_effective_threads` + warning            |
| Pickle de positions_z é grande (>10 MB)          | Baixa         | n_pos=10k → 80 KB OK; positions_z compartilhada via shared memory se necessário |
| Worker death (OOM, segfault)                     | Baixa         | `as_completed` + `try/except` por future          |
| Conflito entre `cfg.num_threads` e env var       | Média         | Doc clara + `_resolve_effective_threads` autoritativo |
| API breaking change                              | Baixa         | Adicionar kwargs `n_workers`, `models` opcionais (default `n_workers=1, models=None`) — backward-compat |
| Custo de spawn em N pequeno                      | Média         | Auto-fallback Modo B se `n_models < 4 × n_workers` |

**Conclusão**: viável, com baixo risco se for implementado com cuidado nos
detalhes de threading. **Pré-requisito**: documentação clara da hierarquia
`n_workers × num_threads × cpu_count`.

## 5. Numba CUDA vs JAX — Recomendação

### 5.1 Veredicto rápido

**Não recomendo migrar para Numba CUDA**. JAX é a escolha estrategicamente
correta para este projeto pelas razões abaixo. Numba CUDA seria útil em
um cenário muito específico (kernel hot-loop curto e bem definido) que não
é o caso do simulador EM 1D TIV.

### 5.2 Comparação técnica detalhada

| Critério                              | Numba CUDA               | JAX (XLA)                |
|:--------------------------------------|:-------------------------|:-------------------------|
| **Modelo de programação**             | Imperativo (kernels CUDA escritos à mão) | Declarativo (XLA escolhe estratégia) |
| **Sintaxe**                           | `@cuda.jit` + thread/block indexing | Funções Python puras com `jax.numpy` |
| **Gerenciamento de memória**          | Manual (`cuda.to_device`, `.copy_to_host`) | Automático (jax arrays) |
| **Diferenciação automática**          | ❌ Não suporta            | ✅ `jacfwd`, `jacrev`, `grad` |
| **Compilação ahead-of-time**          | Sim (PTX cache)          | Sim (XLA HLO cache)      |
| **Multi-GPU**                         | Manual via streams       | `pmap` automático        |
| **TPU**                               | ❌                        | ✅                        |
| **Tipos complexos (`complex128`)**    | Limitado (sem `cmath` em kernels CUDA) | ✅ Completo               |
| **Scan/loop em kernel**               | Manual (`for i in range`) | `lax.scan` otimizado     |
| **Curva de aprendizado**              | Alta (CUDA architecture: SM, warp, shared mem) | Média (mindset funcional) |
| **Quantidade de código a reescrever** | ~4000 LOC do `_numba/` | Já temos 6400 LOC `_jax/`! |
| **Comunidade científica**             | Pequena (Numba CUDA é nicho) | Grande (DeepMind, Google, geofísica) |

### 5.3 Por que Numba CUDA é problemático para EM 1D TIV

O simulador tem características que **não favorecem Numba CUDA**:

#### 5.3.1 Tipos complexos pesados

A computação central é em `complex128` (cmath). Numba CUDA tem suporte
**parcial** para complex em kernels — funções como `np.exp`, `np.sqrt` em
complex64/128 podem dar erro de compilação ou performance ruim, exigindo
implementação manual de `cexp`, `csqrt` em CUDA C inline. JAX/XLA suporta
complex128 nativamente via cuBLAS/cuSOLVER.

#### 5.3.2 Loops aninhados profundos com dependências

A propagação TE/TM tem recursão sequencial entre camadas
([propagation.py:410-466](geosteering_ai/simulation/_numba/propagation.py#L410-L466)).
Em CUDA isso vira:

- ou um único thread por camada (subutilização de SM);
- ou refatorar para `parallel scan` (algoritmo de Brent-Kung) — não trivial.

JAX já resolve isso com `lax.scan`, que XLA compila para um padrão
parallel-scan eficiente.

#### 5.3.3 Diferenciação automática

PINN, treinamento e inversão diferenciável (`∂H/∂ρ`) requerem **autodiff**.
Numba CUDA **não tem autodiff** — você precisaria implementar gradientes
manualmente (forward-mode dual numbers ou backward via tape recording).
Esforço estimado: ~6 semanas.

JAX entrega `jacfwd` em **uma linha**:

```python
J = jax.jacfwd(forward_pure_jax, argnums=(0, 1))(rho_h, rho_v, ctx)
```

#### 5.3.4 Esforço de migração

Reescrever os 4000 LOC de `_numba/` para `_cuda/` (kernels CUDA C-style):

- Propagação TE/TM: ~3 semanas (loops scan)
- Dipolos HMD/VMD com 6 casos: ~4 semanas (lax.switch → if-else CUDA)
- Hankel quadratura: ~1 semana (einsum → manual reduction)
- Geometria + rotação: ~1 semana
- Validação (paridade <1e-12): ~3 semanas
- **Total: ~12 semanas (3 meses)**

Para comparação, o JAX `_jax/` (Sprints 1.1–12) levou ~6 meses para chegar
ao estado atual, e **já roda em GPU**. Phase 2 (PR #24) é apenas **mais 2
semanas**.

### 5.4 Quando Numba CUDA SERIA recomendado

Se o projeto tivesse:

- **Kernel hot único < 50 LOC** com paralelismo embaraçoso;
- **Sem necessidade de autodiff**;
- **Sem necessidade de TPU/multi-GPU**;
- **Times com expertise CUDA C++**.

Nada disso aplica ao Geosteering AI v2.0.

### 5.5 Projeções de performance — comparação 4-way

Para o cenário `oklahoma_28` (22 camadas TIV, 30 posições, 1 freq), em
hardware comum:

| Backend           | Hardware        | Throughput estimado | Speedup vs Numba CPU 8t |
|:------------------|:----------------|:-------------------:|:-----------------------:|
| Numba CPU 1 thread | i9 8c           | 7k mod/h            | 0.06×                   |
| Numba CPU 8 threads | i9 8c          | 60k mod/h           | 1.0× (baseline)         |
| **Numba CPU 4w × 2t** | i9 8c        | **~85k mod/h**      | **~1.4×**               |
| JAX CPU (jit)     | i9 8c           | ~50k mod/h          | 0.83× (overhead trace)  |
| JAX GPU T4 (Sprint 10 P2) | T4 16 GB | **~300k mod/h**     | **~5.0×**               |
| JAX GPU A100      | A100 40 GB     | **~800k mod/h**     | **~13×**                |
| Numba CUDA (hipotético) | T4 16 GB | ~250k mod/h (otimista) | ~4.2×                |

**Observação 1**: Numba CUDA *poderia* alcançar throughput similar a JAX GPU se
fosse implementado idealmente, mas exigiria 12 semanas de trabalho para algo
que JAX já entrega.

**Observação 2**: o ganho real de JAX GPU vem de **batch processing massivo**
— quando você simula 1000+ modelos com mesmo perfil estático, o XLA fusion
e `vmap` aproveitam toda a GPU. Para 1 modelo isolado, JAX CPU é mais lento
que Numba CPU pelo overhead de tracing.

### 5.6 Recomendação final

| Cenário                                  | Backend recomendado                  |
|:-----------------------------------------|:-------------------------------------|
| Single model, debug                      | Numba CPU (Modo A — 1 thread)        |
| Single model, n_pos grande               | Numba CPU (Modo B — multi-thread)    |
| Lote 30k modelos, sem GPU disponível     | **Numba CPU (Modo D — 4w × 2t)**    |
| Lote 30k modelos, T4 disponível          | **JAX GPU bucketed**                  |
| Lote 30k modelos, A100 disponível        | **JAX GPU unified (Phase 2)**         |
| PINN / inversão diferenciável            | **JAX CPU/GPU** (jacfwd nativo)       |
| Multi-GPU, cluster                       | JAX (`pmap`)                          |
| Numba CUDA                               | **Não recomendado**                   |
