## §0. Sumário Executivo — TL;DR

O usuário levantou **duas observações estratégicas que invalidam parcialmente o foco atual** do desenvolvimento do simulador JAX GPU e exigem reformulação imediata do roadmap Sprint O:

1. **Objetivo real do simulador GPU**: não é "ser rápido em A100" — é **superar o Numba JIT CPU com paralelismo de threads e workers (i9 8C/16T multi-process, 7970X 32C/64T multi-process), em TODOS os cenários, especialmente os de produção realista**. Esse ganho de throughput deve se traduzir em redução tangível de tempo de treino das redes neurais (incluindo PINNs) e do tempo de inferência online em geosteering.

2. **A obsessão com A100 vicia o design**: a comparação deve ser **GPU vs CPU paralelo absoluto**, e o simulador JAX deve **extrair o máximo de qualquer GPU NVIDIA** (T4, V100, A40, A100, H100, RTX A6000). É um simulador de **caráter universal**, não específico A100.

**Após análise quantitativa rigorosa dos baselines oficiais** (`.claude/perf_baseline.json` + 6 reports de baseline), **ambas as observações se confirmam quantitativamente** e o gate atual (`≥1.5× Numba T4 single-process`) é **metodologicamente inflado**. O simulador JAX GPU **perde** para Numba CPU multi-worker em cenários E (n_pos=600) e G (multi-freq×TR×ang), que são justamente os cenários de geosteering em produção. A reformulação proposta:

- **Novo objetivo primário**: `JAX GPU (qualquer NVIDIA Pascal+) > Numba CPU multi-worker (i9 8C ou superior)` em **TODOS os cenários A-G**, com cenário H tolerado como degradação aceitável em hardware sub-A100.
- **Princípio universal-GPU**: design para T4/V100/A6000 (commodity) PRIMEIRO; A100/H100 é validação adicional. Zero hardware-tuning específico a uma microarquitetura no código de produção (exceto detecção runtime de mem_fraction e cache size).
- **Re-baselining obrigatório**: medir Numba CPU multi-worker oficial (i9 + 7970X) e adicionar ao `perf_baseline.json` como **referência de gate primário**.

**Recomendação imediata**: pausar Sprint O2 Phase 2 (chunking) e reorientar Sprint O2 Phase 1 (Recovery) para fechar primeiro o gap absoluto vs Numba CPU multi-worker em T4, antes de qualquer otimização incremental adicional.

**Diagnóstico inequívoco**:
- ✅ **Wins ambíguos** (cenários "fáceis"): A, C, D — single-pos ou multi-freq overhead-bound; GPU sempre vence aqui.
- ❌ **Losses críticos**: **E, G, H** — cenários **realistas de geosteering** (n_pos=600 = poço completo, multi-freq×TR×ang = tool real LWD multi-spacing). **O simulador JAX GPU perde para Numba CPU multi-worker**.
- ⚠️ **Win/loss dependente de hardware**: B/F — vence em i9 mas pode perder em 7970X. Sinal de fragilidade.

### §1.3. Por que o gate atual mascara o problema

O `_gate_criterion` documentado é:

```json
"min_ratio": 1.5,
"scenarios": ["A", "B", "E"],
"baseline_ref": "ratio_vs_numba_t4_local"
```

O `ratio_vs_numba_t4_local` (campo `2.562` para A, `1.899` para E) compara contra Numba **single-process** rodando em **n1-standard-4** (4 vCPUs Skylake, ~16 GB RAM) — não contra Numba multi-worker em hardware real. Para o cenário E:

- Numba T4 local single-process: ~22.700 mod/h (43.021 / 1.899)
- Numba i9 multi-worker (8C): ~122.000 mod/h
- **Inflação metodológica**: 122.000 / 22.700 = **5,4× otimismo**

O gate "PASS" registrado em todos os cenários é tecnicamente correto **dentro da metodologia escolhida**, mas a metodologia escolhida **não reflete o cenário real de competição CPU vs GPU**. A nota `_historical_i9_comparison` no JSON reconhece isso ao oferecer "normalização per-core" como defesa — o que é **um sintoma do problema, não a solução**: o GPU é um device discreto disputando com 8/16/32 cores de CPU agindo em paralelo absoluto.

## §2. Insight #1 — O Verdadeiro Objetivo é Bater Numba CPU Multi-Worker

### §2.1. Recontextualização do projeto

O simulador EM 1D TIV existe para **três cargas-de-trabalho de Geosteering AI**:

| Carga | Volume | Tolerância latência | Backend preferido (atual) |
|:--|:--|:--|:--|
| Geração de dataset (treino DL) | 100k–1M+ modelos | dias (offline) | Numba multi-worker |
| Treino PINN (auto-diff sobre ρ_h, ρ_v, esp) | 10k+ iter × forward + grad | ~horas | JAX (`jax.grad` obrigatório) |
| Inferência online (geosteering realtime) | 1 modelo × multi-pos × multi-freq | <50 ms | Numba ou JAX |

**Por que GPU?** Não é dogma — é porque **a CPU paralela atinge teto físico** para certos workloads:

- **Geração de dataset**: o Numba 7970X 32C atinge ~400k mod/h em E. Para 1M modelos, isso são **2,5 horas**. Se a GPU faz 10× mais (4M mod/h), são **15 minutos**. Esse delta é o ROI da GPU.
- **Treino PINN**: requer `jax.grad` (Numba não diferencia automaticamente). Aqui a GPU **não é opcional** — é arquitetural. Mas a GPU também precisa ser eficiente, senão um treino de 24h vira 1 semana.
- **Inferência online**: 1 modelo, latência crítica. Aqui a GPU pode perder por overhead de transferência (PCIe). Numba pode ser superior.

**Quando GPU deve ganhar (objetivo do projeto)**:
1. **Geração de dataset em escala (batch ≥50 modelos)**: GPU deve ser ≥3× Numba 7970X 32C em todos os cenários A-G.
2. **Treino PINN (forward+grad)**: GPU deve completar 1 iter em <50ms para perfis típicos (n_pos≈300, multi-freq).
3. **Inferência online**: GPU deve ter latência total (incl. transferência) ≤2× Numba single-thread.

**Quando CPU multi-worker deve ganhar (aceitável)**:
1. Workloads serial-bound puro (cenário A, single-position, n_models=1).
2. Workloads dominados por I/O (filtros Hankel, FilterLoader bootstrap).

### §2.2. Reformulação do gate de aceitação

O gate **deve mudar de**:

```
PASSA SE: JAX GPU / Numba T4 single-process ≥ 1.5× em A, B, E
```

**para**:

```
PASSA SE: JAX GPU (qualquer NVIDIA Pascal+) / Numba CPU multi-worker (i9 8C ou ≥) ≥ R em TODOS cenários A-G:

| Cenário | R mínimo (T4) | R mínimo (A100) | Rationale |
|:--:|:--:|:--:|:--|
| A | ≥1.0× | ≥2.0× | Overhead-bound, GPU sempre vence se warmup OK |
| B | ≥1.0× | ≥2.0× | Bandwidth-bound, depende de VRAM |
| C | ≥1.5× | ≥3.0× | Multi-freq sweet spot vmap |
| D | ≥1.0× | ≥2.0× | Multi-TR sweet spot |
| E | ≥1.5× | ≥2.5× | **CRÍTICO** — poço completo n_pos=600 |
| F | ≥1.5× | ≥3.0× | Multi-freq×TR realista |
| G | ≥1.0× | ≥2.0× | Triple-nested vmap stress test |
| H | aceito OOM | ≥1.0× | Stress 8×8×8, sub-A100 OOM aceitável |

E adicionalmente: Numba 7970X (32C/64T):
- Cenário E: JAX A100 deve ser ≥1.5× (não falhar em hardware HEDT)
- Demais: pode ficar até 0.8× (aceito como gap de hardware)
```

Este gate é **honesto** sobre o que o projeto persegue: superar a alternativa real (Numba multi-worker), não uma alternativa enfraquecida (Numba single-process Colab).

## §3. Insight #2 — Princípio Universal-GPU

### §3.1. Diagnóstico do A100-centrism atual

Auditoria de `geosteering_ai/simulation/_jax/` revela tuning hardware-específico:

| Local | Padrão | Diagnóstico |
|:--|:--|:--|
| `_jax/__init__.py:_setup_xla_environment` | Detecta A100/H100 via `nvidia-smi`; aplica `latency_hiding_scheduler=true` SÓ se A100-class | Patch O1-fix-2 — necessário porque Sprint O1 quebrou T4. Sintoma de A100-tuning vazado para produção. |
| `_jax/dipoles_unified.py:_get_max_layer_unroll` | Retorna `8` se A100/H100, `1` caso contrário | Sprint O1 introduziu `unroll=8` como otimização A100 que regrediu T4 30-47%. |
| `_jax/propagation.py:_effective_unroll_for_scan` | Idem — A100 unrolla, T4 não | Mesma regressão de origem. |
| `perf_baseline.json` | Seções `jax_gpu_t4` e `jax_gpu_a100` separadas | Aceito (medidas independentes), mas reforça narrativa "A100 é a baseline oficial". |

**A consequência**: o caminho hot tem **branches hardware-específicos** que:
- Aumentam complexidade do código (cada branch é mais 1 ponto de regressão).
- Distorcem o foco de design (escolhas otimizadas para A100 propagam para T4 mesmo com runtime check).
- Requerem manutenção de baselines duplicadas (T4 + A100 + futuramente V100).

### §3.2. Princípio universal-GPU — formulação

**Princípio rector** (a partir de Sprint O2):

> **Todo código em `_jax/` deve performar dentro de ±20% do ótimo absoluto para a GPU em uso, sem branches específicos a microarquitetura no caminho hot.**

Operacionalmente:
1. **Algoritmos uniformes**: `lax.fori_loop` em vez de `lax.scan(unroll=K)` para recursões com `K` dependente de hardware. `lax.scan` só quando há ganho universal (sem `unroll` específico).
2. **Memory layout uniforme**: shapes contíguas, `complex128` por default, `complex64` opt-in via `cfg.dtype="complex64"` (Sprint O4 — opt-in universal, não A100-only).
3. **XLA flags conservadoras**: zero `XLA_FLAGS` específicas no `__init__.py`. Se necessário (raro), aplicar via env var documentada e nunca via runtime branch.
4. **Mem fraction única**: `XLA_PYTHON_CLIENT_MEM_FRACTION=0.80` para todos (suficiente em T4 15 GB e A100 40 GB). Eliminar branch A100=0.85 vs T4=0.75.
5. **Cache size único**: `lru_cache(maxsize=64)` ou compilation cache do JAX — tamanho universal, não tuned per device.
6. **Detecção runtime mínima**: apenas para logging e métricas, nunca para mudar comportamento numérico/de fusão.

**Test universal**: rodar `pytest -m gpu tests/test_simulation_jax_perf_baseline.py` em T4, A100, V100 (quando disponível), A6000 (local). Mesmo binário, mesmas flags, ≤20% de variância normalizada vs ratio CPU.

### §3.3. Implicações imediatas no código

A reformulação invalida 2 patches recentes:

| Patch | Status atual | Sob Princípio Universal-GPU |
|:--|:--|:--|
| O1-fix-1 (unroll tier-aware) | Mantido como mitigação | **Substituir** por `lax.fori_loop` sempre (Sprint O2 R1) |
| O1-fix-2 (XLA flags A100-only) | Aplicado em produção | **Remover** o branch — voltar a setdefault padrão sem `latency_hiding_scheduler` |

| `lax.map(batch_size=K)` chunking (Sprint O2 Phase 2) | Benefício universal (limita VRAM, evita OOM) |

## §6. Implicações Arquiteturais

### §6.1. SimulationDispatcher — backend selection automático

O reconhecimento de que **GPU não vence sempre** justifica formalizar um dispatcher:

```python
# geosteering_ai/simulation/dispatcher.py (novo, Sprint O5)

@dataclass
class SimulationDispatcher:
    cfg: PipelineConfig

    def select_backend(self, request: SimRequest) -> Backend:
        """Seleciona backend com base em (workload, hardware, requirements).

        Regras explícitas:
        - PINN training (needs jax.grad): JAX (única opção)
        - Cenários A/D (overhead-bound, single-pos): JAX GPU se disponível
        - Cenários E/G (n_pos large, multi-dim): Numba multi-worker se ≥8 cores;
          senão JAX GPU.
        - Inferência online single-model: Numba se latência crítica, JAX se batch
        - dtype=complex64 explícito: JAX (Numba ainda não tem path complex64)
        """
        if request.needs_grad:
            return Backend.JAX
        if request.n_models == 1 and request.latency_critical:
            return Backend.NUMBA
        if request.is_realistic_workflow():  # E ou G
            return Backend.NUMBA if cpu_cores >= 8 else Backend.JAX
        return Backend.JAX  # Default para batch grande
```

Esse dispatcher é honesto sobre o que cada backend faz melhor.

### §6.2. PINN — JAX continua mandatório

Mesmo com Numba CPU dominando E em throughput puro, **PINN training requer `jax.grad`**. O simulador JAX permanece **arquiteturalmente obrigatório** para:
- Treino de PINNs com física embutida (Maxwell EM)
- Treino com auto-diff sobre parâmetros físicos (rho_h, rho_v, esp)
- Investigação de gradiente para sensibilidade ∂H/∂ρ

A reformulação **não desfaz** o JAX backend — apenas redireciona seu foco de "competir com Numba em throughput sempre" para "ser a única opção quando precisa de grad + ser competitivo em casos batch onde Numba CPU não escala".

## §7. Riscos e Ressalvas

### §7.1. Risco de over-correction

**Risco**: ao priorizar T4/V100, o A100 fica subutilizado. Em workloads que exigem A100 (treino PINN com batches grandes, datasets sintéticos massivos), perderemos ganho.

**Mitigação**: o gate revisado **inclui** thresholds A100 (≥2× Numba). A100 é **medido e gateado**, apenas não é o **driver de design**. Otimizações A100-específicas (TF32, Tensor Cores) podem ser feitas em `cfg.dtype="complex64"` (Sprint O4) como opt-in que rodam em qualquer GPU com Tensor Cores (T4 v100 a100 h100).

### §7.2. Risco de baseline obsoleto

**Risco**: a baseline Numba i9 (`E_n200_warm = 105.423 mod/h` em v2.36) tem mais de 4 meses. Hardware/SO podem ter mudado.

**Mitigação**: re-medir como parte de **Sprint O2b**. Re-medição deve fixar versão Numba (`numba==0.61.0`), versão Python (3.13), flags (`NUMBA_NUM_THREADS=2`, workers=4) e gravar git commit do código no JSON.

### §7.3. Risco de quebrar paridade Fortran

**Risco**: reverter `unroll` para `fori_loop` poderia mudar tolerância numérica (ordem de soma de ponto flutuante).

**Mitigação**: o pytest existente (`tests/test_simulation_jax_fortran_parity.py` + 164 testes em `-m gpu`) tem tolerância `<1e-12`. `fori_loop` e `scan(unroll=1)` são bit-exatos em CPU; em GPU pode haver diferença no nível ULP (assoc. flutuante XLA), absorvida pela tolerância existente. **Verificar** antes de mergear Sprint O2.

### §7.4. Risco de Sprint O2 demorar mais que estimado

**Risco**: a reformulação adiciona Sprint O2b (baseline) ao caminho crítico. Tempo total Sprint O2 estimado: ~3 dias (era ~1 dia para Recovery puro).

**Mitigação**: aceitar. A baseline correta é mais valiosa que velocidade de iteração — sem ela, **Sprint O3-O8 continuariam míopes**.

### §7.5. Risco político — "estamos regredindo o A100"

**Risco**: medições pós-Sprint O2 podem mostrar A100 -5 a -10% vs baseline atual (remover `latency_hiding_scheduler` pode custar isso).

**Mitigação**: documentar como **trade-off explícito**. A100 perde 5-10% (ainda passa gate), T4 ganha 30-47% (recupera regressão O1), universalidade ganha 100% (próximo desenvolvedor não precisa pensar em hardware). ADR-0002 formaliza essa escolha.

## §9. Conclusão

Os dois insights do usuário **estão corretos e revelam um viés metodológico fundamental** no roadmap Sprint O atual. O ajuste proposto é:

1. **Trocar o gate**: de `1.5× Numba T4 single-process` para `≥R× Numba CPU multi-worker (i9 8C ou ≥)`, com R diferenciado por cenário.
2. **Princípio Universal-GPU**: zero hardware-tuning no caminho hot — design para T4/V100/A6000 PRIMEIRO; A100 é apenas validação adicional.
3. **Reformular Sprint O2-O8**: O2 = Recovery + Universal Cleanup; O2b = Baseline Numba multi-worker; O3-O8 = otimizações universais sequenciais.
4. **Reconhecer trade-offs**: A100 pode regredir 5-10% (formalizado em ADR-0002); T4/V100 ganham 30-47% (recupera Sprint O1) + universalidade.
5. **PINN inalterado**: JAX backend continua mandatório para `jax.grad`. A reformulação não desfaz JAX — redireciona seu foco.

### Anexo B — Pseudo-código fix R1 (propagation.py)

```python
def _scan_or_fori_loop(body_fn, init, xs, length):
    """Universal-GPU: usa fori_loop sempre (zero unroll branching).

    Justificativa: lax.scan(unroll=K) gera XLA While op com carry overhead
    quando K=1 (T4/V100/CPU); fori_loop é mais enxuto. Para K>1, scan ainda
    seria melhor em A100, mas removido sob Princípio Universal-GPU (ADR-0002).
    """
    def fori_body(i, carry):
        x_i = jax.tree.map(lambda x: x[i], xs)
        carry_new, _ = body_fn(carry, x_i)
        return carry_new
    return lax.fori_loop(0, length, fori_body, init)
```

### Anexo C — Pseudo-código fix R2 (kernel.py)

```python
# Module-scope cache pinada em device
_FILTER_JAX_CACHE: dict[str, tuple[jnp.ndarray, jnp.ndarray, jnp.ndarray]] = {}

def get_hankel_filter_jax(name: str) -> tuple:
    """Retorna (kr, w_j0, w_j1) já em device, cache singleton.

    Substitui pattern antigo `loader = FilterLoader(); filt = loader.load(name)`
    em todo call, que custava 1-5ms de overhead por chamada (R2).
    """
    if name not in _FILTER_JAX_CACHE:
        loader = FilterLoader()
        filt = loader.load(name)
        _FILTER_JAX_CACHE[name] = (
            jax.device_put(jnp.asarray(filt.kr_j0j1)),
            jax.device_put(jnp.asarray(filt.weights_j0)),
            jax.device_put(jnp.asarray(filt.weights_j1)),
        )
    return _FILTER_JAX_CACHE[name]
```
