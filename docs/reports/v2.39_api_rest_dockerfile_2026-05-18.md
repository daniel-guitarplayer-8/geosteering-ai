Conforme `docs/reports/v2.39_proximos_passos_roadmap_2026-05-17.md` §2.1, a ausência
de API REST era o **gargalo crítico** da Fase 2:

```
┌─────────────────┐
│  API REST MVP   │  I2.7 ← BLOQUEIO RESOLVIDO em v2.39
│  (era AUSENTE)  │
└────────┬────────┘
         │ desbloqueia
   ┌─────┼──────────┬──────────┐
   ▼     ▼          ▼          ▼
Docker  MCP        Studio    Fase 3
I2.8    colab      ALPHA     MLOps
```

A Sprint adota um contrato **`raw_data` 22-colunas** (alinhado ao `InferencePipeline.predict()`
existente), em detrimento da proposta inicial `{z_obs, H_re, H_im}` do roadmap. Justificativa:
o formato 22-col é canônico no projeto (validado por `config.n_columns=22`), evita uma camada
de tradução, e mantém 1:1 com a função real. A versão decomposta fica como endpoint futuro
`POST /predict/components` (v2.40+).

## 4. Contratos HTTP

### 4.1 `GET /health` (custo <1 ms, sem TF)

```json
Response 200:
{
  "status": "ok",
  "version": "2.39.0",
  "model_loaded": false,
  "model_path": null
}
```

Adequado para Kubernetes liveness/readiness probes. Reflete o estado do singleton sem chamar `get_pipeline()`.

### 4.2 `POST /predict` (lazy load TF no 1º request)

**Request**:
```json
{
  "raw_data": [[[...22 floats]×seq_len]×N],
  "theta": 15.0,                    // opcional, [0, 90] graus
  "freq": 20000.0,                  // opcional, [100, 1e6] Hz
  "return_uncertainty": false,      // opcional, default false
  "mc_samples": 30                  // opcional, [2, 200]
}
```

**Response 200 (sem UQ)**:
```json
{
  "predictions": [[[rho_h, rho_v], ...], ...],
  "shape": [N, seq_len, 2],
  "latency_ms": 12.4,
  "model_type": "ResNet_18"
}
```

**Response 200 (com UQ)**: idem + `"uncertainty": [...]` + `"mc_samples": 30`.

### 4.3 Erros padronizados

| Código | `type` | Causa | Resolução |
|-------:|:-------|:------|:----------|
| 413 | `payload_too_large` | Content-Length > `max_body_bytes` (16 MiB default) | Dividir em batches |
| 422 | (Pydantic) | Shape inválido, ranges físicos, extra fields | Corrigir request |
| 503 | `model_not_loaded` | `GEOSTEERING_MODEL_PATH` não setada | Configurar env var + montar volume |
| 503 | `model_load_failed` | Path inválido / pipeline corrompido | Verificar artefatos do pipeline |
| 500 | `internal` | Exceção não-categorizada | Consultar logs do servidor |

Mensagens sanitizadas — **não vazam** nome de env var, paths absolutos ou stack traces.

## 5. Decisões de Design (D1-D8)

| ID | Decisão | Justificativa |
|:--:|:--------|:--------------|
| D1 | Contrato `raw_data` 22-col (não `{z_obs, H_re, H_im}`) | Alinha com `InferencePipeline.predict()` real; evita camada de tradução |
| D2 | Singleton lazy + `threading.Lock` (sem `Depends`) | `/health` funciona sem TF; ordem Pydantic→Pipeline garantida |
| D3 | Extras `[api]` opcional + imports lazy | Pacote `geosteering_ai/` continua usável sem FastAPI |
| D4 | Dockerfile.cpu multi-stage, SEM Fortran | MVP inferência; Fortran fica para `Dockerfile.fortran` v2.40+ |
| D5 | Workflow CI `paths-filter` + `concurrency.cancel-in-progress` | Economiza minutos GH Actions; sem rebuild desnecessário |
| D6 | Entry point `geosteering-api` via `argparse + uvicorn.run` | Padrão alinhado com `geosteering-cli/_main.py` |
| D7 | Exceções tipadas `ModelNotLoadedError`/`ModelLoadFailedError` | Cliente diferencia env var ausente vs pipeline corrompido |
| D8 | Mega-header D1-D14 + docstrings Google-style + PT-BR | Conformidade absoluta CLAUDE.md |
