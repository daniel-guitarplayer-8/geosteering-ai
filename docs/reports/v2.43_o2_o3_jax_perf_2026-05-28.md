de baseline (v2.43_paridade_jax_gpu_e_bench_numba_vs_jax_2026-05-28.md):

em JAX GPU**, **3.3× mais rápido que Numba 4w×16t** (3.95M mod/h). Este é o regime
PINN típico (treino com batches de muitas amostras pequenas), onde JAX agora
**vence claro** Numba.

Em E (600 posições, single forward) e H (multi-combo), o workload é
**memory-bound, não FP-bound** — `complex64` não acelera porque o gargalo é
banda de memória GPU e overhead de dispatch XLA. Ganho marginal observado

**Bug crítico encontrado e corrigido**: a wiring inicial de `complex_dtype`
em Sprint O2 (9 commits) tinha 3 vazamentos silenciosos para `complex128`
(escalar `zeta`, kwarg ausente em `_dipoles_impl`, `H_tensor` numpy
hardcoded). Após fix, `forward_pure_jax(ctx_c64).dtype == complex64` ✓.

### 4.2 Arquitetura

`SimulationConfig.dtype` (já existe em `config.py:176-181, 361`) → string
`"complex128"|"complex64"`. Default `"complex128"` (paridade Fortran sagrada).

**Helper canônico** (`forward_pure.py` topo):
```python
_COMPLEX_DTYPE_MAP = {"complex128": jnp.complex128, "complex64": jnp.complex64}
```

**Propagação por pipeline**:
```
SimulationConfig.dtype
    → build_static_context(complex_dtype=...)
    → ForwardPureContext.complex_dtype
    → common_arrays_jax(complex_dtype=...)  [7 sites]
    → common_factors_jax(complex_dtype=...)
    → native_dipoles_full_jax_unified(complex_dtype=...)  [8 sites]
    → _single_position_jax(complex_dtype=...)  [via _dipoles_impl]
    → H_out via jnp.zeros(dtype=_COMPLEX_DTYPE_MAP[...])
```

**Float64 imutável**: `rho_h`, `rho_v`, `freqs`, `positions_z`, `esp` permanecem
`float64` (parâmetros reais). Apenas tensor H complexo e arrays propagadores
TE/TM (`u`, `s`, `uh`, `sh`, `RTEdw`, `RTEup`, `Mxdw`, `Mxup`, etc.) viram `complex64`.

### 4.5 Bug crítico encontrado e corrigido (commit 7ee9dbe)

Investigação pós-implementação revelou que `forward_pure_jax(ctx_c64).dtype`
retornava `complex128` em vez de `complex64`. **3 vazamentos para c128**:

1. **Escalar `zeta` promotion** em `kernel.py:330`:
   ```python
   # ANTES (bug): zeta = 1j*omega*_MU_0  → Python complex128 default
   # DEPOIS (fix): zeta = jnp.asarray(1j*omega*_MU_0, dtype=complex_dtype)
   ```
   Propagava para `u, s, uh, sh` via `common_arrays_jax::_per_layer::kh2=-zeta*sigma_h`.

2. **`_dipoles_impl` sem `complex_dtype` explícito** em `kernel.py:377`:
   ```python
   # ANTES (bug): matH = _dipoles_impl(Tx, Ty, ..., u, s, ...)
   # DEPOIS (fix): matH = _dipoles_impl(Tx, Ty, ..., u, s, ..., complex_dtype=complex_dtype)
   ```
   Dependia de derivação silenciosa de `u.dtype` — frágil porque qualquer
   promoção implícita em operações intermediárias revertia para c128.

3. **`H_tensor` numpy hardcoded** em `multi_forward.py:535`:
   ```python
   # ANTES (bug): H_tensor = np.empty((nTR, nAng, n_pos, nf, 9), dtype=np.complex128)
   # DEPOIS (fix): H_tensor = np.empty(..., dtype=_NP_DTYPE_MAP[cfg.dtype])
   ```

**Validação pós-fix** (GPU, oklahoma_5):
```
c128 forward: dtype=complex128, shape=(600, 1, 9)
c64  forward: dtype=complex64, shape=(600, 1, 9)
parity c64 vs c128: max_abs=6.112e-05, max_rel=3.875e-04
```

## 5. Sprint O3p1 — `_CTX_CACHE` LRU

### 5.1 Motivação

Diagnóstico empírico baseline mostrou `build_static_context` (NumPy puro)
custando **60 ms/modelo** (38% do wall-time em Cenário E). Em PINN típico,
varia-se ρ mantendo geometria `(esp, positions_z, freqs, tr_spacing, dip,
filter, strategy, chunk_size, complex_dtype)` constante. Contexto é
totalmente cacheável.

### 5.2 Implementação

`forward_pure.py` (topo, junto a `_BUCKET_JIT_CACHE`):

```python
_CTX_CACHE: OrderedDict[bytes, ForwardPureContext] = OrderedDict()
_CTX_CACHE_MAXSIZE: int = 32  # cobre ~32 configs PINN típicas

def _hash_ctx_key(esp, pos_z, freqs, tr, dip, filter, strategy, chunk, dtype) -> bytes:
    h = hashlib.blake2b(digest_size=24)
    h.update(esp.tobytes()); h.update(pos_z.tobytes()); h.update(freqs.tobytes())
    h.update(struct.pack("dd", tr, dip))
    h.update(filter.encode()); h.update(strategy.encode()); h.update(dtype.encode())
    h.update(struct.pack("q", chunk if chunk else -1))
    return h.digest()

def build_static_context_cached(...):
    key = _hash_ctx_key(...)
    if key in _CTX_CACHE:
        _CTX_CACHE.move_to_end(key)  # LRU touch
        return _CTX_CACHE[key]
    ctx = build_static_context(...)
    if len(_CTX_CACHE) >= _CTX_CACHE_MAXSIZE:
        _CTX_CACHE.popitem(last=False)  # LRU evict
    _CTX_CACHE[key] = ctx
    return ctx
```

### 5.3 API pública (paralela a `_BUCKET_JIT_CACHE`)

| Função | Propósito |
|:-------|:----------|
| `set_ctx_cache_maxsize(maxsize: int)` | Ajusta tamanho máximo (e evicta se excede) |
| `clear_ctx_cache() -> int` | Limpa cache, retorna nº entries removidas |
| `get_ctx_cache_info() -> dict` | Diagnóstico `{n_entries, maxsize}` |
| `build_static_context_cached(...)` | Wrapper LRU principal |

### 5.4 Integração

- `multi_forward.py::simulate_multi_jax` (linha ~612): chama `build_static_context_cached`
- `simulate_multi_jax_batched`: NÃO integrado (path já otimizado; ganho marginal — futura otimização)

## 10. Análise: por que c64 não acelerou E e H?

### 10.1 Workload memory-bound, não FP-bound

**Hipótese teórica** (pré-Sprint): A6000 tem 32× mais FLOPS em FP32 que FP64
(38.7 TFLOPS vs 1.2 TFLOPS). Espera-se 2-4× speedup real para complex64.

**Medição empírica** (Cenário E, single-call): c128 = 100ms, c64 = 100ms.

**Diagnóstico via decomposição**:
- `build_static_context` (NumPy puro): ~60ms (não muda com dtype)
- `forward_pure_jax` XLA execution: ~40ms (idêntico c128/c64)

Conclusão: a parte XLA gasta seu tempo em **memory bandwidth** (load/store de
tensores complexos) e **kernel launch overhead**, não em compute. complex64
reduz bandwidth por 2× mas o ganho é absorvido por overhead constante.

### 10.2 Por que Cenário A vence?

A é único caso onde batching paga: 50 modelos pequenos (n_pos=1) fundidos em
1 chamada JAX via vmap externo. GPU saturada com computação paralela; o
overhead por modelo (que dominava o per-model loop) é amortizado em 1
dispatch. Isso é exatamente o regime PINN: muitos forward passes pequenos
por gradient step.

### 10.3 Por que E não vence batched?

E já tem n_pos=600 (single model fundido em 1 vmap). Adicionar batching por
modelo não melhora — cada modelo já é "rico" o suficiente para saturar GPU.
O bottleneck é a **propagação multicamada sequencial** (lax.scan sobre 5
camadas TE/TM), que é serial por natureza.

### 10.4 Por que H OOM em batched?

VRAM: 50 modelos × 8 freq × 8 TR × 8 dip × 100 pos × n × npt × c128 = ~97 GB
de tensores intermediários. A6000 tem 48 GB. Sprint O4 implementaria
`chunk_size_models` para tiling (e.g., 4 modelos/chunk × 12 chunks).
