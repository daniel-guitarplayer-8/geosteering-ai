## Pergunta 1 — GUI como Trilha Separada no Roadmap

### 1.1 O Estado Atual

O roadmap possui 6 trilhas (A–F). O único item relacionado à interface gráfica
está em:

```
Trilha F (Governance) | F-mvc-split | Status: BACKLOG P4
"MVC split físico do Simulation Manager (refator package)"
```

Atualmente, tudo relacionado a UI/UX vive em Trilha F (Governance) ou em
Trilha E (Infrastructure), o que é **semanticamente incorreto**:

- **Governance (F)** cobre hooks, skills, ADRs, documentação — não código de
  interface gráfica
- **Infrastructure (E)** cobre CLI, API REST, Docker, MCP — não widgets Qt

O Simulation Manager GUI tem >10.000 linhas em um único arquivo
(`simulation/simulation_manager.py`). A análise arquitetural completa
(`docs/reports/mvc_simulation_manager_studio_analysis_*.md`) documenta o
escopo substancial do trabalho de UI. O "Geosteering AI Studio" (versão
distribuível para geofísicos) é um produto de frontend completo planejado.

### 1.2 A Proposta: Trilha G — Frontend/GUI

A sugestão de adicionar Trilha G faz sentido por **4 razões técnicas**:

**Razão 1 — Escopo distinto e substancial**

O trabalho de frontend envolve:

| Componente | LOC estimado | Tecnologia |
|:--|:--:|:--|
| `simulation_manager.py` (monolítico) | >10.000 | PyQt6/PySide6 |
| GUI pytest-qt (testes) | ~500 atual | pytest-qt |
| Geosteering AI Studio (futuro) | 5.000-15.000 | PyQt6 ou Electron/Tauri |
| Widgets de visualização EM | ~2.000 | PyQt6 + Matplotlib/Plotly |
| UX geofísico (fluxos de usuário) | design | User research |

Este é um domínio técnico separado, com seus próprios padrões (MVC/MVP, signal-
slot Qt, thread safety entre UI e workers, teste headless com xvfb).

**Razão 2 — Perfil de habilidade diferente**

Trabalho nas Trilhas A-F requer: geofísica, Numba JIT, TensorFlow, FastAPI,
shell scripting. Trabalho na Trilha G requer: PyQt6, design de UX para domínio
científico, acessibilidade, teste de UI, empacotamento cross-platform. São
skillsets diferentes que não se sobrepõem significativamente.

**Razão 3 — Confusão semântica atual**

Com `F-mvc-split` em Trilha F, um colaborador novo que lê "Trilha F —
Governance" esperaria encontrar hooks e ADRs, não refatoração de widgets Qt.
A Trilha G resolve essa confusão: basta olhar `Trilha G` para encontrar todo o
trabalho relacionado a GUI/UX.

**Razão 4 — Itens de backlog específicos**

Os itens que pertenceriam naturalmente a Trilha G:

| Code (proposto) | Item |
|:--|:--|
| `G-mvc-split` | MVC split do Simulation Manager (migrado de F-mvc-split) |
| `G-studio-alpha` | Geosteering AI Studio ALPHA (distribuição standalone) |
| `G-pytest-qt-expand` | Expansão dos 16 testes GUI para 40+ (parcialmente em B-numba-O4-O6) |
| `G-ux-geofisico` | UX research para fluxo de uso de geofísicos de campo |
| `G-dark-mode` | Tema dark mode + acessibilidade (contraste para monitores de campo) |
| `G-plot-export` | Exportação de plots EM para PDF/SVG (relatórios de campo) |

### 1.3 Contra-Argumentos

A proposta não é trivialmente correta. Há contra-argumentos legítimos:

**Contra 1 — Complexidade de governança**

7 trilhas são mais difíceis de manter do que 6. O ganho semântico de separar F
de G precisa superar o custo de atualizar ROADMAP.md, CLAUDE.md, e todos os
documentos que citam "Trilhas A-F".

**Contra 2 — Volume atual é baixo**

Hoje existe 1 item GUI no backlog (`F-mvc-split`). Criar uma trilha para 1 item
pode ser prematuro. A convenção poderia ser: criar Trilha G quando ≥3 itens GUI
estiverem no backlog ativamente.

**Contra 3 — F-mvc-split é P4 (longo prazo)**

O MVC split é prioridade P4 — não será executado nas próximas 6-8 sprints.
Criar uma trilha agora para trabalho de P4 pode ser overengineering.

### 1.4 Recomendação

**Criar Trilha G é a decisão correta, mas não agora.**

O momento certo é quando o "Geosteering AI Studio" entrar em CANDIDATE (P2/P3).
Nesse ponto, a carga de trabalho justifica uma trilha dedicada. Por ora:

- **Mover `F-mvc-split` para Trilha E** (Infrastructure é mais semanticamente
  próximo de "refatoração de aplicativo" do que "governance")
- **Anotar em ROADMAP.md** que Trilha G existe conceitualmente e será formalizada
  quando Studio ALPHA entrar em CANDIDATE
- **Criar ADR-0002** quando a decisão de formalizar Trilha G for tomada (R7
  da ADR-0001 recomenda ADR para decisões de escopo de sprint)

## Pergunta 2 — SurrogateNet Dataset Apenas Dip=0°

### 2.1 Resultado da Investigação

Esta pergunta revelou um **mal-entendido parcial**. A realidade é mais nuançada:

| Afirmação | Verdade |
|:--|:--|
| "Dataset atual é só dip=0°" | ✅ Verdade — os arquivos `.dat` em disco foram gerados apenas com dip=0° |
| "O simulador Python não suporta multi-dip" | ❌ Falso — suporta completamente desde Sprint 11 (PR #15) |
| "Precisamos do Fortran para gerar multi-dip" | ❌ Falso tecnicamente — mas há nuances práticas importantes |
| "JAX GPU pode gerar o dataset" | ✅ Verdade — e é a abordagem recomendada |

### 2.2 Por Que o Dataset É Só Dip=0°

O módulo `surrogate_data.py` **não executa simulação**. Ele extrai pares de um
arquivo `.dat` pré-gerado:

```python
# geosteering_ai/data/surrogate_data.py — lógica central
def extract_surrogate_pairs(
    data: np.ndarray,  # shape (N_models, seq_len, 22) — vem do .dat
    config: PipelineConfig,
    *,
    apply_decoup: bool = True,
) -> SurrogateDataset:
    """
    Extrai pares (rho_h, rho_v) → H_EM de arquivo .dat pré-gerado.
    NÃO executa simulação — apenas processa dados já calculados.
    """
    x_rho = data[:, :, OUTPUT_TARGETS]   # log10(rho_h), log10(rho_v)
    y_em  = data[:, :, INPUT_FEATURES]   # Re/Im das componentes selecionadas
    ...
```

O arquivo `.dat` de 22 colunas é o formato de saída do **simulador Fortran**
(`tatu.x`). Os arquivos existentes em disco foram gerados em sessões anteriores
de simulação, com parâmetros `dip=0°`. Como o `surrogate_data.py` lê esses
arquivos, o dataset resultante só tem dip=0° — não por limitação do código, mas
por limitação dos **dados em disco**.

Em outras palavras: o problema não é técnico, é de **dados históricos**.

### 2.3 O Simulador Python Suporta Multi-Dip?

**Sim, completamente.** O suporte foi implementado em Sprint 11 (PR #15).

```python
# multi_forward.py:661-682 — API completa
def simulate_multi(
    rho_h, rho_v, esp, positions,
    frequencies_hz,
    tr_spacings_m,
    dip_degs: Optional[Sequence[float]] = None,  # ← SUPORTE COMPLETO
    backend: str = "numba",
    config: Optional[SimulationConfig] = None,
) -> MultiSimulationResult:
    """
    Aceita dip_degs=[0.0, 30.0, 60.0, 90.0] — qualquer array de ângulos.
    """
```

O simulador Python implementa a mesma lógica geométrica do Fortran:

```python
# Geometria de dip — mesma física do Fortran
dip_rad = np.radians(dip_deg)
hordist = L × |sin(dip_rad)|      # distância horizontal TR
dz_half = 0.5 × L × cos(dip_rad) # offset vertical por dip
pz = p_med × cos(dip_rad)         # spacing vertical adaptativo
```

**Validação de paridade:** O teste `test_fortran_byte_exact_dat_nonzero_dip`
confirma paridade <1e-12 vs Fortran para dip=30°, incluindo componentes
off-diagonal (Hxz, Hzx) que só são não-nulas quando dip≠0°.

### 2.4 Por Que Então Não Usamos o Simulador Python Para Gerar Multi-Dip?

A resposta honesta é: **podemos, mas há trade-offs de performance que explicam
por que ainda não foi feito.**

**Throughput comparativo do simulador Python em CPU:**

| Cenário | Throughput medido | Configuração |
|:--|:--:|:--|
| Cenário A (simples) | ~1.4M mod/h | single-process |
| Cenário E (produção, n=600 pos) | ~95-150k mod/h | 4 workers × 2 threads |
| Cenário F (multi-TR, multi-freq) | ~300k mod/h | 4 workers × 2 threads |

Para um dataset de treinamento útil (ex: 100.000 modelos geológicos × 4 dips ×
600 posições), o tempo estimado em CPU:

```
100.000 modelos × 4 dips = 400.000 simulações de modelo completo
Throughput Cenário E: ~120.000 mod/h
Tempo estimado: ~3.3 horas em CPU (4 workers × 2 threads, M-series Mac)
```

Isso é viável — mas há um fator adicional: **cada "modelo" no dataset é uma
sequência de 600 posições × N_componentes EM**, o que é pesado para armazenar.
100.000 modelos × 4 dips × 600 pos × 22 colunas × 8 bytes (float64) =
~422 GB em disco (sem compressão).

Na prática, o dataset de treinamento atual provavelmente é uma fração disso
(subconjunto de modelos, posições amostradas mais esparsamente). O ponto é que
a geração não é bloqueada por capacidade técnica — é uma questão de decidir
o tamanho do dataset e o tempo de geração aceitável.

### 2.5 Por Que Não Usar o Simulador JAX GPU?

**Esta é a abordagem correta para geração em larga escala**, e está
tecnicamente disponível.

O simulador JAX (`backend="jax"`) tem as mesmas capacidades multi-dip:

```python
# _jax/multi_forward.py — JAX com vmap aninhado
def simulate_multi_jax(
    ...,
    dip_degs: Optional[Sequence[float]] = None,
    jax_vmap_real: bool = False,  # Sprint 12 — vmap aninhado sobre (iTR, iAng)
) -> MultiSimulationResult:
    ...
```

Com GPU A100 no Colab Pro+, JAX compila os kernels via XLA e obtém speedups
de **10-100× vs CPU Numba** para operações matriciais paralelizáveis:

| Backend | Hardware | Throughput estimado |
|:--|:--|:--|
| Numba | CPU local (8C) | 95-150k mod/h (Cenário E) |
| JAX | CPU local | ~similar ao Numba (sem aceleração GPU) |
| JAX + XLA | Colab T4 (16GB VRAM) | ~500k-2M mod/h (estimado) |
| JAX + XLA | Colab A100 (40GB VRAM) | ~2M-10M mod/h (estimado) |

Note que esses são **valores estimados** — o benchmark formal de JAX GPU para
geração de dataset ainda não foi executado (é um dos objetivos do template
`benchmark_tfdata_mp16.ipynb`).

**Fluxo proposto para gerar dataset multi-dip via JAX GPU:**

```python
# notebooks/colab_templates/generate_dataset_multidip.ipynb (a criar)

from geosteering_ai.simulation import simulate_multi
from geosteering_ai.config import SimulationConfig

sim_cfg = SimulationConfig(
    backend="jax",
    jax_vmap_real=True,         # vmap aninhado para multi-dip
    jax_position_chunk_size=64, # chunking de posições (evita OOM em A100)
)

# Gerar para 4 ângulos em uma única chamada
result = simulate_multi(
    rho_h=rho_h_batch,          # shape (N_models, N_layers)
    rho_v=rho_v_batch,
    esp=esp_batch,
    positions=positions,        # 600 posições
    frequencies_hz=[20000.0],
    tr_spacings_m=[1.0],
    dip_degs=[0.0, 30.0, 60.0, 90.0],  # 4 dips em batch
    backend="jax",
    config=sim_cfg,
)
# result.H: shape (N_models, 4_dips, 600_pos, 1_tr, 1_freq, 3, 3)
```

### 2.6 Síntese: Por Que Só Dip=0° Hoje?

```
Causa raiz: os arquivos .dat históricos foram gerados pelo Fortran com dip=0°
           (decisão de dataset, não limitação técnica do simulador)

Bloqueios técnicos: NENHUM
  ✅ Simulador Python Numba suporta multi-dip (Sprint 11)
  ✅ Simulador Python JAX suporta multi-dip com vmap aninhado (Sprint 12)
  ✅ Paridade Fortran <1e-12 para dip=30° validada em testes

Próximos passos para multi-dip:
  1. Gerar dataset multi-dip via JAX GPU no Colab A100
     (após C-surrogate-train baseline com dip=0°)
  2. Criar notebook generate_dataset_multidip.ipynb
  3. Treinar SurrogateNet v3 com 4 ângulos concatenados como canal adicional
```

A SurrogateNet com dip=0° (item `C-surrogate-train`) é a **primeira etapa**
de uma trajetória planejada: treinar baseline → validar → expandir para multi-dip.
Não é uma limitação permanente, é uma decisão de sequenciamento.

## Pergunta 3 — TensorFlow Não Está Instalado no Venv

### 3.1 Estado Atual Verificado

```bash
# Investigação realizada em 2026-05-19
$ source ~/Geosteering_AI_venv/bin/activate
$ python --version
Python 3.13.5  # venv criado com Anaconda Python 3.13
$ uname -m
x86_64  # macOS Intel (não Apple Silicon)
$ pip install tensorflow --dry-run
ERROR: No matching distribution found for tensorflow
$ pip install tensorflow-macos --dry-run
ERROR: No matching distribution found for tensorflow-macos
```

**Keras está instalado** (v3.14.0) mas **falha ao importar sem TF**:

```bash
$ python -c "import keras"
ModuleNotFoundError: No module named 'tensorflow'
# Keras 3.14 tenta importar tensorflow.python.trackable em seu __init__

$ KERAS_BACKEND=jax python -c "import keras; print(keras.backend.backend())"
jax  # ← funciona corretamente com backend JAX
```

### 3.2 Por Que TensorFlow Não Pode Ser Instalado

A causa-raiz tem **duas camadas**:

**Camada 1 — Plataforma não suportada: macOS x86_64 (Intel Mac)**

O TensorFlow abandonou o suporte a macOS Intel (x86_64) após a versão 2.12.
Isso não é uma decisão arbitrária — é consequência da política da Google e da
Apple:

- Apple descontinuou suporte ativo a Macs Intel em 2020 (transição para Apple
  Silicon ARM)
- O Google TensorFlow priorizou recursos de engenharia para Linux x86_64 (onde
  rodam os servidores e clusters de treinamento) e Linux ARM64 (onde rodam TPUs
  e Apple Silicon)
- O pacote `tensorflow-macos` (mantido pela Apple) **só funciona em Apple
  Silicon** (M1/M2/M3/M4 — arquitetura ARM64) — Intel Macs não estão em seu
  escopo

A última versão disponível para macOS Intel foi:
- `tensorflow==2.12.0` para Python ≤3.11

**Camada 2 — Python 3.13 sem wheel de TensorFlow para nenhuma plataforma Mac**

Mesmo em Apple Silicon, `tensorflow-macos` não tem wheel para Python 3.13 ainda:

```bash
# Em qualquer Mac (Intel ou Apple Silicon), com Python 3.13:
pip install tensorflow      → ERROR: No matching distribution
pip install tensorflow-macos → ERROR: No matching distribution
pip install tensorflow-metal → ERROR: No matching distribution
```

O PyPI exige que o pacote tenha sido compilado para a ABI Python específica
(cp313 para Python 3.13). TF é uma biblioteca com extensões C++ compiladas —
não é puro Python. Cada versão de TF precisa ser recompilada e empacotada para
cada combinação de (OS, arquitetura, versão Python). A equipe TF libera wheels
gradualmente; Python 3.13 em macOS ainda não tem release oficial.

### 3.3 A Solução Atual: Keras 3.x com Backend JAX

Keras 3.x (a versão instalada, 3.14.0) é **multi-backend** por design. Quando
`KERAS_BACKEND=jax`, o Keras usa JAX como motor de cálculo em vez de TensorFlow:

```
TensorFlow 2.x
  └── keras legado (tf.keras) — fortemente acoplado ao TF

Keras 3.x (novo, standalone)
  ├── Backend TensorFlow (padrão quando TF instalado)
  ├── Backend JAX ← usado localmente neste projeto
  ├── Backend PyTorch (opt-in)
  └── Backend NumPy (apenas CPU, para debug)
```

Com `KERAS_BACKEND=jax`, todas as operações de tensor usam `jax.numpy` em vez
de `tf`. O modelo Keras roda identicamente — a diferença é apenas o motor de
cálculo subjacente:

```python
# O código Keras é o mesmo — muda apenas o motor
KERAS_BACKEND=jax python -c "
import keras
model = keras.Sequential([keras.layers.Dense(64, activation='relu')])
print(model.compute_output_shape((None, 10)))  # funciona com JAX backend
"
# TensorShape([None, 64])
```

### 3.4 Implicações Práticas para o Projeto

| Contexto | TensorFlow | Keras Backend | Estado |
|:--|:--:|:--|:--|
| Venv local (macOS Intel) | ❌ Impossível | JAX (`KERAS_BACKEND=jax`) | Funcional |
| GitHub Actions CI (Linux x86_64) | ✅ Disponível | TF (padrão) | Funcional |
| Google Colab (Linux x86_64, Python 3.10-3.11) | ✅ Disponível | TF (padrão) | Funcional — ambiente de treino |
| Apple Silicon Mac | ⚠️ `tensorflow-macos` (Python ≤3.12) | TF ou JAX | Parcial |

**A diferença que importa para o projeto:**

Em Colab (ambiente de treino), `KERAS_BACKEND` não é setado, então Keras usa TF
automaticamente. Isso significa que o código treinado em Colab usa TF como
backend, e o modelo salvo (`.keras`) usa operações TF.

Localmente, `KERAS_BACKEND=jax` permite desenvolver, testar e fazer debug —
mas os modelos gerados localmente podem ter pequenas incompatibilidades se
carregados em ambiente TF (e vice-versa), porque algumas camadas customizadas
usam `tf.*` direto em vez de `keras.ops.*`.

Isso é exatamente a motivação do item de backlog `C-keras-ops` (P4): auditar
todo o código que usa `tf.*` e migrar para `keras.ops.*`, tornando o código
100% portável entre backends.

### 3.5 Seria Possível Instalar TF? Opções e Limitações

| Opção | Viável? | Detalhes |
|:--|:--:|:--|
| `pip install tensorflow` (atual) | ❌ | Sem wheel para macOS Intel |
| `pip install tensorflow-macos` | ❌ | Só funciona em Apple Silicon + Python ≤3.12 |
| Build do TF from source | ⚠️ Teórica | Requer Bazel + compilação C++ de ~6h. Risco de instabilidade. Não recomendado. |
| Usar Python 3.11 + TF 2.15 no Intel Mac | ⚠️ Parcial | Não é suportado pela Google mas pode funcionar via `tensorflow` do conda-forge |
| **Docker Linux x86_64** | ✅ Prático | Rodar container Docker com Python 3.12 + TF 2.15 + CUDA (sem GPU no Mac) |
| **Continuar com KERAS_BACKEND=jax** | ✅ Recomendado | Funciona, CI usa TF, Colab usa TF — consistência suficiente |

**Recomendação concreta:** não tentar instalar TF localmente. A arquitetura
atual (JAX local + TF em Colab/CI) é a solução correta para este ambiente.

O único ajuste técnico necessário é a variável de ambiente:

```bash
# Adicionar em ~/.zshrc ou ~/.zprofile
export KERAS_BACKEND=jax

# Ou criar .env na raiz do projeto (carregado pelo pyproject.toml via python-dotenv)
echo "KERAS_BACKEND=jax" > /Users/daniel/Geosteering_AI/.env
```

E verificar que todos os lugares que fazem `import keras` estão protegidos:

```python
# Em qualquer módulo que usa Keras — adicionar no topo
import os
os.environ.setdefault("KERAS_BACKEND", "jax")  # fallback se não setado no shell
```

### 3.6 Impacto nos Testes Locais

Os 1.670 testes atuais com 0 FAIL indicam que o projeto funciona corretamente
com backend JAX. No entanto, existem cenários em que TF e JAX podem divergir:

1. **Comportamento de `tf.data`:** os 4 flags de `PipelineConfig` introduzidos
   em v2.40 (`tf_shuffle_buffer_size`, etc.) usam `tf.data` diretamente — isso
   só funciona quando TF está disponível. Localmente, os testes que cobrem
   `build_tf_dataset()` provavelmente **skipam ou são ignorados** em ambiente
   sem TF.

2. **Mixed precision:** `tf.keras.mixed_precision` é uma API TF-específica.
   Com backend JAX, `setup_mixed_precision_policy()` pode ter comportamento
   diferente ou não-operacional localmente.

3. **Exportação SavedModel/TFLite:** depende de TF — não funciona localmente.

Estes cenários são aceitáveis porque o código de produção (treinamento,
exportação) roda em Colab/CI com TF disponível.
