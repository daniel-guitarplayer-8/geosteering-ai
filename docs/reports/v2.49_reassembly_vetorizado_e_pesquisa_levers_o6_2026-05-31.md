### Achado honesto (correção do relatório do Sprint C)

O relatório v2.48 atribuiu a morte do run de 30k (a 4:38) ao **`np.stack` O(n_models)
Python**. A medição isolada **refuta** essa atribuição:

- `res.H_tensor` **já é numpy** (convertido em [forward](../../geosteering_ai/simulation/_jax/multi_forward.py#L531) `np.asarray`), então o `np.stack` antigo é **memcpy-bound (~1 s a 30k)**, não Python-bound.
- O **ganho real da vetorização é PICO DE MEMÓRIA (~2×)**: o padrão antigo retinha *views*
  de **todos os K grupos** vivos até o `np.stack` final (pico ≈ 2× a saída = 4995 MB); o
  scatter copia e **libera** cada grupo na iteração seguinte (pico ≈ 1× a saída = 2532 MB).
  É isso que evita o **OOM** para tensores multi-config (nTR·nAng·nf > 1).
- O **custo dominante de ESCALA** é o **nº de GRUPOS** (overhead de launch JAX por grupo),
  confirmado pelo teste K-fixo: com K=16, µs/modelo **cai** 59k→17k→7k conforme n cresce
  (256→1024→4096) — a reassembly **não** é o gargalo; o nº de chamadas JAX é.

> **Conclusão**: o fix é correto e válido (memória −49 %, código mais limpo, fail-fast),
> mas a alavanca de *throughput* a 30k é **poucas geometrias** (templates/quantize), não a
> reassembly. Ambas as recomendações continuam válidas.

## 2. Pesquisa O6+ — levers #3, #4, #5

**Método**: workflow de 3 agentes de pesquisa (leitura estática do kernel) + 3 céticos
adversariais + **microbenchmarks empíricos no A6000** (ground-truth). Os números abaixo
são MEDIDOS, não estimados.

### Lever #3 — Fatorar `exp(-2uh)` recomputado → **REJECT**

| Evidência (medida) | Valor | Conclusão |
|:-------------------|:------|:----------|
| Transcendentais no jaxpr source (1 pos×1 freq) | **52 exp + 6 sqrt = 58** | confirma exatamente o teto O6 |
| **Argumentos exp DISTINTOS** | **52 de 52 (0 duplicados)** | **nada a fatorar** — CSE-removível = 0 |
| `exp(-2uh)`/`exp(-2sh)` por-camada | **1 op `(5,201)` batched** | já computado UMA vez sobre as 5 camadas |
| Demais 50 exp | offsets de profundidade distintos | redundância **APARENTE** (sinais/offsets diferentes) |

**Decisivo**: extraídos os operandos de cada uma das 52 eqns `= exp` do jaxpr — **todos
distintos**. A hipótese de recomputação (mesmo `exp(-2uh)` em `_per_layer` + `common_factors`
+ 12 sites de dipolo) **não se materializa** em ops duplicadas: ou o tracer já deduplicou,
ou os argumentos são genuinamente distintos. Um refactor manual de fatoração reduziria
**0 ops**. Risco de paridade de tentar mesmo assim: **MÉDIO-ALTO** (a única variante que
reduz ops teria de fundir `exp(-2uh)` com `exp(-uh)` via quadrado → perturba o `tghuh`
bit-casado com Fortran `utils.f08:212`). **Veredito: REJECT.**

### Lever #4 — complex64 SELETIVO nos transcendentais → **DEFER**

| Evidência (medida) | Valor | Conclusão |
|:-------------------|:------|:----------|
| `jnp.exp(c128)` vs `jnp.exp(c64)` **isolado** (n=2²²) | **10.97×** (c64 mais rápido) | premissa FP64-SFU **CONFIRMADA** |
| idem n=2¹⁸ / 2²⁰ | 2.26× / 9.00× | speedup cresce com tamanho |
| Kernel completo c64 vs c128 (medido v2.43) | **~0×** (igual) | exp **não satura** no kernel (bucketed = tiles pequenos, scan-bound) |
| Paridade c64 full (v2.43, shape 600×9) | `max_rel = 3.88e-4` | **~7 ordens acima** de `<1e-13` |

**Reconciliação**: o exp c64 é **11× mais rápido em isolamento** (SFU FP64 é lenta), mas no
**regime de produção (bucketed)** o kernel é limitado pelas **recursões seriais `lax.scan`
e latência de tiles pequenos**, não pelo throughput de exp — por isso o full-kernel mede
c64≈c128. O ganho isolado **não aflora** end-to-end. Além disso é **paridade-fatal**
(disqualificado como default por construção). **Veredito: DEFER** — só faz sentido como
**modo inexato opt-in/gated** (PINN `<1e-4`) e **apenas** se um regime saturado (batch
grande) for necessário — que hoje é OOM-proibido em high-config.

### Lever #5 — Tensor-core / TF32 nas reduções Hankel → **REJECT**

| Evidência (medida) | Valor | Conclusão |
|:-------------------|:------|:----------|
| TF32 matmul vs FP64 `jnp.sum` (npt=201, 4096 lin) | **1.36×** | redução pequena, latency-bound |
| Erro TF32 — vetor sintético (dinâmica 10¹⁰) | `max 1.07e-5` | PINN-grade na melhor hipótese |
| Erro TF32 — **filtro Werthmuller-201 REAL** (κ 400–800) | **37–78 %** | **catastrófico** (cancelamento alternado) |
| Amdahl (reduções ~1.5 % do FLOP) | teto **1.05×** | irrelevante mesmo se instantâneo |
| complex128 em tensor-core | **não suportado** | downcast obrigatório → paridade-fatal |

**Decisivo**: tensor-core exige downcast (TF32 ~10 bits) — incompatível com `<1e-13`. No
filtro **real** (pesos alternados, 9.2 ordens de dinâmica), o cancelamento leva a **37–78 %
de erro** (o vetor sintético subestimava). Speedup marginal (1.36×) sobre ~1.5 % do FLOP →
**ganho líquido ~0 a negativo**. **Veredito: REJECT.**
