## 2. Bugs Detectados e Corrigidos

### Bug B1 — `model_name` vs `canonical_model_name` no teste de paridade

**Origem**: commit `5f6a96a` (Sprint O0 pre-flight). A assinatura de
`compare_fortran_python()` foi renomeada de `model_name` para
`canonical_model_name`, mas `tests/test_simulation_jax_fortran_parity.py`
não foi atualizado e 4 testes ficaram bloqueados via
`pytest.mark.skip(reason="Bug B1...")`.

**Fix**: substituído `model_name=` por `canonical_model_name=`. Também
ajustado uso da API — `compare_fortran_python` retorna `list[FortranComparisonResult]`,
não dict; testes usavam `result.get(...)` (interface dict inexistente).

### Bug B2 — `simulate_multi` não roteava `backend='jax'`

**Origem**: arquitetural. `compare_fortran_python` chamava `simulate()` que
delegava a `simulate_multi()`, mas este rejeitava `backend='jax'` com
`NotImplementedError`. Resultado: todas as chamadas com backends
`jax_hybrid` ou `jax_native` falhavam silenciosamente com `notes=...`.

**Fix**: roteamento explícito em `compare_fortran.py` — para
backends JAX, chama `simulate_multi_jax()` (que vive em `_jax/multi_forward.py`)
e desembrulha a saída `(nTR=1, nAng=1, n_pos, nf, 9)` → `(n_pos, nf, 9)` para
comparação com Fortran. O caminho Numba permanece inalterado.

### Bug B3 — Range dip JAX (`[0,89]`) ≠ Numba (`[0,90]`)

**Estado**: documentado, não corrigido. JAX `simulate_multi_jax` rejeita
`dip=90°` mas Numba aceita. No bench Cenário H, ajustamos dip=89° para
manter comparabilidade.

## 6. Análise: por que JAX GPU perde para Numba CPU?

### 6.1 Diagnóstico empírico (Cenário E, single-call breakdown)

Medições standalone para 1 modelo, n_pos=600, com warmup quente:

| Etapa | Tempo |
|:------|------:|
| `build_static_context()` (NumPy puro, prepara `camad_t_array` etc.) | ~60 ms |
| `forward_pure_jax()` (XLA execution) | ~100 ms |
| **Total per_model** | **~160 ms** |

Numba completa o mesmo trabalho em ~9 ms — uma vantagem de **18×** no tempo
de wall-clock por chamada.

### 6.2 Hipóteses (não-exclusivas)

| # | Hipótese | Suporte empírico |
|:-:|:---------|:----------------|
| 1 | **Workload CPU-friendly**: ~5 camadas × 600 posições × 1 freq = trabalho pequeno por elemento; GPU não satura | A6000 tem 10 752 CUDA cores; 600 posições com complex128 não preenchem warps eficientemente |
| 2 | **Per-call XLA dispatch overhead** (~1-2 ms por launch) | Cenário A (n_pos=1): JAX bottoms out em ~2 ms/call independente do strategy |
| 3 | **complex128 não é GPU's sweet spot** | A6000 nominal: FP64 = 1.2 TFLOPS vs FP32 = 38.7 TFLOPS (32× pior) |
| 4 | **Python loop em `simulate_multi_jax`** sobre (iTR, iAng) | vmap_real elimina o loop mas só ganha 1.4× em H |
| 5 | **`build_static_context` é NumPy puro** chamado por modelo | Ainda 60 ms cada (poderia ser jit-cached ou batched) |
| 6 | **`simulate_multi_jax_batched` (vmap modelos)** explode VRAM em H | 97 GB pedidos para `(10, 8, 8, 100, 8, 9)` complex128 intermediate |

### 6.3 Onde JAX GPU venceria

| Caso de uso | Por quê |
|:-----------|:--------|
| **Differentiable forward** (`jax.grad`, `jacfwd`) | Numba não tem autodiff — JAX é único nesta capacidade |
| **Batches **enormes** (10⁴+ modelos) com tensores de input grandes** | vmap fusiona em 1 XLA program; amortiza overhead |
| **Pipeline DL** (treino on-the-fly com physics-loss) | Integra natural com TF/Keras stack |
| **Look-ahead inversion** com Gauss-Newton via jacfwd | Jacobiano analítico GPU-acelerado |
| **Posições muito altas (10⁴+ por modelo)** | GPU saturação reduz overhead relativo |
| **complex64 / float32** (precisão reduzida) | 32× mais throughput FP32 na A6000 |

### 6.4 Onde Numba CPU vence (este projeto, hoje)

- Geração de datasets de treinamento (30k+ modelos sequenciais) — Numba 4w×16t bate todos os outros backends.
- Inversão LWD tempo-real (latência <10 ms por sample) — Cenário E Numba 9 ms vs JAX 100 ms.
- Stress multi-configuração (8³ scenarios) — Numba 280 ms vs JAX 2 s.
- Paridade científica é Fortran-bit-exata sem custo de software extra.

## 7. Recomendações

### 7.1 Produção (atual)

- **Manter Numba (4w × 16t) como default em produção** para geração de
  datasets, inversão real-time e stress-test multi-combo.
- **Validade científica preservada** — JAX é apenas mais lento, não menos
  preciso (paridade <1.53e-13 vs Fortran nos 7 modelos canônicos).

### 7.2 JAX como path de pesquisa (mantido)

- Manter `_jax/` como path opt-in para uso em DL/PINN onde diferenciabilidade
  é mandatória (`jax.grad`, `jacfwd`).
- `simulate_multi_jax` com `strategy="unified"` + `vmap_real=True` é o
  melhor setup hoje para multi-combo.

### 7.3 Backlog técnico para fechar gap JAX (não-crítico)

1. **`build_static_context` jit-batched** — vetorizar sobre modelos (NumPy
   puro hoje custa ~60 ms/modelo).
2. **complex64 path** — adicionar config opt-in `dtype="complex64"` em
   `simulate_multi_jax`; espera ganho 2-4× em A6000.
3. **simulate_multi_jax_batched** com `chunk_size` para evitar OOM em H
   (paralelizar nTR×nAng em micro-batches).
4. **Wire JAX em `simulate_multi`** — unificar API (atualmente requer rotear
   pela validation module). Backlog F2.
5. **Alinhar range dip JAX [0,89]→[0,90]** (B3) — pequena correção
   `range validation` em `_jax/multi_forward.py`.

## 10. Conclusões

1. **Paridade científica JAX GPU ↔ Numba CPU ↔ Fortran tatu.x: confirmada**.
   Erro máximo 1.53e-13 — 3 ordens de magnitude abaixo do gate de produção
   (1e-10). Todos os 7 modelos canônicos (incluindo oklahoma_28 com ρv=2ρh e
   alta resistividade até 200 Ω·m) passam.

2. **Fidelidade física e geofísica: confirmada**. 65/65 testes analíticos
   cobrem decoupling factors (ACp, ACx), skin depth, wavenumber TIV, VMD
   axial/broadside, HMD TIV, estabilidade até ρ=10⁶ Ω·m, reciprocidade
   T↔R, e simetria Maxwell sob rotação Euler.

3. **Performance: Numba CPU domina** neste workload de geosteering AI.
   JAX GPU é 3.7×–9.8× mais lento que Numba 4w×16t em A/E/H. A razão é
   arquitetural: o workload tem poucos elementos por chamada (5-28 camadas
   × ≤600 posições × complex128) e não satura a A6000.

4. **JAX continua sendo valioso** em capacidades únicas — diferenciabilidade
   (`jax.grad`, `jacfwd` para Jacobianos analíticos e PINN), integração com
   stack DL (Keras/TF), e batches massivos futuros (com `chunk_size` para
   evitar OOM).

5. **Sem ações corretivas físicas necessárias** — paridade está garantida.
   As correções aplicadas (B1, B2) eram de **harness de teste**, não de
   física do simulador.
