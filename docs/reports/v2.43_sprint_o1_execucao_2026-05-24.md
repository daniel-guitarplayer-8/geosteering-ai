## 2. Otimizações Aplicadas

### #1 (CRÍTICA) — Vetorização do loop Python `rho_h_at_obs`

**Localização**: `multi_forward.py:641-650 (single bucketed)`, `854-866 (vmap_real)`, `1192-1221 (batched API)`

**Problema**: o loop Python `for i_model × for i_ang × for i_pos: layer_at_depth(...)` executava na CPU **depois** que a GPU sincronizava — 60.000 chamadas Numba em Cenário E, desperdiçando 15-60 ms enquanto a GPU permanecia ociosa.

**Solução**: novo helper `_layer_at_depth_vec(n, z, prof_mid)` em `multi_forward.py:318-349` usando `np.searchsorted(prof_mid[1:n], z, side="right")` — bit-exato com a convenção `layer_at_depth` Numba (RX-inclusive, `>=`).

**Ganho estimado**: +40-60% no tempo total dos cenários B e E (gargalo dominante eliminado).

**Validação**: T1.2 (paridade JAX batched vs Numba direta) PASSOU em oklahoma_3 e oklahoma_5.

### #2 — `lax.scan(unroll=K)` substitui `fori_loop` em propagação/dipoles

**Localização**: `dipoles_unified.py:588, 629, 938, 970` (4 sites HMD/VMD descent + ascent) + `propagation.py:221, 279` (2 sites bottom-up/top-down TE/TM)

**Problema**: `jax.lax.fori_loop` sobre n_cam emitia 1 kernel CUDA por iteração (overhead 5-10 µs por launch). Para oklahoma_5 (n=5 camadas, 4 iterações): ~28 kernel launches por modelo×posição×frequência. Para oklahoma_28: ~108 launches.

**Solução**: `lax.scan(length=n, mask=(j>=ct)&(j<=cr), unroll=K)` funde iterações em 1 kernel CUDA único quando `K=n_cam`. Constante `_MAX_LAYER_UNROLL = 8` evita stack overflow XLA em n_cam > 8 (oklahoma_28 usa unroll parcial K=8).

**Flag opt-out**: `SimulationConfig.unroll_layer_loop: bool = True` (default ON).

**Ganho estimado**: oklahoma_5 → 7× redução em kernel launches (28 → 4); oklahoma_28 → 14× redução. Esperado +50% a +200% no hot path para n_cam ∈ [5, 8] em GPU.

**Risco residual**: a transformação `fori_loop(start, end, ...)` → `scan(length=n, mask=(j≥start)&(j≤end))` muda iterações condicionais por máscara. **Matematicamente equivalente**, mas precisa de validação rigorosa em GPU real (testes JAX GPU ficam skipped em CPU dev).

### #3 — Persistent XLA Compilation Cache

**Localização**: `_jax/__init__.py` (função `_setup_xla_environment()`)

**Problema**: cada sessão Colab/notebook/pytest recompilava todos os JITs do zero (cold-start 30-180s).

**Solução**: configura `JAX_COMPILATION_CACHE_DIR` antes do primeiro `import jax`. Cache em `$TMPDIR/jax_compilation_cache_geosteering`. Aplicado em 2 camadas: env var (lida pelo jaxlib internamente) + `jax.config.update("jax_compilation_cache_dir", ...)` após `import jax` (gracioso em JAX < 0.4.25 via try/except).

**Backward compatibility**: opt-out global via `GEOSTEERING_JAX_NO_XLA_SETUP=1`. Cache dir criado on-demand (OSError → log warning + segue sem cache).

**Ganho estimado**: −95% do cold-start (30-180s → ~3s) em sessões repetidas. Não impacta steady-state benchmarks.

### #4 — XLA_FLAGS otimizadas para A100

**Localização**: mesma função `_setup_xla_environment()` em `_jax/__init__.py`

**Flags aplicadas** (idempotente — preserva XLA_FLAGS pré-existentes):

```
--xla_gpu_enable_latency_hiding_scheduler=true
--xla_gpu_enable_triton_softmax_fusion=true
```

**VRAM allocation** (`setdefault` — preserva override):
- `XLA_PYTHON_CLIENT_PREALLOCATE=false` (evita alocação eager de 75% VRAM via BFC allocator)
- `XLA_PYTHON_CLIENT_MEM_FRACTION=0.85` (deixa 15% para CUDA runtime + cuDNN handles)

**Ganho estimado**: +5-15% steady-state (latency_hiding_scheduler sobrepõe compute/copy). VRAM disponível para batches ~30% maior em F/G/H.

### #5 (parcial) — `donate_argnums` — SKIP INTENCIONAL

**Decisão**: NÃO aplicar `donate_argnums` em `_forward_unified`, `_forward_unified_chunked`, `_forward_bucket` (Agente C) nem em hot JITs de `multi_forward.py` (Agente A).

**Justificativa documentada**:
- `rho_h`/`rho_v` vêm de `ctx` e são reusados em múltiplas chamadas (warmup → simulate, cenários A-K iterativos)
- Quebra `jax.jacfwd` / `jax.grad` que reusa entradas para o Jacobiano (PINN)
- O hot path de `multi_forward.py` não tem `@jax.jit` direto — tudo passa por `_get_unified_jit` em `forward_pure.py`

**Recomendação para Sprint futuro**: auditar `multi_forward.py` e construir wrappers donate-safe APENAS em fluxos batched onde o caller já clonou os arrays. Não é Quick Win — requer análise cuidadosa.

### #6 — FilterLoader cached process-wide

**Localização**: `_jax/kernel.py:158-255` (helper + wrapper público); wired em `_jax/forward_pure.py:274` e `_jax/multi_forward.py:751,1083`

**Problema**: cada call ao kernel JAX recarregava o filtro Hankel (`Werthmüller 201pt` por default), executando `np.asarray + jnp.asarray` desnecessariamente — 1-5 ms por chamada em I/O + 1 HostToDevice copy extra.

**Solução**: novo helper `get_hankel_filter_jax(filter_name)` com `@functools.lru_cache(maxsize=8)` retorna `(krJ0J1, wJ0, wJ1)` como `jax.Array` já no device. Normaliza aliases via `FilterLoader.resolve_name` antes de chavear o LRU (evita entradas duplicadas para `'wer'`/`'werthmuller_201pt'`/`'0'`).

**Cache invalidation**: chaveado por nome canônico. 3 filtros catalogados (Werthmüller 201pt, Kong 61pt, Anderson 801pt) compartilham `maxsize=8`. `cache_clear()` exposto para uso futuro.

**Thread-safety**: `functools.lru_cache` atômico sob GIL + `FilterLoader` interno com double-checked locking. Workers spawn: cada processo tem seu próprio LRU (correto — handles JAX não são shared-memory).

**Ganho estimado**: +1-5 ms/call eliminados em hot paths + 1 HtoD por call eliminado.

### #7 — Remove `block_until_ready()` redundante

**Localização**: `multi_forward.py:850` (vmap_real) + `multi_forward.py:1191` (batched API)

**Problema**: `result.block_until_ready()` seguido de `np.asarray(result)` — o segundo já sincroniza implicitamente, então o primeiro era 1 RTT GPU/CPU desperdiçado.

**Solução**: remove `block_until_ready()`; sync deferido ao caller `np.asarray()`.

**Ganho estimado**: −0.3-0.5 ms por call em T4 (mais em A100 com PCIe Gen4).

### #8 — Agrupa `jnp.asarray` em `jax.tree.map`

**Localização**: `multi_forward.py:764-790` (vmap_real, 6 transfers) + `multi_forward.py:1093-1118` (batched API, 6 transfers)

**Problema**: até 9 chamadas `jnp.asarray` separadas para arrays de input (cada uma um HostToDevice copy individual com overhead ~3 µs).

**Solução**: agrupa em 1 único `jax.tree.map(lambda x: jnp.asarray(x, dtype=jnp.float64), (...))`. Filtros Hankel (krJ0J1/wJ0/wJ1) agora já vêm como `jax.Array` do cache LRU (#6), reduzindo de 9 → 6 transfers.

**Ganho estimado**: redução de 27 µs/call para 18 µs/call + menor overhead Python entre transfers.

### #9 — Vetoriza `_sanitize_profile_batch`

**Localização**: `multi_forward.py:355-421`

**Problema**: loop Python `for i_model: _sanitize_profile_kernel(esp[i_model])` — overhead Numba per-model em chamadas batched (5-25 ms para n_models ≥ 50).

**Solução**: substitui por `np.cumsum(h_arr_batch, axis=1, out=prof_arr_batch[:, 1:])` único + tratamento explícito de n=1 (semi-espaço único).

**Ganho estimado**: −5 a −25 ms por call batched para n_models ≥ 50.

## 3. Bug Crítico Encontrado e Corrigido

### Regressão n=2 (dois semi-espaços, esp shape (n_models, 0))

**Sintoma**: teste `test_t5_edge_n_2_dois_semi_espacos` falhou na primeira execução do pytest pós-O1:

```
ValueError: non-broadcastable output operand with shape (2,1) [remapped to (1,2)]
            doesn't match the broadcast shape (2,0)
```

**Causa-raiz**: Agente A alocou `prof_mid_batch` shape `(n_models, n)` na versão batched, mas a convenção bit-exata do single-model path (`_simulate_multi_jax_vmap_real:852`) é shape `(n-1,)` (i.e., `(n_models, n-1)` no batched). Para n=2, isso significa `(n_models, 1)` — incompatível com `np.cumsum(esp shape (n_models, 0), out=prof[:, 1:1])`.

**Fix aplicado** (`multi_forward.py:1198-1209`):

```python
if n >= 2:
    prof_mid_batch = np.empty((n_models, n - 1), dtype=np.float64)
    prof_mid_batch[:, 0] = 0.0
    if n > 2:  # cumsum só faz sentido quando esp tem ≥1 coluna (n-2 ≥ 1)
        np.cumsum(esp_batch_np, axis=1, out=prof_mid_batch[:, 1:])
else:
    prof_mid_batch = np.zeros((n_models, 1), dtype=np.float64)
```

Agora bit-exato com `_simulate_multi_jax_vmap_real:852-854` para todos os n.

**Validação pós-fix**: `test_simulation_jax_batched_api.py` → 26 PASS / 1 SKIP.
