### 3.1 Topologia do paralelismo — antes vs depois

```
┌────────────────────────────────────────────────────────────────────┐
│  v2.21 (Sprint 13.3 + 21.1 fix de KB-013):                         │
│    prange(nTR × nAng × n_pos)                                       │
│       └── range(nf) SERIAL dentro de _fields_in_freqs_kernel_cached │
│                                                                     │
│    Pontos negativos:                                                │
│    • nf não paralelizado ⇒ ganho desperdiçado em multi-freq         │

└────────────────────────────────────────────────────────────────────┘
                              ↓ Sprint v2.22
┌────────────────────────────────────────────────────────────────────┐
│  v2.22 (FLAT prange):                                              │
│    prange(nTR × nAng × n_pos × nf)                                 │
│       └── _fields_at_single_freq (1 freq por tarefa)               │
│                                                                     │
│    Vantagens:                                                       │
│    • Paralelismo natural sobre 4 dimensões                          │
│    • Cada tarefa paralela faz 1 cálculo Hankel (sem loop interno)   │
```

#### 3.2.1 `geosteering_ai/simulation/_numba/kernel.py` (+179 linhas)

Nova função `_fields_at_single_freq` extraída do corpo do `for i_f in range(nf)` em `_fields_in_freqs_kernel_cached`:

```python
@njit(cache=True, nogil=True)
def _fields_at_single_freq(
    Tx, Ty, Tz, cx, cy, cz, dip_rad,
    n, h_arr, prof_arr, eta, freq_hz,
    krJ0J1, wJ0, wJ1,
    u, s, uh, sh, RTEdw, RTEup, RTMdw, RTMup, AdmInt,  # slices 2D [npt, n]
    camad_t, camad_r,                                  # pré-calculados
) -> np.ndarray:  # (9,) complex128
    omega = 2.0 * math.pi * freq_hz
    zeta = 1j * omega * _MU_0
    Mxdw, ..., FEupz = common_factors(...)
    Hx_hmd, Hy_hmd, Hz_hmd = hmd_tiv(...)
    Hx_vmd, Hy_vmd, Hz_vmd = vmd(...)
    matH = np.empty((3, 3), dtype=np.complex128)
    # ... (montagem matH 3×3 idêntica a kernel.py:840-861)
    tH = rotate_tensor(dip_rad, 0.0, 0.0, matH)
    out = np.empty(9, dtype=np.complex128)
    # ... (flatten tH em (9,))
    return out
```

**Decorador**: `@njit(cache=True, nogil=True)` — sem `parallel=True` (função folha chamada de prange outer; evita o anti-pattern KB-013).

**Garantias**: bit-exato com o corpo do loop em `_fields_in_freqs_kernel_cached` (validado por 27/27 testes).

#### 3.2.2 `geosteering_ai/simulation/forward.py` (+179 linhas)

Nova função `_simulate_combined_prange_flat`:

```python
@njit(parallel=True, cache=True, nogil=True)
def _simulate_combined_prange_flat(
    dz_halfs, r_halfs, dip_rads_flat, cache_indices, nAngles,
    u_unique, s_unique, ..., AdmInt_unique,
    positions_z, n, rho_h, rho_v, h_arr, prof_arr, eta,
    freqs_hz, krJ0J1, wJ0, wJ1,
    H_tensor,  # [nTR, nAngles, n_pos, nf, 9] pré-alocado
) -> None:
    n_combos = dz_halfs.shape[0]
    n_pos = positions_z.shape[0]
    nf = freqs_hz.shape[0]
    n_combo_pos_f = n_pos * nf
    n_total = n_combos * n_combo_pos_f

    for k in _prange(n_total):
        # Decomposição flat 4D: k → (i_combo, j, i_f) → (i_tr, i_ang, j, i_f)
        i_combo = k // n_combo_pos_f
        rem = k % n_combo_pos_f
        j = rem // nf
        i_f = rem % nf
        i_tr = i_combo // nAngles
        i_ang = i_combo % nAngles

        # Geometria do combo, coordenadas T/R, find_layers_tr...
        # ...

        # Computa tensor para frequência i_f apenas (slices 2D = views)
        out_9 = _fields_at_single_freq(
            ..., u_unique[ci, i_f], s_unique[ci, i_f], ...,
            camad_t, camad_r,
        )

        # Write exclusivo por (i_tr, i_ang, j, i_f) — sem race
        for c in range(9):
            H_tensor[i_tr, i_ang, j, i_f, c] = out_9[c]
```

### 3.3 Decomposição do índice flat (correção matemática)

```
n_combos       = nTR × nAngles
n_combo_pos_f  = n_pos × nf
n_total        = n_combos × n_combo_pos_f
               = nTR × nAngles × n_pos × nf

Para cada k ∈ [0, n_total):
    i_combo = k // n_combo_pos_f       ∈ [0, n_combos)
    rem     = k %  n_combo_pos_f       ∈ [0, n_combo_pos_f)
    j       = rem // nf                ∈ [0, n_pos)
    i_f     = rem %  nf                ∈ [0, nf)
    i_tr    = i_combo // nAngles       ∈ [0, nTR)
    i_ang   = i_combo %  nAngles       ∈ [0, nAngles)
```

## 5. Performance — Benchmarks Cenário E/B/F

### 5.1 Resultados (single-process, M-series 8C/16T, 3 runs medianos)

| Cenário | Descrição | Legacy v2.21 | FLAT v2.22 | Speedup |
|:-------:|:----------|:------------:|:----------:|:-------:|
| **E** | n_pos=600, nf=1, 1TR, 1ang | 214 511 mod/h | 212 132 mod/h | 0.99× |
| **B** | n_pos=200, nf=1, 3TR×4ang | 48 417 mod/h | 53 830 mod/h | **1.11×** |
| **F** | n_pos=600, nf=4, 1TR, 1ang | 52 532 mod/h | 57 245 mod/h | **1.09×** |

### 5.2 Análise dos resultados

**1. Cenário E sem regressão** — speedup de 0.99× está dentro do ruído de medição (stdev ±5%). Quando `nf=1`, FLAT degenera matematicamente para `prange(n_combos × n_pos × 1)` ≡ `prange(n_combos × n_pos)`, idêntico ao legacy.

**2. Cenários B e F com ganho modesto (~10%)** — FLAT amortiza o overhead de fork/join do scheduler Numba que o legacy paga implicitamente em cada chamada de `_fields_in_freqs_kernel_cached` (que aninhava prange em prange). O ganho relativo é menor que a estimativa otimista (2-3×) do documento de análise porque:
- 80-87% do custo total do kernel está em `hmd_tiv` + `vmd` (Hankel 201pt × n_layers operações)
- Estes kernels já são internamente eficientes (LLVM SIMD)
- FLAT só remove overhead do scheduler, que é ~5-15% do total

**3. Baseline histórico não-direta** — os números 303k (Cenário B) e 600k+ (meta) do documento de análise se baseiam em **execução em batch via `ProcessPoolExecutor` 4 workers × 2 threads** (~8× throughput vs single-process). Este benchmark é single-process; a comparação relativa FLAT vs legacy é a métrica significativa.

### 5.3 Benefício arquitetural (não-quantitativo, mais importante)

A Sprint v2.22 estabelece o **padrão FLAT** como base para as próximas otimizações:

```
Sprint v2.23 (fastmath em hmd_tiv/vmd):
  Ganho esperado +5-10% — melhor mensurável quando 100% das tarefas
  estão paralelizadas (FLAT habilita).

Sprint v2.24 (Hankel pré-cômputo avançado):
  Ganho esperado +10-15% — depende de FLAT para que o cache Hankel
  seja indexado por (ci, i_f) consistentemente.

Sprint v2.25 (cache de contexto inversão iterativa):
  Ganho esperado +30% em geosteering tempo-real — depende de FLAT
  para invalidar/restaurar cache por frequência.
```
