## 2. Sprint 13.1 — Vetorização de Frequências

### 2.1 Arquivos modificados

`geosteering_ai/simulation/_numba/kernel.py`:

```diff
+ from geosteering_ai.simulation._numba.propagation import HAS_NUMBA, ...
+
+ if HAS_NUMBA:
+     from numba import prange as _prange
+ else:
+     _prange = range

- @njit(cache=True)
+ @njit(cache=True, parallel=True, nogil=True)
  def precompute_common_arrays_cache(...):
      ...
-     for i_f in range(nf):
+     for i_f in _prange(nf):
          ... common_arrays(...) ...

- @njit(cache=True)
+ @njit(cache=True, parallel=True, nogil=True)
  def _fields_in_freqs_kernel_cached(...):
      ...
-     for i_f in range(nf):
+     for i_f in _prange(nf):
          ... hmd_tiv + vmd ...
```

### 2.2 Análise de race condition

Cada iteração `i_f` escreve em índice exclusivo:
- `precompute_common_arrays_cache`: `u_cache[i_f] = u`, etc. → **safe**
- `_fields_in_freqs_kernel_cached`: `cH[i_f, k] = tH[...]` → **safe**

`matH` é `np.empty((3, 3))` local por iteração → Numba aloca per-thread no
stack. **No race condition.**

### 2.3 Comportamento sob nested prange

Quando `_fields_in_freqs_kernel_cached` é chamado de
`_simulate_positions_njit_cached` (que já tem `prange(n_pos)`), o `prange(nf)`
interno é **automaticamente serializado** pelo Numba. Sem regressão para o
caso production (n_pos=600 satura cores).

Para PINN (n_pos=1-10, nf=10), o prange(nf) ativa quando chamado fora do
contexto paralelo externo.

## 3. Sprint 13.2 — Cache Cross-Call

### 3.1 Estrutura do cache global

Em `multi_forward.py`:

```python
_GLOBAL_HORDIST_CACHE: Dict[Tuple[Any, ...], Tuple[np.ndarray, ...]] = {}
_CACHE_LOCK = threading.Lock()


def _build_cache_key(hordist, freqs_hz, n, h_arr, eta) -> Tuple:
    """Chave única e bit-exata para o cache."""
    return (
        round(float(hordist), 12),                       # hordist
        tuple(int(round(float(f))) for f in freqs_hz),    # freqs_signature
        int(n),                                          # n_layers
        np.ascontiguousarray(eta, dtype=np.complex128).tobytes(),    # eta
        np.ascontiguousarray(h_arr, dtype=np.float64).tobytes(),     # h_arr
    )


def release_numba_cache() -> int:
    """Libera cache, retorna count."""
    with _CACHE_LOCK:
        n = len(_GLOBAL_HORDIST_CACHE)
        _GLOBAL_HORDIST_CACHE.clear()
    return n
```

### 3.2 Fluxo no dispatcher `_build_unique_hordist_caches`

Refatoração:

```python
for L in tr_spacings_m:
    for dip_rad in dip_rads:
        hordist = float(L) * abs(math.sin(float(dip_rad)))
        key = round(hordist, _HORDIST_KEY_DECIMALS)
        if key in caches:
            continue  # dedup local clássico

        # NOVO v2.13: tenta cache global persistente
        if cache_persistent:
            global_key = _build_cache_key(hordist, freqs_hz, n, h_arr, eta)
            with _CACHE_LOCK:
                cached = _GLOBAL_HORDIST_CACHE.get(global_key)
            if cached is not None:
                caches[key] = cached
                continue  # global hit — reusa sem recomputar

        cache_tuple = precompute_common_arrays_cache(...)
        caches[key] = cache_tuple

        if cache_persistent:
            with _CACHE_LOCK:
                _GLOBAL_HORDIST_CACHE[global_key] = cache_tuple
```

### 3.3 Backward-compat preservado

- Default `cache_persistent=False` → comportamento idêntico v2.12 (sem cache global)
- `cache_persistent=True` opt-in → nova funcionalidade
- Cache local-por-chamada (dedup hordist) preservado em ambos os caminhos

### 3.4 UI integration

`closeEvent` em `simulation_manager.py:7284` agora chama 3 releases:

```python
release_numba_pool()        # v2.10 — pool da UI
release_pool()              # v2.12 — pool nativo do core
release_numba_cache()       # v2.13 — cache global de common_arrays (NOVO)
```

## 4. Sprint 13.4 — `nogil=True` Universal

### 4.1 Wrapper modificado

`_numba/propagation.py:172`:

```python
def njit(*args, **kwargs):
    if len(args) == 1 and callable(args[0]):
        return _numba_njit(
            cache=True,
            fastmath=False,
            error_model="numpy",
            nogil=True,           # ← NOVO v2.13
        )(args[0])
    kwargs.setdefault("cache", True)
    kwargs.setdefault("fastmath", False)
    kwargs.setdefault("error_model", "numpy")
    kwargs.setdefault("nogil", True)  # ← NOVO v2.13
    return _numba_njit(*args, **kwargs)
```

### 4.2 Impacto

- **Custo zero em performance** (Numba já libera GIL automaticamente em código nativo;
  `nogil=True` documenta intenção e habilita otimizações adicionais)
- **Benefício direto** em uso multi-thread:
  - Notebooks com `concurrent.futures.ThreadPoolExecutor`
  - UI Qt com workers Python orchestrando chamadas
  - Treino offline com data pipelines paralelos

### 4.3 fastmath=True deferido

`fastmath=True` pode causar diferenças <1e-15 em operações sensíveis
(recursões TE/TM, common_arrays). Deferido para v2.14 com aplicação
**seletiva** apenas em:
- Hankel quadratura (sums escalares — FMA seguro)
- Decoupling factors (multiplicações complexas)

Após validação caso-a-caso de paridade Fortran <1e-12 em 4 modelos canônicos.

## 7. Pendências e Riscos

### 7.1 Pendências v2.14

| Item | Esforço | Crítico? |
|:-----|:-------:|:--------:|
| Sprint 13.3 (prange TR×ang) | 1.5 dias | Médio (ganho 30% multi-TR) |
| Sprint 13.4 fastmath seletivo | 0.5 dias | Baixo (ganho 5-10% Hankel) |
| Benchmark formal 4 cenários | 0.5 dias | Médio (validação numérica) |
| Smoke test 30k em hardware real | 0.25 dias | Alto (validação UX) |

### 7.2 Riscos identificados

| Risco | Prob. | Impacto | Mitigação aplicada |
|:------|:-----:|:-------:|:-------------------|
| `nogil=True` invalida JIT cache | Alta | Baixo | Recompila na 1ª chamada (esperado) |
| `cache_persistent=True` causa memory leak | Baixa | Médio | Default OFF + `release_numba_cache()` exposto |
| Cache global colide entre perfis | Baixa | Alto | Key inclui `eta_bytes` + `h_bytes` (bit-exato) |
| Nested prange degrada perf | Baixa | Médio | Numba serializa nível interno (documentado) |

### 7.3 Limites conhecidos

- **Cache global cresce indefinidamente**: usuário deve chamar
  `release_numba_cache()` manualmente. UI já chama no `closeEvent`.
- **Sprint 13.3 não entregue**: TR×ângulo continuam sendo Python serial.
  Para batches com nTR×nAngles altos (>20 combinações), o overhead Python
  ainda domina ~5-10% do tempo total.
