## §2 Hot Path Quantificado

### Hot path: `_numba/dipoles.py::hmd_tiv` linhas 325–420

```
Por posição × por frequência × por TR × por ângulo:
  ┌──────────────────────────────────────────────────────────────┐
  │  hmd_tiv (recursão TE/TM, geometric case camad_r ≷ camad_t) │
  │    Loop range(camad_t, camad_r+1) ~ n_layers iterações      │
  │      cada: ~8 ops complexos (exp, mult, div) × npt          │
  │    = ~n_layers × 8 × npt ≈ 8·22·201 ≈ 35k ops/dipolo        │
  │                                                              │
  │  vmd (recursão TE only)                                     │
  │    similar mas TE-only ≈ ~25k ops/dipolo                    │
  │                                                              │
  │  Kernel assembly (linhas 654–695)                           │
  │    Σ Ktm × wJ0, Σ Ktedz × wJ1, etc.                         │
  │    ~6 reduções × 201 elementos ≈ ~1200 ops                  │
  │                                                              │
  │  TOTAL ~62k ops/posição                                     │
  └──────────────────────────────────────────────────────────────┘

Para 600 pos × 22 layers × 201 pts × 1 freq × 1 TR × 1 ang × 1 modelo:
  600 × 62k = 37M ops/modelo (lower-bound, sem cache misses)
```

### Verificação empírica (Cenário E pós-fix v2.16)

```
Cenário E (Sprint 15.3 v2.16, hardware 8C HT):
  600 modelos × 600 pos × 1 freq × 1 TR × 1 ang × 1 modelo
  = 19.40 s (4 workers × 2 threads = 8 threads ativas)
  = 32.3 ms/modelo
  = ~30s @ 1 modelo single-threaded (extrapolando 8× speedup)

Razão observada vs predição:
  37M ops × ~5 cycles/op (cache misses + branches) = ~185M cycles
  / 2.4 GHz = 77 ms predito (single-thread)
  vs. 32.3 ms × 8 threads = 258 ms total — agrega 3.4× over-prediction
  (consistente: predição era lower-bound)
```

## §3 Complexidade Total

```
T(modelo) = O(n_freqs × n_TR × n_ang × n_pos × n_layers × n_filter_pts)
                        │      │       │       │           │
                        │      │       │       │           201 (Werthmüller)
                        │      │       │       └─ 22 (Oklahoma_28)
                        │      │       └─ 600 (production LWD)
                        │      └─ 5 (multi-ângulo)
                        └─ 3 (multi-TR)
            10× para multi-freq

Casos práticos:
  Single-freq (PINN inversão online):
    1 × 1 × 1 × 600 × 22 × 201 = 2.65M elementos · ~3 ops = 8M flops/mod
  Multi-freq (TIV inversion 10 freqs):
    10× = 80M flops/mod
  Multi-TR×ang (deep-imaging tool):
    3 × 5 × 600 × 22 × 201 × 10 freqs = 1.2G flops/mod (~600× mais)
```

**Linearidade `n_pos`:** confirmada empiricamente — Cenário A (30 pts) atinge
**1.74M mod/h**, Cenário E (600 pts) atinge **123k mod/h** com 2000 modelos.
Razão exata: 1740/123 ≈ **14.1×** (subllinear porque overhead de spawn/JIT é
amortizado em ambos os casos).

**Observação crítica:** No Cenário E (600 pts × 1 freq × 1 TR × 1 ang),
o caminho ativo é `_simulate_positions_njit_cached::prange(n_pos)`. Com
8 threads ativas, 600 pos / 8 = 75 pos/thread. Cada thread processa ~75
posições serialmente. Sem locks ou atomics — paralelismo ideal.

## §5 Top 3 Oportunidades de Otimização

### Oportunidade 1 — Tile/Block Processing (15–25%, baixo risco)

**Problema:** Cada thread processa 75 posições contíguas, mas internamente
acessa o cache `(nf, npt, n_layers)` em padrão strided. Para Oklahoma_28:
- 1 freq × 201 pts × 22 layers × 16 B = **70 KB** (working set para 1 freq)
- 8 threads × 70 KB = 560 KB (excede L1=32KB, cabe em L2=256KB por thread? Não — L2 é por core)

**Solução:** agrupar M=4 posições contíguas em tile interno, processar
todas usando o mesmo trecho de cache, antes de migrar para próximo tile.

```python
# Antes:
for j in prange(n_pos):
    # corpo (lê cache strided)

# Depois (Sprint 15.6 condicional):
TILE = 4
n_tiles = (n_pos + TILE - 1) // TILE
for t in prange(n_tiles):
    j_start = t * TILE
    j_end = min(j_start + TILE, n_pos)
    for j in range(j_start, j_end):
        # corpo (cache permanece quente em L1 entre 4 pos)
```

**Ganho estimado:** 15–25% (cache L1 hit rate 60% → 85%).
**Risco:** Baixo — semântica idêntica, apenas reordering. Paridade Fortran
preservada (validação obrigatória).

**Status v2.16:** **NÃO aplicado** — Cenário E pós-fix já entrega 123k mod/h
(4× speedup confirmado). Tile/block deferido para v2.17 caso necessário.

### Oportunidade 2 — Pré-compute Hankel kernels TE/TM (10–15%, médio risco)

**Problema:** Dentro de `hmd_tiv`/`vmd`, `Ktm`, `Kte`, `Ktedz` são
recalculados a cada chamada (por posição). Operações dominantes:
- `np.sqrt(kr_squared - kh2)` × n_layers × npt = ~4400 sqrt/dipolo
- Cada sqrt custa ~5 ciclos SIMD (vs 1 para add)

**Solução:** pré-computar kernels uma vez por `(freq, layer)` em
`precompute_common_arrays_cache`, armazenar em cache extra de shape
`(nf, n_layers, npt)`. Dipoles consumem read-only.

**Ganho estimado:** 10–15%.
**Risco:** Médio — novo array no cache; paridade Fortran requer revalidação
em todos os 7 canônicos (`oklahoma_28` é o caso crítico).

**Status v2.16:** **NÃO aplicado** — risco/benefício desfavorável dado o
ganho da Sprint 15.1 (4×). Adiar para v2.18.

### Oportunidade 3 — SIMD Vectorization via NumPy ufuncs (20–40%, alto risco)

**Problema:** Loop de soma Hankel (linhas 660–666) usa `np.sum(Ktedz_J1)`
que é vetorizado, mas multiplicação ponto-a-ponto `Ktm * wJ0` é elemento
por elemento dentro de loops Numba.

**Solução:** delegar para `np.multiply` + `np.sum` com SIMD AVX2/AVX-512
processando 4–8 complex em paralelo.

**Ganho estimado:** 20–40%.
**Risco:** **ALTO** — NumPy ufuncs usam IEEE 754 + FMA com reordering
permitido. Pode quebrar paridade Fortran <1e-12 (gate inviolável). Erro
acumulado FMA estimado em recursão TE/TM: ~2e-13 — **0.83× do gate**.

**Status v2.16:** **REJEITADO** — risco de quebrar paridade física.
Reavaliar apenas se análise rigorosa (10k casos aleatórios) confirmar
estabilidade <1e-12.

## §6 Constraints Físicas e Geofísicas (invioláveis)

| Constraint | Valor | Como validar |
|:-----------|:------|:-------------|
| Paridade Fortran v10.0 | `max_abs_error < 1e-12` em 7 canônicos | `pytest -k fortran_python_numba` |
| Paridade JAX vs Numba | < 1e-10 (relax 100×) | `tests/test_simulation_jax_*` |
| Estabilidade alta-ρ | finite em `oklahoma_28` (ρ > 1000 Ω·m) | `test_compare_fortran_high_rho_stability` |
| Estabilidade contraste | finite em `hou_7` (camada condutiva 0.3 Ω·m) | `test_compare_fortran_python_numba[hou_7]` |
| Convenção temporal | e^(−iωt), k² = iωμ₀σ | invariante no kernel TE/TM |
| Convenção T/R | T em +x/+z (ABAIXO), R em −x/−z (ACIMA) | `tests/_fortran_helpers::write_model_in_multi` |

**Regra de ouro v2.16:** qualquer otimização em `_numba/` (especialmente
`dipoles.py`, `kernel.py`, `propagation.py`) deve passar nos 7 canônicos
com `max_abs_error < 1e-12` ANTES de merge. Erros >1e-12 invalidam o
modelo para uso em inversão (gradientes mal-condicionados).
