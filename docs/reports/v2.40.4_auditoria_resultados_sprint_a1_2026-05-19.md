## 1. Sumário Executivo

Sprint A1 executou o notebook `validate_jax_gpu_v240.ipynb` em GPU T4 do Colab Pro+ e
gerou dois resultados antagônicos:

| Aspecto | Status | Evidência |
|:--|:-:|:--|
| **Paridade Fortran < 1e-12** (correção) | OK | 163 PASS / 0 FAIL em 32 min (pytest `-m gpu`) |
| **Performance JAX GPU ≥ 1.5× Numba CPU** (gate) | **FALHA** | A 0.38×, B 0.37×, E 0.61× (todos invertidos) |

A auditoria confirmou **causa raiz dupla**:

1. **Bucket cache explosion na primeira iteração** — `_BUCKET_JIT_CACHE` é chaveado em
   `(camad_t, camad_r, n, npt)`; 50 modelos com perfis aleatórios geram ~50 entradas
   distintas que compilam em série na Run 1 (146 s para Cenário B, ~3 s por bucket).
   O warmup com `models[0]` cobre apenas O(1) entradas — insuficiente.

2. **Overhead Python serial domina sobre throughput GPU** — `simulate_multi_jax`
   itera modelos um a um em Python loop, chamando `build_static_context` (5–20 ms
   de overhead Python) + `np.asarray()` (sync GPU→CPU bloqueante) por modelo. Para
   problemas pequenos (n_pos ≤ 600, n=5 camadas), a GPU fica ociosa > 90% do tempo.

Adicionalmente, **a metodologia do benchmark possui 4 bugs**: warmup por cenário
desalinhado, `statistics.median` mascara cold-start, baseline Numba sourced de
hardware diferente (Intel i9-9980HK 8C/16T Mac Intel vs T4 n1-standard-4 4vCPU), e ausência de
`block_until_ready` explícito.

**Conclusão técnica**: o resultado "JAX 2–5× mais lento que Numba" é **consistente com
o histórico Sprint 7** (oklahoma_28 ainda em 113 ms JAX vs 7 ms Numba após bucketing+JIT)
e **não invalida a paridade física**. O simulador JAX está correto; o que está incorreto
é (a) a forma como o benchmark mede e (b) a arquitetura atual de `simulate_multi_jax`
que itera Python loop em vez de batch sobre eixo de modelos via `vmap`.

**Decisão recomendada**: NÃO promover `A-jax-gpu-validate` para DONE. Renomear a
sprint para `A-jax-gpu-batch-redesign` e adicionar Sprint A1.5 (batched API)
ANTES de A2 (dispatcher). Detalhes em §7.

## 3. Causa Raiz Confirmada (Auditoria Multi-Agente)

### 3.1 Patologia 1 — Bucket Cache Explosion

**Local**: [geosteering_ai/simulation/_jax/forward_pure.py:573](geosteering_ai/simulation/_jax/forward_pure.py#L573)

```python
key = (int(ct), int(cr), int(n), int(npt))
```

`_BUCKET_JIT_CACHE` (OrderedDict global, default `maxsize=64` per
[forward_pure.py:424](geosteering_ai/simulation/_jax/forward_pure.py#L424)):

| Evento | Comportamento |
|:--|:--|
| Per-scenario warmup com `models[0]` | Compila buckets de `models[0]`: O(n_pos) entradas no pior caso, mas dedup colapsa para `unique(ct, cr)` ≈ 1–3 entradas |
| Run 1 itera 50 modelos | Cada modelo tem perfil aleatório → `prof_arr` distinto → `(ct, cr)` distintos → **até 50 novas compilações XLA seriais** (~2–5 s cada em T4) |
| Run 2-5 | Cache LRU contém todos os 50 buckets (≤ 64 limit) → reuso, hot-cache, ~2 s/run |
| Cenário H (512 combos × 50 modelos) | Excede LRU=64 → eviction durante Run 1 → mais recompilação → 191 s |

**Per-scenario warmup é insuficiente por design**. O warmup compila para
`models[0]` apenas, mas Run 1 itera 50 modelos com `(ct, cr)` distintos.

### 3.2 Patologia 2 — Hot-Cache Performance ainda Inferior a Numba

**Locais**: [multi_forward.py:400-422](geosteering_ai/simulation/_jax/multi_forward.py#L400) (loop interno sobre `(iTR, iAng)` por modelo) + `notebooks/colab_templates/validate_jax_gpu_v240.ipynb` célula `a1-bench-defs` (loop externo sobre modelos)

> **Errata Sprint A1.5 (2026-05-19)**: a tabela abaixo originalmente afirmava que
> o "loop Python serial sobre modelos" estava em `multi_forward.py:400-422`. Esta
> referência é **parcialmente incorreta**: as linhas 400-422 contêm o loop sobre
> `(iTR, iAng)` para **um único modelo**. O loop externo sobre modelos ocorre no
> benchmark notebook (`validate_jax_gpu_v240.ipynb`, célula `a1-bench-defs`), que
> itera `for model in models:` chamando `simulate_multi_jax()` 50 vezes em Python.
> A causa-raiz arquitetural permanece a mesma: a **assinatura** de `simulate_multi_jax`
> aceita apenas um modelo por chamada, forçando o usuário a iterar em Python — não
> existe API batched sobre o eixo de modelos. A Sprint A1.5
> (`A-jax-gpu-batched-api`) resolve isso introduzindo `simulate_multi_jax_batched()`.

Mesmo após cache JIT quente (Runs 2-5), o throughput JAX permanece 1.6–8× inferior
ao Numba CPU por **três fatores estruturais**:

| # | Fator | Local | Custo estimado por modelo |
|:-:|:--|:--|:--|
| 1 | Loop Python serial sobre modelos (notebook chama API 1× por modelo) | notebook `a1-bench-defs` + assinatura `simulate_multi_jax` aceita 1 modelo | Python GIL impede paralelismo; Numba usa `ProcessPoolExecutor` workers nativos |
| 2 | `build_static_context()` recriado por modelo | `multi_forward.py:403-414` (chamado pelo notebook 50× em Python) | 5–20 ms (I/O filtro + 8× `jnp.asarray` + `find_layers_tr` Numba FFI) |
| 3 | `np.asarray(H_jax)` força sync GPU→CPU | `multi_forward.py:422` (executado 50× em Python) | 50–200 μs barrier + cancela pipelining assíncrono |
| 4 | Ausência de `vmap` sobre eixo de modelos | (não existe API até Sprint A1.5) | GPU subutilizada para n_pos ≤ 600 (kernel < 1 ms vs overhead 5–20 ms) |

**Estimativa de overhead Python por run** (Cenário B, 50 modelos):
- 50 × `build_static_context` ≈ 250–1000 ms
- 50 × `np.asarray` barrier ≈ 2.5–10 ms
- Compute GPU efetivo ≈ 500 ms
- Total observado: ~2 s → overhead Python é 50–67% do wall-clock

**Para problemas pequenos, GPU fica ociosa > 90% do tempo**. Isto é
consistente com o achado de Sprint 7 (PR #14d) de que JAX CPU é 4–15× mais
lento que Numba CPU em modelos pequenos (oklahoma_3 a oklahoma_28).

### 3.3 Patologia 3 — Baseline Numba Enviesado (Diferente Hardware)

| Origem da baseline Numba | Hardware | CPU |
|:--|:--|:--|
| `numba_baseline_mod_h: 1_180_000` (cenário A) | **MacBook Pro Intel i9-9980HK** (Mac Intel — ver `.claude/perf_baseline.json:15`, `docs/reports/v2.36_2026-05-15.md:8`) | 8 cores físicos / 16 threads SMT |
| Colab Pro+ T4 | `n1-standard-4` Google Compute | 4 vCPUs Intel Xeon, sem HT |

A comparação **JAX-GPU-T4 vs Numba-CPU-i9-9980HK** não é hardware-aware. Se Numba
rodasse no mesmo CPU T4 (`n1-standard-4`), o throughput estimado seria 50–60%
do baseline i9-9980HK (4vCPU vs 8C+SMT). Mesmo assim, JAX permaneceria ~2× mais
lento, mas o gate ≥ 1.5× seria potencialmente atingível em A.

## 5. Análise Arquitetural — Por que JAX é Mais Lento?

### 5.1 Comparação Estrutural simulate_multi (Numba) vs simulate_multi_jax (JAX)

| Aspecto | `simulate_multi` (Numba, v2.12+) | `simulate_multi_jax` (JAX, v1.6.0) |
|:--|:--|:--|
| Eixo de batch (modelos) | **Sim** — `models: List[Dict]`, 4 modos A/B/C/D | **Não** — só single-model API |
| Paralelismo inter-modelo | `ProcessPoolExecutor` workers nativos, anti-oversubscription | Python loop serial (GIL preso) |
| Per-call setup | `_workers.py` aquece pool uma vez | `build_static_context` recriado por (modelo × i_tr × i_ang) |
| Sync para timing | Worker retorna NumPy array (sync nativo) | Depende de `np.asarray()` implícito |
| JIT cache | `@njit(cache=True)` persistente em disco | `_BUCKET_JIT_CACHE` em memória, LRU 64 entradas |
| Chave de cache | Especialização Numba por dtypes (~1) | Bucket `(ct, cr, n, npt)` — explode para N modelos diversos |
| Usabilidade em produção | `n_workers=4, threads_per_worker=2` em 1 linha | Requer warmup manual + LRU tuning + chunking |
| Esperado superar GPU? | — | Para n_pos ≤ 600, sim — overhead Python domina |

### 5.2 Por que vmap_real (Sprint 12) é mais robusto

- Cache `_UNIFIED_JIT_CACHE` chaveado em `(n, npt)` apenas — invariante a modelos
- `find_layers_tr_jax_vmap` substitui Numba FFI por JAX nativo (tracer-safe)
- `_block_until_ready` explícito em [multi_forward.py:631](geosteering_ai/simulation/_jax/multi_forward.py#L631)
- 1 compilação amortizada entre todos os modelos da mesma forma geométrica

Mas **ainda perde** para Numba em hot-cache porque:
- Loop Python externo continua serial (cada modelo é uma chamada `vmap_fn(...)` separada)
- `_one_config` retraça `find_layers_tr_jax_vmap` por (L, θ) — não compartilha entre modelos

### 5.3 O que falta para JAX GPU ser competitivo

**Arquitetura proposta** (não implementada):

```python
# API atual (lenta):
for m in models_list:
    H = simulate_multi_jax(m["rho_h"], m["rho_v"], m["esp"], positions_z, ...)  # 1 modelo

# API necessária (batched):
rho_h_batch = jnp.stack([m["rho_h"] for m in models_list])   # (n_models, n_layers)
rho_v_batch = jnp.stack([m["rho_v"] for m in models_list])   # (n_models, n_layers)
esp_batch   = jnp.stack([m["esp"]   for m in models_list])   # (n_models, n_layers-2)

# vmap externo sobre eixo de modelos:
H_batch = jax.vmap(simulate_multi_jax_single, in_axes=(0, 0, 0, None, None, None))(
    rho_h_batch, rho_v_batch, esp_batch, positions_z, freqs, ctx
)  # 1 dispatch GPU, output (n_models, nTR, nAngles, n_pos, nf, 9)
```

**Ganho estimado**: 50× redução de overhead Python (1 chamada vs 50), saturação GPU (compute aumenta de 10% para 80%+ do wall-clock), throughput projetado 5–15× Numba CPU para n_models ≥ 100.
