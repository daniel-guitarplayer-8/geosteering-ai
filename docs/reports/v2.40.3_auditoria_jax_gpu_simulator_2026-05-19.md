## 1. Sumário Executivo

### 1.1 Pergunta do usuário

> "Os datasets a serem utilizados no futuro treino de SurrogateNet deverão
> ser todos gerados através do simulador JAX GPU. Em produção, o treino das
> redes deep learning deverá ser feito através do simulador JAX GPU (por
> padrão), com o simulador Numba JIT sendo apresentado como uma opção a ser
> escolhida pelo usuário. Analise a arquitetura atual, verifique o ROADMAP,
> investigue aprimoramentos e — se viável — trabalhe primeiro o simulador
> JAX GPU para depois partirmos para o SurrogateNet."

### 1.2 Resposta direta

A proposta é **tecnicamente viável**, mas **executar agora seria prematuro**.
O simulador JAX GPU está **~85% pronto** (núcleo forward, multi-dip, multi-TR,
multi-freq, paridade Numba bit-exata em testes), mas tem **8 gaps críticos**
(G1–G8) que bloqueiam usá-lo como default em produção.

**Recomendação**: executar **3 sprints granulares (A1, A2, A3)** antes da
sprint `C-surrogate-train`. Esforço total estimado: **~16–20 h** (1 sessão
Colab + 2 sessões de código local). ROI alto:

- Desbloqueia SurrogateNet com pipeline correto (JAX GPU + ruído on-the-fly + multi-dip)
- Desbloqueia `E-api-simulate` (que precisa do dispatcher unificado)
- **Remove o Fortran do runtime training pipeline** (refinamento 1 do usuário)
- Publica baseline empírico JAX GPU em [PERFORMANCE_BASELINE.md](../PERFORMANCE_BASELINE.md)

## 3. Refinamento 1 — Substituição Fortran → Numba JIT

### 3.1 `SyntheticDataGenerator` já existe (paridade 22-col)

Localização: [geosteering_ai/data/synthetic_generator.py](../../geosteering_ai/data/synthetic_generator.py)

Cabeçalho do módulo (linhas 12–35) declara explicitamente:

> **FINALIDADE**: Substitui `Fortran_Gerador/batch_runner.py` (ProcessPoolExecutor
> chamando `tatu.x` via subprocess) por um gerador 100% Python que respeita
> `config.simulator_backend` (Sprint 6.1) para rotear entre Numba / JAX híbrido /
> JAX nativo / Fortran legacy.
>
> **Compatibilidade de saída 22-col preservada** (ver `io/binary_dat.py`) — os
> `.dat` gerados por este módulo são lidos por `loading.py` do mesmo jeito que
> os do Fortran.

| Comparação | Fortran `batch_runner` | `SyntheticDataGenerator` |
|:--|:-:|:-:|
| Subprocess `tatu.x` | Sim | Não |
| I/O binário 22-col | Sim | Sim (paridade byte-a-byte) |
| Paralelização | ProcessPoolExecutor | `prange` Numba ou `vmap` JAX |
| GPU opcional | Não | **Sim** (`backend="jax"`) |
| In-memory batching | Não | Sim (batch tensor `(n_models, n_pos, nf, 9)`) |

Saída de [GeneratedBatch](../../geosteering_ai/data/synthetic_generator.py#L86):

```python
H_tensor: np.ndarray   # (n_models, n_pos, nf, 9) complex128
rho_h: np.ndarray      # (n_models, n_layers) float64
rho_v: np.ndarray      # (n_models, n_layers) float64
esp: np.ndarray        # (n_models, n_layers-2) float64
dat_22col: np.ndarray  # estruturado com DTYPE_22COL, escrito em .dat binário
```

### 3.2 Consumo em `surrogate_data.py` é agnóstico à origem

[extract_surrogate_pairs()](../../geosteering_ai/data/surrogate_data.py#L249) na linha 249 aceita:

```python
data: np.ndarray  # shape (N_models, seq_len, 22), formato 22-col conforme COL_MAP_22
```

Não há nenhuma checagem que distinga Fortran-generated de Python-generated.
Apenas validação de shape e contagem de colunas (linhas 306–308).

**Conclusão arquitetural**: trocar o gerador do Fortran para Numba JIT é
**transparente** para o pipeline de SurrogateNet. Basta apontar o gerador
para o backend desejado e o `.dat` resultante é byte-compatível.

### 3.3 Hierarquia de backends proposta para produção

```text

│  GERAÇÃO DE DADOS DE TREINO — Hierarquia de Backend             │

│     - Throughput-alvo: T4 ≥200k mod/h; A100 ≥500k mod/h         │
│                                                                 │
│  2. Numba JIT (default em CPU local; opt-in em Colab)           │
│     - Quando GPU indisponível OU usuário escolhe explicitamente │
│     - Fallback automático se `jax.devices() == [cpu]`           │

│     - Throughput atual: Cenário E ~122k mod/h em 8C/16T         │
│                                                                 │
│  3. Fortran .dat manual (apenas reprodução de runs históricos)  │
│     - REMOVIDO do runtime training pipeline                     │
│     - Mantido em `Fortran_Gerador/` para gerar arquivos legados │
│       quando necessário (paridade científica histórica)         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 3.4 Remoção do Fortran do runtime training pipeline

Antes (estado atual):

```text
Fortran tatu.x → .dat (22-col) → loading.py → surrogate_data.py → DataPipeline → TrainingLoop
       ↑
       └─ dependência externa: compilação OpenMP, batch_runner.py, subprocess
```

Depois (proposta):

```text
SyntheticDataGenerator(backend=jax_gpu|numba_cpu) → GeneratedBatch → surrogate_data.py → DataPipeline → TrainingLoop
       │
       └─ alternativa: GeneratedBatch.dat_22col → escrita opcional em disco para auditoria/cache
```

**Ganhos arquiteturais**:

- Independência de toolchain externa (Fortran + OpenMP)
- Geração on-the-fly em batches (sem necessidade de pré-gerar `.dat` em disco)
- Suporte a GPU como cidadão de primeira classe
- Eliminação de subprocess + ProcessPoolExecutor (latência reduzida)
- Logs estruturados Python (vs. parsing de stdout do Fortran)

## 4. Refinamento 2 — Insights Numba JIT Mapeados para JAX GPU

Resumo das 10 lições aprendidas em ~30 sprints de otimização Numba (v2.10 a
v2.35) que devem ser **deliberadamente reaproveitadas** no JAX GPU.

| # | Lição Numba | Sprint | Aplicação no JAX GPU |
|:-:|:--|:-:|:--|
| **L1** | Nested `prange` (`parallel=True` em função chamada de `prange` outer) acumula ~14 s/run em Cenário E (overhead de paralelismo aninhado) | v2.21 | Evitar `vmap` aninhado quando achatar é viável. `_simulate_multi_jax_vmap_real` já achata `(nTR × nAngles)` — promover a default em GPU |
| **L2** | FLAT `prange` 4D (`nTR × nAng × n_pos × nf`) elimina `range(nf)` serial em `_fields_in_freqs_kernel_cached` (+11% B, +9% F em CPU) | v2.22 | `cfg.jax_vmap_real=True` é o análogo direto. Validar se ganho em GPU é ≥3× (paralelismo XLA escala melhor que multi-thread CPU) |
| **L3** | Fastmath seletivo em `geometry.py` (3 funções) + `rotation.py` (2 funções) preserva paridade <1e-12 e elimina conservadorismo do IEEE 754 | v2.23 | `jax.config.update("jax_default_matmul_precision", "tensorfloat32")` em A100 pode equivaler. **Validar paridade <1e-12** antes de promover |
| **L4** | Warmup com dados REAIS (anisotrópico + `dip≠0°`) cobre paths que isotrópico + `dip=0°` não acionam — caso contrário, JIT compila no 1º chunk e infla a medida | v2.28, v2.29 | `simulate_multi_jax` deve ter warmup explícito com shape representativo do dataset alvo (multi-TR + multi-freq + multi-dip) antes do timing. Sem isso, 1ª medida = JIT compile time + execution time |
| **L5** | `NUMBA_NUM_THREADS` deve ser setado no PAI antes de imports (modo `spawn` carrega numba durante bootstrap dos workers) | v2.29.1 | `jax.config.update("jax_enable_x64", True)` deve estar antes de qualquer `import jax.numpy`. Idem em notebooks Colab — primeira célula deve ter config, não import |
| **L6** | Anti-oversubscription HT-aware: usar `physical_cores`, não `logical_cores`, em CPUs SMT (8C/16T → 4w×2t, não 4w×4t) | v2.17, v2.20 | Em 1 GPU não se aplica. Em multi-GPU Colab Pro+ usaria `pmap`, mas Colab não oferece multi-GPU. **Skip por enquanto** |
| **L7** | Pool persistente bloqueava cold-start JIT — reversão para pool efêmero (`with ProcessPoolExecutor`) | v2.29 | JAX não tem pool de processos. `jit` cache é persistente entre chamadas dentro do mesmo processo Python. **Sem decisão a tomar** |
| **L8** | `NUMBA_CACHE_DIR` em tmpfs (`$TMPDIR`) reduz I/O de `.nbc` em 10–30 s na fase de bootstrap | v2.31 | `jax_compilation_cache_dir` em SSD rápido (Colab `/content/jax_cache`). Medir impacto em Sprint A1; pode ser ganho significativo em runs repetidos |
| **L9** | Cache dedup por hordist (`L × |sin(dip)|`) evita re-simulação de geometrias equivalentes | v2.21+ | JAX **já implementa** via `_build_hordist_groups()` em `multi_forward.py:192`. **Sem mudança** — apenas validar que está ativo |
| **L10** | Workers nativos no `simulate_multi` (4 modos A/B/C/D) anti-oversubscription | v2.12 | JAX `vmap` real (Sprint 12) é o equivalente arquitetural — paralelismo expressado declarativamente, não via threads. Promover `jax_vmap_real=True` a default em GPU |

### 4.1 Implicações práticas para Sprint A1 (validação GPU)

| # | Ação obrigatória | Lição-origem |
|:-:|:--|:-:|
| 1 | Pré-compilar `simulate_multi_jax` com shape representativo (multi-TR + multi-freq + multi-dip) antes de medir | L4 |
| 2 | Garantir `jax_enable_x64=True` e `jax_compilation_cache_dir=/content/jax_cache` setados ANTES de qualquer `import jax.numpy` | L5, L8 |
| 3 | Promover `cfg.jax_vmap_real=True` como default quando `jax.default_backend() == "gpu"` | L1, L2, L10 |

| 6 | Validar cache de hordist está ativo com config multi-dip (log de número de simulações reais vs requisitadas) | L9 |

### 6.2 Justificativa

| Argumento | Detalhe |
