## 2. Problemas Atacados

### 2.1 Anomalia: Especialização Dupla em ``hmd_tiv`` e ``vmd``

**Diagnóstico** (Fase 1):

- ``hmd_tiv`` chamado pelo backend Numba puro (``_fields_in_freqs_kernel``):
  arrays writeable=True → especialização ``mutable=True``.
- ``hmd_tiv`` chamado pelo backend JAX (``_dipoles_numba_host`` via
  ``jax.pure_callback``): arrays JAX vêm readonly → ``np.asarray(arr)``
  preserva ``writeable=False`` → especialização ``mutable=False``.

O Numba trata as duas mutabilidades como tipos distintos. Resultado: dois
``.nbc`` em disco por função (~2 MB redundantes), inflando o cold-start
em estimados 20-40 s para carregar e otimizar o segundo bitcode.

**Validação empírica**:

```bash
# Antes (v2.30):
$ ls geosteering_ai/simulation/_numba/__pycache__/dipoles*.nbc
dipoles.hmd_tiv-180.py313.1.nbc   1.275 MB
dipoles.hmd_tiv-180.py313.2.nbc   1.338 MB   ⚠️
dipoles.vmd-708.py313.1.nbc       734 KB
dipoles.vmd-708.py313.2.nbc       734 KB     ⚠️

# Após (v2.31), em produção via CLI:
$ ls /tmp/geosteering_numba_cache/.../dipoles*.nbc
dipoles.hmd_tiv-180.py313.1.nbc   1.275 MB   ✓
dipoles.vmd-708.py313.1.nbc       683 KB     ✓
```

### 2.2 Limitação Fundamental: LLVM Bitcode em ``cache=True``

**Diagnóstico** (relatório §1.2):

- ``cache=True`` no decorador ``@njit`` armazena **LLVM bitcode**
  (``.nbc``), não código de máquina nativo.
- Em cada novo processo Python, o backend LLVM ainda precisa recompilar
  bitcode → assembly x86-64 (~111 s para 16 funções no projeto).
- Não é um bug — é limitação estrutural da arquitetura JIT do Numba
  0.61.2. AOT via ``numba.pycc`` foi descartado por incompatibilidade
  com ``parallel=True`` (relatório §3.6).

**Mitigação aplicada**: ``NUMBA_CACHE_DIR`` em tmpfs (``/tmp``) mantém
os ``.nbc`` em memória após primeira leitura, reduzindo I/O de
carregamento em invocações subsequentes do mesmo processo Python e
em forks rápidos (impacto estimado −10-30 s em SSD/HDD).

## 3. Mudanças Aplicadas

### 3.1 ``geosteering_ai/simulation/_jax/kernel.py``

Adicionada helper privada ``_to_writeable`` e substituída em todas as
chamadas a ``hmd_tiv`` e ``vmd`` dentro de ``_dipoles_numba_host``:

```python
def _to_writeable(arr) -> np.ndarray:
    """Converte para np.ndarray contíguo e writeable (Sprint v2.31)."""
    out = np.ascontiguousarray(arr)
    if not out.flags.writeable:
        out = out.copy()
    return out
```

- Substituídos **28 sites** de ``np.asarray(...)`` por ``_to_writeable(...)``.
- Adicionado cache de ``u_np`` para evitar conversão dupla
  (``hmd_tiv`` + ``vmd`` ambos usam ``u``).
- ``AdmInt`` calculado a partir de ``u_np`` já convertido (preserva
  semântica e evita conversão redundante).

### 3.2 ``geosteering_ai/cli/main.py``

Adicionado bloco de setup ``NUMBA_CACHE_DIR`` antes de qualquer import
pesado:

```python
if "NUMBA_CACHE_DIR" not in os.environ:
    _default_numba_cache_dir = os.path.join(
        tempfile.gettempdir(), "geosteering_numba_cache"
    )
    try:
        os.makedirs(_default_numba_cache_dir, mode=0o700, exist_ok=True)
        os.chmod(_default_numba_cache_dir, 0o700)
        os.environ["NUMBA_CACHE_DIR"] = _default_numba_cache_dir
    except OSError:
        pass
```

**Características**:

- **Permissões 0o700** (CodeRabbit major finding aplicado): apenas o dono
  acessa o cache — evita injeção de ``.nbc`` maliciosos em sistemas
  multi-tenant.
- ``os.chmod`` explícito após ``makedirs`` blinda contra ``umask`` ou
  diretório pré-existente com permissões frouxas.
- Override do usuário (``export NUMBA_CACHE_DIR=...``) preservado.
- Falhas em ``OSError`` toleradas — Numba cai no default
  ``$CWD/__pycache__``.

Bump da constante ``SIMULATION_MANAGER_VERSION``: ``"v2.30"`` →
``"v2.31"``.

## 6. Trabalho Futuro Sugerido

### 6.1 Investigação: Mutabilidade da Especialização sob pytest

Em uso direto (Python script), a especialização compilada tem
``mutable=True``. Em pytest, a especialização compilada tem
``mutable=False`` (mesmo recebendo arrays ``writeable=True`` via
``_to_writeable``). A divergência foi confirmada por rastreamento de
chamadas (``_dipoles_numba_host`` recebe os arrays corretos), mas o
COUNT permanece em 1 em ambos os casos — a correção v2.31 funciona.

Esta discrepância sugere uma característica do ambiente pytest (env
vars, ``sys.settrace``, ou ordem de import via PySide6/pytest-qt)
afetando heurísticas internas do Numba. **Não é regressão**, apenas
diferença estética no flag interno. Investigar em sprint futura
(baixa prioridade).
