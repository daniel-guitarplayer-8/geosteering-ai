## §2 Diagnóstico da Regressão (v2.15)

### Causa raiz

A combinação de **dois commits da v2.15** (mesmo dia, 1h diferença) criou um buraco silencioso:

1. **Commit `0f92035`** (12:48) — `multi_forward.py:880-886`:
   ```python
   if cfg.num_threads > 0 and HAS_NUMBA:
       import numba as _numba
       try:
           _numba.set_num_threads(cfg.num_threads)
       except RuntimeError:
           pass  # ❌ silencia falha sem fallback observável
   ```

2. **Commit `e1c8864`** (13:53) — removeu `os.environ["NUMBA_NUM_THREADS"]`
   de `_numba_init_worker` em `sm_workers.py` e de `_simulate_worker_init`
   em `_workers.py`. Justificativa do commit: evitar `RuntimeError: Cannot
   set NUMBA_NUM_THREADS to a different value` durante JIT. Mas isso deixou
   workers spawn sem dimensionamento explícito do pool de threads Numba.

### Caminho do bug em produção

```
GUI (SimulationManager) → SimulationThread.run()
  → ProcessPoolExecutor.submit(run_numba_chunk, chunk, n_threads=N)
    → _numba_init_worker(n_threads=N) [pool initializer]
      • OMP_NUM_THREADS=N ✓ (controla apenas BLAS/MKL)
      • Numba já foi importado com NUMBA_NUM_THREADS=cpu_count() (16 HT)
    → run_numba_chunk(): for m in chunk: simulate_multi(...)
       → multi_forward.py:880 numba.set_num_threads(N)
         • try/except: pass silencia falhas → threads ativas em estado
           indefinido entre chamadas
         • Resultado: throughput cai 4–8× em produção
```

### Impacto medido

| Configuração | Pré-fix | Esperado | Discrepância |
|:-------------|:-------:|:--------:|:------------:|
| GUI 4w × 2t × 600 pos | 25–38k mod/h | 150k mod/h | **4–6×** |
| Bench Cenário A 30 pos | 753k mod/h¹ | ≥1.5M mod/h | 2× (dominado por JIT amortizado) |

¹ O Cenário A do benchmark setava env vars antes do spawn dos workers
(`_configure_threads(args.threads_per_worker)`), mascarando parcialmente
a regressão na medição automatizada.

## §3 Implementação do Fix (Sprint 15.1)

### 3.1 — `multi_forward.py:877-907`

Substituição de `try/except RuntimeError: pass` por logger.warning observável:

```python
# Sprint 15.1 (v2.16): mascaramento de threads observável
if cfg.num_threads > 0 and HAS_NUMBA:
    import numba as _numba
    current_active = _numba.get_num_threads()
    if current_active != cfg.num_threads:
        try:
            _numba.set_num_threads(cfg.num_threads)
        except RuntimeError as exc:
            pool_size = _numba.config.NUMBA_NUM_THREADS
            logger.warning(
                "numba.set_num_threads(%d) falhou (threads ativas atual=%d, "
                "pool size NUMBA_NUM_THREADS=%d): %s. Performance pode "
                "degradar significativamente; verifique se o env var "
                "NUMBA_NUM_THREADS foi setado antes do spawn dos workers.",
                cfg.num_threads, current_active, pool_size, exc,
            )
```

### 3.2 — `sm_workers.py::_acquire_numba_pool`

Setar `NUMBA_NUM_THREADS` no env do **processo pai** ANTES do spawn:

```python
# ── FIX v2.16 ─────────────────────────────────────────────────────
# Workers spawn herdam env do pai; Numba lê NUMBA_NUM_THREADS na
# primeira import (durante o spawn). Resultado: pool de threads do
# Numba dentro de cada worker nasce dimensionado em n_threads.
os.environ["NUMBA_NUM_THREADS"] = str(n_threads)
os.environ["OMP_NUM_THREADS"] = str(n_threads)
os.environ.setdefault("OMP_MAX_ACTIVE_LEVELS", "2")
os.environ.setdefault("KMP_WARNINGS", "FALSE")
```

### 3.3 — `_workers.py::_acquire_pool`

Mesma estratégia aplicada ao pool nativo do core (espelha `sm_workers.py`).

## §5 Vetorização I/O (Sprint 15.4)

`geosteering_ai/simulation/tests/sm_io.py::write_dat_from_tensor` foi
reescrita substituindo 5 loops Python aninhados por broadcast NumPy:

```python
# ANTES (v2.15): ~1.8M iterações Python para 600 modelos × 600 pos × 1 freq
for m in range(n_models):
    for itr in range(nTR):
        for kt in range(nAng):
            for fi in range(nf):
                for jm in range(n_pos):
                    rec = buf[idx]
                    rec["col0"] = ...

# DEPOIS (v2.16): single np.broadcast_to + np.transpose + ravel
H_perm = np.ascontiguousarray(np.transpose(H, (0, 1, 2, 4, 3, 5)))
buf["col0"] = np.broadcast_to(model_ids[:,None,None,None,None], shape).ravel()
buf["col1"] = np.broadcast_to(z_view, shape).ravel()
# ... (4 colunas + 9 componentes complexos × 2 = 22 colunas total)
```

### Validação bit-exatness

Teste `tests/test_sm_workers_io.py` valida 4 cenários:
1. Bit-exatness vetorizada vs loop em modelo pequeno (5 mod × 2 TR × 3 ang × 7 pos × 2 freq)
2. Bit-exatness com `z_obs` 1D (alternativa de schema)
3. Bit-exatness com `rho_h/v_at_obs=None` (col2/col3 zerados)
4. Performance ≥3× speedup vs loop (escala média 20 mod × 2 TR × 3 ang × 200 pos × 2 freq)

**Resultado:** 4/4 PASS em 2.73s.
