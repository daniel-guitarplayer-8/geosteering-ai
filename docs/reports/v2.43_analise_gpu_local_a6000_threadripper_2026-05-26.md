## 2. Pergunta 1 — Migração para Linux Ubuntu

### 2.1 Resposta Direta

**Sim, totalmente viável. A migração é direta e o ambiente Linux é, em vários aspectos, superior ao macOS para este projeto específico.**

### 2.2 Análise Componente a Componente

#### JAX + CUDA

| Item | Detalhe |
|:-----|:--------|
| `jax[cuda12]` | RTX A6000 (CC 8.6) é suportada: JAX 0.4.30+ suporta CC 8.0+ oficialmente |
| CUDA 12.x | RTX A6000 Ampere suporta CUDA 11.x e 12.x; recomenda CUDA 12.3 para máxima compatibilidade |
| `jax_enable_x64 = True` | Plataforma independente |
| Detecção em `_setup_xla_environment()` | A função verifica `"a100" in gpu_name` ou `"h100" in gpu_name`. RTX A6000 retorna `"NVIDIA RTX A6000"` — **não detectada como A100/H100**, portanto recebe config conservadora (unroll=1, MEM_FRACTION=0.75). É seguro e correto — a A6000 tem FP64 ratio 1:128 (similar ao T4), então unroll=1 é a escolha certa. Porém QW#4 `latency_hiding_scheduler` pode beneficiar a A6000 (Ampere, 10k+ CUDA cores). Detalhado em §6 |
| XLA compilation cache | Persistente entre runs locais: elimina cold-start de 30-180s a cada sessão |

#### TensorFlow / Keras

| Item | Detalhe |
|:-----|:--------|
| TF 2.15+  | Wheels oficiais Ubuntu + CUDA 12.2+. Atualizar `tensorflow>=2.15` em `pyproject.toml` |
| TF 2.13 (atual) | Requer CUDA 11.8; funciona mas subótimo com A6000 em CUDA 12 |
| Mixed precision FP16 | 3ª gen Tensor Cores (311 TFLOPS FP16 Tensor) — idêntico ao A100 em throughput FP16 |
| BF16 | Suportado nativamente (Ampere CC 8.0+) |
| `tf.config.list_physical_devices("GPU")` | Detecta A6000 automaticamente; conftest.py responde corretamente |

#### Numba JIT

| Item | Detalhe |
|:-----|:--------|
| `@njit(parallel=True, prange)` | Linux é o ambiente nativo do LLVM — compile times menores |
| `detect_cpu_topology()` | `psutil` em Linux: retorna 32 cores físicos, 64 lógicos |
| `recommend_default_parallelism()` | Retornará 32 workers (phys_cores) — cenário de escala nunca antes testado |
| Thermal throttling | Sem thermal throttling de laptop; workstation com cooling adequado mantém clock estável |
| `NUMBA_CACHE_DIR` | Idêntico ao CI (Ubuntu) — elimina divergências macOS vs Linux |

#### Fortran (tatu.x / PerfilaAnisoOmp)

| Item | Detalhe |
|:-----|:--------|
| `gfortran` | `apt install gfortran` — padrão Ubuntu; suporte OpenMP completo |
| Makefile | Workaround `ld-classic` é `ifeq ($(shell uname -s),Darwin)` — **não executado em Linux**; build direto sem wrapper |
| OpenMP (`-fopenmp`) | Escala para 32 threads/64 threads sem limitações macOS GCD |
| Performance Fortran | 32-thread OpenMP em Threadripper 7970X pode ser **4-8× mais rápido** que macOS 8C |
| PCIe 5.0 x80 | Acesso ultra-rápido a filesystem para I/O de modelos `.dat`/`.out` |

#### PyQt6 (Simulation Manager)

| Item | Detalhe |
|:-----|:--------|
| PyQt6 | `pip install PyQt6` no Ubuntu — suportado nativamente |
| X11/Wayland | GUI completa com display físico; Wayland: `QT_QPA_PLATFORM=xcb` |
| `xvfb-run` headless | Idêntico ao CI; `pytest-qt` PASS garantido |

#### CI/CD Paridade

O CI GitHub Actions roda em `ubuntu-latest`. Com desenvolvimento local em Ubuntu, **o ambiente de dev = ambiente de CI**:

| Problema (macOS dev → CI Ubuntu) | Com Linux dev |
|:----------------------------------|:-------------|
| Linker `ld-classic` workaround | ✅ Eliminado |
| `tatu.x` exec format error (v2.15) | ✅ Eliminado |
| GPU `@pytest.mark.gpu` skip (sem CUDA) | ✅ **GPU real: 164+ testes rodam localmente** |
| Numba LLVM variações macOS vs Linux | ✅ Eliminado |
| Comportamento `xvfb` divergente | ✅ Idêntico ao CI |

## 5. Pergunta 4 — RTX A6000 + JAX GPU vs Numba JIT CPU Paralelo

### 5.1 O Desafio Central: FP64 vs Consumer GPU

Esta é a questão mais crítica do relatório. O simulador **opera exclusivamente em
`complex128` (2× float64)** por exigência de paridade Fortran <1e-12. Isso cria uma
tensão fundamental com as GPUs de workstation:

```
GPU IDEAL para FP64:    A100 (ratio 1:2) → 38.8 TFLOPS FP64
GPU REAL (A6000):       ratio 1:128      → ~0.31 TFLOPS FP64
CPU Threadripper 7970X: 32C × ~4 GFLOPS  → ~128 GFLOPS FP64 (estimado, AVX-512)
```

> **A RTX A6000 tem FP64 similar ao T4** (0.31 vs 0.25 TFLOPS). A CPU 7970X
> pode ter FP64 comparável ou superior em cargas de trabalho com alta localidade
> de dados e paralelismo de nível de tarefas (Numba prange).

### 5.2 Projeção de Throughput Numba 7970X

**Base de referência** (dados reais do projeto):

| Referência | Hardware | Cenário E | Contexto |
|:-----------|:---------|:---------:|:---------|
| Colab VM Numba LOCAL | n1-standard-4, 4 vCPU | 22,653 mod/h | 4 workers, 1 thread |
| Intel i9-9980HK histórico | 8C/16T, DDR4-2933 dual | 122,000 mod/h | 8 workers |
| macOS M-series (atual) | 8P/4E cores | ~95,000-145,000 mod/h | ~8 workers efetivos |

**Projeção Threadripper 7970X** (metodologia: scaling linear + eficiência + IPC + bandwidth):

| Fator de ganho | Magnitude | Justificativa |
|:---------------|:---------:|:--------------|
| Cores físicos (32 vs 8) | **4.0×** | Numba prange escala quase linearmente até ~16-20 workers |
| Eficiência de escala | ~0.7× | Lock overhead + NUMA effects acima de 16 workers |
| IPC Zen 4c vs Skylake | **1.15×** | Zen 4c ~10-15% melhor IPC em cargas numéricas |
| DDR5-4800 8ch vs DDR4-2933 2ch | **~2.0×** | 600 vs 47 GB/s — simulador E é parcialmente bandwidth-bound |
| L3 cache 128 MB vs 16 MB | **~1.2×** | Reduz cache misses em `common_arrays` (nf × npt × n) |

**Estimativa conservadora cenário E**:
```
Base i9: 122,000 mod/h
× 4.0 (cores) × 0.70 (eficiência) × 1.15 (IPC) × 1.50 (bandwidth/cache) = ~497,000 mod/h
```

**Estimativa otimista cenário E**: ~640,000 mod/h (sem overhead NUMA, DDR5 bandwidth máximo)

### 5.3 Projeção RTX A6000 JAX GPU (FP64)

Usando o T4 como baseline (43,021 mod/h em cenário E) e escalando pelas diferenças de hardware:

| Fator | A6000 vs T4 | Impacto em E |
|:------|:-----------:|:------------:|
| FP64 compute | 0.31 / 0.25 = **1.24×** | Parcial (E é bandwidth+compute misto) |
| Memory bandwidth | 768 / 300 = **2.56×** | Alto (E: 600 posições = muita memória) |
| VRAM 48 GB vs 15 GB | **3.2×** | Elimina OOM e fragmentação |
| XLA cache persistente | **+cache** | Elimina cold-start |

**Estimativa RTX A6000 JAX FP64, cenário E**:
```
T4 baseline: 43,021 mod/h
× 1.2 (FP64) × 1.8 (bandwidth efetivo + sem OOM) = ~93,000 mod/h
```

**Faixa realista**: **65,000–110,000 mod/h** (cenário E, float64)

### 5.5 Cenários onde JAX GPU A6000 VENCE Numba 7970X

Apesar da limitação FP64 em cenário E, o JAX GPU A6000 tem vantagens em:

| Cenário | Por que GPU vence | Estimativa GPU | Estimativa CPU |
|:--------|:-----------------|:--------------:|:--------------:|
| **A** (1 pos) | Alta concorrência de modelos em paralelo via vmap; sem position loop | ~8M-12M mod/h | ~3M-4M mod/h |
| **C** (4 freq) | vmap sobre freq (4×) amortiza overhead por batch | ~180k-250k mod/h | ~120k-180k mod/h |
| **D** (4 TR) | vmap sobre TR (4×) — ganho exponencial vs CPU linear | ~4M-6M mod/h | ~1M-2M mod/h |
| **F** (16 combos) | vmap composto — 16× paralelismo em GPU vs 16× serial + parallel CPU | ~50k-80k mod/h | ~40k-60k mod/h |
| **G** (64 combos) | 64× vmap — GPU domina completamente | ~15k-30k mod/h | ~10k-15k mod/h |

> **Padrão**: GPU vence em cenários com **muitas combinações de configuração
> (nf×nTR×nAng >> 1)** porque o vmap paralelo escala com o número de combinações.
> CPU vence em cenários com **muitas posições e poucas configurações** (cenários
> E-like) porque o prange de posições escala com os 32 cores.

### 5.6 O Caminho Real para GPU Superar CPU em Todos os Cenários

Existem três caminhos para GPU A6000 superar Numba 7970X em **todos** os cenários:

#### Caminho 1 — Complex64 (Sprint O4)
```
JAX float64 cenário E: ~65-110k mod/h → JAX complex64: ~130-220k mod/h (2× teórico)
Threadripper Numba cenário E: ~500-640k mod/h
Resultado: Ainda insuficiente (3-5× abaixo)
Mas: complex64 ativa FP16 Tensor Cores → pode ser 4-8× em cenários com Tensor Cores!
```

#### Caminho 2 — Mudança de paradigma: vmap agressivo sobre modelos
```
Batches grandes (n_models=1000+) × vmap → GPU mantém alta ocupância
Em cenário E com n_models=5000: GPU pode recuperar throughput via batching
Custo: VRAM (48 GB A6000 suporta n_models=5000 × 600 pos em complex64)
```

#### Caminho 3 — Numba GPU (@cuda.jit)
```
Reescrita do kernel crítico (common_arrays, Hankel integration) em CUDA puro
Vantagem: controle total do kernel, podem explorar shared memory, stream parallelism
Desvantagem: reescrita significativa; FP64 hardware limitation permanece
Estimativa: ~150-250k mod/h em cenário E com kernels otimizados
Ainda inferior ao Numba 7970X (~500-640k mod/h)
```

> **Conclusão**: Para este hardware específico (A6000 + 7970X), a estratégia ótima é:
> - **CPU (Numba 7970X)** para cenários E-like (poucas configs, muitas posições)
> - **GPU (JAX A6000 FP64)** para cenários C/D/G-like (muitas configs)
> - **GPU (JAX A6000 FP32/complex64)** para geração de dados em escala (relaxado >1e-6)
> - **GPU (TF/Keras)** para treino do pipeline ML (FP32/FP16 domina)

### 6.4 Lógica de Detecção Proposta (extensão de `_setup_xla_environment`)

```python
def _classify_gpu_tier(gpu_name: str, compute_capability: float) -> str:
    """Classifica GPU em tier para configuração XLA ótima.

    Tiers:
      datacenter_fp64: A100, H100, L40S, RTX 6000 Ada Pro (ratio FP64 1:2)
      workstation: A6000, RTX 4000/5000/6000 Ada (ratio ~1:64-128, alta VRAM)
      inference: T4, A10, RTX consumer (ratio 1:32-128, menor VRAM)
    """
    name = gpu_name.lower()
    # Datacenter FP64 (ratio 1:2): vantagem clara para float64
    if any(x in name for x in ["a100", "h100", "h200", "l40s", "b100", "b200"]):
        return "datacenter_fp64"
    # Workstation com VRAM alta mas FP64 workstation ratio
    if any(x in name for x in ["a6000", "a5000", "a4000", "rtx 6000", "rtx 5000"]):
        return "workstation"
    # RTX Ada Pro-level (alto FP32, VRAM alta, ratio consumer)
    if "ada" in name and any(x in name for x in ["6000", "5000", "4000"]):
        return "workstation"
    # Default: inference tier (T4, A10, consumer RTX)
    return "inference"

def _get_jax_config_for_tier(tier: str, vram_gb: float) -> dict:
    return {
        "datacenter_fp64": {
            "unroll": 8,
            "mem_fraction": 0.85,
            "latency_hiding": True,
            "preferred_dtype": "complex128",
        },
        "workstation": {
            "unroll": 2,           # A6000: 2 é bom trade-off (maior que T4=1)
            "mem_fraction": 0.80,  # 48 GB: 0.80 = 38.4 GB para JAX
            "latency_hiding": True,  # A6000 tem 84 copy engines (testar)
            "preferred_dtype": "complex128",  # mas complex64 como opt-in
        },
        "inference": {
            "unroll": 1,
            "mem_fraction": 0.75,
            "latency_hiding": False,
            "preferred_dtype": "complex128",
        },
    }[tier]
```
