### 3.3 F9 — Auto-tuning de `flat_prange_min_combos`

**Situação atual**: Hardcoded em 2 (heurística v2.37 F1). Para perfis muito grandes (n_pos ≥ 1000) ou hardware com muitos cores (32+), o valor ótimo pode diferir.

**Implementação planejada**:
```python
def recommend_flat_prange_min_combos(n_pos: int, n_threads: int) -> int:
    """Heurística: evita FLAT quando tasks < 4 × n_threads."""
    min_tasks_for_balance = 4 * n_threads
    for min_combos in range(1, 64):
        if n_pos >= min_tasks_for_balance // min_combos:
            return min_combos
    return 2  # fallback conservador
```

**Esforço estimado**: 4–6h

### 4.1 tf.data + Mixed Precision (Fase I da Arquitetura)

**Situação**: `DataPipeline` existe e está correto fisicamente, mas provavelmente usa `.numpy()` loop ou `tf.data` básico sem AUTOTUNE/prefetch.

**O que implementar** em `geosteering_ai/data/pipeline.py`:

```python
def build_tf_dataset(self, split: str) -> tf.data.Dataset:
    """Constrói tf.data.Dataset otimizado com AUTOTUNE."""
    ds = tf.data.Dataset.from_tensor_slices((x_raw, y))

    if split == "train":
        ds = ds.shuffle(min(len(x_raw), 10_000), seed=self.config.seed)
        ds = ds.batch(self.config.batch_size)
        ds = ds.map(self._train_map_fn,           # ruído on-the-fly + FV + GS + scale
                    num_parallel_calls=tf.data.AUTOTUNE)
        ds = ds.prefetch(tf.data.AUTOTUNE)
    else:
        ds = ds.batch(self.config.batch_size)
        ds = ds.map(self._eval_map_fn,            # FV + GS + scale (sem ruído)
                    num_parallel_calls=tf.data.AUTOTUNE)
        ds = ds.cache()                           # val/test são estáticos
        ds = ds.prefetch(tf.data.AUTOTUNE)

    return ds
```

**Mixed Precision** em `geosteering_ai/training/loop.py`:

```python
if config.use_mixed_precision:
    tf.keras.mixed_precision.set_global_policy("mixed_float16")

```

class TIVConstraintLayer(tf.keras.layers.Layer):
    """Projeta saída do modelo para range físico válido [ρ_min, ρ_max]."""
    def call(self, x):
        rho_h = tf.nn.softplus(x[..., 0]) * self.scale + self.rho_min
        rho_v = rho_h * (1.0 + tf.nn.softplus(x[..., 1]))  # ρ_v ≥ ρ_h
        return tf.stack([rho_h, rho_v], axis=-1)

## 7. Trilha F — Geosteering AI Studio (Longo Prazo)

### 7.1 Arquitetura da Aplicação

O **Geosteering AI Studio** é uma aplicação desktop PyQt6 separada (repositório privado), com 6 camadas:

```
┌──────────────────────────────────────────────────────────────────┐
│  S1 — Presentation Layer     (PyQt6 UI: MainWindow + Dialogs)    │
│  S2 — Controller Layer       (MVC: SessionController, etc.)      │
│  S3 — Service Layer          (StudioSession, InversionService)   │
│  S4 — Domain Layer           (geosteering_ai/ package — core)    │
│  S5 — Infrastructure Layer   (REST API, Colab, Cloud Storage)    │
│  S6 — Data Layer             (SQLite/PostgreSQL + File System)   │
└──────────────────────────────────────────────────────────────────┘
```

### 7.2 Relação Hexagonal vs MVC

```
SISTEMA (Hexagonal):
  Core (geosteering_ai/) → Port (API/CLI/GUI) → Adapter (REST/Qt/Colab)

DENTRO DO ADAPTER GUI (MVC):
  Model    = StudioSession (estado da sessão)
  View     = MainWindow + PlotWidgets (PyQt6)
  Controller = SessionController (evento → lógica)
```

A arquitetura Hexagonal define a **topologia do sistema** (o `geosteering_ai/` core é independente de qualquer UI). O MVC organiza a **estrutura interna do adapter GUI**. São complementares, não concorrentes.
