## 2. Implementação A.1 — Fastmath Dual-Mode

### 2.1 Funções modificadas

| Arquivo | Função | Linha | Decorador antigo | Decorador novo |
|:--------|:-------|:-----:|:-----------------|:---------------|
| `_numba/geometry.py` | `_sanitize_profile_kernel` | 186 | `@njit(cache=True)` | `@njit(cache=True, fastmath=True)` |
| `_numba/geometry.py` | `find_layers_tr` | 222 | `@njit` | `@njit(fastmath=True)` |
| `_numba/geometry.py` | `layer_at_depth` | 309 | `@njit` | `@njit(fastmath=True)` |
| `_numba/rotation.py` | `build_rotation_matrix` | 107 | `@njit` | `@njit(fastmath=True)` |
| `_numba/rotation.py` | `rotate_tensor` | 168 | `@njit` | `@njit(fastmath=True)` |

### 2.2 Funções intencionalmente NÃO modificadas

Funções "raiz" e da cascata crítica permanecem `fastmath=False` por design:

- `_simulate_positions_njit*`, `_simulate_combined_prange_flat`, `_fields_*_kernel*`
  (forward.py, kernel.py): saída direta comparada contra Fortran <1e-12
- `common_arrays`, `common_factors` (propagation.py): raiz da cadeia (sqrt complexo,
  exponenciais — sensíveis a ordem de FMA)
- `hmd_tiv`, `vmd` (dipoles.py): integração Hankel + acumulação produto-soma

### 2.3 Justificativa física do escopo

| Categoria | Risco FMA reorder | Justificativa |
|:----------|:------------------|:--------------|
| `find_layers_tr`, `layer_at_depth` | **Zero** | Apenas comparações `≥`, `>` — fastmath não afeta operações lógicas |
| `_sanitize_profile_kernel` | **Quase zero** | Acumulação escalar serial (`cumulative += h[i]`); Numba não reordena reduções escalares serial |
| `build_rotation_matrix` | **Baixo (~1e-15 ULP)** | Produtos `cos·sin` em 9 elementos da matriz R |
| `rotate_tensor` | **Baixo (~1e-14 acumulado)** | Matmul `Rᵀ·H·R` (3×3 complex128); 2-3 ordens de margem ao gate <1e-12 |

**Validação empírica**: A suite `test_simulation_compare_fortran.py` passou 13/13
após aplicação. Modelos canônicos: oklahoma_3, oklahoma_5, oklahoma_28 (alta
resistividade), hou_7, devine_8, oklahoma_15, viking_graben_10.

## 3. Implementação A.2 — Threads Adaptativos

### 3.1 Auto-detect em `__post_init__`

Quando o usuário NÃO especifica `n_workers` nem `threads_per_worker`, o
`SimulationConfig.__post_init__` chama `recommend_default_parallelism()` do
módulo `_workers.py` (infraestrutura existente desde v2.17, validada
empiricamente em v2.20):

```python
# Sprint v2.23 A.2 — auto-detect threads quando ambos None
if self.n_workers is None and self.threads_per_worker is None:
    try:
        from geosteering_ai.simulation._workers import (
            recommend_default_parallelism,
        )
        rec_workers, rec_threads = recommend_default_parallelism()
        object.__setattr__(self, "n_workers", rec_workers)
        object.__setattr__(self, "threads_per_worker", rec_threads)
        logger.info(
            "Sprint v2.23 A.2 — auto-detect: n_workers=%d, threads_per_worker=%d",
            rec_workers, rec_threads,
        )
    except (ImportError, Exception) as exc:
        logger.warning(
            "Sprint v2.23 A.2 — auto-detect falhou (%s). "
            "Mantendo n_workers=None (legado single-process).",
            exc,
        )
```

**Try-except robusto** (fix do perf-reviewer): se `_workers.py` falhar a importar
(e.g., psutil ausente em ambiente mínimo), config NÃO quebra — fallback para
comportamento legado.

### 3.2 Defesa em camadas KB-019 (anti-oversubscription)

Quando `n_workers` e `threads_per_worker` foram setados (explícita ou
automaticamente), valida invariante:

```python
if self.n_workers is not None and self.threads_per_worker is not None:
    try:
        from geosteering_ai.simulation._workers import detect_cpu_topology
        _, physical_cores, _ = detect_cpu_topology()
        total_threads = self.n_workers * self.threads_per_worker
        limit = 4 * physical_cores
        if total_threads > limit:
            logger.warning(
                "Sprint v2.23 — oversubscription severa detectada: "
                "n_workers=%d × threads_per_worker=%d = %d threads totais > "
                "4× physical_cores (%d). Pode degradar throughput (KB-019).",
                ...
            )
    except Exception:
        pass  # detect_cpu_topology raramente falha
```

**Margem 4×**: tolera casos legítimos de I/O-bound + configs experimentais.
Anti-oversubscription severa (e.g., 256w × 256t = 65k threads) emite warning.

### 3.3 Backward-compat preservada

- `SimulationConfig(n_workers=1)` → `n_workers=1` (single-process, NÃO dispara auto-detect)
- `SimulationConfig(threads_per_worker=4)` → `threads_per_worker=4` preservado
- Apenas o caso AMBOS `None` dispara auto-detect

### 3.4 Hook `setup-environment.sh`

Adicionada exibição de topologia + recomendação:

```bash
CPU: 8P/16L (HT=sim) - default v2.23: 4w x 2t
```

Tolerante a falhas: se import falhar, omite a linha (não bloqueia setup).

## 6. Impacto Esperado

### 6.1 Performance

- **A.1 Fastmath**: ganho marginal sistêmico (~0.5-2%) — geometry.py e rotation.py
  são <1% do tempo total (kernel.py + dipoles dominam). Aplicação é
  defensável pela ausência de risco e pela preparação para sprints futuras.
- **A.2 Threads adaptativos**: ganho potencial de **+25-50% em throughput**
  para usuários que rodavam com `SimulationConfig()` sem configurar paralelismo
  manualmente. Defaults hoje (4w × 2t em 8C/16T HT) são empiricamente ótimos
  conforme validação v2.20 (5 runs Cenário E).

### 6.2 Robustez

- Try-except no lazy import: NÃO quebra ambientes mínimos sem psutil
- Defesa em camadas KB-019: warning informativo (não bloqueia config)
- Hook tolerante a falhas: omite info de CPU se módulo não importável

### 6.3 Próximas sprints habilitadas

- **v2.24** pode introduzir kernels condicionais usando o flag `use_fastmath`
  (atualmente documental)
- **v2.25** Hankel pre-compute + Kong UI opt-in
- **v2.27** JAX `vmap_real` flip default após validação Colab T4/A100
