## 1. Sumário Executivo

Usuário relatou que após v2.29.2 (cache LRU configurável), o throughput da
**configuração padrão** (Cenário E com `n=2000` modelos) PIOROU e o computador
ficou mais lento. Esta sprint investigou empiricamente a regressão, aplicou
correções de qualidade e implementou infraestrutura de **prevenção** de
regressões futuras.

**Veredito empírico**: **NÃO há regressão por v2.29.2**. Benchmarks lado-a-lado
(commit `24fba72` vs `e3a0617`) com cache JIT limpo + 3 runs cada mostram
que v2.29.2 é **+3.8% MELHOR** que v2.29.1 (95,124 vs 91,620 mod/h mediana).

**Achados secundários**:

1. **Bug minor `is_paused()`** em `_resume_simulation()` (não afeta throughput)
   — corrigido
2. **Smoke test usa `gen.is_cancelled` corretamente** — `ModelGenerationThread.is_cancelled`
   é `@property` enquanto `SimulationThread.is_cancelled` é método (inconsistência
   API documentada)
3. **Discrepância 150k mod/h vs medições atuais**: relatório v2.29.1 cita
   145,202 mod/h com `n=1000`, mas medições atuais em `n=2000` chegam a apenas
   95k mod/h. Explicação: warmup amortizado diferente + possível thermal
   throttling em sequência de simulações

**Entregáveis arquiteturais**:

- Hook `.claude/hooks/check-perf-regression.sh` — anti-regressão automático
- `.claude/perf_baseline.json` — baseline numérico canônico
- `docs/PERFORMANCE_BASELINE.md` — tabela de cenários + processo de atualização
- `.claude/commands/geosteering-perf-baseline.md` — skill reviewer perf-aware

## 2. Investigação Empírica de Regressão

### 2.1 Metodologia

1. Backup do estado atual (`v2.29.2_2026-05-11_114920/`)
2. `git stash` working tree
3. `git checkout 24fba72` (v2.29.1 baseline)
4. `find geosteering_ai -name __pycache__ -exec rm -rf {} +` (cold cache)
5. 3 runs: `python -m geosteering_ai.cli benchmark --scenario E --n 2000`
6. `git checkout main` + `git stash pop` (v2.29.2 atual)
7. Limpar cache + 3 runs
8. Comparação de mediana

### 2.2 Resultados

**Configuração padrão canônica (esclarecida pelo usuário)**:

- 1 freq 20 kHz · 1 dip 0° · 1 TR · 600 pos · tj 120 m · **n=2000 modelos**

| Run | v2.29.1 (24fba72) | v2.29.2 (e3a0617) | Δ |
|:---:|:-----------------:|:-----------------:|:-:|
| Cold | 91,620 mod/h | 94,563 mod/h | +3.2% |
| Warm 1 | 93,510 mod/h | 96,737 mod/h | +3.5% |
| Warm 2 | 88,732 mod/h | 95,124 mod/h | +7.2% |
| **Mediana** | **91,620** | **95,124** | **+3.8%** |

### 2.3 Por que NÃO há regressão

Análise estática (via 2 Explore agents) confirma que todas as 4 mudanças
v2.29.2 são **periféricas** ao caminho crítico Numba JIT:

- `MainWindow.__init__` (linhas 7494–7509): leitura QSettings — **STARTUP ONLY**
- `PreferencesPage._build_cache_lru_group` (7062–7159): UI widgets — **STARTUP**
- `_on_cache_settings_changed` (8082–8136): dispara **APENAS** quando usuário
  altera QSpinBox em Preferências, NUNCA durante simulação
- Smoke test fix `is_paused()`/`is_cancelled()` (linha 10595): não afeta runtime

Caminho crítico Numba (`SimulationThread.run` → `ProcessPoolExecutor` →
`run_numba_chunk` → `simulate_multi` → `_simulate_combined_prange_flat`)
**INTOCADO** em todos os 4 commits v2.29.2.

### 2.4 Por que o usuário percebeu "lentidão"?

Hipóteses plausíveis (não-mutuamente-exclusivas):

1. **Cache JIT em disco em estado intermediário** — após mudanças no código
   v2.29.2, o cache `.nbc` pode ter sido invalidado parcialmente; a primeira
   execução paga ~30s de LLVM compile
2. **Thermal throttling do Mac** após várias simulações consecutivas;
   throughput pode cair 10–20% após 5 min contínuos
3. **Outros processos no sistema** (browser, editor, Spotlight) consumindo
   recursos compartilhados
4. **Percepção do warmup inline** (~30s na 1ª simulação) interpretado como
   "computador lento"

### 2.5 Discrepância 150k mod/h vs medições atuais

| Versão | n_models | Throughput reportado | Notas |
|:------:|:--------:|:--------------------:|:------|
| v2.21 | 1000 | 121,957 | doc canônico |
| v2.29.1 | 1000 | 145,202 | report v2.29.1 |
| **v2.29.1** | **2000** | **91,620** | **medido hoje** |
| **v2.29.2** | **2000** | **95,124** | **medido hoje** |
| v2.29.2 | 1000 | 92,395 | medido hoje |
| v2.29.2 | 500 | 83,590 | medido hoje |
| v2.29.2 | 200 | 65,576 | medido hoje |

**Conclusão**: a meta histórica de **150k mod/h** parece ter sido medida em
contexto específico — possivelmente:

- Cache JIT 100% warm (todos `.nbc` pré-existentes)
- Sem thermal throttling (Mac frio)
- Sem outros processos competindo
- Hardware em pico de clock

A configuração atual entrega **~95k mod/h estável** com `n=2000`. Para
atingir 150k consistentemente, seriam necessárias otimizações adicionais
(Sprint v2.30+) ou melhor controle das condições de medição.

## 3. Mudanças Aplicadas (v2.29.3)

### 3.1 Fix `is_paused()` em `_resume_simulation`

**Antes** ([simulation_manager.py:8472](../../geosteering_ai/simulation/tests/simulation_manager.py#L8472)):

```python
if sim is not None and sim.isRunning() and sim.is_paused:  # ← BUG
    sim.request_resume()
```

`sim.is_paused` (sem `()`) é referência ao método bound, sempre truthy. O
bloco sempre executava `request_resume()`, mas `request_resume` é idempotente
(linha 535–540 de `sm_workers.py`), então não causava regressão funcional.

**Depois**:

```python
if sim is not None and sim.isRunning() and sim.is_paused():  # ← FIX
    sim.request_resume()
```

## 4. Respostas às 4 Perguntas Conceituais

### 4.1 Pergunta 1 — Workers no SM ou no simulador Python Numba JIT?

**Resposta**: existem **duas camadas complementares** de workers.

| Camada | Arquivo | Propósito |
|:-------|:--------|:----------|
| **GUI (SM)** | [`geosteering_ai/simulation/tests/sm_workers.py`](../../geosteering_ai/simulation/tests/sm_workers.py) | Orquestração Qt-aware. `SimulationThread` (QThread) + `run_numba_chunk()` picklable. Pool **efêmero** (`with ProcessPoolExecutor`). Sinais Qt: `progress_update`, `paused`, `resumed`, `cancelled` |
| **Core (Simulador)** | [`geosteering_ai/simulation/_workers.py`](../../geosteering_ai/simulation/_workers.py) | Paralelismo puro do simulador. `simulate_multi()` API pública + pool **persistente** singleton. Reutilizável em CLI, notebooks, integração externa |

**Diagrama de fluxo**:

```text
GUI Path                          CLI Path                      Notebook Path
─────────                         ─────────                     ──────────────
SimulationThread (QThread)        argparse main()              import simulate_multi
  └─ pool efêmero                   └─ simulate_multi()           └─ simulate_multi()
      └─ run_numba_chunk            └─ _workers._acquire_pool      └─ _workers._acquire_pool
          └─ simulate_multi             └─ _run_simulate_chunk         └─ _run_simulate_chunk
                                            (pool persistente)              (pool persistente)
```

**Diferença chave**: `sm_workers.py` é GUI-specific (Qt signals, control
flow, pause/cancel, NumbaPrimer); `_workers.py` é core puro. Migração de
core para `_workers.py` em v2.12 desacoplou paralelismo da UI.

### 4.2 Pergunta 2 — Como executar o CLI?

**Entry point** ([`pyproject.toml:62-64`](../../pyproject.toml#L62-L64)):

```toml
[project.scripts]
geosteering-cli = "geosteering_ai.cli:main"
```

**Instalação** (editable):

```bash
cd ~/Geosteering_AI
source ~/Geosteering_AI_venv/bin/activate
pip install -e .
```

**Subcomandos disponíveis**:

| Comando | Arquivo handler | Descrição |
|:--------|:----------------|:----------|
| `simulate` | [`geosteering_ai/cli/simulate.py`](../../geosteering_ai/cli/simulate.py) | Gera N modelos 1D 5-camada randomizados e executa `simulate_multi` em batch |
| `benchmark` | [`geosteering_ai/cli/benchmark.py`](../../geosteering_ai/cli/benchmark.py) | Executa cenários canônicos (A/B/C/D/E/F) com diferentes combinações npos/freq/TR e reporta mod/h |
| `version` | [`geosteering_ai/cli/main.py`](../../geosteering_ai/cli/main.py) (inline 218–221) | Exibe versão atual |

**Uso**:

```bash
# Help geral
geosteering-cli --help

# Benchmark configuração padrão canônica (n=2000)
geosteering-cli benchmark --scenario E --n 2000

# Simulate 100 modelos randomizados
geosteering-cli simulate --n 100

# Versão
geosteering-cli version

# Override workers/threads
geosteering-cli benchmark --scenario A --n 500 --workers 4 --threads 2
```

**Via módulo** (sem entry point instalado):

```bash
python -m geosteering_ai.cli benchmark --scenario E --n 2000
```

### 4.3 Pergunta 3 — Refactor do código Numba JIT e refactor SM em MVC?

**Análise do estado atual**:

| Componente | LOC | Modularização | Qualidade |
|:-----------|:---:|:-------------:|:----------|
| Simulador Numba JIT (`geosteering_ai/simulation/`) | ~12,000 | **Boa** — sub-pacotes `_numba/`, `_jax/`, `filters/`, `validation/`, `io/`, etc. | Estável; bem testado; paridade Fortran <1e-12 |
| Simulation Manager GUI (`simulation_manager.py`) | **10,748** | **Fraca/Implícita** — page classes (`ParametersPage`, `SimulatorPage`, etc.) misturam controller embutido | Monolítico; difícil testar; pouco MVC explícito |

**Recomendação — Simulador Python Numba JIT**: **NÃO re-escrever**.

- Está estável (37/37 testes PASS, paridade Fortran <1e-12)
- Modularização adequada (sub-pacotes bem definidos)
- Otimizações documentadas em `analise_cenarios_otimizacao_simulador_numba.md`
- Re-escrita arriscaria perder microoptimizações acumuladas em 13+ sprints
- Foco: **melhorar incrementalmente** via Sprint v2.30+ (FLAT prange dispatch,
  cache LRU de contexto, vmap pmap JAX em Cenário A)

**Recomendação — Simulation Manager**: **Refactor MVC viável em Sprint v3.0+**.

Proposta de arquitetura:

```text
geosteering_ai/
├── simulation/                     # core simulador (intocado)
└── gui/                            # NOVO pacote MVC
    ├── views/                      # widgets PyQt puros
    │   ├── parameters_view.py
    │   ├── simulator_view.py
    │   ├── benchmark_view.py
    │   ├── results_view.py
    │   └── preferences_view.py
    ├── controllers/                # lógica de orquestração
    │   ├── simulation_controller.py
    │   ├── benchmark_controller.py
    │   └── preferences_controller.py
    ├── models/                     # dataclasses + state
    │   ├── experiment_state.py
    │   ├── simulation_snapshot.py
    │   └── plot_bundle.py
    └── main_window.py              # glue
```

**Custo**: ~6 sprints (v3.0–v3.5) com testes pytest-qt completos.
**Benefício**: testabilidade, paralelismo de desenvolvimento, manutenibilidade.
**Risco**: regressão funcional durante migração (mitigável com pytest-qt
golden path tests + comparação visual side-by-side).

**Sugestão final**: Sprint v2.27 (já planejada) deve adicionar
**pytest-qt golden path** ANTES de iniciar refactor MVC. Sem cobertura
de testes UI, refactor MVC é risco alto demais.
