## Sumário Executivo (TL;DR)

A investigação por **profiling empírico** (não teórico) revelou que o teto de
**42.8k mod/h NÃO é um limite do JAX GPU** — é um artefato de **duas escolhas de
implementação**:

1. **`strategy="unified"` é 6.9× mais lenta que `strategy="bucketed"`** no
   Cenário E. O `lax.fori_loop` da unified, com bounds dinâmicos por posição,
   impede o XLA de fazer unroll/vetorização. Com `bucketed`, **JAX GPU faz
   369k–427k mod/h no Cenário E — empatando/superando Numba 4w×16t (389k)**.

2. **`simulate_multi_jax_batched` hardcoda `unified`** (linhas 746, 1063 de
   `multi_forward.py`), ignorando `cfg.jax_strategy`. Por isso o path batched
   (usado no bench de on-the-fly) ficou preso em ~67k mod/h com 600 posições.

**Descoberta central**: o JAX GPU **já tem 2 regimes onde domina**:
- **PINN com pontos de colocação (poucas posições): 33.1M mod/h** (256 modelos × 1 pos) — imbatível.
- **Cenário E single-model com bucketed: 369k mod/h** — empata Numba.

O gargalo real está em **um único regime**: muitos modelos × muitas posições
(on-the-fly dataset gen com 600 pos), onde o batched-unified colapsa para 67k.
A solução é um **path batched-bucketed** (Sprint O4).

## Q1 — Por que o JAX GPU não passava de 42.8k mod/h? Como chegar à metade do Numba?

### 1.1 Metodologia: profiling empírico (decomposição de tempo)

Em vez de teorizar, decompus o tempo de cada etapa do Cenário E (600 posições,
1 freq, 1 TR, dip=0, oklahoma_5) na RTX A6000:

| Etapa | Tempo | Observação |
|:------|------:|:-----------|
| `build_static_context` (NumPy puro) | 53.60 ms | **Já eliminado por `_CTX_CACHE` (Sprint O3)** |
| `build_static_context_cached` (HIT) | 0.007 ms | Cache hit economiza 53.6 ms |
| `forward_pure_jax` (XLA puro, unified) | 66.25 ms | **O gargalo real** |
| `np.asarray` transfer GPU→CPU | 0.005 ms | Desprezível |
| **E2E `simulate_multi_jax` (unified)** | **66.42 ms** | = 54k mod/h |

**Conclusão parcial 1**: o `_CTX_CACHE` (Sprint O3) já removeu os 53.6 ms de
NumPy. O transfer GPU→CPU é desprezível. O gargalo é 100% a **execução XLA do
`forward_pure_jax`**.

### 1.2 A descoberta decisiva: unified vs bucketed

Comparei as duas estratégias de paralelismo no mesmo Cenário E:

| Strategy | forward_pure_jax | E2E `simulate_multi_jax` | Throughput E2E |
|:---------|----------------:|-------------------------:|---------------:|
| **bucketed** | **9.03 ms** | **9.74 ms** | **369,468 mod/h** |
| unified | 66.50 ms | 67.45 ms | 53,372 mod/h |
| **speedup** | **7.4×** | **6.9×** | — |

**Paridade bucketed vs unified: 6.44e-15** (bit-exato — mesma física).

**O bench original usava `unified`** (via `simulate_multi_jax_batched` que
hardcoda unified, e via `--jax-strategies unified` no headline). Trocar para
`bucketed` resolve o gargalo imediatamente.

### 1.3 Scaling analysis (confirma o diagnóstico)

| Eixo | Comportamento | Interpretação |
|:-----|:--------------|:--------------|
| **n_pos** (unified) | Linear: ~109 µs/pos constante (1→1200 pos) | Gargalo é per-position, sem overhead fixo após compile |
| **nf** (multi-freq) | **Sublinear**: 8 freqs = 2× de 1 freq (11.4→22 ms) | vmap sobre freqs é **eficiente** — multi-freq favorece GPU |
| **dtype** | c128 == c64 (66 ms ambos, unified) | Gargalo NÃO é memory bandwidth nem FP throughput |
| **bucket count** | dip 0°–85°: sempre 4 buckets, ~9 ms | Geometrias típicas geram poucos buckets |
| **modelo denso** | oklahoma_28: bucketed 13 ms vs unified 75 ms | Mesmo 28 camadas → só 4 buckets (posições varrem poucas camadas) |

### 1.4 Mecanismo: por que unified é lenta

O `fori_loop` da unified (em `dipoles_unified.py::_vmd_descent_body` e
equivalentes HMD) executa, **para cada camada iterada**:

```python
# 5 candidatos calculados SEMPRE (mesmo que só 1 seja usado):
TEdwz_first        = zeta * _MZ / (2.0 * u[:, j])
TEdwz_next_last    = TEdwz[:, j_prev] * exp_arg_nl
TEdwz_next_internal= TEdwz[:, j_prev] * exp_arg_nl / denom_ni
TEdwz_internal     = TEdwz[:, j_prev] * (1.0 + RTEdw[:, j_prev]) * ... / denom_ni
TEdwz_last         = TEdwz[:, j_prev] * (1.0 + RTEdw[:, j_prev]) * ...
# seleção via jnp.where aninhado 4 níveis:
TEdwz_new = jnp.where(is_first, ..., jnp.where(is_next_last, ..., ...))
# scatter dinâmico por camada:
return TEdwz.at[:, j].set(TEdwz_new)
```

Problemas que impedem o XLA de otimizar:
1. **Bounds dinâmicos** (`camad_t`, `camad_r` são tracers int32) → `fori_loop`
   não pode ser unrollado.
2. **5× compute redundante** por camada (calcula todos os branches, descarta 4).
3. **Scatter dinâmico** `.at[:, j].set()` por camada (vs slice estático).
4. **Serial por dependência** (cada camada depende da anterior — recursão TE/TM).

A `bucketed` evita tudo isso: `camad_t`/`camad_r` são **constantes Python** por
bucket, então o XLA faz **unroll completo do loop**, **constant-fold da seleção
de branch** (zero `jnp.where` no código gerado), e usa **slicing estático**.

### 1.5 Resposta a Q1

> **Já chegamos à metade do Numba — na verdade, ao empate.** Com `bucketed`,
> o JAX GPU faz **369k mod/h (E2E) a 427k mod/h (forward puro)** no Cenário E,
> contra ~389k do Numba 4w×16t. Em **multi-freq (4 freqs) faz 234k mod/h**.

**Ações imediatas para destravar (sem código novo)**:
1. Usar `cfg.jax_strategy="bucketed"` (já é o default em `SimulationConfig`).
2. No bench, parar de forçar `unified`.

**Ação de código (Sprint O4)**: fazer `simulate_multi_jax_batched` respeitar
`cfg.jax_strategy` (hoje hardcoda `_get_unified_jit`) — ver Q2/Q3.

## Q2 — Próximos passos: Sprints e etapas de otimização JAX GPU

Roadmap priorizado por **ROI medido** (throughput ganho / esforço):

### Sprint O4 — Batched Bucketed + dispatcher correto (PRIORIDADE MÁXIMA)

**Problema**: `simulate_multi_jax_batched` hardcoda `_get_unified_jit` (slow
per-position). Para on-the-fly com 600 pos, isso dá 67k mod/h (5.5× pior que o
loop bucketed single-model de 369k).

**Entregas**:
1. **Dispatcher honra `cfg.jax_strategy` no batched** — quando `bucketed` e
   geometria compartilhada (mesmo `esp` entre modelos), vmap sobre modelos
   reusando os kernels estáticos por bucket.
2. **Caso `esp` variável** (on-the-fly real): agrupar modelos por estrutura de
   buckets idêntica, OU padding-to-max-layers + máscara estática (evita fori_loop
   dinâmico mantendo 1 kernel).
3. **`chunk_size_models`** para evitar OOM no Cenário H (97 GB → tiles de 4-8
   modelos).

**Meta**: on-the-fly 600 pos de 67k → ≥250k mod/h (3.7×); Cenário H deixar de OOM.

**Esforço**: ~1 sprint (3-4 sessões). **Risco**: médio (paridade via gate F1).

### Sprint O5 — Otimizar a propagação per-position (raiz do per-position cost)

**Problema**: mesmo bucketed gasta ~109 µs/pos (forward), dominado pela
recursão TE/TM serial sobre camadas + integração Hankel (201 pts).

**Entregas**:
1. **`lax.scan(unroll=K)`** na recursão TE/TM (já existe parcialmente no Sprint
   O1 #2 — auditar se está ativo no bucketed).
2. **Fundir os 6 casos geométricos** (`lax.switch`) que ainda têm overhead.
3. **Vetorizar a quadratura Hankel** sobre posições (einsum batched em vez de
   loop por posição).
4. **Avaliar `donate_argnums`** para reuso de buffer (reduz alocação).

**Meta**: ≤60 µs/pos (1.8×) → Cenário E ~700k mod/h, **superando Numba**.

**Esforço**: ~1-2 sprints. **Risco**: alto (toca o kernel físico — gate F1
obrigatório).

### Sprint O6 — Multi-stream / pmap + persistent JIT cache

**Entregas**:
1. **`jax.pmap`** se houver 2ª GPU (ou multi-stream numa A6000 via
   `jax.experimental` streams) para on-the-fly em larga escala.
2. **Persistent JIT cache** (`jax.experimental.serialize_executable`) — elimina
   warmup de 5-15s em CI/cron/notebooks.
3. **CUDA Graphs** (XLA 0.7+ auto) — reduz launch overhead no Cenário A.

**Meta**: amortizar warmup; escalar on-the-fly linearmente com GPUs.

**Esforço**: ~1 sprint. **Risco**: baixo.

### Sprint O7 — `complex64` real end-to-end + Tensor Cores

**Problema**: medimos c128 == c64 no forward porque o gargalo era estrutural
(fori_loop), não FP. **Após O4+O5** (kernels eficientes), o c64 deve finalmente
render 1.5-2× (memory bandwidth).

**Entregas**:
1. Re-benchmark c64 após O4/O5 (esperado: ganho real materializa).
2. Avaliar `bfloat16`/TF32 em sub-operações tolerantes (não no path de paridade).

**Esforço**: ~1 sprint. **Risco**: médio (precisão).

### Tabela-resumo do roadmap

| Sprint | Foco | Meta throughput | Esforço | Risco |
|:------:|:-----|:----------------|:-------:|:-----:|
| **O4** | Batched bucketed + dispatcher | on-the-fly 67k→250k | 1 sprint | Médio |
| **O5** | Per-position kernel | E 369k→700k | 1-2 sprints | Alto |
| **O6** | Multi-stream + JIT cache | warmup amortizado | 1 sprint | Baixo |
| **O7** | c64 real + Tensor Cores | +1.5-2× pós-O5 | 1 sprint | Médio |

## Q3 — Otimização para PINNs e geração on-the-fly (alto throughput)

### 3.1 Medições por regime de uso (RTX A6000, batched, c64)

| Caso de uso | Config | Throughput | Status |
|:------------|:-------|-----------:|:-------|
| **PINN collocation** | 256 mod × 1 pos | **33,116,324 mod/h** | ✅ Imbatível |
| **PINN small** | 128 mod × 16 pos | **2,405,419 mod/h** | ✅ Excelente |
| PINN medium | 64 mod × 64 pos | 613,445 mod/h | 🟡 Bom |
| on-the-fly | 32 mod × 600 pos | 66,416 mod/h | ❌ Gargalo |
| on-the-fly big | 128 mod × 600 pos | 67,568 mod/h | ❌ Gargalo (plateau) |

### 3.2 Análise dos dois contextos

#### Contexto A — PINN (Physics-Informed Neural Networks)

PINNs avaliam a física em **pontos de colocação** (poucas posições por amostra,
muitas amostras por batch). Este é o regime onde o batched-unified **já domina**:

- **256 modelos × 1 posição = 33.1M mod/h**. O vmap sobre modelos funde tudo em
  1 kernel XLA massivamente paralelo → GPU saturada.
- Diferenciável: `jax.grad`/`jacfwd` funcionam end-to-end (Sprint 5.1b).
- **Recomendação**: para PINN, **JAX GPU batched + c64 é a escolha definitiva**.
  Nenhuma otimização adicional é necessária para o caso colocação. Numba não
  tem autodiff — JAX é insubstituível aqui.

#### Contexto B — Geração on-the-fly durante treino (600 posições/perfil)

Aqui o batched-unified **colapsa para 67k mod/h** porque cada modelo tem 600
posições e o `fori_loop` unified é serial/ineficiente por posição. **Pior que o
loop bucketed single-model (369k) e que o Numba 4w×16t (~1M).**

Estratégia de otimização (Sprint O4 + O5):
1. **Batched bucketed** (O4): quando `esp` é compartilhado entre modelos do
   batch, vmap sobre modelos com kernels estáticos por bucket → esperado
   ~250-370k mod/h.
2. **Per-position kernel** (O5): reduz o custo de 109 µs/pos → ~60 µs/pos.
3. **Combinado**: on-the-fly 600 pos pode chegar a **300-500k mod/h** — metade
   a paridade com Numba, com o bônus de diferenciabilidade.

### 3.3 Recomendação prática híbrida (hoje, sem código novo)

| Necessidade | Backend recomendado HOJE | Throughput |
|:------------|:-------------------------|-----------:|
| PINN colocação (≤16 pos/amostra) | **JAX GPU batched + c64** | 2.4M–33M mod/h |
| PINN perfil completo (600 pos) | Numba 4w×16t (gerar) + JAX `grad` (loss) | ~1M gen |
| On-the-fly dataset gen (600 pos) | **Numba 4w×16t** | ~1M mod/h |
| Inversão look-ahead (Jacobiano) | **JAX `jacfwd` + c64 + ctx_cache** | autodiff |
| Forward único E (600 pos) | JAX bucketed (369k) ou Numba (389k) | empate |

### 3.4 Caminho para o objetivo "alto throughput universal"

O objetivo (alto throughput em PINN **e** on-the-fly) é alcançável:
- **PINN: JÁ ATINGIDO** (33M mod/h colocação).
- **On-the-fly: requer Sprint O4 (batched bucketed)** para sair de 67k → 250-370k.
  Com O5 (per-position), 300-500k. Isso é metade-a-paridade com Numba, mas com
  diferenciabilidade — viabilizando treino end-to-end com física no loop.

## Q4 — Garantia: tatu.x sempre versão Linux

### 4.1 Problema raiz

`Fortran_Gerador/tatu.x` é **git-tracked** e o histórico de `main` contém a
versão **Mach-O (macOS)**. Operações git (`checkout`, `reset --hard`, worktree
de commit antigo) restauram o binário macOS → quebra paridade Fortran no Linux
(`OSError: Exec format error`).

### 4.2 Solução defense-in-depth (implementada)

1. **Binário Linux ELF commitado na branch de trabalho** (`feat/jax-perf-o2-o3`,
   commit 7ee9dbe) — checkouts normais nesta branch trazem Linux.

2. **Hook `ensure-tatu-linux.sh`** (NOVO) — rede de segurança:
   - Detecta `tatu.x` ausente / Mach-O / corrompido via `file`.
   - Reconstrói automaticamente via `make -C Fortran_Gerador` (gfortran 13.3).
   - Silencioso quando já é ELF Linux; nunca bloqueia (exit 0).
   - Wired em **SessionStart (startup)** em `settings.json` (timeout 60s).

3. **Backup do macOS** preservado em `Fortran_Gerador/tatu.x.macos.bak`.

### 4.3 Validação

| Teste | Resultado |
|:------|:----------|
| tatu.x já ELF → hook silencioso | ✅ exit 0, sem output |
| tatu.x Mach-O plantado → hook reconstrói | ✅ ELF Linux em ~5s |
| Paridade pós-reconstrução (`compare_fortran`) | ✅ 10/10 PASS |
| `settings.json` válido | ✅ JSON parse OK |

**Status Q4: RESOLVIDO** — defense-in-depth garante tatu.x Linux em qualquer
sessão futura, com auto-rebuild se algo reverter o binário.

## Q5 — Recomendação: Opus 4.8 + Ultracode (xhigh) + Workflows?

### 5.1 O que cada componente oferece

| Componente | O que é | Quando ajuda |
|:-----------|:--------|:-------------|
| **Opus 4.8 (1M ctx)** | Modelo mais capaz, contexto 1M tokens | Raciocínio cross-arquivo profundo, manter 46k LOC + docs em contexto |
| **Ultracode (xhigh)** | Effort máximo + workflows por padrão, custo de tokens não é restrição | Tarefas exaustivas, correção máxima, cobertura ampla |
| **Workflows** | Orquestração multi-agente determinística (fan-out, verify adversarial, pipeline) | Decompor + paralelizar + verificar com perspectivas independentes |

### 5.2 Análise para os próximos passos do JAX GPU (Sprints O4-O7)

**Recomendação: SIM, com nuance por sprint.**

#### Onde Opus 4.8 + Ultracode + Workflows AJUDA muito

- **Sprint O5 (per-position kernel)** — toca o kernel físico, exige:
  - Raciocínio sobre 6 casos geométricos × recursão TE/TM × paridade <1e-13.
  - Workflow ideal: fan-out (1 agente por otimização candidata: scan unroll,
    switch fusion, Hankel vectorize) → verificação adversarial de paridade por
    candidato → síntese do que preserva <1e-13. **Aqui o multi-agente com
    verify adversarial vale ouro** (paridade é sagrada; um bug passa
    silenciosamente sem verificação independente).

- **Sprint O4 (batched bucketed)** — design com trade-offs (esp compartilhado vs
  variável, grouping, padding). Workflow: judge panel de N abordagens de design
  → score → síntese. Opus 4.8 1M context mantém todo o `_jax/` (≈6k LOC) +
  histórico de sprints em contexto.

#### Onde é OVERKILL

- **Sprint O6 (JIT cache, CUDA graphs)** — mudanças localizadas, bem
  documentadas na API JAX. Opus 4.8 normal (sem ultracode) basta. Workflow seria
  desperdício de tokens.

- **Ações triviais** (flip de default strategy, ajuste de bench) — edição
  mono-arquivo. Sonnet ou Opus normal.

### 5.3 Recomendação calibrada

| Sprint | Modelo | Ultracode | Workflows | Justificativa |
|:------:|:-------|:---------:|:---------:|:--------------|
| **O4** | Opus 4.8 | xhigh | ✅ Sim | Design com trade-offs + paridade; judge panel + verify |
| **O5** | Opus 4.8 | xhigh | ✅ **Essencial** | Kernel físico; verify adversarial de paridade por candidato |
| **O6** | Opus 4.8 | Não | Não | Mudanças localizadas, API bem documentada |
| **O7** | Opus 4.8 | xhigh | 🟡 Parcial | Precisão c64; verify de paridade nos casos limítrofes |

**Razão fundamental para usar workflows nos sprints de kernel (O4/O5)**: a
**paridade Fortran <1e-13 é inviolável** e bugs de paridade são silenciosos
(passam nos smoke tests, falham só em modelos edge-case alta-ρ). O padrão
**adversarial-verify** dos workflows (N céticos independentes tentando refutar
cada finding de paridade) é exatamente a salvaguarda que o projeto precisa.

**Mas atenção ao custo**: workflows podem gastar centenas de milhares de tokens.
Use-os **apenas nos sprints O4/O5** (alto risco, alto valor). Para O6 e ações
mecânicas, Opus 4.8 normal é mais econômico e suficiente.

### 5.4 Padrão de workflow recomendado para O5 (exemplo concreto)

```
Fase 1 (Design):    judge panel — 3 abordagens de per-position kernel
                    (scan-unroll / switch-fusion / Hankel-einsum), scored
Fase 2 (Implement): pipeline — cada otimização em worktree isolada
Fase 3 (Verify):    para cada otimização, 3 céticos adversariais tentam
                    refutar "paridade <1e-13 preservada" em 7 modelos canônicos
                    + alta-ρ (10⁶ Ω·m). Kill se ≥2 refutam.
Fase 4 (Synthesize): merge só das otimizações que sobrevivem à verificação
```

## Conclusões Gerais

1. **O teto de 42.8k mod/h era artefato de implementação, não limite do JAX.**
   `strategy="unified"` (6.9× mais lenta que `bucketed`) + `batched` hardcodando
   unified. Com bucketed, **Cenário E faz 369k mod/h — empata Numba 4w×16t**.

2. **JAX GPU já domina o regime PINN colocação: 33.1M mod/h** (256 mod × 1 pos).
   Para PINN, é a escolha definitiva (diferenciável, GPU-saturada).

3. **O único gargalo remanescente é on-the-fly com 600 posições (67k mod/h)** —
   requer Sprint O4 (batched bucketed) para chegar a 250-370k.

4. **Multi-freq é eficiente na GPU** (8 freqs = 2× de 1 freq) — favorece
   cenários multi-frequência reais (ARC/PeriScope dual-freq).

5. **Q4 resolvido**: hook `ensure-tatu-linux.sh` garante tatu.x Linux com
   auto-rebuild.

6. **Q5**: usar Opus 4.8 + Ultracode + Workflows nos **sprints de kernel
   (O4/O5)** pela salvaguarda de paridade adversarial; Opus normal para O6 e
   tarefas mecânicas.
