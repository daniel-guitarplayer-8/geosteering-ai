## 1. Pergunta central (op 1): por que o JAX GPU é mais rápido no CLI que no SM?

### 1.1 Causa-raiz — tempo de vida do processo (distinta do cold-compile do PR #59)

| Aspecto | CLI | SM (antes desta entrega) |
|:--|:--|:--|
| Processo JAX | **1 processo de longa duração** (in-process) | **subprocesso `spawn` NOVO a cada run** |
| Onde | `geosteering_ai/cli/_exec.py` (in-process) | `gui/services/base.py::_pool_run` → `with ProcessPoolExecutor(spawn) as pool:` (criado/destruído por chamada) |
| `CUDA/cuDNN init` | pago **1×**, quente entre `--repeat` | **re-pago todo run** (~10-15 s) |
| Reload do executável do cache XLA de disco | 1× | **todo run** |
| Cache JIT em-processo (`_BUCKET_JIT_CACHE`) | quente entre runs | descartado a cada run |

O SM roda o JAX num **subprocesso isolado** por uma razão correta: importar JAX/inicializar
CUDA numa `QThread` estoura o TLS estático (`_dl_allocate_tls_init`) — o processo da GUI
**nunca** pode importar JAX (TLS-safe). O problema não era o isolamento, e sim o subprocesso
ser **efêmero** (destruído após cada run), forçando o re-init.

### 1.2 Evidência empírica (caminho FIEL ao SM, config do report: 1000 modelos, 20 camadas, 600 pos)

| Experimento | Tempo | Interpretação |
|:--|--:|:--|
| Subprocesso fresco (efêmero), cache de disco quente | **29.2 s** | custo por-run do SM ANTES |
| Tudo quente no MESMO processo (= worker persistente) | **12.4 s** | custo por-run com o fix |
| 2ª chamada in-process (probe n=64) | 11.8 s | compute puro é trivial |
| Numba (baseline) | 36.8 s | — |

➡️ **O cache de disco (PR #59) só elimina o *compile* XLA único (~145 s na 1ª vez de cada
forma); ele NÃO elimina o init-CUDA + reload por-processo. Só um worker persistente elimina.**

### 1.3 Fix — worker JAX persistente (PR #60)

Um único subprocesso `spawn` **vivo e reutilizado** entre runs (espelha o pool persistente
do Numba em `simulation/_workers.py`).

- `base.py`: `_JAX_POOL` singleton + `_JAX_POOL_LOCK` + `_acquire_jax_pool()` (registra
  `atexit` 1×) + `release_jax_pool()` (público, idempotente) + `_pool_run_persistent()`
  (self-heal). O `_pool_run` **efêmero permanece intocado** (default seguro + contrato dos
  testes diretos). Opt-in via `persistent=True`.
- `simulation_service.py`: o ramo `jax/auto` passa `persistent=True`. O **Numba in-thread
  fica intocado** (QThread + cancel/pause cooperativos).
- Teardown defense-in-depth: `SM_MainWindow.closeEvent` + `app.aboutToQuit` + `atexit` →
  **sem processo GPU órfão** (contexto CUDA + VRAM liberados). Validado: `child_alive_after=False`.

**Resultado medido (A6000): SM 2ª+ sim = 12.7 s — ~3× mais rápido que o Numba (36.8 s).**

### 1.4 Robustez (revisão adversarial — race teardown × in-flight)

`_discard_failed_pool(pool)` distingue:
- **release externo** (encerramento trocou/nulou o singleton): ABORTA limpo
  (`_SHUTDOWN_CANCEL_MSG`), sem re-rodar/recriar pool no shutdown;
- **crash do nosso pool** (`BrokenProcessPool`): self-heal — descarta o pool ESPECÍFICO e
  re-tenta 1×; 2ª falha → `_SUBPROCESS_DIED_MSG`.

Também trata `CancelledError` (future pendente cancelado por `cancel_futures=True`) → sem
erro vazio na UI. Exceções do PRÓPRIO `fn` (ValueError/RuntimeError da sim) **propagam
intactas** (só `BrokenProcessPool`/`CancelledError` são tratados como falha de infra).

## 2. Como obter throughput > Numba no SM (op 2 — doc `docs/reference/sm_jax_persistent_worker.md`)

1. **Backend** = `jax` ou `auto`.
2. Aceite o **cold-start** da 1ª sim de cada *forma* nova (compila XLA, ~min, único, cacheado).
3. **Sature a GPU**: **N grande** (≥256-1000) · **`n_layers fixo`** (1 grupo) · **`Geometrias (K)`
   pequeno** (knob do PR #59 — menos compilações). ρₕ/ρᵥ/λ continuam por-modelo.
4. **Mantenha o app aberto** entre sims (worker quente → 2ª+ sim ~12 s).
5. **Pré-aqueça** (opcional): boot warmup (Preferências) ou `geosteering-warmup --jax`.

| Cenário | Backend recomendado |
|:--|:--|
| 1 sim avulsa, N pequeno (< ~256) | **Numba** (sem cold-start; GPU subocupada) |
| Muitas sims na sessão, N grande | **JAX** (worker persistente: 2ª+ sim ~12 s) |
| Lote único enorme, máxima ocupação | **JAX** + `n_layers fixo` + K pequeno + boot warmup |

## 3. Boot warmup opt-in (op 3 — PR #61)

Pré-aquece o worker persistente no boot, tirando o cold-start XLA da 1ª sim.

- `geosteering_ai/gui/services/jax_warmup_service.py`: `JaxWarmupService.warmup()` guarda via
  `importlib.util.find_spec("jax")` (**não importa jax na GUI** — TLS-safe) e submete a forma
  canônica ao **MESMO** pool persistente de `base.py`.
- `apps/sim_manager/boot_warmup.py`: `schedule_boot_warmup(ctx, window, enabled)` — no-op se
  desligado/sem jax; feedback transitório no status bar; falha **nunca alarma**; guard
  anti-double-schedule.
- `app.py`: lê a pref `jax_boot_warmup` (**default False** — opt-in) após `window.show()`.
  Preferências: chave + `_safe_bool` round-trip + QCheckBox.

**Default OFF**: CI/Studio/usuários Numba-only **não pagam** GPU no boot.

## 4. `geosteering-warmup --jax/--jax-only/--jax-auto` (op 4 — PR #62)

Flags **aditivas** que aquecem a forma canônica do SM no cache XLA de disco (compartilhado
com o worker persistente e o `geosteering-cli`). Reusa `warmup_jax_simulator` (**não toca o
kernel**). Default **inalterado** (só Numba — contrato v2.44 byte-a-byte; o CI atual segue
numba-only; JAX é opt-in).

- `--jax` (Numba+JAX) · `--jax-only` (só JAX) · `--jax-auto` (JAX só com GPU visível — no-op +
  exit 0 no CI CPU). Mutuamente exclusivos (exit 2).
- Taxonomia: backend ausente → warning + exit 0 (observabilidade, não gating, v2.44); falha
  real → `logger.error` + exit 1.

> ⚠️ **Descoberta empírica:** para a GPU use **`export JAX_PLATFORMS=cuda,cpu`** (cuda p/
> compute **+ cpu** p/ o `jax.pure_callback` do kernel — bridge Numba host). Só `cuda` QUEBRA
> (`failed to find a local CPU device`). O **SM não seta `JAX_PLATFORMS`** (auto cuda+cpu),
> então o worker persistente já roda na GPU corretamente. Validado: `--jax-auto` (cuda,cpu) →
> **39 buckets, persisted=True, OK em 105 s**.
