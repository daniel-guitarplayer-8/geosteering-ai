## Sumário Executivo

O projeto Geosteering AI é um sistema de **inversão 1D de resistividade via
Deep Learning** para geosteering de poços petrolíferos. Em termos operacionais:
dado um conjunto de medições eletromagnéticas (leituras LWD de instrumento no
fundo do poço), o sistema estima as resistividades das camadas de rocha ao redor
do poço, em tempo real.

O estado atual (v2.40.2) marca a **conclusão da fase de fundação de software**.
A infraestrutura de testes, CI/CD, automação via MCP Colab, mixed precision,
tf.data parametrizado e governança documental estão completos. O projeto entra
agora na **fase de treinamento e validação científica** — o trabalho técnico que
transforma a arquitetura em produto.

Este relatório mapeia os 15 itens de backlog em ordem de prioridade, explica
o processo de execução de cada sprint, e traça a sequência temporal recomendada.

## Parte I — Estado Atual do Projeto

### 1.1 O Que Está Implementado

O projeto tem **três subsistemas principais** completamente implementados:

#### Subsistema A — Simulador Físico (Fortran + Python)

O coração científico. O simulador Fortran `tatu.x` (6.859 LOC em 6 módulos F08)
calcula o tensor eletromagnético 3×3 complexo a partir de parâmetros geológicos:
resistividades horizontais e verticais por camada, espessuras, ângulo de dip,
espaçamento transmissor-receptor e frequências de operação. É a fonte de todos
os dados de treinamento e a referência de validação de paridade.

O simulador Python (JAX + Numba) é um espelho em desenvolvimento que permitirá
simulação em GPU. A **paridade Fortran <1e-12 é invariante sagrada** — nenhum
commit pode quebrá-la. 10 modelos canônicos geológicos são testados a cada CI.

Features implementadas no Fortran:
- Multi-TR (múltiplos espaçamentos transmissor-receptor em uma única chamada)
- Frequências arbitrárias (flag `use_arbitrary_freq`, até 16 frequências)
- Antenas inclinadas (flag `use_tilted_antennas`, β e φ configuráveis)
- Compensação midpoint (flag `use_compensation`, atenuação e phase difference)
- Filtro Hankel adaptativo (3 opções: Werthmuller 201pt, Kong 61pt, Anderson 801pt)

#### Subsistema B — Pipeline de Deep Learning

Módulo completo para treinamento, validação e inferência de modelos neurais:

| Componente | Detalhe |

| Arquiteturas | 48 redes em 9 famílias (ResNet, TCN, WaveNet, ModernTCN, INN, ResNeXt, etc.) |
| Funções de perda | 26 (4 categorias: MSE, física, anisotropia, incerteza) |
| Tipos de ruído | 34 implementados (gaussiano, curriculum 3-fases, físicos parciais) |

#### `C-surrogate-train` — SurrogateNet Training em GPU

| Campo | Valor |
|:--|:--|
| **Trilha** | C — Deep Learning |
| **Esforço** | 4-6h prep local + 4-8h GPU em Colab Pro+ |
| **Status** | CANDIDATE |
| **Dependência** | Desbloqueado por v2.40 (fix mixed precision D5) |
| **Modelo Claude** | Opus 4.7 (effort=extra-high) para prep; Sonnet 4.6 para commits |

**O Que É**

Treinar as redes SurrogateNet — substitutos neurais rápidos do simulador
Fortran — em Google Colab Pro+ com GPU A100, usando mixed precision (fp16) e
compilação XLA. Duas arquiteturas serão treinadas:

- **SurrogateNet TCN:** Temporal Convolutional Network, 127M parâmetros.
  Especializada em capturar dependências de longo alcance no perfil de poço
  (camadas espessas, variações graduais de resistividade).
- **SurrogateNet v2 ModernTCN:** Arquitetura moderna com atenção multi-escala,
  204M parâmetros. Maior capacidade representacional para perfis complexos
  (contrastes de resistividade >100×).

**Por Que Agora**

A Sprint v2.40 corrigiu o bug crítico de mixed precision: `set_global_policy()`
era chamado **depois** de `build_model()`, deixando todas as camadas em fp32
mesmo com `use_mixed_precision=True`. Com o bug corrigido via
`build_model_with_mp_policy(config)`, o treinamento em fp16 está desbloqueado
e é ~1.5-2× mais rápido em A100.

Os templates de notebook foram implementados em v2.40 e ajustados em v2.40.1
(com `git rev-parse HEAD` para rastreabilidade imutável do run de treinamento).

**Impacto Esperado**

SurrogateNet treinada deve ser ≥10.000× mais rápida que o simulador Fortran em
modo de inferência. Isso habilita:
1. Treinamento do modelo de inversão com datasets muito maiores (não limitado
   pelo tempo de simulação)
2. Inferência em tempo real (<10ms por poço) em campo
3. Otimização bayesiana de hiperparâmetros com Optuna (centenas de trials)
4. Artigo científico com comparação quantitativa de performance

#### `B-flat-prange-default` — Promover FLAT prange como Default

| Campo | Valor |
|:--|:--|
| **Trilha** | B — Performance |
| **Esforço** | 1-2h |
| **Status** | BACKLOG |
| **Dependência** | Re-bench 5 runs Cenário E (investigar root-cause da regressão F10) |

**O Que É**

Mudar `use_flat_prange: bool = False` para `True` em `geosteering_ai/config.py`.
O FLAT prange (implementado em v2.22) reorganiza o loop interno do simulador de
nested loops para loop 4D flat (`nTR × nAng × n_pos × nf`), eliminando a
serialização que ocorre quando loops Numba prange são aninhados.

Benchmarks de v2.22 mostraram:
- Cenário B: +11% throughput
- Cenário F: +9% throughput
- Cenário E: sem regressão

**Por Que Está em BACKLOG e Não Executado**

A Sprint F10 (antes de v2.30) registrou uma **regressão catastrófica** ao
promover como default: 2s→33s em Cenário E. A hipótese atual é que foi causada
por **Numba cache invalidation** — a mudança de assinatura do loop forçou
recompilação de todos os kernels JIT durante o benchmark, não por defeito na
lógica de tiling. O cache limpo não foi considerado na medição original.

**Como Será Executado**

```bash
# 1. Limpar cache Numba completamente
python -c "from geosteering_ai.simulation import release_numba_cache; release_numba_cache()"

# 2. Benchmark 5 runs com cache limpo DEPOIS do warmup JIT
geosteering-warmup --verbose
geosteering-cli benchmark --scenario E --n 1000 --workers 4 --threads 2

# 3. Se mediana 5 runs sem regressão vs baseline:
#    Mudar default em config.py + commitar
# 4. Se houver regressão: investigar mais fundo (nova sprint B-flat-debug)
```

É a sprint mais curta do backlog. Se a hipótese for confirmada, é uma vitória
rápida de +10-20% throughput com 1-2h de trabalho.

#### `E-api-simulate` — Endpoint POST /simulate

| Campo | Valor |
|:--|:--|
| **Trilha** | E — Infrastructure |
| **Esforço** | 6-8h |
| **Status** | BACKLOG |
| **Dependência** | Desbloqueado por v2.39 (API REST MVP) |

**O Que É**

Adicionar endpoint `POST /simulate` à API REST FastAPI existente. Recebe
parâmetros geológicos em JSON e retorna o tensor EM calculado pelo simulador
JAX/Numba, sem necessidade de CLI local ou instalação do pacote.

**Schema do Endpoint**

```python
# Request — Pydantic v2
class SimulateRequest(BaseModel):
    rho_h: list[float]          # resistividade horizontal por camada (Ω·m)
    rho_v: list[float]          # resistividade vertical — anisotropia TIV
    esp: list[float]            # espessuras das camadas (metros)
    positions_z: list[float]    # posições z do instrumento no poço
    tr_spacings_m: list[float]  # espaçamentos transmissor-receptor
    dip_degs: list[float]       # ângulos de dip da formação
    frequencies_hz: list[float] # frequências de operação (Hz)
    backend: Literal["numba", "jax"] = "numba"

# Response
class SimulateResponse(BaseModel):
    H_tensor: list              # tensor EM 3×3 complexo (JSON-serialized)
    shape: list[int]            # [n_pos, n_tr, n_dip, n_freq, 3, 3]
    backend_used: str
    duration_ms: float
```

**Desafios Técnicos**

1. **Schema 4D:** Validators Pydantic para garantir consistência de dimensões
   (ex: `len(rho_h) == len(rho_v) == len(esp)+1`)
2. **Timeout:** Simulações longas podem exceder 30s. Configurar timeout
   FastAPI + retornar 408 Request Timeout com estimativa de duração
3. **Payload grande:** Tensor 4D complexo pode ser vários MB. Configurar
   compressão gzip no response via `GZipMiddleware`
4. **Paridade garantida:** `POST /simulate` deve retornar resultado idêntico
   à chamada direta de `simulate_multi()` (<1e-12 para Numba, <1e-10 para JAX)

**Impacto**

Habilita automação completa de ciclos simulação→dataset→treinamento via scripts
Python ou notebooks Colab sem instalação local. Abre caminho para clientes
externos (web app, SDK Python) que consumam o simulador como serviço.

#### `C-noise-35` — Catálogo de Ruído 35 Tipos

| Campo | Valor |
|:--|:--|
| **Trilha** | C — Deep Learning |
| **Esforço** | 8-10h |
| **Status** | BACKLOG |
| **Dependência** | Nenhuma |

**O Que É**

Expansão de `geosteering_ai/noise/functions.py` com tipos adicionais de ruído
físico realista para instrumentos LWD. Hoje existem 34 tipos implementados.
Esta sprint adiciona os tipos que simulam condições físicas específicas do poço.

**Por Que Importa**

Ruído realista é fundamental para generalização do modelo. Uma rede treinada
apenas com ruído gaussiano falha em campo, onde existem: eccentricity
(descentramento da antena no poço), shoulder bed effects (influência de camadas
adjacentes finas), e spike noise (picos impulsivos por vibração mecânica). Cada
tipo adicionado aumenta a robustez da inversão em condições reais de LWD.

**Tipos de Ruído por Categoria**

*Mecânicos (4 tipos):*

| Tipo | Descrição física |
|:--|:--|
| `eccentricity` | Descentramento da antena — distorce a geometria de medição. Antena fora do centro do poço vê assimetria no campo EM. Modelado como perturbação radial aleatória na posição da antena. |
| `mandrel_effect` | Efeito do mandrel metálico na leitura EM. O mandrel condutivo cria uma resposta parasita que varia com frequência e temperatura. |
| `borehole_rugosity` | Rugosidade da parede do poço — cavidades irregulares criam variações de alta frequência no campo EM. Modelado como ruído correlacionado com `z_obs`. |
| `cable_stretch` | Estiramento de cabo em medições LWD — offset temporal entre amostras consecutivas. Resulta em desalinhamento de profundidade. |

*De Formação (5 tipos):*

| Tipo | Descrição física |
|:--|:--|
| `formation_invasion_filtrate` | Filtrado de lama de perfuração penetra a formação porosa, criando zona de invasão com resistividade intermediária entre lama e formação virgem. |
| `formation_invasion_nonfiltrate` | Invasão não-filtrada (formação de baixa permeabilidade): o perfil radial de resistividade tem transição mais abrupta. |
| `shoulder_bed_effect` | Camadas adjacentes (shoulders) influenciam a leitura de instrumentos com profundidade de investigação rasa. Medições em interfaces de camada sofrem efeito de shoulder bed. |
| `thin_bed_aliasing` | Camadas mais finas que a resolução vertical do instrumento aparecem como gradiente suave em vez de contraste abrupto. |
| `anisotropy_perturbation` | Variações aleatórias na razão de anisotropia TIV (ρ_v/ρ_h) entre profundidades adjacentes — simulam heterogeneidade sub-vertical. |

*De Instrumentação (6 tipos):*

| Tipo | Descrição |
|:--|:--|
| `temperature_drift` | Deriva de temperatura ao longo do poço — instrumentos LWD operam em altas temperaturas (>150°C em poços profundos). |
| `gain_imbalance` | Desbalanço de ganho entre canais de medição — canais Hxx, Hyy, Hzz podem ter ganhos ligeiramente diferentes. |
| `phase_offset` | Offset de fase entre transmissor e receptor — variações de calibração entre passagens. |
| `sampling_jitter` | Variações no tempo de amostragem — o instrumento não amostra exatamente nas profundidades nominais. |
| `depth_mismatch` | Desalinhamento de profundidade entre passagens distintas (entrada e saída do poço). |
| `spike_noise` | Picos de ruído impulsivo por vibração mecânica, cavitação do bit, ou interferência elétrica. |

*Geológicos (5 tipos):*

| Tipo | Descrição |
|:--|:--|
| `fracture_noise` | Fraturas que alteram permeabilidade e fluxo — criam anomalias locais de resistividade. |
| `fault_proximity` | Proximidade de falhas geológicas — zona de dano reduz resistividade localmente. |
| `karst_features` | Cavidades kársticas — anomalias de resistividade muito alta (ar) ou baixa (água). |
| `washout` | Cavidades na parede do poço — instrumento perde contato com a formação. |
| `salt_proximity` | Alta condutividade de depósitos de sal — mascara a resposta da formação de interesse. |

**Invariância Obrigatória**

Para todos os 35 tipos: ruído **nunca** afeta colunas de metadados (`z_obs`,
`theta`, `freq`). Validado por asserts em `tests/test_noise.py`.

**Curriculum Expandido**

```python
# 3 fases do curriculum de treinamento
Fase 1 (0-30%):   apenas ruídos gaussianos suaves (σ < 0.01)
Fase 2 (30-70%):  + ruídos físicos moderados (invasion, eccentricity, temperature_drift)
Fase 3 (70-100%): todos os 35 tipos, incluindo spikes e geológicos
```

#### `B-numba-O3` — Selective Fastmath + Pairwise Summation

| Campo | Valor |
|:--|:--|
| **Trilha** | B — Performance |
| **Esforço** | 4-6h |
| **Status** | BACKLOG |
| **Contexto técnico** | `docs/reference/analise_cenarios_otimizacao_simulador_numba.md` |

**O Que É**

Aplicar 5 flags fastmath "seguras" nos kernels Numba mais críticos. As flags
seguras não causam perda de precisão em modelos geofísicos típicos:

```python
# Flags SEGURAS para kernels EM (excluem reassoc e nsz)
SAFE_FASTMATH = {'nnan', 'ninf', 'contract', 'arcp', 'afn'}

@njit(fastmath=SAFE_FASTMATH)
def _hmd_tiv_propagation(rho_h, rho_v, esp, freq, ...):
    ...
```

Flags excluídas e por quê:
- `reassoc`: permite reordenar somas — causa cancelamento catastrófico em
  modelos de alta resistividade (ρ > 1kΩ·m, ex: carbonato seco, evaporita)
- `nsz`: signed zero — altera comportamento de divisão em bordas de camada

Também inclui **pairwise summation** manual em 7 reduções `np.sum` em `hmd_tiv`
(linhas 665-671): substituição de acumulação linear por soma hierárquica em pares,
reduzindo erro numérico acumulado de O(N·ε) para O(log(N)·ε).

**Gate de Paridade**

3 modelos canônicos alta-ρ como gate obrigatório:
- `carbonato_seco_5c`: ρ = 5kΩ·m (diagnóstico de zona paga)
- `evaporita_3c`: ρ = 100kΩ·m (selante de sal)
- `gas_seco_8c`: ρ = 10kΩ·m (reservatório de gás)

Paridade <1e-12 vs Fortran deve ser preservada nos 3 modelos.

**Ganho esperado:** +6-10% throughput global em Cenários E e F.

#### `C-inversion-alt` — Métodos Alternativos de Inversão

| Campo | Valor |
|:--|:--|
| **Trilha** | C — Deep Learning |
| **Esforço** | 1 semana |
| **Status** | BACKLOG |
| **Dependência** | Pesquisa científica (literatura 2024-2026) |

**O Que É**

Implementar 3 métodos clássicos de inversão geofísica como baseline científico
para comparação quantitativa no artigo:

**Occam Regularizado (Constable et al., 1987)**

Busca pelo modelo mais simples (mínima rugosidade de resistividade) consistente
com os dados observados. Formula como minimização de:

```
min_m  ||W_d(d - F(m))||² + λ ||∂m/∂z||²

onde:
  d     = dados medidos (tensor EM)
  F(m)  = simulador Fortran aplicado ao modelo m
  λ     = parâmetro de regularização (busca de Occam)
  ∂m/∂z = derivada da resistividade em profundidade (rugosidade)
```

**Look-Up Table (LUT)**

Biblioteca pré-computada de respostas EM para grid discreto de modelos
geológicos. Inversão via interpolação trilinear no espaço (ρ_h, ρ_v, z_bed).
Muito rápido em produção, mas requer memória substancial e não generaliza além
do grid pré-computado.

**Tikhonov Regularization**

Generalização do Occam que permite operadores de regularização L mais gerais:
`min_m ||d - F(m)||² + λ||Lm||²`. Implementar 3 variantes de L (identidade,
primeira derivada, segunda derivada) e comparar com Occam.

**Objetivo Final**

Tabela comparativa para artigo científico:

| Método | MAE ρ_h | MAE ρ_v | DTB (p50) | Tempo/poço | Robusto ao ruído |
|:--|:--|:--|:--|:--|:--|
| Occam | ? | ? | ? | ~5min | Médio |
| LUT | ? | ? | ? | <1ms | Baixo |
| Tikhonov | ? | ? | ? | ~2min | Alto |
