## 1. Causas-raiz do warmup (investigação)

O run de geração 30k×600 em processo FRIO custou **1962s vs 491s morno** (~1471s de
compilação XLA). Três causas confirmadas lendo o código:

| # | Causa-raiz | Evidência |
|:-:|:-----------|:----------|
| 1 | **Warmup não toca o kernel GPU** | `_warmup_numba_tier2_sync` usa `use_native_dipoles=False` ([cli/_main.py:146](../../geosteering_ai/cli/_main.py#L146)) → aquece só Numba `pure_callback`, nunca `simulate_multi_jax_batched`, nunca o `_BUCKET_JIT_CACHE` |
| 2 | **Cache persistente sub-configurado** | `jax_compilation_cache_dir` setado, mas `jax_persistent_cache_min_compile_time_secs` NÃO (default ~1.0s → compiles rápidos não persistem); dir em `/tmp` (efêmero) |
| 3 | **Shape-explosion** | `_get_bucket_jit` recompila por SHAPE de `z_bucket` (posições-por-bucket variam com a geometria) → data-gen multi-geometria aleatória não é cache-cobrível cross-run |

## 2. Soluções construídas

### A — Warmup JAX-nativo ([_jax/warmup.py](../../geosteering_ai/simulation/_jax/warmup.py))
`warmup_jax_simulator(...)` + `warmup_jax_simulator_from_config(cfg)`: dispara um
`simulate_multi_jax_batched` tiny SINTÉTICO (modelo homogêneo → caminho bucketed de
produção, `use_native_dipoles=True`) → compila os buckets + popula o cache de disco.
**Decisão**: dispatch REAL (não AOT `.lower().compile()`) → HLO bit-idêntico ao de
produção → **cache-hit garantido** (provado pelo teste de zero-recompilação).

### B — Hardening do cache persistente ([_jax/__init__.py](../../geosteering_ai/simulation/_jax/__init__.py))
- `jax.config.update("jax_persistent_cache_min_compile_time_secs", 0.0)` → **todo**
  compile de bucket persiste (version-guarded).
- Dir default ESTÁVEL `~/.cache/geosteering/jax_compilation_cache` (sobrevive reboot)
  com fallback `$TMPDIR` (gracioso OSError); respeita override `JAX_COMPILATION_CACHE_DIR`.

### C — CLI ([cli/warmup.py](../../geosteering_ai/cli/warmup.py)) + CI
- `geosteering-warmup --jax`/`--gpu`/`--auto` + overrides (`--jax-n-pos/--jax-dtype/...`).
- **Lift de `JAX_PLATFORMS`**: o módulo força `cpu` (para o Numba warmup); quando `--jax`/
  GPU é pedido, LIBERA a GPU removendo o force-CPU default — **só se o usuário não setou
  explicitamente** (`_USER_JAX_PLATFORMS` capturado antes do `setdefault`). Lógica extraída
  em `_resolve_jax_warmup` (pura, testada por tabela-verdade).
- CI: `geosteering-warmup --verbose --auto` (runner CPU → JAX warmup auto-skip).

## 4. Deliverable D — Shape-explosion: análise + design de flag (NÃO implementado)

### Mecanismo
No caminho batched bucketed (`_forward_config_buckets_over_models` →
`_get_bucket_jit(...)` chamado com `z_bucket = positions[indices]`), o `jax.jit`
recompila por SHAPE de entrada. Como `indices` (posições por bucket `(ct,cr)`) depende da
geometria, geometrias diferentes → comprimentos de `z_bucket` diferentes → compilações
XLA distintas.

| Caso de uso | Comportamento | Warmable? |
|:------------|:--------------|:----------|
| Inversão / inferência (1 grid fixo) | #shapes limitado | **SIM** (warmup A + cache B) |
| Data-gen multi-geometria aleatória | shapes distintos por run | **NÃO** cross-run (cache MISS p/ geometrias novas) |

### Mitigação proposta (FUTURA, gated, NÃO implementada)
**Shape-bucketing**: padding de `z_bucket` ao próximo múltiplo de N (ex.: 32) → #shapes
distintos limitado a `n_pos/N` → compilações bounded + cache-hits cross-geometria.

```python
# DESIGN da flag futura (NÃO implementada neste sprint):
SimulationConfig.jax_pad_buckets_to: Optional[int] = None   # None = bit-idêntico (atual)
# kernel: pad z_bucket → ceil(len/N)*N; roda; slice [:len] ANTES do scatter
#   → saída bit-idêntica às linhas reais (linhas padded descartadas)
```

**Risco** (por que NÃO neste sprint): toca o KERNEL QUENTE
(`_forward_pure_jax_bucketed_impl`/`_forward_config_buckets_over_models`) → risco de
**throughput** (<N posições padded/bucket de compute extra) + **paridade** (qualquer
off-by-one no slice-antes-do-scatter corrompe `H_out`). **Exigiria** gate de paridade
`<1e-13` + gate DG aprovados na A6000 ANTES de qualquer default-flip. Decisão do usuário:
escopo conservador → documentar, não implementar.
