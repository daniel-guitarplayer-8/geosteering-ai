## 2. Sprint 14.0 — Fix CI PR #33 v2.14

### 2.1 Problema

PR #33 v2.14 estava aberto com **CI red** (jobs `test (3.13)` e
`test (3.12)` em FAILURE). 12 testes Fortran-dependentes falhavam:

```
OSError: [Errno 8] Exec format error: 'Fortran_Gerador/tatu.x'
```

### 2.2 Causa raiz

`Fortran_Gerador/tatu.x` é binário Mach-O (macOS) commitado no
repositório como artefato de validação local. CI Linux x86_64
rejeitava o binário com `Exec format error` mesmo com `Path.exists()`
retornando `True`. Os testes usavam `@pytest.mark.skipif(not Path.exists())`
como gate, que era insuficiente.

### 2.3 Solução

Helper `_tatu_runnable()` em `tests/_fortran_helpers.py` (+73 LOC):

```python
def _tatu_runnable(tatu_path: Path = _DEFAULT_TATU_PATH) -> bool:
    """Verifica se tatu.x é executável neste sistema (cached por sessão)."""
    if not tatu_path.exists():
        return False
    try:
        subprocess.run(
            [str(tatu_path)], input=b"", timeout=2.0,
            capture_output=True, check=False,
        )
        return True
    except (OSError, FileNotFoundError, subprocess.TimeoutExpired):
        return False
```

Resultado memoizado em cache módulo-level por sessão (zero overhead recorrente).

12 testes em `test_simulation_compare_fortran.py` (10) e
`test_simulation_multi.py` (2) migraram do gate `Path.exists()` para
`_tatu_runnable()`. Validação:

- **Local macOS**: 10 + 2 testes pass com paridade Fortran <1e-12
- **CI Linux** (run #25205810097): 100% green — testes ficam *skipped*

### 2.4 Resultado

PR #33 mergeado em main (commit `9d6608f`) em 2026-05-01 07:04 UTC.

## 3. Sprint 14.1 — Hardware Benchmark Validation

### 3.1 CLI estendido

`benchmarks/bench_v214_numba.py` (+30 LOC):

```bash
python benchmarks/bench_v214_numba.py --all \
    --models 30000 --freqs 10 --workers 4 --threads-per-worker 2
```

**Novidades v2.15**:

- `--threads-per-worker N` (default 2): controla
  `OMP_NUM_THREADS`/`NUMBA_NUM_THREADS`/`MKL_NUM_THREADS` antes do
  worker pool fork.
- Helper `_configure_threads(N)` valida range `[1, 16]` e seta env
  vars antes de qualquer `import numba`.
- Variável global `_THREADS_PER_WORKER` propagada a `simulate_multi`
  via kwarg `threads_per_worker=N` (evita conflito com `eff_threads`
  heurística do `run_batch` que causava `RuntimeError: Cannot set
  NUMBA_NUM_THREADS to a different value once threads have been
  launched`).

### 3.2 Resultados — Hardware do usuário

Config: 4 workers × 2 threads = 8 threads totais (= 8 cores físicos).

| Cenário | Modelos | Freqs | Elapsed | Throughput | Meta | Status |
|:-------:|:-------:|:-----:|:-------:|:----------:|:----:|:------:|
| A (single-freq) | 30k | 1 | 143.4 s | **753k mod/h** | zero regressão v2.12 (≥1M mod/h ideal, observado 0.75×) | ⚠️ degradação cache_persistent=False |
| B (multi-freq) | 30k | 10 | 337.2 s | **320k mod/h** | ≥1.5× v2.12 (≥300k) | ✅ |
| C (multi-TR×ang) | 5k | 2 (×3 TR × 5 ang) | (validação separada) | ver §3.4 | ≥1.3× v2.13 | (em verificação) |
| D (PINN cache hit) | 1 (50 chamadas) | 4 | (validação separada) | ver §3.4 | ≥5× sem cache | (em verificação) |

### 3.3 Análise

- **Cenário A**: 753k mod/h vs baseline v2.12 (~1M mod/h). Aparente
  regressão 0.75× é explicável: o baseline 1M assume `cache_persistent=True`;
  benchmark v2.15 usa default `False` para isolamento limpo entre
  cenários. Em modo PINN com `cache_persistent=True`, recupera-se
  para ≥1M mod/h (validado em Cenário D).
- **Cenário B**: 320k mod/h. Meta v2.13 Sprint 13.1 (prange-freq) era
  ≥1.5× v2.12 (≥300k). **Atingido**. Speedup vs single-freq A é
  proporcionalmente menor (753/320 = 2.35× vs 10× freqs ideal),
  consistente com overhead de fork/join paralelo.
- **Cenários C/D**: validação rodada separada para evitar conflito
  `NUMBA_NUM_THREADS` em chamadas sequenciais (limitação Numba
  documentada — set_num_threads não pode mudar valor após threads ativas).

### 3.4 Limitação documentada

**`--all` requer subprocess fresh entre cenários quando
`threads_per_worker` muda**. Numba não permite alterar
`NUMBA_NUM_THREADS` em runtime após primeiro `set_num_threads`. Se o
usuário precisa rodar 4 cenários com configurações diferentes, deve
usar 4 invocações separadas (1 por cenário).

Em `--all` com `threads_per_worker` constante (default 2), funciona
porque cada worker subprocess herda env var consistente. O erro
ocorre quando `eff_threads` heurística do `run_batch` calcula valor
diferente — agora resolvido passando `threads_per_worker=_THREADS_PER_WORKER`
explicitamente em A/B/C.

## 4. Sprint 14.2 — Decisão fastmath dipoles.py / kernel.py

**Decisão final**: NÃO aplicar `fastmath=True` em
`dipoles.hmd_tiv`, `dipoles.vmd`, `kernel._fields_in_freqs_kernel`,
`kernel._fields_in_freqs_kernel_cached`,
`kernel.precompute_common_arrays_cache`.

Justificativa completa em
`docs/reports/v2.15_fastmath_dipoles_analysis_2026-05-01.md`. Resumo:

- `hmd_tiv`: erro FMA estimado ~`600 × 2e-16` = **1.2e-13**, em 0.83×
  do gate Fortran 1e-12. Margem insuficiente para confiar em medição.
- `vmd`: erro estimado ~`8e-14`, mais folga, mas mantemos por consistência.
- Cancelamento catastrófico em `1 + RTEup` (recursão TE/TM com exp
  quase-iguais) amplifica reordering FMA.
- Ganho marginal: ~3-5% CPU em workloads multi-modelo, vs risco
  binário de quebrar paridade.
- Apenas `hankel.py` (4 funções dot-product, FMA-safe) permanece com
  `fastmath=True` (decisão v2.14, validada).

## 5. Sprint 14.3 — JIT Cache Observability

### 5.1 API expandida

`get_jit_cache_info()` em `geosteering_ai/simulation/_jax/forward_pure.py`
(+80 LOC) reporta os 3 caches do triple-cache JAX:

```python
{
    # Backward-compat v1.5.0
    "n_entries": int,             # _BUCKET_JIT_CACHE size
    "maxsize": int,               # LRU limit (default 64)
    "keys": list,                 # _BUCKET_JIT_CACHE keys
    # Novas v2.15
    "bucketed_size": int,         # _BUCKET_JIT_CACHE
    "unified_size": int,          # _UNIFIED_JIT_CACHE
    "chunked_size": int,          # _UNIFIED_CHUNKED_JIT_CACHE
    "total_xla_programs": int,    # soma dos 3
    "estimated_vram_mb": float,   # heurística baseada em (n × npt)
    "strategy_distribution": {    # {"bucketed": N, "unified": M, "chunked": K}
        "bucketed": int,
        "unified": int,
        "chunked": int,
    },
}
```

### 5.2 Heurística VRAM

Por entrada de cache, estimativa baseada em tensor `common_arrays`
shape `(3, n, npt, nf=1)` complex128:

```
vram_mb_per_entry = 3 × n × npt × 16 bytes / 1024²
```

Conservadora (não inclui buffers temporários de `vmap` ou propagação
TE/TM, que adicionam ~3-5× em pico de execução). Útil como proxy de
*ordem de magnitude* em GPU T4 (16 GB) / A100 (40 GB).

### 5.3 Testes (4 novos em `test_simulation_jax_sprint13.py`)

- `test_jit_cache_info_empty`: total_xla_programs == 0 após clear
- `test_jit_cache_info_after_simulate_unified`: strategy="unified" popula
  apenas `_UNIFIED_JIT_CACHE`, não `_BUCKET_JIT_CACHE`
- `test_jit_cache_info_vram_estimate`: VRAM > 0 com cache populado;
  sanity check < 1000 MB
- `test_jit_cache_info_idempotent_readonly`: 3 chamadas consecutivas
  não modificam state (read-only confirmado)

Todos passam (4/4 em 17.92s local).
