## 1. Resumo Executivo

A v2.29 ("Back to Basics") restaurou a arquitetura `ProcessPoolExecutor`
efêmero do `old_geosteering_ai/`, mas reintroduziu um bug latente: setar
`os.environ["NUMBA_NUM_THREADS"]` **dentro** do worker `run_numba_chunk`
falha quando o ambiente do pai já tem `NUMBA_NUM_THREADS` setado para
valor distinto. O resultado em produção (com `NUMBA_NUM_THREADS=2` herdado
do shell do usuário):

```
[ERRO] Cannot set NUMBA_NUM_THREADS to a different value once the threads
have been launched (currently have 2, trying to set 16)
```

**Fix v2.29.1**: mover o setting de `NUMBA_NUM_THREADS` (e `OMP_NUM_THREADS`)
do worker para o **pai** (`SimulationThread.run` e
`_run_numba_parallel_sync`), antes de criar o `ProcessPoolExecutor`. Workers
spawned herdam o env var correto. Restauração em `try/finally` evita
vazamento para outras partes da GUI.

## 2. Sintomas Reportados (pré-fix)

1. **RuntimeError em multi-freq + multi-dip + spawn pool** durante warmup do worker
2. **Throughput regredido**: 40k mod/h cold + 120k mod/h warm (baseline 150k)
3. **Python travava ao fechar** o programa antes da simulação terminar

## 3. Causa-Raiz

### 3.1 Por que setar `NUMBA_NUM_THREADS` dentro do worker falha

Quando `ProcessPoolExecutor` cria workers em modo **spawn** (default macOS
Python 3.13), cada worker precisa `unpickle` da função `run_numba_chunk`
referenciada pela submissão. Para isso, Python importa o módulo que
contém a função: `geosteering_ai.simulation.tests.sm_workers`.

A cadeia de imports do módulo `sm_workers` resolve o pacote pai
`geosteering_ai.simulation` (via `from .sm_qt_compat import` ou via
namespace package resolution), que importa numba **antes** de qualquer
linha de `run_numba_chunk` rodar.

Resultado:
1. Worker spawn herda `NUMBA_NUM_THREADS=2` (env do pai)
2. Bootstrap importa numba → pool inicializado com 2 threads
3. `run_numba_chunk` linha 170 chama `os.environ["NUMBA_NUM_THREADS"]=16`
4. `_simulate_combined_prange_flat` compila → reload_config detecta
   discrepância env vs pool → `RuntimeError: Cannot set NUMBA_NUM_THREADS
   to a different value once the threads have been launched`

Essa armadilha está documentada em
[`_workers.py:472-484`](../../geosteering_ai/simulation/_workers.py#L472-L484):

> "Quando este inicializador é chamado, geosteering_ai.simulation já foi
> importado como efeito colateral do unpickle do próprio worker — Python
> importa o pacote pai para resolver o módulo desta função. Nesse ponto
> Numba já foi inicializado..."

### 3.2 Por que OLD funcionava em alguns sistemas

`old_geosteering_ai/simulation/tests/sm_workers.py:138` tem **o mesmo
bug**. A diferença era ambiente:

| Cenário | Pai (env) | Worker tenta | Resultado |
|:--------|:---------:|:------------:|:----------|
| OLD em shell sem `NUMBA_NUM_THREADS` | default os.cpu_count()=16 | 16 | No-op, OK |
| OLD em shell `NUMBA_NUM_THREADS=2` | 2 | 16 | RuntimeError |
| v2.29 com NumbaPrimer no pai | 2 herdado do shell | 16 | RuntimeError |

A v2.18–v2.28 tinha o `_acquire_numba_pool` no `sm_workers.py` que setava
env var **no pai antes do spawn**. A v2.29 removeu junto com `PoolWarmupThread`,
mas não percebeu que essa parte era essencial.

### 3.3 Throughput 40k cold cache

Quando o `RuntimeError` é levantado e capturado em
[`multi_forward.py:1010-1028`](../../geosteering_ai/simulation/multi_forward.py#L1010-L1028)
como `logger.warning`, os workers continuam com pool de 2 threads em vez
dos 4 desejados. Resultado: throughput ~50% do esperado (≈40k em vez de
80k cold).

## 4. Solução Implementada

### 4.1 Arquivos Modificados

| Arquivo | Mudança |
|:--------|:--------|
| `geosteering_ai/simulation/tests/sm_workers.py` | Remove `os.environ["NUMBA_NUM_THREADS"]` de `run_numba_chunk`; adiciona `try/finally` em `SimulationThread.run` antes do `ProcessPoolExecutor` |
| `geosteering_ai/simulation/tests/sm_benchmark.py` | Mesmo padrão em `_run_numba_parallel_sync` |
| `tests/test_simulation_workers_ephemeral.py` | +3 guard tests v2.29.1 |

### 4.2 Diff Conceitual

**ANTES (v2.29 — bug)**:
```python
def run_numba_chunk(...):
    os.environ["NUMBA_NUM_THREADS"] = str(n_threads)  # ← TARDE DEMAIS
    os.environ["OMP_NUM_THREADS"] = str(n_threads)
    from geosteering_ai.simulation import simulate_multi  # numba JÁ importado
    ...
```

**DEPOIS (v2.29.1 — fix)**:
```python
# Em SimulationThread.run (PAI, antes do pool):
_prev_nnt = os.environ.get("NUMBA_NUM_THREADS")
_prev_omp = os.environ.get("OMP_NUM_THREADS")
os.environ["NUMBA_NUM_THREADS"] = str(req.n_threads)
os.environ["OMP_NUM_THREADS"] = str(req.n_threads)
try:
    with ProcessPoolExecutor(max_workers=n_workers) as pool:
        # workers spawned herdam env var correto ✓
        ...
finally:
    # Restaura env vars (evita vazamento para outras simulações/benchmarks)
    if _prev_nnt is None:
        os.environ.pop("NUMBA_NUM_THREADS", None)
    else:
        os.environ["NUMBA_NUM_THREADS"] = _prev_nnt
    ...

# Em run_numba_chunk (WORKER):
def run_numba_chunk(...):
    # v2.29.1: env var NÃO é setada aqui (numba já importado pelo bootstrap)
    from geosteering_ai.simulation import simulate_multi
    ...
```

## 6. Lições Aprendidas

1. **Em modo spawn, `os.environ[X]` no worker é tardio demais** se `X`
   afeta uma biblioteca importada durante o bootstrap (numba, MKL, OMP).
   Pai DEVE setar antes do pool.

2. **A documentação canônica em `_workers.py:472-484`** já alertava sobre
   isso desde a v2.16. A v2.29 ignorou ao copiar a estrutura do OLD sem
   incluir o setup do pai (que estava no `_acquire_numba_pool` removido).

3. **CodeRabbit (`coderabbit review --agent`)** confirma 0 findings após
   o fix — bom sinal de que o try/finally está estruturalmente correto.

4. **Reproducer empírico é mais eficaz que análise estática** para validar
   fix de regressão de performance. Setar `NUMBA_NUM_THREADS=2` antes do
   teste reproduz exatamente o cenário do usuário.

5. **Throughput 150k é viável neste hardware** (Mac Apple Silicon),
   confirmado pelo run 3 do reproducer (149,984 mod/h pool-only).
