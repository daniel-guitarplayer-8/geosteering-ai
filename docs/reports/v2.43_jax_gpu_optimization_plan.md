## Sumário Executivo

A investigação multi-agente identificou **convergência total dos 3 maiores gargalos** (todos os 3 agentes técnicos chegaram independentemente ao mesmo top-3):

1. **🔴 Loop Python `rho_h_at_obs`** (40-60% do tempo em B/E) — fix TRIVIAL
2. **🟠 5× eager eval em `_hmd_tiv_descent_body`** (40% compute desperdiçado) — fix ALTO
3. **🟠 `complex128` hardcoded** (2-8× compute potencial) — fix ALTO RISCO paridade

O 4º agente (estado da arte JAX/XLA) confirmou que **a A100 está sendo subutilizada por causa de 4 fatores estruturais** que JAX/XLA modernos têm soluções documentadas para:

1. Sem persistent compilation cache (cold-start 30-180s desnecessário)
2. Sem XLA_FLAGS A100 (latency_hiding_scheduler desligado)
3. Sem `donate_argnums` (pressão de memória extra)
4. `fori_loop` sobre n_cam=5 (deveria ser unrolled)

**Estimativa de ganho composto** após Sprint Quick-Wins (1 semana, ~150 LOC):
- Cenário E: 47.812 → **~95.000-120.000 mod/h** (2-2.5× ganho)
- Cenário G: 16.835 → **~30.000-40.000 mod/h** (1.8-2.4× ganho)

Isso **iguala ou supera o histórico i9 per-device** em E (122.000) e G (45.000). Sprints subsequentes (lax.map chunking, complex64 opt-in) podem dobrar isso novamente.

A investigação também identificou **3 lacunas BLOQUEANTES** na cobertura de testes que precisam ser fechadas **ANTES** de qualquer modificação no código de produção.

### O1.3 — Fix Loop Python `rho_h_at_obs` (commit C03 — O MAIOR GANHO)

**Arquivo**: `geosteering_ai/simulation/_jax/multi_forward.py:1120-1135`
**Esforço**: ~30 LOC
**Ganho**: **40-60% redução de tempo total em B/E** (estimativa de 25-45 ms recuperados em E_50)
**Risco**: BAIXO

```python
# ANTES (lines 1120-1135):
for i_model in range(n_models):
    rho_h_m = rho_h_batch_np[i_model]
    rho_v_m = rho_v_batch_np[i_model]
    if n >= 2:
        esp_m = esp_batch_np[i_model]
        prof_mid = np.concatenate([np.array([0.0]), np.cumsum(esp_m)])
    else:
        prof_mid = np.array([0.0])
    for i_ang in range(nAngles):
        for i_pos in range(n_pos):
            lay = layer_at_depth(n, float(positions_z_np[i_pos]), prof_mid)
            rho_h_at_obs[i_model, i_ang, i_pos] = rho_h_m[lay]
            rho_v_at_obs[i_model, i_ang, i_pos] = rho_v_m[lay]

# DEPOIS — vetorização NumPy completa:
if n >= 2:
    # prof_mid_batch: (n_models, n-1) — fronteiras de camada por modelo
    prof_mid_batch = np.concatenate(
        [np.zeros((n_models, 1)), np.cumsum(esp_batch_np, axis=1)],
        axis=1,
    )
else:
    prof_mid_batch = np.zeros((n_models, 1))

# searchsorted vetorizado: (n_models, n_pos) → índice de camada
# Como prof_mid é monotônico por modelo, fazemos searchsorted batched:
layer_indices = np.empty((n_models, n_pos), dtype=np.int64)
for i_model in range(n_models):
    layer_indices[i_model] = np.clip(
        np.searchsorted(prof_mid_batch[i_model], positions_z_np, side='right') - 1,
        0, n - 1,
    )

# Indexação fancy: (n_models, n_pos) → (n_models, n_pos) com rho_h
rho_h_per_pos = rho_h_batch_np[np.arange(n_models)[:, None], layer_indices]
rho_v_per_pos = rho_v_batch_np[np.arange(n_models)[:, None], layer_indices]

# Broadcasting para nAngles (todos os ângulos veem mesma camada por z):
rho_h_at_obs = np.broadcast_to(
    rho_h_per_pos[:, None, :], (n_models, nAngles, n_pos)
).copy()
rho_v_at_obs = np.broadcast_to(
    rho_v_per_pos[:, None, :], (n_models, nAngles, n_pos)
).copy()
```

**Validação**: paridade com versão anterior bit-exata (mesmo `layer_at_depth` semanticamente).

## Parte V — Sprint O2: Chunking + Async (Ganho Adicional 1.2-1.4×)

**Duração estimada**: 1-2 semanas
**Objetivo**: Reduzir pressão de VRAM via chunking e ocupar GPU 100% via async dispatch.

### O2.1 — `lax.map(batch_size=K)` Sobre `n_models` (commit C10)

**Arquivos**: `multi_forward.py` (`simulate_multi_jax_batched`) + `config.py` (adicionar `jax_map_batch_size`)
**Esforço**: ~200 LOC
**Ganho**: 30-50% redução VRAM, throughput similar ou +20%
**Risco**: BAIXO (matemática idêntica)

```python
# Em SimulationConfig:
jax_map_batch_size: Optional[int] = None  # None = vmap puro; int = chunking

# Em simulate_multi_jax_batched:
if cfg.jax_map_batch_size is None:
    # caminho atual: vmap puro
    H_flat_batched = jax.vmap(_one_model, in_axes=(0, 0, 0, 0))(...)
else:
    # caminho novo: lax.map(vmap(...), batch_size=K)
    H_flat_batched = jax.lax.map(
        lambda chunk: jax.vmap(_one_model, in_axes=(0, 0, 0, 0))(*chunk),
        (rho_h_batch_jnp, rho_v_batch_jnp, h_arr_batch_jnp, prof_arr_batch_jnp),
        batch_size=cfg.jax_map_batch_size,
    )
```

**Sweep empírico em A100**: `batch_size ∈ {16, 32, 64, 128, 256, 512}` × cenários E/F/G/H. Documentar valor ótimo por cenário.

### O2.2 — Async Dispatch + Prefetch (commit C11)

**Arquivos**: `multi_forward.py` (novo wrapper async)
**Esforço**: ~50 LOC
**Ganho**: 5-30% (depende de tempo prep CPU)
**Risco**: ZERO

```python
from concurrent.futures import ThreadPoolExecutor
from collections import deque

def simulate_multi_jax_batched_async(
    chunks_iter, *, cfg, n_prefetch=2
):
    """Generator que sobrepõe prep CPU (chunk N+1) com compute GPU (chunk N)."""
    with ThreadPoolExecutor(max_workers=n_prefetch) as ex:
        futures = deque()
        for chunk in chunks_iter:
            futures.append(ex.submit(_prepare_chunk, chunk))
            if len(futures) >= n_prefetch:
                prepared = futures.popleft().result()
                result = simulate_multi_jax_batched(**prepared, cfg=cfg)
                yield result
        while futures:
            prepared = futures.popleft().result()
            yield simulate_multi_jax_batched(**prepared, cfg=cfg)
```

### O3.1 — Roofline Analysis Definitiva

- Se **memory-bound**: investir em layout (O3.2) ou complex64 (reduz pressão 2×)

### O3.2 — Memory Layout Optimization (commit C12)

**Pré-requisito**: profile confirma strided access em algum eixo.
**Esforço**: ~100 LOC
**Ganho**: 0-5× em hot loop afetado

Aplicar `jax.lax.transpose` para forçar layout coalescido no eixo crítico (provavelmente `nf` ou `n_pos`).

### O3.3 — Eliminar Overcompute em `lax.switch` (commit C13)

**Pré-requisito**: profile mostra distribution skewed (1-2 casos dominantes).
**Esforço**: ~300 LOC
**Ganho**: 1.2-2× nos kernels HMD/VMD
**Risco**: MÉDIO (NaN poisoning em branches "inválidos")

Substituir `lax.switch` por arquitetura unificada com `jnp.where`:
```python
# Em vez de switch 6 casos:
result = lax.switch(case_idx, [case_0, case_1, ..., case_5], operand)

# Computar tudo em forma unificada:
common_part = unified_kernel(operand)  # trabalho compartilhado
diff_per_case = compute_case_diffs(operand)  # diferenças específicas
result = jnp.where(
    case_idx == 0, common_part + diff_per_case[0],
    jnp.where(
        case_idx == 1, common_part + diff_per_case[1],
        # ... cuidado: usar jnp.nan_to_num para evitar NaN poisoning
    )
)
```
