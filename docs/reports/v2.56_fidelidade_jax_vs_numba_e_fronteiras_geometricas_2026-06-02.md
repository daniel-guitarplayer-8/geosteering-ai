## 1. Pergunta 1 — O Conceito de "Variedade de Fronteiras Geométricas"

### 1.1 Definição geofísica: de `esp` à profundidade de interface

Um modelo geológico 1D TIV é descrito por um vetor de **espessuras de camada** `esp` (em metros) e pelos vetores de resistividade horizontal/vertical (`rho_h`, `rho_v`). A geometria das **fronteiras** (interfaces entre camadas) não é dada diretamente: ela emerge da **soma cumulativa** das espessuras.

```
  esp        = [ esp_0,           esp_1,                 esp_2,        ... ]   (espessuras)
                   │                 │                       │
  cumsum(esp)→ z_1 = esp_0    z_2 = esp_0+esp_1   z_3 = esp_0+esp_1+esp_2  ...   (profundidades
                   │                 │                       │                    de interface)
  ┌────────────────┴─────────────────┴───────────────────────┴──────────────┐
  │ Camada 0 │     Camada 1         │      Camada 2          │   Camada 3 ... │
  └──────────┴──────────────────────┴───────────────────────┴────────────────┘
   z=0       z_1                   z_2                      z_3
```

Cada `z_k = Σ_{i<k} esp_i` é a profundidade de uma **fronteira de camada** (Depth-To-Boundary, DTB). No código, essa conversão é **explícita e verificada**:

| Etapa | Código | Evidência |
|:------|:-------|:----------|
| Fronteiras por modelo (CPU/produção) | `acc = np.cumsum(esp_all, axis=1)  # fronteiras por modelo` | `synthetic_generator.py:677` |
| Mapear `z` → camada que o contém | `_find_layer_for_z` / `_layer_at_batch` (twin vetorizado; "menor índice tal que fronteira cumsum(esp) > z") | `synthetic_generator.py:643,686` |
| Vetor de fronteiras alimentado ao kernel JAX | `prof_arr = np.concatenate([np.array([0.0]), np.cumsum(esp_np)])` | `multi_forward.py:622,821,1155` |

A "variedade de fronteiras geométricas" é, portanto, **quão diferentes são esses conjuntos de profundidades de interface entre os modelos do dataset** — modelos com camadas finas/espessas, fronteiras rasas/profundas, mais ou menos interfaces.

### 1.2 Por que importa em geosteering

O objetivo do geosteering é manter o poço dentro de uma camada-reservatório navegando pela resposta EM. O sinal medido (tensor magnético 3×3) é dominado pela **proximidade e contraste das fronteiras** — a resposta perto de uma interface raso/profundo difere radicalmente. Se o dataset de treino só contém uma faixa estreita de geometrias de fronteira, o modelo de inversão **não generaliza** para a variedade real encontrada no poço. Cobrir uma boa "variedade de fronteiras geométricas" é o que dá ao modelo robustez para localizar interfaces em profundidades não vistas.

### 1.3 Os dois eixos ortogonais: geometria vs resistividade

A diversidade do dataset tem **duas dimensões independentes**, amostradas separadamente:

| Eixo | Grandeza | Como é amostrado | O que controla |
|:-----|:---------|:-----------------|:---------------|
| **Geometria das fronteiras** | `esp` → `cumsum` → profundidades de interface | `sample_geometry(rng, ..., mode)` / `geometry_mode` | **Onde** estão as fronteiras (DTB) |
| **Resistividade** | `rho_h`, `rho_v` (e razão de anisotropia) | `rng.uniform(...)` por modelo | **Quão forte** é o contraste EM entre camadas |

Os dois são **ortogonais**: pode-se ter a mesma geometria com contrastes de resistividade muito distintos, ou geometrias diferentes com a mesma resistividade. Ambos são sorteados por um RNG determinístico (`np.random.default_rng(seed)`) **antes** de qualquer escolha de backend (`synthetic_generator.py:290-356`; CLI `simulate.py:152-159`). A taxonomia de modos de geometria é a mesma em ambos os caminhos:

| Modo (`geometry_mode` / `--geometry`) | Diversidade de fronteiras | Efeito no batching JAX |
|:--------------------------------------|:--------------------------|:-----------------------|
| `per_model` / `per-model` | **Máxima** — `esp` único por modelo | Cada modelo = 1 grupo → dispara fallback Numba |
| `templates` (default da CLI) | **Média** — K geometrias compartilhadas (cap K=4, v2.56) | Agrupável → batching eficiente |
| `quantize` | Intermediária — `esp` quantizado em passos | Agrupamento parcial |

> **Nota crítica para a P2:** a "variedade de fronteiras geométricas" é decidida **neste estágio de amostragem**, totalmente **antes** e **independentemente** do backend. O backend apenas **reage** à agrupabilidade da geometria já sorteada — ele nunca a produz.

## 2. Pergunta 2 — O JAX GPU é "Inferior" ao Numba JIT?

A pergunta tem duas facetas, respondidas separadamente: **(A)** as *amostras* (modelos geológicos) geradas para o JAX são menos diversas? e **(B)** os *valores de campo* computados pelo JAX são menos fiéis fisicamente?

### 2.1 Faceta A — A amostragem é BACKEND-AGNÓSTICA (mesmos modelos vão a ambos)

O backend **não** participa da geração dos modelos. A diversidade é decidida antes — provado por construção em **três níveis independentes**:

| Nível de prova | Evidência (`file:line`) | O que demonstra |
|:---------------|:------------------------|:----------------|
| **1. Ordem temporal no fonte** | `simulate.py:443` (`models = _build_random_models(...)`) → `:460` (`resolve_backend_preflight(...)`) → `:495` (`run_once(...)`); idem `benchmark.py:311/341/418` | Modelos construídos **antes** de o backend ser sequer resolvido |
| **2. Assinaturas sem backend** | `_build_random_models(n_models, seed, *, geometry, n_geometries, quantize_step)` (`simulate.py:84-91`); `sample_geometry(rng, n, ..., mode)` (`_exec.py:76-162`) | As funções de amostragem **não têm parâmetro de backend**; o RNG nunca o recebe |
| **3. Mesmo objeto Python aos dois** | `--compare-backends`: o loop `for be in (numba, jax)` passa o **mesmo** `models` a `run_once` (`_exec.py:620-649`) | Identidade dos modelos é mais forte que "mesma seed": é literalmente o mesmo array |

O que `run_once` faz é apenas escolher **como** computar o campo, sem tocar os valores dos modelos:

```
run_once(backend, models, ...)          (_exec.py:458)
   ├─ backend == 'jax':  models_to_batch(models) → simulate_batch(...)   (_exec.py:519-545)
   │                     └─ models_to_batch só faz np.stack + astype(float64) → no-op numérico (_exec.py:194-214)
   └─ backend == 'numba': simulate_multi(models=list(models), ...)        (_exec.py:548-568)
```

Componentes que poderiam (mas **não**) alterar os valores:

- **`group_by_geometry`** (`multi_forward.py:1257-1261`): retorna apenas **listas de índices `int64`** agrupados por `esp.tobytes()`, reassembla na ordem original (`:1293`). Particiona, não muta — cobertura exaustiva e disjunta de `[0, n_models)`.
- **Preflight / dispatch gate** (`_exec.py:260-270`; `dispatch.py:147-153`, `_GROUPABLE_RATIO_MAX=0.5`): só **lê** `esp.tobytes()` para contar grupos e **rotear** (jax → numba se `n_groups > 0,5·n_models`). Nenhum site reatribui `esp`/`rho`.
- **Cap K=4 da v2.56** (`_exec.py:73`, `_TEMPLATE_GEOMETRIES_CAP=4`): afeta a amostragem que alimenta **ambos** os backends igualmente. O Numba é **indiferente** ao modo (cada modelo é independente no pool); só o JAX-grouped tira proveito do compartilhamento para batching.

> **Conclusão da Faceta A:** com a mesma seed, `--geometry templates --backend numba` e `--backend jax` recebem **exatamente** os mesmos `esp`/`rho`. O JAX **não** gera amostras menos diversas — a diversidade é função **só** do modo de amostragem. *(Os 8 achados desta dimensão sobreviveram integralmente à verificação adversarial.)*

### 2.2 Faceta B — Fidelidade física no config DEFAULT (complex128): paridade, não inferioridade

O **ground-truth** é o simulador Fortran `tatu.x`. Numba e JAX são portes validados contra ele. No default, **ambos rodam em complex128** e batem o Fortran no mesmo patamar.

#### 2.2.1 Tolerâncias EXATAS dos testes (não rótulos)

| Par comparado | Gate (tolerância de teste) | Erro medido (pior caso) | Evidência |
|:--------------|:---------------------------|:------------------------|:----------|
| Numba ↔ Fortran | `max_abs < 1e-6` | ~1,29e-13 (hou_7) | `test_simulation_compare_fortran.py:126` |
| JAX ↔ Numba (arrays internos) | `assert diff < 1e-12` | dentro do gate | `test_simulation_jax_propagation.py:131` |
| **JAX ↔ Fortran (direto, fecha o triângulo)** | `tol_abs = 1e-10` | `max_abs_error < 1e-10` | `test_simulation_jax_fortran_parity.py:113-130` |
| JAX-c128 ↔ Numba (gate anti-regressão) | `max_abs < 2,301e-13` ("Path c128 INVIOLÁVEL") | 1,53e-13 (JAX) vs 1,29e-13 (Numba) | `test_simulation_jax_complex64_parity.py:189` |

A faixa global medida em c128 é **5,7e-14 ≤ max_abs ≤ 1,53e-13** — três ordens de magnitude abaixo do gate JAX↔Fortran (1e-10).

#### 2.2.2 Por que nenhum dos dois é "mais correto"

A diferença residual JAX↔Numba (~1e-13) é **nível-ULP do float64, sem viés sistemático**: o JAX fica entre **−13% e +30%** do erro do Numba conforme o modelo (em alguns é menor, em outros maior). A causa atribuída é reordenação de reduções pelo XLA (aritmética FP64 não-associativa), não perda de precisão. Adicionalmente, **JAX_hybrid e JAX_native em c128 são bit-identical entre si** (mesmo programa XLA no caminho bucketed), confirmando que a escolha de caminho JAX não altera a fidelidade em c128.

> **Ressalva de honestidade (correção adversarial):** o fechamento **direto** do triângulo (teste JAX-vs-Fortran) cobre o caminho **jax_hybrid/bucketed** — que é o caminho default. O caminho `jax_native` **não** é exercitado direto contra o Fortran por nenhum teste; sua paridade é garantida por ser bit-identical ao hybrid em c128.

### 2.3 Faceta B (cont.) — O ângulo complex64: trade-off OPT-IN, NÃO inferioridade não-intencional

#### 2.3.1 c64 não é o default em lugar nenhum

| Local | Default | Evidência |
|:------|:--------|:----------|
| CLI `--dtype` | `complex128` ("paridade") | `cli/_main.py:227-230` |
| `run_once` / warmup | `dtype = "complex128"` | `cli/_exec.py:412` |
| `SimulationConfig` | `dtype = "complex128"` | `config.py:361` |
| **Geração de dados de produção** | `complex128` **hardcoded** | `synthetic_generator.py:403` |
| Numba (todos os 9+ arrays de propagação) | `np.complex128` hardcoded — **não existe modo c64 no Numba** | `_numba/propagation.py:358-371`; `kernel.py:424-425` |

O preset `production_gpu()` usa `complex64`, mas está **morto no código real** — só referenciado em docstrings, nunca invocado em produção. Como o **Numba é c128-only**, a comparação "c64 vs Numba" sequer existe: **c64 é exclusivamente um caminho JAX opt-in**.

#### 2.3.2 Quando ativado, c64 É menos preciso — por design e quantificado

| Métrica | Valor medido | Gate | Evidência |
|:--------|:-------------|:-----|:----------|
| c64 vs c128 — erro absoluto | `max_abs = 6,112e-05` | `< 1e-4` | `v2.43_o2_o3_jax_perf_2026-05-28.md:159`; `test_simulation_jax_complex64_parity.py:131` |
| c64 vs c128 — erro relativo | `max_rel = 3,875e-04` | `< 1e-4` (relativo borderline) | mesmo relatório, linha 159/320 |

Isso é **~9 ordens de magnitude** pior que o c128 (~1e-13 → ~6e-5), exatamente o esperado de aritmética FP32 (~1e-6/1e-7 por operação, amplificada pela cadeia de propagação). É **degradação deliberada e quantificada** — alcançada **apenas** por `--dtype complex64` explícito — trocada conscientemente por menor VRAM e maior velocidade em GPU/PINN.

> **Distinção rigorosa exigida:**
> - **Inferioridade não-intencional** = um backend que, sem que o usuário escolha, produz pior resultado que o outro. **Não ocorre** aqui: no default ambos são c128 e equivalentes.
> - **Trade-off OPT-IN** = o usuário ativa explicitamente um modo de menor precisão para ganhar velocidade/memória, com erro conhecido e dentro de tolerância publicada. **É o caso do c64.**

#### 2.3.3 Correção adversarial — o que NÃO é verdade sobre a validação do c64

Um achado preliminar afirmava que o caminho native/c64 seria "validado **direto** contra o Fortran (~8,7e-06, tolerância 1e-4)". **Isto foi REFUTADO** pela verificação adversarial (`survives=false`) e **não** deve ser usado:

- O valor `8,7e-06` (`validation/compare_fortran.py:67`) é um **exemplo de docstring**, não uma medição asserida por teste.
- **Nenhum** teste roda backend `native` (muito menos c64) **direto contra o Fortran** — só `jax_hybrid` (`test_simulation_jax_fortran_parity.py:110`, `backends=['jax_hybrid']`) e `numba`.
- `SimulationConfig` **não possui** campo `use_native_dipoles` (o construtor lança `TypeError`); a rota "jax_native" em `compare_fortran` cai silenciosamente para **c128 unified** — a medição real rotulada "jax_native vs Fortran" no relatório (linhas 308/311) é **~1,26e-13 = nível complex128**, não 8,7e-6 nem c64.

**Versão corrigida:** o modo c64 é validado **apenas indiretamente** (c64 vs c128 default, ~6e-5 abs); **não existe** medição direta c64-vs-Fortran na suite. A validação direta contra o *ground-truth* Fortran cobre **apenas** o caminho `jax_hybrid` em c128. Isto **enfraquece um sub-claim de robustez do c64**, mas **não** altera o veredito geral — c64 continua sendo um trade-off opt-in com erro conhecido (~6e-5), e o default c128 permanece em paridade plena.

### 2.4 Faceta B (cont.) — Os demais caminhos divergentes NÃO degradam fidelidade

| Caminho | Natureza | Paridade em c128 | Evidência |
|:--------|:---------|:-----------------|:----------|
| **Fallback grouped → Numba** (geometria não-agrupável) | Roteamento de performance; recai no **próprio Numba c128 (ground-truth-validado)** | Exato (é o Numba) | `_exec.py:265-266`, `:172` |
| **OOM → chunk** (`jax_chunk_size_models`) | Fragmenta só o eixo de modelos; mesmo programa XLA por chunk | `< 1e-13` (inclui chunk=5, não-divisor de 16) | `test_simulation_jax_o4_batched_bucketed.py:368-395` |
| **unified vs bucketed** | unified é "correto, mais lento" (fallback) | `< 1e-13` | `test_..._o4_batched_bucketed.py:157`; `multi_forward.py:1090` |
| **fallback heterogêneo ≠ unified** | partição alternativa | `< 1e-13` | `test_..._o4_batched_bucketed.py:239` |

Todos são decisões de **dispatch/performance** em c128, equivalentes dentro de 1e-13 — nenhuma degrada fidelidade. Notavelmente, o fallback recai justamente no Numba c128, o *ground-truth*-validado.

## 4. Conclusão

**Pergunta 1.** "Variedade de fronteiras geométricas" é a diversidade das **profundidades de interface de camada** dos modelos geológicos — obtidas pela **soma cumulativa** do vetor de espessuras `esp` (`np.cumsum`, `synthetic_generator.py:677` / `multi_forward.py:622`). É um dos **dois eixos ortogonais** de diversidade do dataset (geometria das fronteiras × valores de resistividade), e importa porque a resposta EM em geosteering é dominada pela proximidade/contraste das fronteiras: sem variedade geométrica suficiente, o modelo de inversão não generaliza para profundidades de interface não vistas.

**Pergunta 2.** **Os resultados do simulador JAX GPU NÃO são inferiores aos do Numba JIT no config default.** A resposta tem três pilares, todos sobreviventes à verificação adversarial:

1. **Amostragem backend-agnóstica** — a diversidade dos modelos (incluindo a variedade de fronteiras geométricas da P1) é decidida **antes** e **independentemente** do backend; com a mesma seed, JAX e Numba recebem exatamente os mesmos modelos. O JAX não gera nada "menos diverso".
2. **Paridade no default (complex128)** — ambos os backends são portes do mesmo algoritmo e batem o *ground-truth* Fortran no mesmo patamar de ~1e-13 (nível-ULP, sem viés sistemático). Nenhum é "mais correto".
3. **c64 é trade-off, não inferioridade** — a única forma de o JAX ser numericamente menos preciso é o modo **complex64 opt-in**, ativado **apenas** por escolha explícita do usuário, com erro **conhecido e quantificado** (~6e-5 abs / ~3,9e-4 rel, gate <1e-4) trocado deliberadamente por velocidade/VRAM. **Ressalva honesta:** esse modo tem validação **apenas indireta** (c64 vs c128), não medição direta contra o Fortran — a validação direta cobre só o caminho hybrid/c128.

**Veredito final:** Não, no default c128 — paridade plena, nenhum inferior. Em c64, é um **trade-off deliberado de precisão opt-in**, não uma inferioridade não-intencional. Os demais caminhos de dispatch (fallback, chunk, unified) preservam fidelidade em c128 dentro de 1e-13.
