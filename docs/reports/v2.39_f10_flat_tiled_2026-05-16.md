## Resumo Executivo

Sprint **F10 — Tile/Block no Path FLAT** implementa `_simulate_combined_prange_flat_tiled`,
variante tile-aware do hot path FLAT 4D (`_simulate_combined_prange_flat`, v2.22), com o
objetivo de manter as slices de cache `u_unique[ci, i_f]` (~280 kB por par combo/freq)
quentes em L2 durante múltiplas posições consecutivas. Benchmarks empíricos no hardware
alvo (Intel Core i9-9980HK, 16T HT) revelaram **resultado negativo**: o agrupamento em
tiles reduz o número de tarefas `prange` em `tile_size×`, causando inanição de threads
que supera o benefício de localidade de cache. A paridade bit-exata com o path não-tiled
foi **confirmada em 17/17 testes**. A função está disponível para cenários de
larga escala futuros; o default `use_tiled_positions=False` é preservado.

## 7. Arquitetura da Função F10

### 7.1 Decomposição de Índice FLAT-tiled

```
v2.22 — _simulate_combined_prange_flat (não-tiled):
  prange(n_combos × n_pos × nf)       ← n_total tarefas
    └── _fields_at_single_freq(1 pos, 1 freq)

F10 — _simulate_combined_prange_flat_tiled:
  prange(n_combos × n_tiles × nf)     ← n_total / tile_size tarefas
    └── range(tile_len) serial         ← tile_size posições por tarefa
          └── _fields_at_single_freq
              ← slices u_unique[ci, i_f] REUTILIZADAS pelo tile inteiro
```

Fórmula de decomposição (k → i_combo, i_tile, i_f):

```python
n_tiles  = (n_pos + tile_size - 1) // tile_size   # ceil sem divisão float
n_tile_f = n_tiles * nf
i_combo  = k // n_tile_f
rem      = k %  n_tile_f
i_tile   = rem // nf
i_f      = rem %  nf
```

### 7.2 Diagrama de Localidade de Cache

```
┌─────────────────────────────────────────────────────────────────────┐
│  Intel i9-9980HK — Hierarquia de memória relevante                  │
│  L1d = 32 kB/core  ·  L2 = 256 kB/core  ·  L3 = 16 MB (shared)   │
├─────────────────────────────────────────────────────────────────────┤
│  Slice de cache: u_unique[ci, i_f]                                  │
│    shape = (npt, n) = (201, 10)   →   ~31.4 kB (float64)           │
│  9 slices por (ci, i_f):          →   ~280 kB total                │
│                                                                     │
│  Hipótese F10:                                                      │
│    tile_size=8 → 8 posições reusam mesmas 280 kB → quentes em L2   │
│    → ~8× menos fetch de main memory por posição                    │
│                                                                     │
│  Realidade medida:                                                  │
│    L2 = 256 kB/core < 280 kB → ~90% das slices cabem (caso nf=1)  │
│    Mas: Numba JIT já usa prefetch linear → L2 naturalmente quente  │
│    E:   tiling divide tasks → threads ficam ociosas                │
└─────────────────────────────────────────────────────────────────────┘
```

### 8.3 Análise da Causa Raiz

```
Hipótese inicial:
  tiling mantém ~280 kB de slices quentes em L2
  → ~8× menos fetch de main memory por posição
  → ganho de throughput proporcional

Realidade observada (análise de inanição de threads):
  ┌──────────────────────────────────────────────────────────────────┐
  │  Cenário: nf=3, n_pos=200, tile=8, 12 combos                    │
  │                                                                  │
  │  FLAT não-tiled:                                                 │
  │    n_total = 12 × 200 × 3 = 7.200 tarefas prange               │
  │    16 threads → ~450 tarefas/thread → balanceamento ótimo       │
  │                                                                  │
  │  FLAT tiled (tile=8):                                            │
  │    n_tiles = ceil(200/8) = 25                                    │
  │    n_total = 12 × 25 × 3 = 900 tarefas prange                  │
  │    16 threads → ~56 tarefas/thread → threads ficam ociosas      │
  │    Redução 8× nas tarefas > benefício de localidade L2          │
  └──────────────────────────────────────────────────────────────────┘

Motivos adicionais para o benefício de cache não se materializar:
  1. L2 = 256 kB/core; 9 slices × ~31 kB = ~280 kB → marginal para nf=1
  2. Numba JIT usa acesso linear sequencial → prefetch de hardware já eficaz
  3. Overhead de agendamento de threads domina para n_tiles pequeno
  4. Benefício real só emerge quando slice >> L2, i.e.:
     - n >> 22 camadas  (shape npt×n cresce: 201×10 → 201×50+)
     - filtro Anderson 801pt (npt=801 vs npt=201 Werthmuller)
```
