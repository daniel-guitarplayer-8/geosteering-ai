## 3. Investigação — Causa-Raiz

### 3.1 Inspeção empírica das assinaturas Numba

Comparação direta dos dois caminhos via `hmd_tiv.signatures`:

```
Path A — Numba direto (test_simulation_numba_dipoles.py)
  krJ0J1: readonly array(float64, 1d, C)   ← FilterLoader.setflags(write=False)
  wJ0   : readonly array(float64, 1d, C)
  wJ1   : readonly array(float64, 1d, C)
  h     : array(float64, 1d, C)             ← np.array local writeable
  prof  : array(float64, 1d, C)
  ...
  TODOS os demais: array(..., C)            (writeable)

Path B — JAX callback (após v2.31 fix _to_writeable)
  krJ0J1: array(float64, 1d, C)             ← _to_writeable forçou writeable
  wJ0   : array(float64, 1d, C)
  wJ1   : array(float64, 1d, C)
  h     : array(float64, 1d, C)
  prof  : array(float64, 1d, C)
  ...
  TODOS: array(..., C)                      (writeable)
```

### 3.2 Causa precisa

O fix v2.31 (`_to_writeable`) tentou eliminar a especialização duplicada
forçando arrays JAX-callback para `writeable=True`. **Funcionou para
arrays criados localmente** (`h_arr`, `prof_arr`, `eta`, `u`, `s`...)
mas **introduziu nova divergência nos filtros de Hankel**:

| Array              | Numba direto         | JAX callback (v2.31) | Convergência |
|:-------------------|:---------------------|:---------------------|:------------:|
| `krJ0J1, wJ0, wJ1` | readonly (FilterLoader) | writeable (_to_writeable forçou) | **DIVERGENTE** |
| `h_arr, prof_arr…` | writeable (np.array)    | writeable (_to_writeable)         | OK           |

`FilterLoader.load()` faz `arr.setflags(write=False)` propositalmente em
`filters/loader.py:451-453` para proteção contra mutação acidental
compartilhada — invariante intencional do design.

Numba trata `readonly array(...)` e `array(...)` como **tipos distintos**
no system de especialização, gerando 2 entradas em `.signatures` (e 2
arquivos `.nbc` quando `cache=True`).

### 3.3 Por que passa isolado?

Quando `test_simulation_numba_specializations.py` roda sozinho:
1. Path B (JAX callback) compila **primeira** — gera especialização
   "writeable everything"
2. Não há Path A subsequente → permanece 1 signature

Quando roda em suite com `test_simulation_numba_dipoles.py` antes:
1. Path A (Numba direto) compila **primeira** — gera "readonly filters"
2. Path B (JAX callback) compila depois — gera "writeable filters"
3. Total: 2 signatures distintas → FAIL

## 4. Fix Aplicado

### 4.1 Estratégia

**Convergir os dois caminhos espelhando a mutabilidade do caminho
Numba direto** (que é a fonte da verdade, dado que `FilterLoader` é o
único produtor de filtros em produção):

- Filtros (`krJ0J1`, `wJ0`, `wJ1`): preservar `readonly` (matching
  FilterLoader).
- Demais arrays (`h_arr`, `prof_arr`, `eta`, `u`, `s`, `uh`, `sh`, RT*,
  M*, E*, F*): forçar `writeable=True` (matching código local de
  produção e testes).

### 4.2 Implementação

Arquivo modificado: `geosteering_ai/simulation/_jax/kernel.py`

**(1) Novo helper `_to_readonly_contig`** complementa `_to_writeable`:

```python
def _to_readonly_contig(arr) -> np.ndarray:
    """Converte para ``np.ndarray`` contíguo preservando ``readonly`` (v2.38)."""
    out = np.ascontiguousarray(arr)
    if out.flags.writeable:
        out = out.copy()
        out.setflags(write=False)
    return out
```

**(2) `_dipoles_numba_host` reescrito** para usar o helper correto por
parâmetro:

```python
kr_ro = _to_readonly_contig(krJ0J1)   # filtros: readonly (espelha FilterLoader)
w0_ro = _to_readonly_contig(wJ0)
w1_ro = _to_readonly_contig(wJ1)
u_np  = _to_writeable(u)              # demais: writeable

Hx, Hy, Hz = hmd_tiv(..., kr_ro, w0_ro, w1_ro, _to_writeable(h_arr), ...)
Hx, Hy, Hz = vmd   (..., kr_ro, w0_ro, w1_ro, _to_writeable(h_arr), ...)
```

**(3) Docstring de `_to_writeable`** atualizada com nota de uso seletivo
(remetendo a `_to_readonly_contig` para os filtros).

## 5. Débito D3 — `NUMBA_CACHE_DIR` em conftest.py (fechado)

### 5.1 Contexto

O plano Sprint v2.36 listava D3 como pendente:
> **D3** — `NUMBA_CACHE_DIR` ausente do `tests/conftest.py` →
> LLVM recompile na suite, risco BAIXO

O fix v2.31 setava `NUMBA_CACHE_DIR` somente em `cli/_main.py` (em tempo
de import). Testes que importam diretamente `simulation/_numba/*` (a
maioria dos `test_simulation_numba_*.py`) **não acionavam o setup**, e
geravam `.nbc` em local inconsistente entre runs.

### 5.2 Implementação

Arquivo: `tests/conftest.py` (extensão do conftest raiz)

```python
# ── NUMBA_CACHE_DIR em tmpfs (v2.36 D3) ──────────────────────────────────
# DEVE executar antes de qualquer ``import numba`` ou
# ``from geosteering_ai.simulation._numba import ...``. Idempotente:
# preserva override do usuário (CI, debug).
if "NUMBA_CACHE_DIR" not in os.environ:
    _cache_dir = os.path.join(tempfile.gettempdir(), "geosteering_numba_cache_test")
    try:
        os.makedirs(_cache_dir, mode=0o700, exist_ok=True)
        os.environ["NUMBA_CACHE_DIR"] = _cache_dir
    except OSError:
        pass
```

### Trade-offs Avaliados

| Decisão                                        | Justificativa                                                              |
|:-----------------------------------------------|:--------------------------------------------------------------------------|
| Novo helper vs flag boolean em `_to_writeable` | Helper explícito é mais legível e auto-documentado. Custo trivial.        |
| Cópia + `setflags(write=False)` para filtros writeable | Garante semântica readonly mesmo se o caller passar mutável. Cópia de 201 floats é negligível (<1 µs). |
| Mutabilidade espelha caminho Numba direto      | FilterLoader é a fonte da verdade em produção. JAX adapta-se ao consumidor. |
| Sem mudança em código de produção (forward.py) | Hot path Numba intocado; fix isolado ao callback JAX. Zero risco a paridade. |

### Riscos Residuais

| Risco                                                    | Severidade | Mitigação                                  |
|:---------------------------------------------------------|:----------:|:-------------------------------------------|
| Futuro caller passar filtros mutáveis no caminho Numba direto | Baixo | FilterLoader é singleton + readonly; convenção robusta. |
| Cópia em `_to_readonly_contig` quando input já writeable | Trivial   | 201 doubles × ~1 ns/elem = ~200 ns; uma vez por simulação. |
| Test ordering ainda frágil em outros caminhos não cobertos | Baixo   | Cobertura atual (1762 PASS) inclui paths críticos. |
