## 1. Contexto e Problema Relatado

Após a entrega de v2.26 (que moveu o warmup do inicializador para um `Future` cancelável,
eliminando o hang no shutdown), o usuário reportou que o problema persistia com **piora significativa**:

| Sintoma | v2.25 | v2.26 |
|:--------|:------|:------|
| Tempo de warmup (cache frio) | 35–38 s (masked por OOM/SIGKILL) | **110+ s** |
| Tempo de warmup (cache quente) | ~6 s | ~6 s |
| Hang ao fechar | Sim (handler atexit aguarda init) | Não (futures canceláveis) |
| Warmup falha silenciosamente | Às vezes (OOM/SIGKILL) | Não observado |

O usuário também perguntou:
1. **Por que 110+ s?** (era ~35 s antes)
2. **Acontece fora do SM?** (simulações diretas via CLI/testes)
3. **Um warmup com 1 modelo ajudaria?** (configuração mínima)

## 2. Investigação — Causa-Raiz

### 2.1 Anatomia do tempo de warmup

O Simulation Manager possui ~22 funções JIT compiladas via Numba/LLVM:

| Módulo | Funções `@njit` |
|:-------|:----------------|
| `_numba/hankel.py` | 4 (`prepare_kr`, `integrate_j0`, `integrate_j1`, `integrate_j0_j1`) |
| `_numba/geometry.py` | 3 (`_sanitize_profile_kernel`, `find_layers_tr`, `layer_at_depth`) |
| `_numba/kernel.py` | 5 (`_fields_in_freqs_kernel`, `_compute_zrho_kernel`, `_simulate_positions_njit`, `_simulate_positions_njit_cached`, `_simulate_combined_prange_flat`) |
| `_numba/propagation.py` | 2 (`common_arrays`, `common_factors`) |
| `_numba/dipoles.py` | 2 (`hmd_tiv`, `vmd`) |
| `_numba/rotation.py` | 2 (`build_rotation_matrix`, `rotate_tensor`) |
| `forward.py` | 4 (`_simulate_positions_njit`, `_simulate_positions_njit_cached`, `_simulate_combined_prange`, `_simulate_combined_prange_flat`) |
| **Total** | **~22 funções** |

Cada função é compilada via **LLVM** na primeira chamada com um dado tipo de assinatura.
Com `cache=True`, o bytecode é gravado em disco (`__pycache__/*.nbi`/`*.nbc`) após a
primeira compilação. Runs subsequentes carregam do disco em < 1 s por função.

**Tempo de compilação LLVM cold (sem cache em disco, 1 processo):** ~35–38 s.

### 2.2 Bug v2.26: N workers simultâneos → saturação LLVM

Em v2.26, `PoolWarmupThread.run()` submetia **n_workers futures simultâneos**:

```python
# BUG v2.26 — N compilações LLVM simultâneas
futs = [
    pool.submit(_run_numba_warmup_task, self._n_threads, self._hankel_filter)
    for _ in range(self._n_workers)  # ← N futures para N workers
]
for f in futs:
    f.result(timeout=120)
```

Com `n_workers = 4` (default típico em máquinas com 4+ cores físicos), os 4 workers
recebiam o future de warmup ao mesmo tempo e iniciavam compilação LLVM simultaneamente.

**Mecanismo de saturação:**
- Numba usa LLVM internamente para compilar JIT. LLVM utiliza múltiplas threads durante
  passes de otimização (loop unrolling, vectorization, etc.).
- Com 4 workers cada um rodando LLVM multi-thread simultaneamente: **4 × N_LLVM_threads**
  disputam os **N_cores** físicos.
- Em uma máquina com 8 cores físicos, cada compilação LLVM que normalmente levaria ~1.7 s
  passa a levar ~5–7 s (3–4× mais lenta por contenção de CPU + thrashing de cache L3).
- 22 funções × 5 s cada (serializado dentro de cada worker) = **110+ s por worker**.
- O `for f in futs: f.result()` aguarda o mais lento → **110+ s total**.

**Por que v2.25 reportava apenas 35–38 s?**
Em v2.25, o warmup ocorria no **inicializador** do pool (`_numba_init_worker`). O macOS,
ao detectar alta pressão de memória durante compilação LLVM simultânea, enviava **OOM/SIGKILL**
para os workers. O processo principal recebia `BrokenProcessPool`, emitia `warmup_error` e
terminava o ciclo — **nunca vendo o tempo real de 110+ s** que os workers teriam levado.
O tempo de 35–38 s observado em v2.25 correspondeu a runs com **cache quente** (~6 s de load
do disco × overhead mínimo), não a compilação LLVM fria.

### 2.3 Segundo bug: `_WORKER_INITIALIZED` nunca setado em `run_numba_chunk`

O bloco de warmup secundário em `run_numba_chunk` existia para workers que não haviam
recebido o future de warmup:

```python
# BUG implícito — _WORKER_INITIALIZED jamais setado após warmup secundário
if chunk and not _WORKER_INITIALIZED:
    m0 = chunk[0]
    _ = simulate_multi(...)
    # ← FALTA: _WORKER_INITIALIZED = True
```

Como `_WORKER_INITIALIZED` permanecia `False` após o warmup secundário, o bloco rodava
**a cada chamada de `run_numba_chunk`** para o mesmo worker na mesma sessão — descartando
o resultado do primeiro modelo de cada chunk desnecessariamente.

### 2.4 Fora do SM?

**Não.** O problema é específico ao `PoolWarmupThread` do SM. Quando `simulate_multi` é
chamado diretamente (testes, CLI `geosteering-cli benchmark`, benchmarks em notebook), a
compilação JIT ocorre no **processo principal** (inline, sem pool de workers). Nenhuma
contenção LLVM multi-processo é possível. O cold JIT é ~35–38 s na primeira chamada e
< 1 s em runs subsequentes (cache de disco).

### 2.5 Um warmup com 1 modelo ajudaria?

**Não de forma significativa.** O tempo de warmup é dominado pela **compilação LLVM**,
não pela execução do modelo em si. LLVM compila o bytecode das funções com base nos
**tipos** dos argumentos (assinatura de tipo), independentemente do tamanho dos arrays.
Usar `n_layers=1` vs `n_layers=10`, ou `n_pos=1` vs `n_pos=50`, resulta na **mesma
compilação LLVM** — o bytecode é idêntico para ambos. A solução correta é reduzir o
**número de compilações paralelas** (de N workers para 1), não o tamanho do input.

## 3. Solução Implementada (v2.27)

### 3.1 `PoolWarmupThread.run()` — 1 future (não n_workers)

```python
# FIX v2.27 — 1 compilação LLVM (sem contenção)
fut = pool.submit(
    _run_numba_warmup_task, self._n_threads, self._hankel_filter
)
result = fut.result(timeout=120)
if not result:
    self.warmup_error.emit("warmup JIT retornou False — ...")
    return
self.warmup_done.emit(time.perf_counter() - t0, self._n_workers, self._n_threads)
```

**Fluxo com o fix:**

```
GUI abre → PoolWarmupThread inicia
  └─ pool.submit(warmup_task) → Worker 0 recebe o future
       └─ Worker 0 compila 22 funções JIT (~35 s cold / < 2 s quente)
            └─ Bytecode gravado em __pycache__/*.nbi/*.nbc
                 └─ warmup_done emitido → GUI atualiza label verde

Usuário clica "Simular" → SimulationThread submete chunks
  └─ Worker 0: já warm → processa chunk imediatamente
  └─ Workers 1..N: secondary warmup em run_numba_chunk:
       └─ _WORKER_INITIALIZED == False → simulate_multi() 1 chamada
            └─ Numba lê bytecode do cache em disco (< 2 s por worker)
                 └─ _WORKER_INITIALIZED = True → chunks subsequentes: sem overhead
```

**Tempo total:**
- Cache frio (1ª execução após instalação/update): **~35–38 s** (Worker 0 compila)
- Cache quente (execuções subsequentes): **< 2 s** (Worker 0 carrega do disco)
- Workers 2..N: **< 2 s overhead na 1ª task** (carrega do disco via warmup secundário)

### 3.2 `run_numba_chunk()` — `_WORKER_INITIALIZED = True` após warmup secundário

```python
global _WORKER_INITIALIZED
if chunk and not _WORKER_INITIALIZED:
    m0 = chunk[0]
    _ = simulate_multi(
        rho_h=np.asarray(m0["rho_h"], dtype=np.float64),
        ...
    )
    _WORKER_INITIALIZED = True  # ← FIX v2.27
```

Evita o warmup secundário em **todos** os chunks subsequentes do mesmo worker.

### 3.3 Compatibilidade com mecanismo `t0_sim` (noop-based)

O mecanismo de `t0_sim` em `SimulationThread.run()` usa `n_workers` noops para sincronizar
o pool antes de medir o throughput. Com apenas 1 warmup future:

- Workers 1..N recebem noops imediatamente (sem warmup future pendente) → noops completam
  em < 0.1 s.
- Worker 0 recebe 1 noop APÓS o warmup future (FIFO por worker) → noop completa quando
  warmup termina.
- `t0_sim` é setado após **o mais lento** dos noops → aguarda Worker 0 completar warmup.

A garantia de t0_sim permanece correta: throughput mede apenas a fase de simulação real.
