## §1. Q1 — Pré-Mortem: Módulos JAX Otimizados Exclusivamente por Cenário

### §1.1. Premissa em análise

> Construir 8 módulos JAX dedicados, um por cenário, cada um com layout de memória, vmap pattern, dtype default e XLA flags otimizados especificamente para aquele cenário. Dispatcher heurístico no `simulate_multi_jax_batched()` escolhe rota em runtime.

**Arquitetura hipotética**:

```
geosteering_ai/simulation/_jax/scenarios/
├── _dispatcher.py          # Heurística (n_pos, nf, nTR, nAng) → cenário
├── scenario_a.py           # Overhead-bound, n_pos=1
├── scenario_b.py           # n_pos=100, single freq/TR/ang
├── scenario_c.py           # 4 freqs vmap sweet spot
├── scenario_d.py           # 4 TR single pos
├── scenario_e.py           # n_pos=600, complex64 default
├── scenario_f.py           # 4×4 multi-dim
├── scenario_g.py           # 4×4×4 triple nested
└── scenario_h.py           # 8×8×8 stress + chunking forçado
```

#### §1.2.3. Pontos Fortes da Ideia (não-niilismo metodológico)

Mesmo rejeitada, a ideia tem **2 mérito reais** que devem ser preservados na alternativa:

| Mérito | Por que vale a pena preservar |
|:--|:--|
| **M1** Reconhecimento de que cenários têm perfis distintos | Cenário A é overhead-bound; E é bandwidth-bound; G é launch-bound. Tratá-los como "o mesmo workload" é ingênuo |
| **M2** Desejo de extrair máximo de cada configuração | Especialização **é desejável** — só não na forma "1 arquivo por cenário". Forma correta é via `static_argnames` + cache JIT por configuração |

#### §1.2.4. Síntese: 3 Problemas Raiz

Consolidando os 8 pontos fracos:

1. **Combinatória implacável**: 8 cenários × 4 hardware × 2 dtypes × N workloads inéditos = explosão de **manutenção**, **testes**, **caches XLA**.
2. **Antithesis a princípios recém-adotados**: viola Princípio Universal-GPU (ADR-0002 draft) antes mesmo dele virar lei. Cria precedente péssimo.
3. **ROI negativo**: cenários "fáceis" (A, D) já vencem trivialmente; cenários "difíceis" (E, G) são limitados por física do hardware (bandwidth, FLOPS), não por falta de especialização lógica.

#### §1.2.5. Alternativa Recomendada (Bloco B — "Especialização Adaptativa")

Em vez de módulos dedicados, usar **JAX nativo** para especialização automática:

```python
# geosteering_ai/simulation/_jax/forward_pure.py (Sprint O3 ampliado)

@functools.partial(jax.jit, static_argnames=(
    "n_pos_tier",       # 1, 100, 600 (tier discreto, não valor exato)
    "n_freq",           # 1, 2, 4, 8
    "n_tr",             # 1, 4, 8
    "n_ang",            # 1, 4, 8
    "chunk_size",       # None ou int (Sprint 12 E4 já existe)
    "use_complex64",    # bool (Sprint O4 opt-in)
))
def forward_pure_unified_adaptive(
    rho_h, rho_v, esp, positions_z,
    freqs, trs, dips,
    n_pos_tier, n_freq, n_tr, n_ang, chunk_size, use_complex64,
):
    """Caminho ÚNICO com especialização via static args.

    JAX gera kernels XLA distintos automaticamente para cada combinação
    de static args, mantendo cache LRU. Equivale a 8 (ou mais) módulos
    distintos sem código duplicado.

    Vantagens vs proposta modular:
    - 1 arquivo de código fonte (manutenção × 1)
    - 1 teste de paridade Fortran (validação × 1)
    - Cache JIT por config (XLA gerencia, não precisamos)
    - Dispatcher trivial: derivar tier de n_pos via heurística simples
    - PINN compatível: jax.grad(forward_pure_unified_adaptive) funciona
    """
    if use_complex64:
        rho_h, rho_v, esp = (x.astype(jnp.float32) for x in (rho_h, rho_v, esp))
    # ... lógica unified existente ...
    # XLA fará fusão/inlining diferente para cada (n_pos_tier, n_freq, ...) combination
```

**Heurística trivial** (substitui dispatcher complexo):

```python
def _n_pos_tier(n_pos: int) -> int:
    """Discretiza n_pos em 3 tiers para JIT cache eficiente."""
    if n_pos <= 10:    return 10
    if n_pos <= 100:   return 100
    return 600  # n_pos > 100 cai sempre em 600
```

JAX automaticamente:
- Gera kernel XLA distinto por `(n_pos_tier=10, n_freq=1, ...)` vs `(n_pos_tier=600, n_freq=4, ...)`
- Cacheia em `_UNIFIED_JIT_CACHE` (já existente)
- Reutiliza em call subsequente com mesma assinatura
- Permite `jax.grad` sobre rho_h sem mudar nada (PINN-friendly)

**Comparação direta**:

| Aspecto | Proposta Modular (8 arquivos) | Alternativa Adaptativa (static_args) |
|:--|:--|:--|
| Linhas de código | ~8 × 600 = 4.800 (proxy) | 1 × 700 (incremento ~50 LOC) |
| Arquivos a manter | 8 + dispatcher = 9 | 1 |
| Testes paridade Fortran | 8 × 164 = 1.312 | 164 (já existe) |
| Cache XLA programs | 8 × N variações = 100+ | 6-12 (3 n_pos_tier × 2-4 dim) |
| Cold-start dispatch | ~30-180s × 8 paths | ~30s × ~10 paths |
| PINN `jax.grad` | Quebrado ou 8× recomp | Direto |
| ADR-0002 compliance | ❌ Viola Universal-GPU | ✅ Mantém path único |
| Onboarding novo dev | 8 contextos similares | 1 contexto + docstring static_args |

### §2.1. Reconhecimento: o que JÁ EXISTE

**Conclusão da investigação**: a dimensão `n_pos` está **plenamente vetorizada**. A questão "podemos paralelizar pontos de medição na GPU?" tem resposta empírica: **já está feito**.

### §2.2. Por que cenário E ainda é lento então?

Se `vmap` sobre n_pos existe, por que cenário E (n_pos=600) é apenas 43k mod/h em T4 e perde para Numba 7970X (390k)?

**Decomposição do gargalo real** (extraído de `v2.43_jax_gpu_optimization_plan.md` Top-10):

| Gargalo | Estimativa do tempo total em E | Natureza |
|:--|:--:|:--|
| Per-position compute (TE/TM propagation + dipoles) | 40-50% | Compute-bound (FP64 FLOPS) |
| Memory bandwidth (loading common_arrays per pos) | 25-35% | Memory-bandwidth bound (320 GB/s teto T4) |
| Dispatch overhead (kernel launches XLA) | 10-15% | Launch-bound (CUDA driver overhead) |
| `rho_h_at_obs` Python loop (multi_forward.py:1220) | 5-10% | Python overhead (não-GPU) |
| `FilterLoader` reinstantiation (R2 fix em Sprint O2) | 1-5% | Python overhead |

**A vmap sobre n_pos está fazendo o que deveria** — entregando 600 posições em paralelo ao XLA. O gargalo é a **densidade computacional por posição** (cada posição requer centenas de muladds para TE/TM propagation) e a **memory bandwidth** ao carregar `common_arrays` (kr=201pt × n_freq × n_pos = 120.600 acessos contíguos).

### §2.3. Oportunidades REAIS de aceleração (não-óbvias)

Dada a paralelização atual, **5 oportunidades de aceleração genuínas** restam:

#### §2.3.1. Reescrita do kernel EM como `jnp.einsum` único

O kernel atual é uma série de operações primitivas (`jnp.multiply`, `jnp.exp`, `jnp.sum`). XLA fusiona algumas, mas não todas. Reescrevê-lo como **1 chamada `jnp.einsum`** massiva pode forçar fusão melhor:

```python
# ATUAL (forward_pure.py, simplificado):
def kernel(z_pos, freq, common_arrays):
    a = jnp.exp(jnp.multiply(common_arrays.u, z_pos - h))
    b = jnp.multiply(reflectance, a)
    integrand = jnp.multiply(b, weights_j0)
    return jnp.sum(integrand, axis=-1)

# PROPOSTO (Sprint O6):
def kernel_einsum(z_pos, freq, common_arrays):
    # Tudo em 1 einsum — XLA otimiza melhor pattern de fusão
    return jnp.einsum(
        "zk,fk,k,k,k->zf",   # z=n_pos, f=nf, k=npt (filtro Hankel)
        z_pos[:, None] - h_bcast,
        common_arrays.u_per_freq,
        common_arrays.reflectance,
        common_arrays.weights_j0,
        common_arrays.kr,
    )
```

**Ganho esperado**: 1.3-1.8× (estimativa baseada em padrões observados em literatura JAX para kernels EM/CFD).

**Risco**: paridade Fortran <1e-12 pode quebrar (reordenamento associativo de soma flutuante). Mitigável com tolerância 1e-10.

#### §2.3.2. `lax.map(batch_size=K)` sobre n_pos com auto-tuning

Já no Sprint O2/O3 do plano original. Permite n_pos=600 sem materializar (n_pos=600, n_freq=4, n=20, npt=201) = 9.6 GB de uma vez. Chunking adaptativo escolhe K ótimo baseado em VRAM disponível.

**Ganho esperado**: elimina OOM (cenário G em T4 com n_models=20 → n_models=50). Throughput: +10-20%.

#### §2.3.3. complex64 em cenário E (Sprint O4)

Atualmente: complex128 forçado em todo o pipeline. Cenário E usa 9.6 GB; cenário G usa 17.8 GB (OOM em T4 com n_models=50).

Mudar para complex64 corta memória pela metade e **dobra** throughput em ops bandwidth-bound (E é bandwidth-bound):

| Cenário | T4 complex128 (atual) | T4 complex64 (Sprint O4) | Ganho |
|:--:|:--:|:--:|:--:|
| E | 43.021 mod/h | ~75.000-90.000 mod/h | 1.7-2.1× |
| G | 9.197 mod/h (n=5) | ~20.000 mod/h (n=20) | 2.2× + n_models maior |

**Paridade Fortran <1e-12 quebra** — mas com gate `≤5e-3 relativo` (aceito para PINN e produção tolerante), o opt-in é viável.

#### §2.3.4. Async dispatch + buffer donation

`jax.jit(donate_argnums=(0,1,2))` permite re-uso in-place de buffers (rho_h, rho_v, esp) — elimina cópia intermediária em batch loops.

**Ganho esperado**: 5-10% em E; viabiliza pipelining de chamadas sequenciais (treino PINN multi-step).

#### §2.3.5. Pre-compute `common_arrays` cache fora do JIT

Como em Sprint 2.10 (Numba) — `common_arrays` depende de (hordist, freq, perfil) mas **não** de z_obs. Recomputar em cada posição é desperdício de 6× (já documentado).

Em JAX, isso traduz-se em: extrair `common_arrays` para **outside** o `vmap` sobre posições e passar como argumento `in_axes=None`. Pode já estar parcialmente feito — investigar em Sprint O6.

**Ganho esperado**: 1.2-1.5× em E (memory bandwidth reduzida proporcionalmente).

| "Onde implementar?" | Sprint O4 (complex64) + Sprint O6 (einsum + cache externalizado + donation). Não cria nova sprint — refina escopo das existentes. |

### §2.5. Cálculo de ganho composto (Sprint O3+O4+O6)

Cenário E (atual T4 = 43.021 mod/h) com **todas** as otimizações:

```
43.021 × 1.5 (einsum)
       × 2.0 (complex64)
       × 1.15 (chunking auto-tuned)
       × 1.30 (common_arrays externalizado)
       × 1.05 (buffer donation)
       ≈ 202.000 mod/h

Numba i9 8C/16T = 122.000 mod/h
Ratio = 202 / 122 = 1.65× ✅ Passa gate ≥1.5×
```

Esse é o **caminho realista** para fechar o gap em E em **um único hardware T4**. Sem precisar multi-GPU.

`★ Insight ─────────────────────────────────────`
**A revelação Q2**: o foco em "paralelizar mais" estava errado — vmap dimension n_pos já está exausta. O gargalo real é **densidade computacional por posição** (FLOPS + bandwidth), não falta de paralelismo. Atacar densidade requer (a) menos ops por posição (einsum fusion), (b) menos bytes por op (complex64), (c) menos roundtrips (cache externalizado). Esse é um tipo de otimização **estruturalmente diferente** de "adicionar mais vmap".
`─────────────────────────────────────────────────`

## §3. Q3 — Numba Multi-Worker vs Treino On-the-Fly

### §3.1. Esclarecimento conceitual (a confusão a desfazer)

O termo "treino on-the-fly" em Geosteering AI v2.0 refere-se especificamente ao **pipeline de transformação de dados em runtime** durante o treino:

```
DISCO                          RAM/GPU                          MODELO
───────                        ───────                          ───────
[H_clean.h5]    →   load   →   H_clean  →  add_noise(A/m)  →  noisy_H
                                              ↓
                                          FV_tf(noisy_H)  →  FV_noisy
                                              ↓
                                          GS_tf(noisy_H)  →  GS_noisy
                                              ↓
                                          scaler.transform → X
                                              ↓
                                          model.fit_step(X, y)
```

**Crítico**: **a simulação EM (forward) NÃO é parte deste pipeline on-the-fly**. Ela é executada **ANTES**, na fase de **geração de dataset**:

```
GERAÇÃO DATASET (offline, dias)        TREINO (on-the-fly, horas)
─────────────────────────────────       ────────────────────────────
profiles_raw                            H_clean.h5 (cached)
   ↓                                       ↓
simulate_multi (Numba/JAX)                load + on-the-fly transforms
   ↓                                       ↓
H_clean (gravado em disk como .h5)      training step
```

### §3.2. Por que minha recomendação NÃO elimina on-the-fly

Na recontextualização (§2.1 do relatório de reformulação), recomendei:

> "Geração de dataset (treino DL): Numba multi-worker"

Isto refere-se à fase **OFFLINE** de geração. Numba multi-worker gera os `H_clean.h5` em massa. O treino DL on-the-fly **carrega** esses `H_clean.h5` e aplica transformações em RAM. Os dois processos:
- Acontecem em **momentos diferentes** (geração: dias antes; treino: hoje)
- Em **hardware diferente** (geração: pode ser CPU/HPC; treino: GPU)
- Com **objetivos diferentes** (geração: produzir dataset; treino: aprender mapping)

**Conclusão**: Numba multi-worker para geração é **complementar**, não substituto, do treino on-the-fly.

### §3.3. Quando JAX é mandatório para on-the-fly

Existe **um caso** em que a recomendação Numba colidiria com on-the-fly: **treino PINN com simulação on-the-fly dentro do loop**.

```python
# PINN training step — simulação ACONTECE NO STEP
def pinn_step(theta, profiles_batch):
    H_predicted_by_model = model.apply(theta, profiles_batch)

    # AQUI: simulação on-the-fly (precisa de grad!)
    H_true_via_physics = forward_jax(profiles_batch)  # JAX mandatório

    loss_data = mse(H_predicted_by_model, H_true_via_physics)
    loss_physics = pinn_constraint(H_predicted_by_model, profiles_batch)
    return loss_data + lambda * loss_physics

grad_fn = jax.grad(pinn_step)  # Precisa diferenciar forward_jax → JAX
```

Neste caso:
- **Não pode** usar Numba (sem auto-diff)
- **Deve** usar JAX, mesmo se Numba CPU multi-worker fosse 3× mais rápido em throughput puro
- Por isso JAX **continua sendo backend obrigatório** do projeto

### §3.4. Tabela canônica de matriz workload × backend

Esta é a tabela que deveria estar em **ADR-0004** (proposto):

| Workload | Volume | Latência | Auto-diff? | Backend recomendado | Razão |
|:--|:--|:--|:--:|:--|:--|
| **Geração dataset offline** | 100k–1M+ modelos | dias OK | Não | **Numba multi-worker** | Throughput puro, sem grad |
| **Treino DL convencional on-the-fly** (carrega H_clean cached + transforma) | 1k batches × 32-128 | horas OK | Sim (sobre weights, não sobre forward) | **TF/Keras nativo** (transforma em GPU) | Forward já cached; transformações são tf.image, tf.random |
| **Treino PINN on-the-fly** (simula no step) | 10k iter × forward+grad | horas OK | Sim (sobre rho_h/rho_v/esp) | **JAX (obrigatório)** | Auto-diff sobre simulador |
| **Inferência online geosteering** (1 forward por step LWD) | 1 modelo × 600 pos | <50 ms | Não | **Numba (CPU)** ou **JAX (GPU)** | Latência crítica; Numba single-thread costuma vencer GPU por overhead PCIe |
| **Inferência batch** (avaliação holdout, 1000+ modelos) | 1000+ × forward | minutos OK | Não | **JAX (GPU se disponível) ou Numba (CPU)** | Depende de hardware; dispatcher escolhe |
| **Uncertainty Quantification (MC Dropout)** | N_mc × 1 forward × M positions | segundos | Não | **Numba ou JAX** | N_mc=20-100, escalável em ambos |
| **Uncertainty Quantification (Ensemble)** | E ensembles × 1 forward | segundos | Não | Mesmo backend do treino (geralmente JAX) | |
| **Uncertainty Quantification (INN)** | sampling posterior latent | segundos | Sim (Jacobiano determinante) | **JAX (preferido)** | Bijector chains diferenciáveis |

### §3.5. O que isso muda no projeto?

**Nada estruturalmente** — apenas formaliza o que já é prática:

1. `geosteering_ai/data/loading.py` carrega `H_clean.h5` (pré-computado)
2. `geosteering_ai/data/pipeline.py` aplica on-the-fly noise+FV+GS+scale
3. `geosteering_ai/training/loop.py` chama `model.fit_step` com batches transformados
4. **Nenhum** desses arquivos chama `simulate_multi_jax` ou `simulate_multi` em runtime — eles carregam do disco

**A confusão que motivou Q3** vinha de **dois usos distintos do termo "on-the-fly"**:

| Uso | Significado | Backend |
|:--|:--|:--|
| "Noise on-the-fly" (v2.0 atual) | Adicionar ruído em RAM antes de cada batch | TF/NumPy (não simula) |
| "Simulation on-the-fly" (PINN training) | Re-simular forward dentro do step de treino | JAX (mandatório) |

O primeiro é o pipeline DL padrão. O segundo é específico de PINNs. **Numba multi-worker para geração não impacta nenhum dos dois** — apenas alimenta o cache `H_clean.h5` que o primeiro usa.

### §3.6. Recomendação Q3

1. **Adicionar ADR-0004** ("Backend Selection by Training Workload") com a tabela §3.4 acima.
2. **Renomear** comentários/docstrings em `geosteering_ai/data/pipeline.py` que digam apenas "on-the-fly" para **"on-the-fly transformations (noise + FV + GS + scale)"** — desambigua de "on-the-fly simulation".
3. **Não mudar nada** no código de produção do pipeline DL. A recomendação Numba é só para `geosteering-cli benchmark` / geração de dataset.
4. **Confirmar com usuário** se PINN training está planejado para entrar em produção (status atual: 8 cenários implementados em `losses/pinns.py` mas treino completo "lambda × L_latent" listado como pendente em MEMORY.md). Se sim, JAX backend é **ainda mais crítico** que o relatório de reformulação reconheceu.

### §4.3. ADR-0004 (esqueleto)

```markdown
# ADR-0004 — Backend Selection by Training Workload

## Status
PROPOSED (Sprint O3 ou earlier)

## Context
O projeto Geosteering AI tem 3 backends de simulação EM (Numba CPU, JAX CPU/GPU,
empymod opt-in) e múltiplas modalidades de treino (DL convencional, PINN, UQ).
Não existe documento canônico que mapeie "qual backend para qual workload".

Confusão observada: "Numba multi-worker para geração" foi interpretado como
"abolir on-the-fly training" — porque "on-the-fly" tem dois usos no projeto:
1. On-the-fly transformations (noise+FV+GS+scale em RAM antes do batch) — TF
2. On-the-fly simulation (forward EM dentro do step PINN) — JAX

## Decision
Tabela canônica (§3.4 do relatório de reflexão 2026-05-27) define backend por workload.

## Consequences
- Geração dataset offline → Numba multi-worker (i9 8C ou 7970X 32C)
- Treino DL on-the-fly → TF/Keras carrega H_clean cached + transforma em GPU
- Treino PINN → JAX backend (forward+grad)
- Inferência online <50ms → Numba single-thread (PCIe overhead descalifica GPU)
- Inferência batch ou UQ → dispatcher escolhe baseado em hardware disponível
```

### §4.4. Resposta consolidada às 3 questões

**Q1** — Modularização por cenário: **NÃO**. Especialização adaptativa via `static_argnames` (Sprint O3 expandido) entrega os mesmos ganhos sem 8 dos 8 problemas identificados na pré-mortem.

**Q2** — Paralelização de pontos de medição: **JÁ FEITO** (vmap sobre n_pos desde Sprint 7.x). Oportunidades restantes são **densidade por posição** (Sprint O4 complex64 + Sprint O6 einsum/cache). Cálculo composto: T4 cenário E ≈ 202k mod/h alcançável (1.65× Numba i9, passa gate ≥1.5×).

**Q3** — Numba multi-worker elimina on-the-fly: **NÃO**. Confusão semântica (on-the-fly transformations ≠ on-the-fly simulation). ADR-0004 formaliza a matriz workload × backend.
