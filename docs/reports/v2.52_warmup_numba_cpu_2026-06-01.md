## 1. Causas-raiz do warmup Numba (investigação)

| # | Achado | Evidência |
|:-:|:-------|:----------|
| 1 | **Todos os ~16 `@njit` já têm `cache=True`** | wrapper aplica `cache=True, fastmath=False, error_model="numpy", nogil=True` → sem alavanca em flags |
| 2 | **AOT (`numba.pycc`) NÃO viável** | removido no Numba ≥0.59 (env 0.65.1) + incompatível com `parallel=True`/prange (batch kernels) |
| 3 | **~111s cold-start = recompile LLVM bitcode→nativo** | `.nbc` guarda bitcode portável; cada processo re-lowering p/ x86-64 (fundamental do Numba) |
| 4 | **GAP PRIMÁRIO: warmup não cobre os kernels de produção** | `_warmup_numba_tier2_sync` ([cli/_main.py:146](../../geosteering_ai/cli/_main.py#L146)) usa o callback JAX (`use_native_dipoles=False`, 3-pos) → aquece só hmd_tiv/vmd; **REQUER JAX**; NUNCA aquece os prange (`_simulate_combined_prange_flat`/`_simulate_positions_njit_cached`/`precompute_common_arrays_cache`) |

## 2. Soluções construídas

### A — Warmup Numba JAX-independente ([_numba/warmup.py](../../geosteering_ai/simulation/_numba/warmup.py))
`warmup_numba_simulator(...)` + `warmup_numba_simulator_from_config(cfg)`: roda o caminho
REAL `simulate_multi(backend="numba")` em **2 chamadas tiny** (a dispatch é mutuamente
exclusiva em `n_combos`): multi-combo `tr=(1.0,2.0)` → `_simulate_combined_prange_flat` +
`precompute`; single-combo `tr=(1.0,)` → `_simulate_positions_njit_cached`. Os dipolos
(`hmd_tiv`/`vmd`) compilam **inlinados** (njit-a-njit) dentro desses kernels. Arrays via
`np.full`/`np.linspace` (writeable C-contíguos) → **nenhuma 2ª especialização** (guard verde).
`from_config` passa o `cfg` REAL via `base_cfg` → honra os seletores de kernel
(`use_flat_prange`/`use_tiled_positions`) → aquece o kernel EXATO de produção.

### B — NUMBA_CACHE_DIR estável ([cli/_main.py](../../geosteering_ai/cli/_main.py) + [cli/warmup.py](../../geosteering_ai/cli/warmup.py))
Default ESTÁVEL `~/.cache/geosteering/numba_cache` (sobrevive reboot) com fallback `$TMPDIR`
(OSError gracioso); `0o700` mantido (segurança v2.31); override respeitado. tmpfs era apagado
no reboot → 1º run pós-reboot re-pagava a compilação COMPLETA; o dir estável persiste o `.nbc`.

### C — CLI ([cli/warmup.py](../../geosteering_ai/cli/warmup.py))
`geosteering-warmup` agora roda o warmup Numba full por DEFAULT (JAX-independente, substitui
o callback-JAX) + `--numba`/`--no-numba`/`--numba-n-pos`/`--numba-n-layers`/`--numba-threads`.
O `--jax`/`--auto` (v2.51) permanece. Background-thread (`geosteering-cli`) aponta p/ o novo
warmup (n_pos=64, best-effort silencioso).

## 4. AOT — por que NÃO perseguir (documentado)

`numba.pycc` (compilador AOT) foi **removido no Numba ≥0.59** (env = 0.65.1) e é
**incompatível com `parallel=True`/prange** — exatamente os batch kernels que dominam o
custo. Já concluído em `warmup_analysis_jit_2026-05-12.md`. O ~111s bitcode→nativo é o piso
fundamental do design Numba (portabilidade do `.nbc`); só o cache persistente o amortiza
cross-process (eixo i: 103×).

## Síntese

- **Gap primário fechado**: o warmup agora aquece os **kernels prange de produção** (antes
  só hmd_tiv/vmd via callback JAX) e é **JAX-independente** (roda em deploy Numba-only).
- **Cache persistente cross-reboot**: dir estável `~/.cache` → **103× cross-process** (.nbc).
- **AOT confirmado inviável**; o ~111s bitcode→nativo é fundamental — amortizado pelo cache.
- **Paridade Fortran `<1e-6` inviolada** (Numba é a referência); **zero regressão** de throughput.
