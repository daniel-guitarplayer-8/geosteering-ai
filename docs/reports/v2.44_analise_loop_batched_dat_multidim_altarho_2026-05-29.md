# Relatório — Geosteering AI v2.44 — JAX Loop vs Batched · Export .dat · Multi-dim · Alta-ρ

> **Data**: 2026-05-29 · **Hardware**: RTX A6000 48 GB + Threadripper 7970X · **Stack**: JAX 0.7.2 cuda12, Numba 0.65.1
> Todas as afirmações **verificadas empiricamente** (scripts em `/tmp/q*_*.py`, dados nesta sessão) ou por leitura de código (file:line).

---

## Resumo Executivo

(1) **Loop vs Batched** são dois pontos de entrada para o MESMO kernel: o *loop*
(`simulate_multi_jax`) processa **1 modelo/chamada** (loop Python sobre configs);
o *batched* (`simulate_multi_jax_batched`) processa **N modelos/chamada** via `jax.vmap`,
amortizando dispatch + 1 único sync GPU→CPU. (2) **Sim, dá para gerar modelos no batched
e salvá-los em `.dat`** — infraestrutura completa de writer 22-col existe e foi
**demonstrada** (batched→.dat→leitura, byte-exato). (3) Há 8 cenários canônicos A–H.
(4) Configurações multi-dim como **2 freq × 2 TR × 2 dip são suportadas e foram testadas
agora** (paridade <1e-13). (5) **Ambos os backends lidam com 1500/5000/10000 Ω·m sem perda
de fidelidade em c128** (Numba-vs-JAX <1e-13, sem NaN/Inf, ancorado em Fortran a 1416 Ω·m);
c64 mantém ~3.8e-4 (erro de bit-width, não de física).

## Q1 — JAX Loop vs JAX Batched

| Aspecto | **Loop** (`simulate_multi_jax`) | **Batched** (`simulate_multi_jax_batched`) |
|:--------|:--------------------------------|:-------------------------------------------|
| Entrada | 1 modelo: `rho_h (n,)`, `esp (n-2,)` | N modelos: `rho_h_batch (n_models, n)`, `esp_batch (n_models, n-2)` |
| Saída `H_tensor` | `(nTR, nAngles, n_pos, nf, 9)` | `(n_models, nTR, nAngles, n_pos, nf, 9)` |
| Container | `MultiSimulationResultJAX` | `MultiSimulationResultBatchedJAX` (+ `.get_model(i)`) |
| Eixo de paralelismo | configs (loop Python) + posições/freq (vmap interno) | **+ modelos (`jax.vmap` externo)** |
| Compilação XLA | 1 por (config-geom) | **1 reutilizada p/ todo o batch** |
| Sync GPU→CPU | **N×configs** (`np.asarray` por config) | **1× por batch** (ou 1×/chunk) |
| `_CTX_CACHE` | amortiza geometria entre chamadas repetidas | geometria computada **1×**, compartilhada no vmap |
| Estratégia | bucketed/unified por config | **bucketed** se geom compartilhada (esp idêntico) → senão fallback **unified** + warning |
| Diferenciável (`jacfwd`) | ✅ sobre rho | ✅ vmap distribui o grad sobre os modelos (PINN) |
| Restrição | — | `n` homogêneo entre modelos; bucketed exige `esp` bit-idêntico |
| **Quando usar** | **inferência single-model**, 1 poço por vez | **treino PINN, geração on-the-fly de datasets, muitos modelos** |

**Mecanismo-chave**: o *loop* paga overhead Python + sync por config (≈30–50 ms/config no
GPU) — bom para 1 modelo. O *batched* **amortiza esse overhead entre os N modelos** e satura
a GPU; com geometria compartilhada usa o kernel **bucketed** (ct/cr estáticos → XLA
especializa) `vmap`-ado sobre o eixo de modelos. Empírico desta sessão: on-the-fly 32×600
**65k (loop/unified) → 1.47M mod/h (batched-bucketed), 22.5×**. Paridade entre os dois:
**bit-exata** (mesmo `_get_bucket_jit`) — medido nesta sessão **5.6e-17** (batched[0] vs loop).

## Q2 — Gerar modelos no Batched e salvar em `.dat` (DEMONSTRADO)

**Sim — totalmente possível e demonstrado nesta sessão.** A infraestrutura de I/O
Fortran-compatível já existe (paridade byte-exata validada vs `tatu.x`):

| Componente | Arquivo | Função |
|:-----------|:--------|:-------|
| Writer single/multi-modelo | `simulation/io/binary_dat.py` | `export_binary_dat(cfg, H_tensor, z_obs, rho_h, rho_v, ..., append=)` |
| Writer multi-TR | `simulation/io/binary_dat_multi.py` | `export_multi_tr_dat(result, base, dir)` (gera `_TR{i}.dat`) |
| Writer modelo Fortran | `simulation/io/model_in.py` | `export_model_in(...)` (`.in` ASCII) |
| Parser | `validation/compare_fortran.py` | `read_fortran_dat_22col(path)` |

**Formato**: binário 22-col, 172 bytes/registro: `i (int32) + z_obs + rho_h + rho_v + 9×(Re,Im)`
(ordem Hxx,Hxy,Hxz,Hyx,Hyy,Hyz,Hzx,Hzy,Hzz).

**Demonstração empírica** (3 modelos batched → 1 `.dat` via `append`):
```
batched H_tensor: (3, 1, 1, 20, 1, 9)
.dat gerado: dataset.dat | bytes: 10320 = 3 modelos × 20 pos × 172 ✓
relido: 60 registros (= 3×20) ✓ — round-trip OK
campos: ('i','z_obs','rho_h','rho_v','Re_Hxx','Im_Hxx',...)
```

**Caso de uso (geração de dataset de treino)**: `simulate_multi_jax_batched` (rho variável,
geom fixa) → para cada modelo: `export_binary_dat(..., append=(i>0))` acumula no mesmo `.dat`.
`MultiSimulationResultBatchedJAX` é 100% NumPy serializável (também `np.savez`/`np.save`).

**Ressalva descoberta na demo**: o atalho `export_multi_tr_dat` chama `export_info_out`, que
acessa `result.H_tilted` (atributo **só do resultado Numba**, F5 tilted-coil) — ausente no
`MultiSimulationResultJAX` → `AttributeError`. **Solução**: usar `export_binary_dat`
diretamente (feito na demo), OU adicionar `H_tilted=None` ao dataclass JAX (fix trivial de
1 linha) para compat total com o wrapper multi-TR. Recomendação: campo `export_per_model`
em `SimulationConfig` (Grupo 7) para integrar nativamente no fluxo batched.

## Q4 — Multi-dim 2 freq × 2 TR × 2 dip: existe e foi testado? (DEMONSTRADO)

**Não existe um cenário CANÔNICO nomeado exatamente 2×2×2**, mas a configuração é
**plenamente suportada** (qualquer lista de freqs/TRs/dips) e **foi testada agora**:

```
Q4 — 2 freq [20k,80k] × 2 TR [1,2] × 2 dip [0°,30°]:
  JAX loop    H_tensor shape: (2, 2, 50, 2, 9)        ✓
  JAX batched H_tensor shape: (4, 2, 2, 50, 2, 9)     ✓
  paridade batched[0] vs loop : 5.55e-17  (bit-exato)
  paridade JAX vs Numba       : 1.75e-13  (< 1e-12)   ✓
  finite: True
```

A suíte O4 já inclui um teste multi-dim permanente (`test_o4_multidim_paridade`: 3 TR × 3 dip
× 2 freq, paridade <1e-13). **Conclusão**: multi-freq + multi-TR + multi-dip combinados
funcionam no JAX (loop e batched), com paridade vs Numba <1e-12. O Cenário **G** (4×4×4=64) é
o caso canônico nomeado de multi-dim completo (também testado no benchmark — JAX batched
empata Numba, 1.05×).

## Q5 — Alta resistividade (1500 / 5000 / 10000 Ω·m): há perda de fidelidade? (DEMONSTRADO)

**Não — em complex128 (default), ambos os backends são estáveis e fiéis. Verificado:**

```
Q5 — modelo [1, ρ, 1] Ω·m, 20 kHz, TR 1m:
  ρ= 1500: |H|max=0.159  Numba-vs-JAX(c128) abs=5.6e-14 rel=3.5e-13  c64-vs-c128 rel=3.8e-4  finite=True
  ρ= 5000: |H|max=0.159  Numba-vs-JAX(c128) abs=5.4e-14 rel=3.4e-13  c64-vs-c128 rel=3.8e-4  finite=True
  ρ=10000: |H|max=0.159  Numba-vs-JAX(c128) abs=7.7e-14 rel=4.8e-13  c64-vs-c128 rel=3.8e-4  finite=True
Âncora Fortran: oklahoma_28 (ρ_max=1416 Ω·m) paridade vs tatu.x PASS (local).
```

**Por que é estável (física)**: alta-ρ → baixa condutividade `σ=1/ρ` → `kh²=-iωμσ` minúsculo
(~1e-5) vs `kr²~1` → `u=√(kr²-kh²) ≈ kr` (regime **quase-estático**). Sem cancelamento
(o termo real domina); `exp(-2uh)` tem argumento `≈ -2·kr·h` limitado pela geometria
(h≤10 m) → **sem overflow/underflow** mesmo a ρ=10⁶ Ω·m. Guards: `tanh` em forma explícita
`(1-e⁻²ˣ)/(1+e⁻²ˣ)` (bit-match Fortran), `eps_singularity=1e-12`, `fastmath=False` (Numba),
`jax_enable_x64=True` (JAX). Observação importante: `|H|` permanece ~0.159 para TODOS os ρ
porque a 1 m/20 kHz o **campo primário (ρ-independente) domina**; a formação é perturbação.

**c128 vs c64**: o erro c64 (~3.8e-4 rel) é **constante** em 1500→10000 — vem do **bit-width
de 32 bits (mantissa 24 vs 53)**, NÃO da física de alta-ρ (não diverge com ρ). Cobertura
existente: `test_high_rho_no_nan` parametrizado **até 10⁶ Ω·m** (passa); `oklahoma_28`
(1416) vs Fortran <1e-6.

| Backend | dtype | Fidelidade alta-ρ |
|:--------|:------|:------------------|
| Numba JIT | sempre c128 | paridade Fortran <1e-13, estável até 10⁶ Ω·m |
| JAX GPU (default) | c128 | idem Numba (<1e-13 entre si) |
| JAX GPU (opt-in PINN) | c64 | ~1e-4 rel (bit-width, estável em ρ) — para treino, não validação |

**Recomendação**: **c128 para ciência/validação** (sem perda em qualquer ρ testado);
**c64 só para treino PINN em GPU**, ciente da tolerância ~1e-4 (que NÃO piora com ρ alto).

---

## Conclusão

- **Loop = inferência single-model; Batched = treino/PINN/geração de datasets** (vence em
  throughput e amortiza dispatch). Mesmo kernel → paridade bit-exata.
- **Geração de dataset .dat a partir do batched: viável e demonstrada** (writer 22-col
  existente; só falta um campo `H_tilted=None` no dataclass JAX para usar o atalho multi-TR).
- **Multi-dim (incl. 2×2×2) suportado e testado**; Cenário G (4×4×4) é o canônico.
- **Alta-ρ sem perda de fidelidade em c128** (até ≥10⁴–10⁶ Ω·m), ancorado em Fortran a 1416 Ω·m.
