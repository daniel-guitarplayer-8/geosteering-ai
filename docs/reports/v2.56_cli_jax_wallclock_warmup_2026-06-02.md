## Resumo Executivo

`benchmark --scenario E --n 1000 --backend jax --geometry templates --warmup --repeat 3` reportava na
tabela "tempo de execução: 2,14 s" mas `time real = 25,1 s`, e o **wall-clock total do JAX (25,1 s)
superava o do Numba (19,5 s)** apesar de o JAX ter ~2× o throughput hot. Causa-raiz (bissecção +
verificação adversarial, **que FALSIFICOU a hipótese de compilação XLA**): (1) a discrepância era de
**observabilidade** — a tabela mostrava só a **mediana hot** (loop `--repeat`), escondendo ~23 s de
startup + warmup, e ainda usava "tempo = última rodada" enquanto "throughput = mediana" (incoerente);
(2) o gargalo NÃO é compilação (o cache XLA de disco está cheio: 284/284 hits, 0 misses; o 2º run
cross-process é **idêntico**) — é **tracing/orquestração Python por-grupo** sobre os **31 grupos de
geometria** (`K = n//32`), re-pago a CADA invocação. **Fix**: reduzir o default de grupos para **poucos
e grandes** (`K = max(1, min(n//256, 4))` → n=1000: 31→3) corta o tracing E satura melhor a GPU; remover
o `warmup_backend` morto no path JAX (compilava shape errado n_models=1); instrumentar warmup/total +
corrigir `elapsed_s` para a **mediana**; e um **auto-chunk** anti-OOM para os cenários high-config (G/H).
**Resultado medido (A6000): JAX warm `real` 12,75 s — mais rápido que o Numba (19,97 s) E que ele mesmo
antes (25,1 s)**, com throughput hot +13% e variância 14× menor.

## 4. Investigação E — Persistência de warmup em disco (pedido do usuário)

Investigação read-only (experimentos no A6000 + leitura de código) da viabilidade de **persistir os
artefatos de warmup** para Numba JIT e JAX GPU. **Conclusão: a persistência já é máxima nos dois
backends; persistir MAIS foi MEDIDO como inefetivo. Sem novo código de persistência.**

| Backend | Persistência atual | `jax.export`/AOT? | Veredito |
|:--------|:-------------------|:------------------|:---------|
| **Numba JIT** | `NUMBA_CACHE_DIR` estável `~/.cache/geosteering/numba_cache` guarda o bitcode `.nbc` cross-reboot (v2.52; 103× cross-process) | **AOT (`numba.pycc`) NÃO viável** (removido no Numba ≥0.59; incompatível com `parallel=True`/prange) | **Já maximal** — o resíduo (LLVM bitcode→nativo, ~1×/processo) é fundamental e já minimizado. Nada a adicionar |
| **JAX GPU** | `jax_compilation_cache_dir` `~/.cache/geosteering/jax_compilation_cache` (148 MB / 2800 arquivos; `min_compile_time=0.0`) | **`jax.export` (AOT, JAX 0.7.2) MEDIDO**: `deserialize+call` = **0,093 s** ≈ `trace+compile(cache-hit)` = **0,094 s** | **Persistir mais NÃO ajuda** — o gargalo é tracing/orquestração Python por-grupo (o 2º run cross-process é idêntico apesar de 284/284 cache hits). A serialização do StableHLO não elimina o tracing |

**Evidência decisiva (falsifica a hipótese de cache):**

```text
1º run (cold codegen, K=3 novo shape):  real 26,0 s · warmup 17,29 s   ← codegen frio dos group-sizes ~333
2º run (codegen em cache, K=3):         real 12,75 s · warmup  3,94 s   ← 2,0× mais rápido, MESMA geometria
```

→ Reduzir **K** (deliverable A) **desloca o gargalo** de tracing-Python-incacheável para
codegen-XLA-cacheável → o cache de disco existente passa a amortizar. **O lever efetivo é o A, não
persistir mais.** A linha "tempo de warmup" (C) torna esse custo residual visível ao usuário; o comando
`geosteering-warmup` pré-popula ambos os caches (`.nbc` + XLA) para a 1ª invocação ser mais barata.
