## 1. Sumário Executivo

A v2.12 entrega **paralelismo inter-modelo nativo** em `simulate_multi`,
migrando a infraestrutura de pool de workers da camada UI para o **core
do simulador**. A nova API permite que notebooks, scripts CLI e treino
offline executem batches de 30k+ modelos sem reimplementar
`ProcessPoolExecutor` manualmente.

| Métrica                              | v2.11 (anterior) | v2.12 (atual)         |
|:-------------------------------------|:----------------:|:---------------------:|
| API `simulate_multi(models=[...])`   | ❌ Não suportado | ✅ 4 modos A/B/C/D    |
| Anti-oversubscription                | Manual           | Automático            |
| `MultiSimulationResultBatch`          | ❌               | ✅ Frozen dataclass   |
| `release_pool()` exportado            | ❌               | ✅                    |
| Backward-compat single-modelo        | n/a              | ✅ 100% preservado    |
| Smoke tests                          | 197              | 202 (+5)              |
| Pytest simulation                    | 730              | 734 (+4) (zero regr.) |
| Testes específicos workers           | 0                | 17 (test_simulation_workers.py) |

## 2. Arquitetura

### 2.1 Novo módulo `geosteering_ai/simulation/_workers.py`

Módulo de **530 LOC** com:

- **`_resolve_effective_threads(n_workers, threads_per_worker, cpu_count=None)`**:
  anti-oversubscription. Quando `threads_per_worker is None`, retorna
  `max(1, cpu // n_workers)`. Garante que `n_workers × eff ≤ cpu_count`.
- **`_detect_mode(n_workers, eff_threads)`**: identifica modo `"A"` (1×1),
  `"B"` (1×N), `"C"` (M×1) ou `"D"` (M×K).
- **`_simulate_worker_init(n_threads, hankel_filter)`**: inicializador
  picklable que configura env vars Numba/OMP **antes** de import e
  aquece JIT com modelo trivial. Idempotente via `_WORKER_INITIALIZED`.
- **`_acquire_pool(n_workers, n_threads, hankel_filter)`**: singleton
  `ProcessPoolExecutor` com `mp_context="spawn"`. Protegido por
  `threading.Lock`. Reusado entre chamadas com mesma config.
- **`release_pool()`**: cleanup explícito (chamado em `closeEvent` UI).
- **`_split_models_uniform(models, n_workers)`**: divide com remainder
  distribuído, preservando índices originais (`(orig_idx, model)`).
- **`_run_simulate_chunk(chunk, sim_kwargs)`**: worker function
  picklable. Itera sequencialmente dentro do worker, retornando
  `(indices, H_chunk_6D)`.
- **`run_batch(models, sim_kwargs, n_workers, threads_per_worker, ...)`**:
  orquestrador. Modos A/B → in-process (`_run_inproc`); Modos C/D →
  ProcessPool com agregação ordenada.
- **`MultiSimulationResultBatch`** (frozen dataclass): container 8-campo:
  `H_stack, z_obs, elapsed_s, throughput_mod_per_h, backend, n_workers,
  n_threads, mode`. Inclui `to_list_of_results()` para retro-compat.

### 2.2 Extensão de `simulate_multi`

Em [multi_forward.py:407](../../geosteering_ai/simulation/multi_forward.py#L407),
3 novos kwargs **opcionais**:

```python
def simulate_multi(
    rho_h=None, rho_v=None, esp=None, positions_z=None,
    *,
    # ... kwargs v2.11 ...
    models: Optional[List[Dict[str, Any]]] = None,
    n_workers: Optional[int] = None,
    threads_per_worker: Optional[int] = None,
) -> Union[MultiSimulationResult, MultiSimulationResultBatch]:
```

**Dispatcher** no início do corpo:

- `models is None` (default v2.11): caminho single-modelo intacto.
  Retorna `MultiSimulationResult`.
- `models is not None`: delega para `_workers.run_batch()`.
  Retorna `MultiSimulationResultBatch`.
- **Mutuamente exclusivo** com `rho_h/rho_v/esp` (raise ValueError).

### 2.3 Os 4 modos de execução

Conforme relatório §3 de
[`v2.11_simulador_python_analise_paralelismo_2026-04-30.md`](v2.11_simulador_python_analise_paralelismo_2026-04-30.md):

| Modo | n_workers | threads_per_worker | Quando usar                            |
|:----:|:---------:|:------------------:|:----------------------------------------|
|  A   |    1      |        1           | Debug, 1 modelo, n_pos < 50             |
|  B   |    1      |        N           | 1 simulação grande, n_pos ≥ 100         |
|  C   |    M      |        1           | Batch grande, n_pos baixo (overhead OK) |
|  D   |    M      |        K           | ★ DEFAULT 30k+ modelos com n_pos ≥ 100  |

### 2.4 Anti-oversubscription

Quando `threads_per_worker is None`, o módulo aplica:

```python
eff_threads = max(1, cpu_count // n_workers)
```

**Exemplo CPU 8-core, n_workers=4**:

- Sem anti-oversub: cada worker tenta `os.cpu_count()=8` threads →
  4 × 8 = 32 threads concorrentes → trashing severo.
- Com anti-oversub: `eff = 8 // 4 = 2` → 4 × 2 = 8 threads totais.

### Sprint 12.3 — Migração Simulation Manager (concluída)

**Decisão arquitetural**: O `sm_workers.run_numba_chunk` **NÃO** foi
substituído por `simulate_multi(models=...)` direto. Motivo:

- A UI requer Pause/Cancel cooperativo (v2.11) com checkpoints
  `_wait_if_paused` + `_cancel_requested`. Esses checkpoints só
  funcionam dentro do loop manual do `run_numba_chunk`.

## 4. Benchmark Empírico

### 4.1 Configuração

- **Hardware**: macOS Darwin 25.4.0, CPU 16 cores
- **Backend**: Numba (parallel=True)
- **Filtro Hankel**: werthmuller_201pt
- **Comando**: `PYTHONPATH=. python benchmarks/bench_v212_workers.py --n N --n-pos K`

### 4.2 Resultados — Batch pequeno (200 modelos, n_pos=50)

| Modo                                    | Tempo (s) | Mod/h     | Speedup vs A |
|:----------------------------------------|:---------:|:---------:|:------------:|
| A — Single (1w × 1t)                    | 3.03      | 237 632   | 1.00×        |
| B — Multi-Thread (1w × Nt)              | 0.39      | 1 832 704 | **7.71×**    |
| C — Workers (4w × 1t)                   | 2.94      | 244 947   | 1.03×        |
| D — Hybrid (4w × Kt)  ★                 | 2.64      | 272 293   | 1.15×        |

**Análise**:

- Modo B vence por margem ampla — `prange` Numba intra-modelo paraleliza
  eficientemente n_pos=50 sem custo de spawn.
- Modos C/D pagam **~3s de spawn** (cold start de 4 processos + import
  Numba + JIT warmup) que domina o benchmark pequeno.
- Modo D > Modo C: o segundo thread por worker amortiza parte do overhead.

**Critério v2.12 (D ≥ 1.3× B)**: NÃO atingido nesta escala — esperado.
O critério é definido para 30k+ modelos onde o overhead de spawn é
desprezível.

### 4.3 Resultado esperado para batch grande (30k modelos)

Conforme relatório `v2.11_simulador_python_analise_paralelismo`:

- Modo B: ~120 mod/h baseline (em 8-core)
- Modo D: ~960 mod/h projetado (8× speedup vs B em CPU 8-core)

A validação empírica desse cenário exige ~30 min de benchmark — pendente
para próxima rodada (v2.12.1 ou validação manual).

### 4.4 Quando usar cada modo (recomendação)

| Cenário                                | Modo recomendado |
|:---------------------------------------|:-----------------:|
| Debug, 1 modelo                        | A                |
| 1 simulação grande (n_pos > 100)       | B                |
| Batch < 1000 modelos                   | B                |
| Batch 1000-10k modelos, n_pos < 50     | C                |
| Batch 1000-10k modelos, n_pos ≥ 100    | D                |
| Batch ≥ 10k modelos                    | **D** (★ default) |

## 7. Pendências e Riscos

### 7.1 Pendências v2.13+

- **Numba optimizations** (relatório §6 v2.11):
  - Vectorize over frequencies (n_f kernels in 1 prange)
  - prange sobre TR × angle (atualmente serial)
  - Cross-call cache (reusa common_arrays entre simulações consecutivas)
- **JAX GPU Phase 2** (PR #24):
  - Completar unified strategy (44 → 1 XLA programs)
  - Flip default `jax_vmap_real=True` após validação T4/A100
- **Streaming H_stack** (v2.15):
  - Para batch >100k modelos, `H_stack` 6-D pode chegar a 50+ GB.
  - Streaming incremental para `.dat` é necessário.
- **Pause persistente** (UI):
  - Salvar estado de pausa em disco para retomar após fechar app.
- **Validação benchmark 30k modelos**:
  - Confirmar empiricamente Modo D ≥ 1.3× Modo B em hardware do usuário.

### 7.2 Riscos identificados

| Risco | Probabilidade | Impacto | Mitigação                              |
|:------|:-------------:|:-------:|:----------------------------------------|
| Pool spawn lento (cold start ~3s) | Alta | Baixo | Reusado entre chamadas (singleton)     |
| Memory blow-up batch 100k+         | Média | Alto  | Documentado; streaming em v2.15        |
| FP non-determinismo entre processos| Baixa | Médio | Tolerância `rtol=1e-12` documentada   |
| `MultiSimulationResultBatch` não retro-compat com APIs upstream | Média | Médio | `to_list_of_results()` provê |
