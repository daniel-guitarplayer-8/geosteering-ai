## Resumo Executivo

Sprint O0/O1 (9 Quick Wins JAX GPU) executou em T4 com **paridade Fortran <1e-12 preservada** (168/168 PASS) e **todos os 10 Quick Wins ativos**. Porém o benchmark mostra **regressão de 30-47% em 7/7 cenários** vs baseline pré-O0/O1 (commit `a06cf12`), com Gate A1.6 reprovado em cenário E (1.21× vs Numba T4, abaixo do mínimo 1.5×). Causa-raiz consolidada: **~25-30% hardware variance** (Numba T4 LOCAL também regrediu 5-30%, confirmando T4 chip do pool Colab + thermal throttling) **+ ~20-40% JAX-específica** com 3 mecanismos identificados via coderabbit independent review: (1) `lax.scan(unroll=1)` produz XLA `While` op que **quebra fusion com vmap externo**, gerando milhares de extra kernel launches em cenários multi-config; (2) `get_hankel_filter_jax` (QW#6) **instancia `FilterLoader()` per call** e buffer pode ser liberado pelo BFC sob pressão; (3) `jax.tree.map` (QW#8) **adiciona overhead Python sem batching real**. Smoking gun: cenário A (menor trace) tem 0% regressão JAX-específica — eliminando hipóteses de overhead per-call (PREALLOCATE, cache lookup). Sprint O1 OK para A100 (otimizações nunca aplicadas em T4 via patch O1-fix-2); Sprint O2 requer reverter `lax.scan` para `fori_loop` quando `unroll==1`, refatorar FilterLoader module-scope, e reverter `tree.map` para inline.

## 4. Causa-Raiz JAX-Específica (refinada via coderabbit review)

### 4.1 Smoking gun — Cenário A não regrediu (0% JAX-específico)

A descoberta crítica que invalida hipóteses iniciais: **Cenário A tem o menor trace (50 mod × 1 pos × 1 config) e não mostra regressão JAX-específica**. Se a causa fosse overhead per-call (`PREALLOCATE=false`, cache lookup), A regrediria MAIS, não menos. Como A preservou ratio (2.58× vs 2.56× pré-O1), a causa é **overhead per-vmap-trace** que escala com número de configs/positions, não per-call.

### 4.2 Padrão da regressão JAX-específica

| Cenário | n_pos | configs (nf×nTR×nAng) | JAX-específico | Comportamento |
|:-------:|:-----:|:----------------------:|:--------------:|:-------------|
| A | 1 | 1 | 0% | **Baseline confirmado** — pior caso para overhead per-call |
| B | 100 | 1 | -33% | Position loop |
| C | 100 | 4 | -44% | Position loop + multi-freq vmap |
| D | 1 | 4 | -30% | Multi-TR vmap (sem position loop) |
| E | 600 | 1 | -36% | Position loop grande |
| F | 100 | 16 | -45% | Position loop + 16 vmap configs |
| G | 100 | 64 | -43% | Position loop + 64 vmap configs |

Regressão escala com **configs × positions** (per-vmap-trace overhead).

### 4.3 Hipóteses ranqueadas

#### H1 (CRÍTICA, alta confiança): `lax.scan` quebra fusion XLA dentro de vmap

**Arquivos**: `propagation.py:218-232,278-298`, `dipoles_unified.py:118-160`.

Pré-O1 usava Python `for` loops desenrolados em trace-time → XLA gerava ~n straight-line ops que fundiam com vmap externo. Sprint O1 substituiu por `lax.scan`. Mesmo com `unroll=1` (nosso patch O1-fix-1 em T4), `lax.scan` produz XLA `While` op (loop primitive) que **não pode ser fundido com vmap externo** (per-frequência, per-posição, per-config).

Impacto quantitativo para Cenário E (n_layers=3-5, n_pos=600, n_freqs=1):
```
extra kernel launches ≈ (n-1) × n_pos × nf × n_configs
                      ≈ 4 × 600 × 1 × 1 = 2,400 extra launches
                      ≈ 2,400 × 7 µs/launch ≈ 17 ms overhead
```

Para Cenário G (3-5 layers, 100 pos, 64 configs): ~25,000 extra launches ≈ 175 ms overhead.

**Measurement proposto**: revert `lax.scan` para `fori_loop` em propagação. Medir Cenário E. Se ganho ≥30%, confirma H1.

#### H2 (CRÍTICA, alta confiança): QW#6 `FilterLoader()` instanciado por call + buffer liberado por BFC

**Arquivos**: `kernel.py:223-255` (`get_hankel_filter_jax`), `kernel.py:158-220` (`_get_hankel_filter_cached`).

Análise coderabbit revelou que `@lru_cache` envolve `_get_hankel_filter_cached(canonical: str)`, mas `get_hankel_filter_jax` chama `FilterLoader().resolve_name(filter_name)` a **cada call** — constrói loader fresh + alias lookup + lock acquisition.

Pior em T4 com `PREALLOCATE=false`: o `jax.Array` cached pode ser **liberado pelo BFC allocator** sob pressão de memória (168 testes prévios + cuDNN handles). `lru_cache` retém Python reference, mas XLA pode silenciosamente re-HtoD o buffer.

**Measurement proposto**: log `id(kr_jax)` antes/depois de Cenário E + `jax.live_arrays()` count. Se id muda OU live_arrays cresce monotonicamente, confirma re-upload silencioso.

#### H3 (MÉDIA): QW#8 `jax.tree.map` adiciona overhead vs inline `jnp.asarray`

**Arquivos**: `multi_forward.py:1095-1112` (batched), `:771-788` (vmap_real).

`jax.tree.map` traversa pytree spec + aplica lambda por leaf + reconstrói tuple. Para 6 leaves: 6 lambda calls + tree-spec walk (~50-150 µs) vs 6 inline `jnp.asarray` (~30-60 µs). A "consolidação" é Python-level abstraction sem batching real no dispatcher — cada `jnp.asarray` ainda produz HtoD task individual.

Em T4 com `PREALLOCATE=false`, cada HtoD aciona `cudaMalloc` → multiplicação do overhead em multi-config cenários.

**Measurement proposto**: micro-benchmark isolado em T4 Colab:
```python
%timeit jax.tree.map(lambda x: jnp.asarray(x, dtype=jnp.float64), (a,b,c,d,e,f))
%timeit (jnp.asarray(a),jnp.asarray(b),jnp.asarray(c),jnp.asarray(d),jnp.asarray(e),jnp.asarray(f))
```

#### H4 (HARDWARE): T4 chip variance + thermal throttling

Numba T4 LOCAL também regrediu 5-30% no mesmo run, comprovando hardware variance independente do código. Esta camada explica **~25-30% da regressão**, não JAX-específica.

### 4.4 Causa-raiz consolidada (revisada)

| Camada | Contribuição estimada | Mecanismo |
|:-------|:--------------------:|:----------|
| Hardware variance (T4 + thermal) | 25-30% | Não controlável |
| H1: `lax.scan` quebra fusion XLA | 15-25% | Sprint O2 fix: gate scan apenas quando `unroll>1` |
| H2: FilterLoader per-call + buffer release | 5-10% | Sprint O2 fix: instância módulo-scope + pin via device_put |
| H3: `tree.map` overhead | 2-5% | Sprint O2 fix: reverter para inline |

**Total**: 47-70% — consistente com observado (47% pior em F).

## 6. Recomendações Sprint O2 (priorizadas via coderabbit + análise)

### O2.1 (CRÍTICO, alta confiança) — Gate `lax.scan` apenas quando `unroll > 1`

**Hipótese H1**: `lax.scan(unroll=1)` produz XLA `While` op que não funde com vmap externo, gerando milhares de extra kernel launches em cenários multi-config.

**Fix proposto**: em `propagation.py:218-232,278-298` + `dipoles_unified.py` (4 helpers HMD/VMD descent/ascent):

```python
def _maybe_scan_or_fori(body, init, n, unroll):
    """Use lax.scan apenas quando unroll > 1; senão fori_loop (XLA-fusable)."""
    if unroll == 1:
        # fori_loop preserva pattern pre-O1 (fundivel com vmap externo)
        return jax.lax.fori_loop(0, n, body, init)
    # Multi-unroll: lax.scan vale a pena (compile-time fusion ganha)
    _, result = jax.lax.scan(body, init, xs=jnp.arange(n), length=n, unroll=unroll)
    return result
```

**Esforço**: 4-6 h (refatorar 4 helpers + 6 testes paridade <1e-12).
**Ganho esperado**: +15-30% em B/C/E (cenários position-loop em T4).
**Validação**: rodar Cenário E + comparar throughput. Se ≥30% melhor, confirma H1.

### O2.2 (CRÍTICO, alta confiança) — Refatorar `get_hankel_filter_jax` para evitar FilterLoader per-call

**Hipótese H2**: cada call em `get_hankel_filter_jax` constrói `FilterLoader()` fresh + sob `PREALLOCATE=false` o buffer pode ser silenciosamente re-uploaded.

**Fix proposto**: em `kernel.py`:

```python
# Module-scope: instância única + registry de buffers pinados
_FILTER_LOADER_INSTANCE = FilterLoader()  # construir 1×, reutilizar
_FILTER_HANDLE_REGISTRY: dict[str, tuple[jax.Array, jax.Array, jax.Array]] = {}

def get_hankel_filter_jax(filter_name: str = "werthmuller_201pt"):
    canonical = _FILTER_LOADER_INSTANCE.resolve_name(filter_name)
    if canonical in _FILTER_HANDLE_REGISTRY:
        return _FILTER_HANDLE_REGISTRY[canonical]
    kr, w0, w1 = _get_hankel_filter_cached(canonical)
    # Pin via device_put (donate=False) — força XLA a manter buffer vivo
    kr_pinned = jax.device_put(kr)
    w0_pinned = jax.device_put(w0)
    w1_pinned = jax.device_put(w1)
    _FILTER_HANDLE_REGISTRY[canonical] = (kr_pinned, w0_pinned, w1_pinned)
    return kr_pinned, w0_pinned, w1_pinned
```

**Esforço**: 1-2 h (refactor + teste de não-regressão LRU speedup).
**Ganho esperado**: +5-10% em todos os cenários (elimina FilterLoader overhead).
**Validação**: log `id(kr)` antes/depois de 10 calls — id deve permanecer constante.

### O2.3 (ALTO) — Reverter QW#8 `jax.tree.map` para inline `jnp.asarray`

**Hipótese H3**: `jax.tree.map` adiciona overhead Python sem batching real no dispatcher.

**Fix proposto**: em `multi_forward.py:1095-1112` + `:771-788` (vmap_real):

```python
# ANTES (QW#8):
(positions_z_jnp, freqs_jnp, rho_h_batch_jnp, ...) = jax.tree.map(
    lambda x: jnp.asarray(x, dtype=jnp.float64),
    (positions_z_np, freqs_arr, ...),
)

# DEPOIS (inline, mais rápido):
positions_z_jnp = jnp.asarray(positions_z_np, dtype=jnp.float64)
freqs_jnp = jnp.asarray(freqs_arr, dtype=jnp.float64)
rho_h_batch_jnp = jnp.asarray(rho_h_batch_np, dtype=jnp.float64)
rho_v_batch_jnp = jnp.asarray(rho_v_batch_np, dtype=jnp.float64)
h_arr_batch_jnp = jnp.asarray(h_arr_batch_np, dtype=jnp.float64)
prof_arr_batch_jnp = jnp.asarray(prof_arr_batch_np, dtype=jnp.float64)
```

**Esforço**: 30 min (2 substituições simétricas + teste paridade).
**Ganho esperado**: +2-5% (overhead Python reduzido).
**Validação**: micro-benchmark em T4 Colab antes/depois.
