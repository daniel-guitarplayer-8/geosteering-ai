## Sumário Executivo

Este relatório responde duas perguntas fundamentais surgidas após a validação T4:

1. **Numba GPU CUDA seria vantajoso?** — Sim, em memória: potencialmente 10–30×
   mais eficiente por ser **lazy e stateful** (não materializa o grafo completo).
   Em performance de throughput puro, comparável ao JAX na maioria dos cenários.
   Na prática, a implementação é **extremamente complexa** para o algoritmo EM
   TIV — e o projeto perde `jax.grad` (inviabiliza PINNs e inversão diferenciável).
   **Veredicto**: vantajoso para geração de dados em escala industrial (>1M modelos),
   desnecessário para os 200k–300k do roadmap atual, inviável sem equipe CUDA dedicada.

2. **Por que JAX consome quase toda a VRAM?** — Duas causas distintas:
   - **Causa A (observação "sempre cheio")**: o alocador XLA BFC pré-aloca
     proativamente 75–90% da VRAM disponível ao inicializar, independentemente
     do que será computado. O monitor de GPU mostra essa reserva, não o uso real.
   - **Causa B (OOM real)**: `jax.vmap` materializa **simultaneamente** todos os
     buffers intermediários para todos os elementos do batch. Para o cenário F
     com n_models=50, a relação intermediário/output é **1.551×** — 17,87 GB
     para 11,52 MB de saída útil. Isso é uma consequência do design do compilador
     XLA, não um bug.

## Parte I — Numba GPU CUDA no Contexto do Simulador EM TIV

### 1.1 O Que é Numba GPU CUDA (Revisão Técnica)

Numba suporta CUDA via `@cuda.jit` — permite escrever kernels GPU em Python com
semântica quase idêntica a C CUDA. A diferença fundamental em relação ao JAX é
**quem gerencia a memória**:

```
JAX/XLA:        compilador decide automaticamente o que alocar e por quanto tempo
                → abstração de alto nível → memória conservadora (aloca mais)

Numba CUDA:     programador decide explicitamente cada array GPU
                → controle granular → memória mínima necessária
```

A hierarquia de memória CUDA relevante:

```
┌─────────────────────────────────────────────────────────────────┐
│  Memória CUDA                    │  Tamanho   │  Latência       │
├──────────────────────────────────┼────────────┼─────────────────┤
│  Registradores (por thread)      │  255 regs  │  1 ciclo        │
│  Shared memory (por bloco SM)    │  48–128 KB │  5–40 ciclos    │
│  L1 Cache (por SM)               │  32–128 KB │  ~30 ciclos     │
│  L2 Cache (global)               │  40–80 MB  │  ~200 ciclos    │
│  Global DRAM (VRAM total)        │  15–80 GB  │  600–800 ciclos │
└──────────────────────────────────┴────────────┴─────────────────┘
```

Em Numba CUDA, você pode usar **shared memory** (`cuda.shared.array`) para
armazenar buffers intermediários **por bloco de threads**, liberá-los ao fim
do kernel e reutilizá-los no próximo modelo. Em JAX, toda essa decisão é feita
pelo compilador XLA com base no grafo de computação estático.

### 1.2 O Algoritmo EM TIV e Sua Paralelizabilidade

O simulador EM TIV (`forward_pure.py`, `propagation.py`, `dipoles_native.py`)
tem a seguinte estrutura de dependências:

```
Para cada modelo m ∈ [0, n_models):           ← PARALELIZÁVEL (vmap/CUDA)
  Para cada TR i ∈ [0, nTR):                  ← PARALELIZÁVEL (vmap/CUDA)
    Para cada dip θ ∈ [0, nAngles):           ← PARALELIZÁVEL (vmap/CUDA)
      Para cada posição z ∈ [0, n_pos):       ← PARALELIZÁVEL (vmap/CUDA)
        Para cada freq f ∈ [0, nf):           ← PARALELIZÁVEL (vmap/CUDA)
          common_arrays(n_cam, npt)            ← depende de z, f, perfil_m
          Para camada L de 0 → n_cam-1:       ← SEQUENCIAL (propagação bottom-up)
            u[L], v[L] = f(u[L-1], v[L-1])
          Para camada L de n_cam-1 → 0:       ← SEQUENCIAL (propagação top-down)
            z[L], z'[L] = g(z[L+1], z'[L+1])
          dipole(camad_T, camad_R, ...)        ← lax.switch (6 casos)
          H[z, f, 9] = Hankel(kernel × weights)  ← integração
```

**A observação crítica**: o loop de camadas (bottom-up + top-down) é **sequencialmente
dependente** — cada camada depende da anterior. No JAX, isso é resolvido com
`jax.lax.fori_loop` (Sprint 10 Phase 2). Em Numba CUDA, você escreveria esse
loop dentro de um kernel, onde cada thread processa **um modelo** (ou um TR/dip)
e faz o loop de camadas sequencialmente.

### 1.3 Modelo de Memória: JAX vs Numba CUDA (Comparação Rigorosa)

#### 1.3.1 JAX/XLA — Materialização Total do Grafo

Quando `jax.vmap(f, in_axes=(0,...))(batch)` é compilado pelo XLA:

```
XLA analisa estaticamente o grafo completo de f para 1 elemento
↓
Identifica todos os buffers intermediários que f necessita
↓
Multiplica cada buffer pelo tamanho do batch (n_models)
↓
Aloca SIMULTANEAMENTE na VRAM: buffer_1×n, buffer_2×n, ..., buffer_k×n
↓
Executa todos os elementos do batch em paralelo (kernels CUDA fusionados)
```

Para o simulador EM TIV no cenário F (n_models=50, nTR=4, nAng=1, n_pos=100, nf=4):

| Buffer intermediário | Shape por modelo | VRAM por modelo | VRAM × 50 |
|:--------------------|:---------------:|:---------------:|:---------:|
| `common_arrays` (TE/TM/dz) | (3, 5, 201, 4) | 188 KB | 9,4 MB |
| Matrizes de propagação TE | (5, 201) × n_pos × nf | ~62 MB | 3,1 GB |
| Matrizes de propagação TM | (5, 201) × n_pos × nf | ~62 MB | 3,1 GB |
| Coeficientes u, v, z, z' | (5, 201) × n_pos × nf | ~50 MB | 2,5 GB |
| Kernels HMD/VMD | (npt,) × n_pos × nf × nTR | ~32 MB | 1,6 GB |
| Buffers `lax.switch` | ~(201,) × 6 casos × n_pos | ~8 MB | 0,4 GB |
| Hankel integration temp | (201,) × n_pos × nf | ~3 MB | 0,15 GB |
| H parcial por TR/Ang | (n_pos, nf, 9) × nTR | ~0,4 MB | 0,02 GB |
| **Total estimado** | | **~257 MB** | **~12,87 GB** |
| **Overhead XLA BFC + frag.** | | | **~5 GB** |
| **Total empírico (OOM)** | | | **17,87 GB** |

**A razão intermediário/output é 1.551×**: para 11,52 MB de resultado útil, o XLA
precisou de 17.870 MB de VRAM intermediária.

#### 1.3.2 Numba CUDA — Modelo Lazy e Stateful

Em um kernel Numba CUDA bem escrito para o mesmo problema, a estratégia seria:

```python
@cuda.jit
def simulate_em_tiv_batch_kernel(
    rho_h_batch, rho_v_batch, esp_batch, positions_z,
    freqs, tr_spacings, dip_degs,
    H_output          # (n_models, nTR, nAng, n_pos, nf, 9)
):
    # Cada thread processa um modelo
    i_model = cuda.grid(1)  # índice global da thread
    if i_model >= rho_h_batch.shape[0]:
        return

    # Shared memory: buffers locais reutilizados por cada modelo
    # Alocados na SM do GPU — 48 KB por SM, compartilhados no bloco
    u    = cuda.shared.array(shape=(MAX_CAM, MAX_NPT), dtype=complex128)
    v    = cuda.shared.array(shape=(MAX_CAM, MAX_NPT), dtype=complex128)
    z_te = cuda.shared.array(shape=(MAX_CAM, MAX_NPT), dtype=complex128)
    # ... (~15 arrays de trabalho)

    # Loop sequencial sobre configurações (TR × Ang × pos × freq)
    for i_tr in range(nTR):
        for i_ang in range(nAng):
            for i_pos in range(n_pos):
                for i_freq in range(nf):
                    # Computa common_arrays → propaga → dipole → Hankel
                    # REUSA os mesmos buffers u, v, z_te para cada pos/freq
                    _compute_common_arrays(u, v, ...)
                    _propagate_up(u, v, z_te, ...)    # loop L: 0→n_cam
                    _propagate_down(z_te, z_te2, ...) # loop L: n_cam→0
                    _hankel_kernel(u, v, z_te, H_output[i_model, i_tr, i_ang, i_pos, i_freq])
                    # u, v, z_te são SOBRESCRITOS na próxima iteração
```

**O que muda fundamentalmente**:
- `u`, `v`, `z_te` são **reutilizados** a cada posição/frequência → O(n_cam × npt) fixo
- Não existe "materialização" do grafo completo → buffers intermediários são efêmeros
- A memória cresce com `n_models` (número de threads ativas) mas **não** com `n_pos × nf`

| Buffer | Numba CUDA (shared mem) | JAX/XLA (vmap materializado) |
|:-------|:-----------------------:|:----------------------------:|
| `common_arrays` | 188 KB × 1 (reutilizado) | 188 KB × n_models × n_pos × nf |
| Prop. matrices | ~50 KB × 1 (reutilizado) | ~57 MB × n_models |
| Kernels HMD/VMD | ~1 KB × 1 | ~32 MB × n_models |
| **Total por execução** | **~250 KB** | **~257 MB × n_models** |

Para F_50: Numba CUDA usaria ~**250 KB** de shared memory + output de **11,52 MB**
vs JAX com **17,87 GB**. Redução teórica: **~700×** no uso de memória intermediária.

**Consequência prática**: com Numba CUDA, o cenário F poderia rodar com
`n_models = 5.000` na T4 (vs n_models = 20 com JAX), e o cenário G com
`n_models = 500+` (vs n_models = 5 com JAX). O throughput por hora seria
proporcional ao aumento de batch size possível.

### 1.4 Performance: Numba CUDA vs JAX

#### 1.4.1 Throughput por FLOP

Para operações `complex128` puras:
- **T4**: 0,26 TFLOPS FP64 → ambas plataformas limitadas pelo hardware
- **A100**: 9,7 TFLOPS FP64 → ambas plataformas ganham proporcionalmente
- JAX/XLA gera código CUDA otimizado automaticamente via cuBLAS/cuDNN onde possível
- Numba CUDA com código bem escrito pode igualar XLA para loops regulares

Para operações com **memória dominante** (bandwidth-bound):
- JAX materializa grandes tensores → pressão alta em HBM bandwidth
- Numba CUDA com shared memory → operações ficam na SM (48 KB/bloco) → sem pressão HBM
- **Numba CUDA potencialmente 2–5× mais rápido em bandwidth-bound scenarios**

#### 1.4.2 Fusão de Kernels

Uma vantagem real do JAX/XLA: o compilador **funde automaticamente** kernels que
seriam separados em Numba CUDA. Exemplo:

```
JAX: vmap(vmap(vmap(f))) → 1 kernel CUDA fusionado (n_models × nTR × nAng × n_pos × nf)
Numba: cada operação = kernel separado → N kernel launches → latência overhead
```

Para o simulador EM TIV, com muitos loops aninhados, a fusão do XLA é uma vantagem
real. Numba CUDA exige programação explícita de warps e coalescência de memória
para conseguir fusão equivalente.

#### 1.4.3 Projeção Quantitativa Comparativa

Baseado nas características do algoritmo e hardware T4:

| Métrica | JAX (atual) | Numba CUDA (estimativa) | Razão |
|:--------|:-----------:|:-----------------------:|:-----:|
| VRAM intermediária (F_50) | 17,87 GB | ~0,3 GB | **60×** |
| max n_models (T4, cenário F) | 20 | ~800 | **40×** |
| Throughput F (T4, max n_models) | 36.468 mod/h | ~200.000 mod/h* | **~5×** |
| Tempo 200k modelos (cenário F) | 5,46h | ~1h | **~5×** |
| Implementação (LOC) | ~2.500 LOC JAX | ~4.000+ LOC CUDA | **−** |
| Diferenciabilidade (`jax.grad`) | ✅ | ❌ | **JAX ganha** |
| Manutenção | Média | Alta | **JAX ganha** |
| Debugging | Médio | Muito difícil | **JAX ganha** |

> *Estimativa conservadora assumindo Numba CUDA com 800 modelos simultâneos no
> mesmo T4 (batch ~40× maior que JAX) e throughput linear com batch size.
> Na prática, bounded por bandwidth e loop sequencial de camadas.

### 1.5 Complexidade de Implementação: O Obstáculo Real

Implementar o simulador EM TIV 1D em Numba CUDA requer:

#### 1.5.1 Desafios Técnicos

**1. complex128 em CUDA**
CUDA nativo usa `cuDoubleComplex` (C99 complex). Numba suporta `complex128` em
`@cuda.jit`, mas com limitações:
- Operações aritméticas `+, *, /` funcionam
- Funções matemáticas `exp, log, sqrt` em complex128 requerem implementação manual
  via decomposição real/imaginária — não existem como primitivas CUDA

**2. Loop sequencial de propagação (bottom-up + top-down)**
A recorrência de propagação TE/TM:
```
u[0] = u0_init
u[L] = f(u[L-1], v[L-1], rho_h[L], rho_v[L], kr[.], freq)  para L = 1..n_cam
```
é inerentemente serial. Em CUDA, cada thread processa UM modelo inteiro, com
o loop de camadas rodando sequencialmente dentro da thread. Isso é correto, mas
significa que threads não podem cooperar na propagação (não há paralelismo intra-camada).

**3. Fator geométrico: 6 casos lax.switch**
`compute_case_index_jax` + `_hmd_tiv_full_jax` implementam 6 casos geométricos
(camad_T vs camad_R vs n_cam) via `jax.lax.switch`. Em CUDA, isso seria um
`if/else if` convencional — mas com **divergência de warp**:
threads no mesmo warp que executam casos diferentes serializam. Para modelos com
perfis variados (diferentes n_cam), a divergência pode ser significativa.

**4. Filtro Hankel 201pt com pesos pré-computados**
A integração Hankel usa 201 pontos de quadratura (filtro Werthmüller). Os pesos
`krJ0J1, wJ0, wJ1` são arrays (201,) que devem ser carregados da VRAM em cada
execução. Em JAX, eles são capturados como closures no JIT. Em Numba CUDA,
precisariam ser passados como argumentos de kernel ou armazenados em memória
constante CUDA (64 KB limit, 201 × 3 arrays × 16 bytes = 9,6 KB — cabe).

**5. Sem equivalente ao `jax.lax.fori_loop` stateless**
O `fori_loop` do JAX garante que o XLA compile apenas 1 cópia do body (não
unrola o loop). Em Numba CUDA, o loop `for L in range(n_cam)` é compilado
como CUDA nativo — naturalmente stateful, sem o problema de unrolling. Isso
é na verdade uma **vantagem** do Numba CUDA.

#### 1.5.2 Estimativa de Esforço de Implementação

| Componente | Sprint JAX original | Sprint equivalente CUDA |
|:-----------|:-------------------:|:-----------------------:|
| Propagação TE/TM (`propagation.py`) | 3 sprints | 5–8 sprints |
| Dipoles HMD/VMD 6 casos | 4 sprints | 6–10 sprints |
| Integração Hankel + filtros | 2 sprints | 3–5 sprints |
| Dispatcher multi-TR/Ang/pos/freq | 2 sprints | 3–5 sprints |
| Paridade Fortran <1e-12 | 1 sprint | 3–6 sprints |
| Testes + documentação | 2 sprints | 3–5 sprints |
| **Total estimado** | **~14 sprints** | **23–39 sprints** |

Isso representa **6–12 meses de trabalho dedicado** com expertise em CUDA,
comparado ao simulador JAX já implementado e validado.

### 1.6 Ausência de Diferenciabilidade — A Perda Estratégica

O maior custo de migrar para Numba CUDA não é o esforço de implementação, mas
a **perda de `jax.grad`**:

```
JAX:
  jax.grad(simulate_multi_jax_batched)(rho_h_batch)
  → ∂H/∂ρ_h para todos os modelos simultaneamente
  → Habilita: inversão diferenciável, PINNs, treinamento de SurrogateNet com
    loss física, meta-learning com gradientes do simulador

Numba CUDA:
  Sem diferenciação automática
  → ∂H/∂ρ_h via diferenças finitas (custo: 2×n_cam×n_calls por Jacobiano)
  → PINNs: inviáveis (gradiente do simulador seria O(100×) mais caro)
  → SurrogateNet: treinável, mas sem loss física diferenciável
```

O roadmap do projeto (Sprint A2+, Fase F7) inclui explicitamente inversão
diferenciável via `jax.grad`. Migrar para Numba CUDA invalidaria essa trajetória.

### 1.7 Veredicto: Quando Numba CUDA Faz Sentido (e Quando Não)

```
┌──────────────────────────────────────────────────────────────────┐
│  Numba CUDA VALE A PENA se:                                      │
│  ✅ Objetivo: geração de dados pura (sem grad)                   │
│  ✅ Volume: >1M modelos (onde o ganho de tempo justifica esforço) │
│  ✅ Equipe: tem engenheiro CUDA dedicado disponível              │
│  ✅ Cenário G/H dominante (JAX é inviável por VRAM)              │
├──────────────────────────────────────────────────────────────────┤
│  Numba CUDA NÃO VALE A PENA se:                                  │
│  ❌ Objetivo inclui inversão diferenciável ou PINNs              │
│  ❌ Volume: 200k–300k modelos (JAX+chunking resolve em <6h)      │
│  ❌ Equipe não tem expertise CUDA                                │
│  ❌ Manutenção a longo prazo (codebase duplicada)               │
└──────────────────────────────────────────────────────────────────┘
```

**Para o roadmap atual (Sprint A3: 200k–300k modelos para SurrogateNet)**:
JAX+chunking é a escolha correta — resolve em 5,5h sem nenhuma implementação
adicional. Numba CUDA seria over-engineering para o volume atual.

**Para escala futura (Sprint A4+: >1M modelos ou cenário G intensivo)**:
considerar um **backend CUDA nativo separado** que coexiste com JAX, ativado
por `cfg.backend = "cuda_native"`, sem substituir o JAX existente.

## Parte II — Por Que JAX Consome Quase Toda a VRAM

### 2.1 Causa A: Pré-Alocação Proativa do Alocador XLA BFC

O comportamento de "consumo quase total da VRAM independente da GPU" observado
no Colab é, em grande parte, **não o consumo real de computação** — é o
comportamento do alocador de memória XLA chamado **BFC (Best-Fit with Coalescing)**.

#### 2.1.1 Como o BFC Funciona

```
Quando jax é importado e a GPU é inicializada:

  1. BFC RESERVA proativamente uma fração da VRAM total
     Padrão: 75% da VRAM (controlável via XLA_PYTHON_CLIENT_MEM_FRACTION)

  2. Exemplo T4 (15 GB):
     BFC reserva 75% × 15 GB = 11,25 GB ao inicializar
     → monitor nvidia-smi mostra "~11 GB used" logo após `import jax`
     → mesmo sem computar absolutamente nada ainda

  3. Exemplo A100 (40 GB):
     BFC reserva 75% × 40 GB = 30 GB ao inicializar
     → monitor mostra "~30 GB used" logo no início
     → também sem nenhuma computação

  4. Internamente: BFC usa essa pré-alocação como pool
     Alocações subsequentes vêm desse pool (rápidas, sem syscalls)
     Desalocações retornam ao pool (não de volta ao sistema CUDA)
```

**Por isso o comportamento parece "independente da GPU"**: em qualquer GPU,
o BFC reserva ~75% do total disponível logo na inicialização. Uma T4 mostra
~11 GB "usados"; uma A100 mostra ~30 GB "usados" — mas em ambos os casos,
o JAX ainda não computou nada útil.

#### 2.1.2 Configuração no Notebook Atual

O notebook `validate_jax_gpu_v240.ipynb` configura três variáveis antes de
importar JAX:

```python
os.environ["JAX_ENABLE_X64"] = "1"
os.environ["JAX_PLATFORMS"] = "cuda,cpu"
os.environ["JAX_COMPILATION_CACHE_DIR"] = "/content/jax_cache"
```

**Não configura `XLA_PYTHON_CLIENT_MEM_FRACTION`** — usa o padrão do JAX.
O padrão atual (JAX 0.4.30+) é 0.75 (75%), que pode ser alterado:

```python
# Antes de importar jax — reduz pré-alocação
os.environ["XLA_PYTHON_CLIENT_MEM_FRACTION"] = "0.50"  # 50% da VRAM
# Ou usa alocação sob demanda (mais lento, mas sem pré-alocação):
os.environ["XLA_PYTHON_CLIENT_ALLOCATOR"] = "platform"  # sem pré-alocação
```

**Consequência**: em qualquer GPU, logo após `import jax`, o monitor mostra
VRAM quase cheia. Isso leva à impressão de que JAX "sempre usa toda a VRAM",
mesmo que a computação real consuma muito menos.

#### 2.1.3 Distinção entre Reserva e Uso Real

```
                         T4 (15 GB)           A100 (40 GB)
                         ──────────           ────────────
Após import jax:        ~11,25 GB            ~30 GB
Durante cenário A_50:   ~11,25 GB +          ~30 GB +
  (n_pos=1, nf=1)       computa ~0,1 GB     computa ~0,1 GB
  Total monitor:        ~11,35 GB            ~30,1 GB

Durante cenário F_50:   17,87 GB → OOM T4
  (n_pos=100, nf=4)     computa 17,87 GB > 15 GB disponível

Durante cenário F_50:   ~30 GB +
  (no A100)             computa ~17,87 GB
  Total monitor:        ~47,87 GB → ainda dentro dos 40 GB pool!
  Na prática: usa dentro do pool pré-alocado → ~30 GB mostrados
```

Isso explica por que o monitor sempre mostra "quase tudo usado": mostra a
**reserva BFC**, não o compute real.

### 2.2 Causa B: Materialização Total de Buffers pelo jax.vmap

Esta é a causa dos **OOM reais** (não da pré-alocação). Quando a computação
real excede o pool pré-alocado, ocorre OOM genuíno.

#### 2.2.1 O Modelo Mental: JAX é um Compilador de Grafos

JAX não executa código Python imperativo — ele **traça** o código, constrói um
grafo de computação, compila esse grafo com XLA, e então executa o grafo compilado.

```python
# O que o programador escreve (parece imperativo):
def _one_model(rho_h, rho_v, h_arr, prof_arr):
    common = compute_common_arrays(rho_h, ...)  # intermediário 1
    u, v   = propagate_up(common, ...)           # intermediário 2, 3
    z, zp  = propagate_down(common, u, v, ...)  # intermediário 4, 5
    dipole = compute_dipole(z, zp, ...)          # intermediário 6
    H      = hankel_integrate(dipole, ...)       # resultado final
    return H

# O que XLA faz (análise estática do grafo):
# 1. Identifica todos os nós intermediários: common, u, v, z, zp, dipole
# 2. Determina lifetime de cada buffer baseado no grafo
# 3. Aloca TUDO de uma vez antes de executar qualquer operação
# 4. XLA não sabe que u, v são "temporários" — não há esse conceito no grafo
```

Quando `jax.vmap(_one_model, in_axes=(0,0,0,0))(batch)` é compilado:

```
XLA replica o grafo n_models vezes (ou usa broadcast de batch)
TODOS os intermediários se tornam tensores com dimensão extra n_models:
  common: (n_models, 3, n_cam, npt, nf) → ~9,4 MB × n_models/modelo
  u:      (n_models, n_cam, npt, n_pos, nf) → ~62 MB × n_models/modelo
  v:      (n_models, n_cam, npt, n_pos, nf) → ~62 MB × n_models/modelo
  z, zp:  (n_models, n_cam, npt, n_pos, nf) → ~50 MB × n_models/modelo
  dipole: (n_models, npt, n_pos, nf, nTR)   → ~32 MB × n_models/modelo
Todos esses buffers existem SIMULTANEAMENTE na VRAM durante a execução
```

**A soma desses intermediários para F_50 é ~12,87 GB teoricamente**,
consistente com o OOM a 17,87 GB (diferença = overhead BFC + fragmentação
+ buffers de saída intermediários não contabilizados).

#### 2.2.2 Por Que o XLA Não Libera Buffers Antes do Fim

Pergunta natural: "por que XLA não libera `u` depois de computar `v` (já que
`u` não é mais necessário)?"

Resposta: **em forward-only mode, poderia**. Mas o XLA é conservador porque:

1. **`jax.grad` precisa de todos os intermediários**: se o usuário chamar
   `jax.grad(vmap(_one_model))`, o XLA precisa guardar todos os intermediários
   para o backpropagation (regra da cadeia). O XLA compila **o mesmo programa**
   para forward e potencial backward — por isso guarda tudo.

2. **Buffer reuse analysis**: XLA faz análise de *buffer liveness* para reaproveitamento,
   mas para vmaps aninhados essa análise se torna conservadora (complexidade O(N²)
   em grafos grandes → XLA prefere ser conservador e não reusar).

3. **Kernel fusion**: quando XLA funde múltiplos kernels em um único lançamento CUDA,
   todos os intermediários precisam estar na VRAM simultaneamente (não há ponto
   de descarte entre operações fusionadas).

#### 2.2.3 Contagem Real de Buffers para o Simulador EM TIV

O grafo de computação de `_one_model` para o cenário F contém:

```
Entradas: rho_h (5,), rho_v (5,), h_arr (5,), prof_arr (6,) — 4 arrays
Intermediários principais:
  - krJ0J1: (201,) complex128 — filtro Hankel
  - common_u: (201,) por (n_cam=5, nf=4) = (5, 4, 201) → ~0,5 MB
  - common_v: idem → ~0,5 MB
  - common_arrays full: (3, 5, 201, 4) → 188 KB
  - Para cada posição z (n_pos=100) × freq (nf=4) × TR (nTR=4):
      u_bottom_up: (5, 201) × 100 × 4 × 4 = ~2,6 GB
      v_bottom_up: (5, 201) × 100 × 4 × 4 = ~2,6 GB
      z_te_down:   (5, 201) × 100 × 4 × 4 = ~2,6 GB
      z_tm_down:   (5, 201) × 100 × 4 × 4 = ~2,6 GB
      dz_te:       (5, 201) × 100 × 4 × 4 = ~2,6 GB  [opcional, caso 3]
  - Kernels HMD: (201,) × 100 × 4 × 4 → ~0,64 MB
  - Kernels VMD: (201,) × 100 × 4 × 4 → ~0,64 MB
  - H parcial: (100, 4, 9) × 4 TR = ~4 MB
Saída: (4, 1, 100, 4, 9) = 225 KB
```

**Total por modelo: ~13 GB de intermediários para 225 KB de saída**.
Com `jax.vmap` + `n_models=50`, esses 13 GB se tornam a pressão que causa
o OOM real no T4 de 15 GB.

### 2.3 Causa C: complex128 — O Multiplicador de 2×

O simulador usa `complex128` (dois float64 = 16 bytes por número complexo)
em todos os tensores intermediários. Isso é necessário para paridade Fortran `<1e-12`.

Impacto:
- Cada buffer custa **2× mais VRAM** que com `complex64` (float32 complex)
- As TPUs e GPUs modernas têm **ratio de throughput FP64/FP32 entre 30:1 e 60:1**
  em favor de FP32 (T4: 0,26 TFLOPS FP64 vs 8,1 TFLOPS FP32 = 31:1)

Implicação: **migrar para complex64** (Sprint 13, mencionada no ROADMAP) reduziria:
- VRAM intermediária: de ~13 GB para ~6,5 GB por F_50
- Permitiria F com n_models=50 na T4 sem OOM
- Throughput: 31× melhora teórica em operações aritméticas (se não for bandwidth-bound)

**O risco é a precisão**: complex64 tem ~7 dígitos significativos vs 15 do complex128.
Isso pode degradar a paridade Fortran de `<1e-12` para `<1e-6` — aceitável para
geração de dados ML, inaceitável para inversão de alta precisão.

### 2.4 Por Que fori_loop Ajuda (mas Não Resolve)

A Sprint 10 Phase 2 introduziu `jax.lax.fori_loop` no loop de camadas. O efeito:

```
Antes (Sprint 7.x — Python loop unrolled pelo trace):
  XLA vê: u[0] = f0(...), u[1] = f1(u[0]), ..., u[n_cam-1] = f_{n_cam-1}(...)
  Resultado: XLA aloca u[0], u[1], ..., u[n_cam-1] TODOS simultaneamente
  → buffers = n_cam × (tamanho de u por camada)

Depois (Sprint 10 — fori_loop):
  XLA vê: body(L, u_L) → u_{L+1}, com L variando 0..n_cam
  Resultado: XLA aloca APENAS u_L e u_{L+1} por vez (reusa buffer)
  → buffers = 2 × (tamanho de u por camada)
  Redução: n_cam / 2 = 5/2 = 2,5× menos memória no loop de camadas
```

Isso é significativo, mas o `fori_loop` reduz apenas os buffers do **loop de camadas**,
não dos loops de posição, frequência, TR e ângulo — que são os dominantes em
cenários com n_pos=100, nf=4, nTR=4. O `vmap` sobre esses eixos ainda materializa
tudo simultaneamente.

## Parte III — Mitigações e Recomendações

### 3.1 Mitigações Imediatas (Sprint A3, zero refatoração)

**Controlar pré-alocação BFC** (adicionar na Célula 1 do notebook):

```python
# ANTES de: os.environ["JAX_ENABLE_X64"] = "1"
os.environ["XLA_PYTHON_CLIENT_MEM_FRACTION"] = "0.65"  # 65% vs 75% padrão
# Efeito T4:  reserva 9,75 GB em vez de 11,25 GB → +1,5 GB para compute
# Efeito A100: reserva 26 GB em vez de 30 GB → +4 GB para compute
# Trade-off: alocações fora do pool são ~2-5 ms mais lentas (syscall extra)
```

**Estratégia alternativa — alocador sob demanda** (para notebooks exploratórios):
```python
os.environ["XLA_PYTHON_CLIENT_ALLOCATOR"] = "platform"
# Sem pré-alocação — monitor mostra uso real
# Custo: cada alocação = syscall CUDA → latência variável (+10-50% por alocação)
# Útil para: diagnóstico de consumo real, notebooks com múltiplos frameworks
```

### 3.2 Mitigação de Médio Prazo (Sprint 13 — complex64)

A migração de `complex128` para `complex64` no caminho de dados do simulador:

| Impacto | Valor |
|:--------|:------|
| Redução de VRAM intermediária | 2× (de ~13 GB para ~6,5 GB por F_50) |
| Cenário F em T4: max n_models | 20 → ~50 (sem chunking) |
| Redução de tempo (T4, bandwidth-bound) | 1,5–2× |
| Degradação de paridade Fortran | <1e-12 → ~1e-6 |
| Impacto em SurrogateNet training | Negligível (ML é insensível a 6ª casa decimal) |
| Impacto em inversão científica | Alto (pode violar critérios de publicação) |

**Recomendação**: implementar `complex64` como modo opt-in via `SimulationConfig.precision = "float32"`, mantendo `complex128` como default para inversão científica.

### 3.3 Mitigação de Longo Prazo (Sprint A4+ — lax.map + pipeline)

Substituir `jax.vmap` por `jax.lax.map` para o eixo de modelos:

```python
# vmap: materializa TODOS os n_models simultaneamente → O(n_models) memória
H = jax.vmap(_one_model, in_axes=(0,0,0,0))(batch)  # ATUAL

# lax.map: processa 1 modelo por vez sequencialmente (intra-XLA)
# → O(1 modelo) memória → zero overhead Python entre modelos
H = jax.lax.map(_one_model, batch)  # FUTURO
```

**Trade-off**: `lax.map` é **sequencial** (sem paralelismo entre modelos no GPU) —
throughput por posição/freq cai para o de um único modelo, mas throughput total
(somando todos os modelos) pode ser comparável ou melhor para cenários onde
o bottleneck é VRAM (G, H), não compute.
