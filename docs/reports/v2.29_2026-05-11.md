## 2. Investigação Empírica

Análise comparativa entre `old_geosteering_ai/simulation/tests/sm_workers.py` (869 LOC,

### 2.1 Comparativo Arquitetural

| Aspecto | OLD (funciona) | CURRENT v2.18–v2.28 (problemas) |
|:--------|:---------------|:---------------------------------|
| **Pool** | `with ProcessPoolExecutor` **efêmero** (criado por simulação) | `_PERSISTENT_POOL` global, reusado |
| **Inicializador** | Nenhum (workers spawn com defaults) | `_numba_init_worker` (env vars) |
| **Warmup background na GUI** | **Nenhum** | `PoolWarmupThread` na abertura da GUI |
| **Warmup task** | **INLINE** em `run_numba_chunk` com dados de `chunk[0]` | `_run_numba_warmup_task` com 4 cenários sintéticos (`n_layers=10` HARDCODED) |
| **Flag de warmup** | Nenhuma | `_WORKER_INITIALIZED` global (race condition) |
| **Medição de tempo** | `t0` único (linha 521) | `t0` + `t0_sim` após NOOPs |
| **Sincronização do pool** | Nenhuma (não necessária — pool ephemeral) | NOOPs submetidos para "garantir" warmup pronto |
| **Shutdown** | Implícito via `with` (context manager `__exit__`) | `release_numba_pool()` + atexit + `cancel_futures=True` |
| **Hang risk ao fechar** | **Mínimo** (workers terminam ao sair do `with`) | **Alto** (workers em JIT não respondem a `cancel_futures`) |
| **LOC** | 869 | 1478 |

### 2.2 Causa-Raiz da Regressão v2.18–v2.28

**Bug #1 — Warmup sintético incompleto** (v2.25–v2.28):
- `_run_numba_warmup_task` executava `simulate_multi` com `n_layers=10` HARDCODED
- Modelos REAIS do usuário têm `n_layers` variável (3, 5, 8, 10, 15, 28)
- Numba compila por assinatura de tipo, mas paths internos (`_sanitize_profile_kernel`,
  `common_arrays`, `find_layers_tr`) podem ter especializações que dependem dos VALORES
  dos parâmetros (não só dos tipos)
- Warmup C+D (v2.28) tentou cobrir paths anisotrópicos/dip≠0 mas ainda usava `n_layers=10`

**Bug #2 — Pool persistente recriado** (v2.27–v2.28):
- Pool é cacheado por `(n_workers, n_threads, hankel_filter)`
- Se `hankel_filter` default em `PoolWarmupThread.__init__` difere do usuário escolhe,
  pool é DESTRUÍDO e RECRIADO → warmup background é DESCARTADO
- Workers da NOVA simulação têm `_WORKER_INITIALIZED=False` → secondary warmup com
  dados reais ocorre APÓS `t0_sim` → JIT compilation é contada como tempo de simulação

**Bug #3 — Hang no shutdown**:
- `release_numba_pool()` chama `shutdown(wait=False, cancel_futures=True)`
- Workers em compilação LLVM **NÃO respondem** a `cancel_futures` (Numba bloqueia GIL)
- Quando Python termina, `ProcessPoolExecutor.__del__` é chamado → `shutdown(wait=True)`
  por padrão → **hang indefinido**

**Bug #4 — Race condition em `_PERSISTENT_POOL_CONFIG`**:
- Setado SEM lock após criação do pool
- Threads concorrentes podem ler config stale → vazamento de pools

### 2.3 Por Que OLD Funciona

```python
# old_geosteering_ai/simulation/tests/sm_workers.py:152-174

# ── Warmup JIT (1 chamada descartada por processo, dados REAIS) ──────
if chunk:
    m0 = chunk[0]
    _ = simulate_multi(rho_h=..., rho_v=..., esp=..., ...)  # ← dados reais!
    # Publica evento de warmup concluído.

total = int(total_hint) if total_hint is not None else len(chunk)
t0 = time.perf_counter()  # ← APÓS warmup: tempo retornado EXCLUI JIT
```

**Insights críticos:**
1. **Warmup usa dados REAIS** de `chunk[0]` → cobre AUTOMATICAMENTE todos os paths
   Numba (n_layers reais, dips reais, anisotropia real)
2. **`t0` é setado APÓS o warmup** dentro do worker → tempo por worker exclui JIT
3. **Pool é efêmero** → `with ProcessPoolExecutor` chama `shutdown(wait=True)` no
   `__exit__` → workers terminam naturalmente sem hang
4. **Sem flag global** → sem race conditions, sem inconsistências entre pool persistente
   e workers

## 7. Riscos e Mitigações

| Risco | Mitigação |
|:------|:----------|
| Pool efêmero adiciona overhead de spawn (~10–30s na 1ª simulação) | NumbaPrimer popula cache JIT em disco no startup. Mensagem honesta ao usuário: "Aquecendo JIT em N workers (~10–30s)". OLD operava assim sem reclamação. |
| Cancelamento/pausa (v2.11) | Preservado: `request_pause`, `request_resume`, `request_cancel`, `_wait_if_paused` + signals `paused`/`resumed`/`cancelled` |
| Multi-frequency / dip≠0 cold compile na 1ª chamada | Warmup INLINE usa dados REAIS de `chunk[0]` — cobre EXATAMENTE os paths que a simulação real usará (superior à abordagem v2.28 com `n_layers=10` hardcoded) |
| Memória residual entre simulações | Pool ephemeral fecha workers no `__exit__` → memória liberada automaticamente. **Resolve por design** o hang do Python ao fechar antes do fim. |
| `NumbaPrimer` ainda existe e roda warmup sintético no startup | Compatível: roda no main process, popula `__pycache__/*.nbi/*.nbc` em disco. Workers spawned leem o cache. Não cria pool persistente. |

## 10. Resumo Executivo

A solução v2.28 (Warmups C+D) abordou um sintoma (cobertura de paths) mas o problema
raiz era arquitetural: **toda a infraestrutura de "warmup background na GUI + pool
persistente + t0_sim" introduzida em v2.18 é incompatível com o cold-start JIT do
Numba**.

A solução definitiva (v2.29) é reverter para a arquitetura comprovadamente funcional
do `old_geosteering_ai/`:
- **Pool efêmero** (`with ProcessPoolExecutor`)
- **Warmup INLINE** com dados reais (`chunk[0]`)
- **Sem flags globais, sem sincronização artificial, sem pool persistente**

416 LOC de complexidade removida. Paridade Fortran <1e-12 inviolada. Hang do Python ao
fechar é resolvido **por design** (context manager `__exit__`).
